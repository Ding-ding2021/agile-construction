WBS 框架布局 — 三种方案

工作分解结构（WBS）需要展示项目 → 工作包 → 任务 → 子任务 四层树形结构。哪种视觉方案更合适？

方案 A：表格式层级展开

沿用现有 SubTaskTree 的 Table 模式，新增 WBS 编码列、进度条列、责任人/日期列。按层级缩进，支持展开/折叠。

WBS Table — 项目「朝阳旗舰店」
WBS编码 任务名称 状态 进度 责任人 截止
📂 CY-01 店面装修 进行中
45% 		张伟 06/30
CY-01.01 地面铺装 已完成
100% 		李工 06/15
CY-01.02 墙面施工 待开始
0% 		王工 06/25
📂 CY-02 设备采购安装 进行中
30% 		赵总 07/10
CY-02.01 厨房设备安装 待开始
0% 		孙班 07/05

优点

  • 沿用现有 Table 模式，开发成本最低
  • 列对齐清晰，适合数据密集场景
  • 与现有 SubTaskTree 完全兼容
  • 筛选/排序/分页都很自然

缺点

  • 深度>3 层时缩进浪费大量水平空间
  • 视觉上像 Excel，不够直观
  • 拖拽排序体验差

方案 B：树形大纲视图

纯树形组件（类似 VS Code 文件树），缩进+图标表达层级。每个节点可展开详情面板。更侧重"浏览和定位"而非"表格对比"。

WBS Tree — 项目「朝阳旗舰店」
▼ 🏗️ CY 朝阳旗舰店 (项目) 45%
▼ 📦 CY-01 店面装修 (工作包) 45%
● ✓ CY-01.01 地面铺装 (任务) 完成
● ⏳ CY-01.02 墙面施工 (任务) 待开始
▶ 📦 CY-02 设备采购安装 (工作包) 30%

优点

  • 空间利用率高，4 层深也不拥挤
  • 视觉层级一目了然
  • 与文件树/大纲视图一致的用户心智模型
  • 适合"快速浏览+点击深入"的工作流

缺点

  • 多列对比（进度/日期/责任人）不够方便
  • 大规模数据（>50 节点）浏览效率下降
  • 需要拖拽排序时交互复杂度高

方案 C：分栏式 WBS + 甘特图

左侧树形 WBS（方案B），右侧时间轴甘特条（显示每个工作包/任务的起止日期和依赖线）。真正的项目管理级 WBS。

WBS + Gantt — 项目「朝阳旗舰店」
WBS
▼ 🏗️ CY 朝阳旗舰店
▼ 📦 01 店面装修
● ✓ 01 地面铺装
● ⏳ 02 墙面施工
▶ 📦 02 设备采购安装
5月6月7月8月
依赖线 (FS)

优点

  • WBS + 时间线一体，专业项目管理体验
  • 依赖关系可视化（FS/SS/FF/SF）
  • 与现有甘特图组件（P2-T6）自然衔接
  • 客户/PM 汇报场景价值高

缺点

  • 开发成本最高（双视图同步滚动）
  • P2-T6 甘特图还未开发
  • 移动端体验差
  • 对数据完整性要求高（必须有起止日期）

推荐策略

推荐 方案 A (核心) + 方案 B (辅助) 组合：

  • 方案 A 作为默认 WBS 视图 — 复用现有 Table + SubTaskTree 基础设施，快速交付
  • 方案 B 作为轻量侧栏视图 — 在任务详情右侧面板中展示迷你WBS
  • 方案 C 留到 P2-T6 甘特图开发时一并实现（WBS → 甘特联动）

第一阶段目标：Table + Tree (递归行) + WBS编码自动生成 + 展开/折叠 + 进度汇总，打掉「范围与任务」标签的 WBS 空白。
