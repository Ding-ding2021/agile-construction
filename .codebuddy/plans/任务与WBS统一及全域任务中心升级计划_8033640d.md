## User Requirements

- 保留“任务管理”菜单与 `#/tasks` 路由，作为全局任务汇总入口，不仅包含项目任务，还要覆盖维修保养等非项目任务。
- 将项目内“任务 + WBS”统一为单一实体，避免多套数据定义导致的不一致。
- 甘特图与WBS使用同一数据源派生，确保结构视图、时间视图、流程视图一致。
- 需要在现有项目架构内给出可执行、可回滚、分阶段的开发计划，而不是一次性重构。

## Product Overview

- 项目详情页定位为“项目内深度治理工作台”：同源呈现结构（WBS）、时间（甘特）、执行（流转）信息。
- 任务管理页定位为“全域任务中心”：跨项目、跨来源统一汇总、筛选和调度。
- 导航保持稳定：侧边栏“任务管理”入口不删除，仅能力升级。

## Core Features

- 统一工作项模型（覆盖项目任务与非项目任务来源）。
- WBS与甘特从统一工作项派生构建。
- 项目详情页计划/执行视图同源联动。
- 全局任务中心新增按任务来源汇总筛选（项目/维修保养/巡检/合规/临时）。
- 渐进式迁移与兼容开关，控制改动风险与回滚边界。

## Tech Stack Selection

- 延用现有前端栈：React + TypeScript + Vite + Hash 路由（已在 `src/App.tsx` 实施）。
- 延用现有本地持久化与仓储模式：`taskRepository`、`personnelRepository`、项目数据在 `src/data/*` 聚合。

## Implementation Approach

采用“数据先并轨、视图后收敛”的分阶段策略：先建立统一 `WorkItem` 域模型与适配层，再逐步替换 WBS/甘特的数据构造器，最后升级项目详情和全局任务中心。该方式可最小化一次性改动面，保证 `#/tasks` 入口持续可用。
关键决策：

1. 保留全局任务中心边界，不把非项目任务强行塞入项目域；
2. WBS/甘特全部改为 selector 派生，消除双模板漂移；
3. 通过兼容层读取旧数据，按 phase 切换写入策略，避免破坏现有状态流转。  
   性能重点：

- WBS树与甘特分组采用 `useMemo` + 纯函数 selector，复杂度控制在 O(n)~O(n log n)；
- 避免重复扁平化/递归遍历，通过节点索引缓存（id->node）减少重复计算；
- 仅对当前项目范围计算项目工作台数据，避免全量任务重算。

## Implementation Notes

- 复用现有状态与日志链路（`TaskManagementPage.tsx` 的流转日志、`projectStatusMachine` 的守卫思想），避免引入第二套状态机。
- 保持 `#/tasks`、`#/projects/:code/...` 兼容；先新增能力，不直接删除旧函数，待 Phase 6 清理。
- 对高风险迁移点（状态映射、依赖关系）加适配函数与兜底默认值，防止渲染空白或状态不可达。
- 日志沿用现有粒度，避免新增高频噪声日志。

## Architecture Design

- **Domain 层**：新增 `workItem` 统一模型与状态映射。
- **Data 层**：新增 `workItems` 聚合与兼容读取；WBS/Gantt 改为基于 `WorkItem[]` 的派生构建。
- **Project UI 层**：`ProjectDetailPage` 的 plan/execute 共享同源工作台组件。
- **Task UI 层**：`TaskManagementPage` 升级为全域任务中心，按 `sourceType` 聚合筛选。
- **Routing 层**：保留 `#/tasks` 路由与侧边栏入口，不改用户主导航习惯。
