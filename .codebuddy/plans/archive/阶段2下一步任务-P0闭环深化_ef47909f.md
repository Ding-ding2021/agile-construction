## User Requirements

- 在已完成任务页主链与路由联动能力基础上，继续推进“阶段2：V1开发”剩余任务，不重复已完成内容。
- 输出下一步开发计划需按 P0/P1/P2 分级，明确依赖顺序、修改文件、验收口径。
- 优先确保一条可端到端演示的闭环链路，并能稳定回归。

## Product Overview

- 当前已具备：任务状态驱动操作、任务页显示修复、`projectCode/taskCode` 路由上下文联动、分页与详情体验优化。
- 当前仍需补齐：模板实例化门禁与一致性、标准快照触发与审计留痕、验收结果到结算建议的跨模块闭环。
- 页面视觉体系已稳定，下一步以流程完整性与数据一致性为主。

## Core Features

- 模板实例化与任务生成仅允许 `active` 模板参与，异常可见且可追踪。
- 在任务进入“待执行/执行链”前生成标准快照并记录审计事件。
- 验收节点、项目状态、结算建议形成可回写的最小闭环。
- 建立阶段化回归场景：模板命中、任务流转、验收整改、结算建议一致性。

## Tech Stack Selection

- 前端：React + TypeScript + Vite（沿用现有工程）
- 路由：Hash 路由（`/Users/dylan/CodeBuddy/20260402092847/src/App.tsx`）
- 数据：本地状态 + `src/data` 与 `src/components/*/*.data.ts`
- 样式：现有 CSS 体系（不引入新 UI 架构）

## Implementation Approach

采用“P0 主链先闭环、P1 增强一致性、P2 治理扩展”策略：先把模板门禁→任务实例化→标准快照→验收回写→结算建议打通，再补充治理与体验增强。核心决策是复用现有 `template-instantiation`、`TaskManagementPage`、`ProjectAcceptanceView`、`App.tsx` 编排，不新增全局状态框架，降低回归风险。

## Implementation Notes

- 复用 `TemplateStatus` 与实例化入口，统一判定 `active` 门禁，避免多处散落判断。
- 审计事件沿用契约类型 `TemplateAuditEvent`，先在前端本地日志容器落地，后续可平滑迁移服务端。
- 关键路径保持 O(n) 列表处理（selector 管道不拆散），避免重复过滤与额外渲染。
- 仅改动主链相关文件，避免无关重构；路由参数保持向后兼容。

## Architecture Design

- 模板域：`standard-template.data.ts` + `template-instantiation.ts` 负责模板命中、依赖校验、门禁与任务种子输出。
- 任务域：`taskManagement.data.ts` + `TaskManagementPage.tsx` 承接快照触发时机、状态推进与页面联动。
- 项目/验收域：`ProjectAcceptanceView.tsx` + `App.tsx` + `data/projects.ts` 承接验收回写与结算建议口径。
- 结算域：`ContractSettlementPage.tsx` 消费项目闭环状态，展示结算建议最小结果。
