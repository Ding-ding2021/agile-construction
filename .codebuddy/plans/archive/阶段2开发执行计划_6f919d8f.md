## User Requirements

- 基于现有《阶段2复盘与后续计划》结论，输出可直接执行的后续开发计划。
- 计划需围绕阶段2收口，不扩展到阶段3范围。
- 重点解决复盘中已确认差距：后端权威数据闭合、审计留痕可靠性、质量门禁、调度台独立化、执行资料链路、Agent可验证触发链。
- 计划应具备明确优先级、依赖顺序、交付物与验收口径。

## Product Overview

- 当前已具备项目、任务、验收、结算的前端演示闭环与跨页跳转能力。
- 本轮计划聚焦把“可演示闭环”升级为“可运营闭环”，保证数据一致性、流程可追溯、质量可发布。
- 视觉层保持现有页面风格与结构，主要补齐流程能力与可靠性，不做大规模界面重构。

## Core Features

- 主链路数据从本地回退模式升级为可验证的后端权威链路。
- 项目/任务/验收/结算关键动作具备稳定审计留痕与追溯能力。
- 调度派单从任务页嵌入能力补齐为可独立运营视角。
- 执行回传补齐资料完整性校验、阻断与提交轨迹。
- 至少3个Agent节点形成“触发输入-结构化输出-人工确认-结果回写”的可验收证据链。

## Implementation Approach

采用“先稳定、再闭环、后收口”的四段策略：

1. 先修复质量门禁与主链稳定性；2) 再闭合前后端接口与审计写入；3) 补齐P0独立调度与执行资料链路；4) 完成P1透明化与阶段评审证据。
   关键决策：复用现有 `App.tsx` 单入口编排与 repository 模式，避免引入新状态框架；高风险动作保持人工确认入口，降低误流转风险。
   性能与可靠性：保持现有列表 O(n)~O(n log n) 管道，避免重复计算；审计写入采用异步+幂等键，减少阻塞与重复提交风险。

- Agent 证据链优先落在现有可追溯节点：模板实例化、快照生成、验收整改、结算建议回写。

## Agent Extensions

- **skill:spec-workflow**
- Purpose: 约束阶段2收口需求、技术方案与任务拆分一致。
- Expected outcome: 形成可评审、可追踪、可验收的阶段计划基线。
- **subagent:code-explorer**
- Purpose: 精准扫描主链调用关系与改动影响面。
- Expected outcome: 降低遗漏回写点和误改风险，保证计划落点准确。
- **integration:tcb**
- Purpose: 承载并验证 `serverAdapter` 对应后端接口与数据落库链路。
- Expected outcome: 项目/任务/验收/结算/审计实现云端权威化闭环。
- **skill:web-development**
- Purpose: 指导 React/Vite 前端主链页面联调与回归。
- Expected outcome: 提升主链稳定性并确保门禁可持续通过。
- **skill:design-to-code-workflows**
- Purpose: 对新增调度区块或资料校验区块快速生成可接入UI草案并保留可访问性。
- Expected outcome: 在不重构整体视觉前提下高效补齐关键交互区块。
