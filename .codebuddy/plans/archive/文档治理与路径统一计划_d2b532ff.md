## User Requirements

- 将当前仓库分散在根目录与 `docs/` 下的文档路径进行系统化治理。
- 目标不是临时整理，而是“彻底治理”：统一入口、统一分层、统一命名、统一维护规则。
- 输出应可直接执行，包含迁移范围、治理规则、执行顺序与可验收标准。

## Product Overview

- 文档体系以 `docs/` 为唯一文档域，`docs/README.md` 作为唯一导航入口。
- 根目录仅保留项目入口类说明文件，不再承载业务规范与需求细则。
- 文档治理结果应体现“可查找、可维护、可审计、可演进”。

## Core Features

- 文档目录分层重构（治理/需求/架构/工程/运营/归档）。
- 根目录文档全量迁移映射与规范化重命名。
- 每篇执行文档统一元数据（负责人、状态、更新时间、关联代码）。
- 建立状态治理（draft/active/deprecated/archived）与 SSOT 规则。
- 建立变更门禁与验收口径，防止后续再次分散。

## Tech Stack Selection

- 复用现有仓库能力：Markdown 文档体系 + Git 版本管理 + 现有前端工程目录结构。
- 不引入新技术栈；以最小工具成本完成文档资产治理。

## Implementation Approach

- 采用“**结构重建 -> 内容迁移 -> 规则固化 -> 门禁校验**”四阶段方案。
- 先建立目标目录与唯一入口，再执行一对一迁移，最后固化元数据与审计规则，避免先改规则后无落点。
- 关键决策：保留短期迁移提示页降低断链风险；执行文档仅保留一个 active 事实源，其他文档降级为 deprecated/archived。
- 复杂度：迁移与索引维护为 O(N)，引用回填近似 O(N+E)（N=文档数，E=链接数）；通过批量检索一次性处理，避免反复扫描。

## Implementation Notes (Execution Details)

- 严格基于已存在路径治理，不改业务代码逻辑。
- 优先处理双规范冲突：`/DESIGN_SPECIFICATION.md` 与 `/docs/DESIGN_SPECIFICATION.md`，保留“总览 + 执行规范”分层。
- 迁移期间旧路径保留“已迁移”提示，避免外部链接立即失效。
- 控制影响面：本次仅治理 `.md` 文档路径与引用，不做无关重写。

## Architecture Design

- 采用文档域分层架构：`00-governance`（规则）→ `01-product`（需求）→ `02-architecture`（设计）→ `03-engineering`（交付）→ `04-operations`（运营）→ `99-archive`（历史）。
- `docs/README.md` 作为单入口目录树与状态看板；所有 active 文档必须在入口登记。
