## 用户需求

- 对现有前端代码进行结构化整理，汇总页面组件与布局组件现状。
- 基于当前项目代码提出可落地的页面组件化方案，重点提升复用率与一致性。
- 方案需分阶段推进，避免一次性大改，保证现有页面功能和视觉表现可持续稳定运行。

## 产品概览

- 当前系统为多业务页面的统一入口应用，包含项目、任务、人员、采购、合同、订单、设施、资源、标准、数字员工等页面。
- 页面视觉风格已形成统一倾向（深色玻璃态、统一头部与导航样式），但组件实现存在重复，导致维护成本偏高。
- 组件化改造后，页面结构将更统一，导航、头部、列表工具栏、统计卡片、分页等视觉区块保持一致交互体验。

## 核心功能

- 建立统一页面骨架组件（侧边栏、头部、内容区插槽化）。
- 收敛多处重复导航逻辑与路由激活判断，统一导航配置。
- 抽象列表页通用能力：搜索、筛选、排序、分页、统计卡片。
- 制定渐进迁移路径与验收标准，确保迁移过程中页面行为一致、回归可控。

## 技术栈选择

- 复用现有技术栈：React + TypeScript + Vite + CSS。
- 复用现有路由方式：Hash 路由（由 `src/App.tsx` 统一编排）。
- 复用现有分层模式：页面组件 + 选择器（selectors）+ 仓储（repositories）+ 本地持久化。

## 实施方案

采用“先统一壳层、再抽象通用能力、后分批迁移页面”的策略：先消除导航与页面骨架重复，再沉淀列表通用组件，最后按业务域逐步替换。
关键决策：保持现有路由与页面功能不变，仅做结构收敛；优先改动重复度最高区域（侧边栏、Header、列表工具栏），降低风险并快速看到收益。
性能方面：列表处理继续走现有 selector 管道（O(n) 过滤/排序为主），通过 `useMemo` 与最小状态更新减少重复渲染；避免引入全局重状态导致额外开销。

## 实施说明

- 保持 `window.location.hash` 语义兼容，不改变现有页面跳转路径。
- 将 `navigateByHash`、激活态判断、菜单配置集中管理，避免跨页分叉。
- 仅迁移高重复页面作为首批试点（项目、任务、标准/系统），稳定后再扩展至其余页面。
- 避免并行重构页面业务逻辑（状态机、仓储、领域模型），控制改造爆炸半径。

## 架构设计

- **共享导航层**：统一菜单配置、跳转工具、激活态工具。
- **共享布局层**：`AppShell` 统一承载 Sidebar/Header/Main。
- **共享列表层**：Toolbar、StatsCards、Pagination、ListState Hook。
- **业务页面层**：保留领域逻辑，仅替换可复用 UI 区块与容器结构。
- **选择器层**：继续沿用 `filter -> search -> advancedFilter -> sort -> paginate` 管道。

## Agent Extensions

- **subagent: code-explorer**
- Purpose: 扫描重复导航、重复工具栏、重复分页逻辑的真实使用点与依赖链。
- Expected outcome: 输出可验证的“重复点清单 + 影响页面清单”，降低迁移遗漏风险。

- **skill: design-system**
- Purpose: 统一共享组件命名、层级边界、组件 API 规范与样式约束。
- Expected outcome: 形成可持续扩展的组件规范，避免二次分叉与风格漂移。
