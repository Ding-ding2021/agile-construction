## User Requirements

- 在本地搭建可运行的后端与数据库，先验证阶段3主链路可行性，暂不依赖云端鉴权。
- 本地联调需覆盖五条核心接口：`/projects/state`、`/tasks/state`、`/acceptance/state`、`/settlement/state`、`/audit/logs`。
- 保持现有前端调用契约不变，包括查询参数与请求/响应结构；写接口继续支持幂等键。
- 在本地可完成读写、幂等重放、异常可定位验证，并沉淀可复测记录。

## Product Overview

- 现有前端继续通过统一 API 层访问状态数据，本地后端提供与当前远端一致的接口行为。
- 本地数据库保存项目、任务、验收、结算建议与审计日志，支持最小可用数据闭环。
- 页面视觉不做改版，用户操作路径与现有页面一致，仅数据来源切换为本地服务。

## Core Features

- 五接口本地读写闭环与统一错误返回。
- `X-Idempotency-Key` 去重与重放一致性。
- 本地数据库初始化、基础样例数据与可重复回归。
- 本地联调指引与阶段3验收记录模板。

## Tech Stack Selection

- 前端：沿用现有 React + TypeScript + Vite（已存在）。
- 本地后端：Node.js + TypeScript 轻量 HTTP API（与现有 TS 项目一致）。
- 本地数据库：SQLite（单机、低成本、便于快速重置与回归）。
- 质量门禁：沿用现有 ESLint + `npm run build` 流程，新增本地联调文档约束。

## Implementation Approach

采用“契约优先、最小闭环”策略：先按 `src/services/api/serverAdapter.ts` 已定义接口实现本地同名端点，再用 SQLite 落地最小状态表与幂等表，最后通过前端环境变量切换到本地地址完成回归。
关键决策：

- 复用现有 API 路径与参数（含 `envId`、`contextKey`、`projectCode`），避免改动页面逻辑。
- 幂等在服务端集中处理，写请求先查键再执行，防止重复脏写。
- 保留前端本地兜底逻辑，后端异常时仍可通过既有回退事件定位问题。  
  性能与可靠性：
- 读取接口均为单次索引查询，写入为事务提交，时间复杂度近似 O(1)~O(log n)。
- 幂等键表按 TTL/最近窗口清理，防止长期膨胀。
- 审计写入异步落库失败可追踪，不阻塞主流程。

## Implementation Notes

- 严格对齐已有调用入口：`src/services/api/serverAdapter.ts` 的五接口签名不改名不改路径。
- `src/services/api/client.ts` 当前已使用 `scenario` 字段，需补齐 `ApiClientOptions` 类型定义，避免构建回归。
- 仅新增本地后端目录与必要脚本，不改业务页面结构，控制变更面。
- 错误返回统一包含可定位信息（状态码、错误码、场景），避免静默失败。

## Architecture Design

- 前端编排仍在 `src/App.tsx` 与 repositories 层。
- repositories 调 `serverAdapter`，由 `VITE_API_BASE_URL` 指向本地 API。
- 本地 API 负责状态读写、幂等校验、审计记录；SQLite 负责持久化。
- 本地与云端切换仅通过环境变量与启动脚本完成，不改上层业务调用。
