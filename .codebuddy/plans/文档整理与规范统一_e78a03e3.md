## User Requirements

- 对现有文档体系进行整理，解决两个设计规范并存导致的口径冲突。
- 明确设计规范的主从关系，避免团队在实现时出现“按哪份文档执行”不一致。
- 输出可执行的文档治理方案，优先做结构与引用治理，不改业务功能定义。

## Product Overview

- 建立统一的文档分层：设计总览、可执行规范、开发检查清单。
- 统一仓库内“设计规范”引用入口，确保研发、规范、指南、规则文档指向同一事实来源。
- 通过版本与状态标记提升可维护性，降低后续规范漂移风险。

## Core Features

- 设计规范主文档确权与次文档角色重定义。
- 全仓设计规范引用路径与版本口径统一。
- 文档状态治理（Active/Overview/Deprecated）与变更记录机制。
- 文档索引页建立（用途、负责人、更新频率、依赖关系）。

## Tech Stack Selection

- 文档格式：Markdown（沿用现有仓库文档体系）
- 版本控制：Git 工作流（仅文档层改动）
- 规范基线：以已实现样式变量 `src/index.css` 的 `--pm-*` 体系为执行依据

## Implementation Approach

采用“先确权、后收敛、再统一引用”的三步策略：

1. 先确定唯一可执行设计规范（SSOT）；2) 将另一份设计规范降级为总览/参考；3) 全仓统一链接与版本口径。
   该方式最小化改动面，避免一次性重写内容导致回归与信息丢失，并可快速验证生效范围。

关键决策：

- 以 `docs/00-governance/design-specification.md` 作为执行规范主源（与 `src/index.css` 的 `--pm-*` 一致）。
- 保留根目录 `DESIGN_SPECIFICATION.md`，但改为“总览文档”，避免历史信息丢失。
- 所有“基于设计规范”声明统一改为主源路径，杜绝双入口。

## Implementation Notes

- 只做文档治理，不触及业务代码、样式实现与路由逻辑。
- 优先修复高影响引用：`README.md`、`docs/00-governance/coding-standards.md`、`docs/03-engineering/development-guide.md`、`CODEBUDDY.md`。
- 对历史文档保留跳转与状态说明，避免外部链接失效。
- 增加“最后校验基线”字段（指向 `src/index.css` 变量体系）以约束后续维护。

## Architecture Design

- **L1 总览层**：`/DESIGN_SPECIFICATION.md`（理念、原则、导航，不承载执行细则）
- **L2 执行层**：`/docs/00-governance/design-specification.md`（Token、组件、布局、示例，唯一执行口径）
- **L3 检查层**：`/docs/00-governance/design-checklist.md`（开发与验收清单，绑定 L2）

文档关系：L1 指向 L2，L3 依赖 L2；研发与规范文档统一引用 L2。
