## User Requirements

- 基于现有周计划，输出“可直接分派、可跟踪、可验收”的下一步任务。
- 任务需围绕上线阻断项优先推进：服务端数据权威化、权限鉴权、审计留痕、质量门禁。
- 采用 AI 全流程开发方式组织执行，任务应包含依赖关系与完成标准。

## Product Overview

- 当前系统已具备模板实例化、任务推进、验收联动、结算联动的前端闭环。
- 下一阶段需把“本地演示闭环”升级为“可上线闭环”：数据统一、权限可控、过程可追踪、发布可回归。
- 视觉上维持现有页面结构与样式，不做大规模界面改版，重点提升流程可靠性与状态一致性。

## Core Features

- 服务端主数据接管项目、任务、验收、结算、审计。
- 角色权限控制与关键动作后端鉴权拦截。
- 全链路审计日志记录与查询。
- 主链路自动化回归与合并门禁。
- 并发冲突与重复提交防护。

## Tech Stack Selection

- 前端沿用现有栈：React + TypeScript + Vite。
- 质量门禁沿用现有工程能力：ESLint + TypeScript build。
- 服务端能力使用已连接平台能力承载，前端通过统一 API 适配层接入，替换本地主存依赖。

## Implementation Approach

- 采用“契约先行、分层替换、灰度切换”的方式：先冻结数据契约与权限矩阵，再补服务端接口，最后将页面读写从 localStorage 切到 API。
- 保持 `src/App.tsx` 单入口编排不变，仅替换数据来源与状态同步链路，避免大范围重构。
- 关键决策：
- 保留前端状态机守卫逻辑，后端增加同口径校验，防止前后口径分叉。
- 引入统一 repository/api-client 层，避免页面直接散落调用。
- 对高频写操作加幂等键与版本字段，降低并发覆盖风险。
- 性能与可靠性：
- 列表查询优先服务端分页，前端仅做轻量筛选；复杂度由前端全量处理降为按页处理。
- 审计写入异步化，避免阻塞主交互；失败重试并限流避免日志风暴。

## Implementation Notes (Execution Details)

- 复用现有路由与页面组件，不改 hash 路由协议，控制改动面。
- `App.tsx` 的 `projectsState` 与 `projectStatusLogs` 改为“初始化拉取 + 操作后回写 + 本地缓存兜底”模式。
- `TaskManagementPage.tsx`、`ProjectAcceptanceView.tsx`、`ContractSettlementPage.tsx` 按“先读 API，失败回退只读提示”迁移，避免静默回退 mock。
- 审计字段沿用已存在事件语义（如模板不匹配、快照生成、实例创建），统一写入服务端并保留查询接口。
- 不做无关重构，优先确保构建、类型、主链路可回归。

## Architecture Design

- 现有分层保持不变：页面层（components）→ 领域层（domain）→ 数据层（data）。
- 新增“API 适配层”与“仓储层”：
- API 适配层：封装鉴权、错误码、重试。
- 仓储层：项目/任务/验收/结算/审计统一读写接口。
- 数据流：页面操作 → repository → 服务端持久化 → 回写页面状态 → 审计落库。

## Agent Extensions

- **spec-workflow**
- Purpose: 约束中大型改造按需求、技术方案、任务拆分执行
- Expected outcome: 任务边界清晰，依赖关系明确，减少返工

- **code-explorer**
- Purpose: 在实施前后快速定位跨文件影响面与调用链
- Expected outcome: 精准改动核心文件，避免遗漏联动点

- **tcb**
- Purpose: 承载服务端数据、鉴权与日志能力接入
- Expected outcome: 完成从本地主存到服务端权威数据的迁移闭环
