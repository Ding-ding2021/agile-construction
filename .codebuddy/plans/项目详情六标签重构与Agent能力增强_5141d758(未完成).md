## User Requirements

- 项目详情页主要服务项目经理，需在单页面完成项目管控闭环。
- 在既有六标签（仪表盘、启动、计划、执行、监控、收尾）中新增和强化 Agent 辅助能力。
- Agent 仅做辅助建议和自动化动作，不替代人工审批与治理责任。
- 在项目详情中支持“制定开发计划”并贯穿执行、监控、收尾，形成可追溯闭环。

## Product Overview

项目详情页将升级为“项目经理闭环工作台”：同屏展示阶段状态、门禁风险、Agent 建议、人工动作与审计结果。视觉保持深色专业风格，提升信息密度与操作直达性，让项目经理在详情页内完成计划制定、偏差处置和归档收口。

## Core Features

- 六标签统一加入 Agent 能力区：建议卡、自动动作结果、人工接管入口、审计留痕。
- 计划标签扩展开发计划编制：进度基线、资料快照、风险应对、采购订单与发布门槛联动。
- 执行和监控标签支持 Agent 触发催办、升级、风险提示，并可一键转人工处理。
- 收尾标签强化 Agent 与人工协同校验：问题闭环、结算确认、归档门禁与双审批可视化。
- 项目日志统一沉淀 Agent 输出摘要、人工决策和状态变化轨迹。

## Tech Stack Selection

- 复用现有前端栈：React + TypeScript + Vite。
- 复用现有 Hash 路由与项目详情结构（`ProjectDetailPage.tsx` + `ProjectTabs.tsx` + `projectTabs.shared.ts`）。
- 样式继续使用现有 `project-detail.css` 深色高密度体系。
- Agent 数据先复用现有项目域数据文件（`projectDigitalEmployee.data.ts`、`projectDigitalEmployee.types.ts`），再预留云端接入点。

## Implementation Approach

采用“先展示闭环、后能力接入”的双阶段方案：

1. 在六标签中补齐 Agent 工作区与人工兜底入口，完成可见闭环；
2. 通过现有回调与日志链路接入可控动作，确保不绕过状态机。
   关键决策：

- 不新增路由标签，避免破坏 `#/projects/:code/:tab` 协议。
- Agent 结果以“建议/推荐动作”呈现，关键动作仍经人工确认。
- 复用 `onAppendActivityLog` 统一沉淀日志，降低改造面。  
  性能与可靠性：
- 主要为列表和卡片渲染，复杂度维持 O(n)。
- 使用 `useMemo` 聚合 Agent 建议与风险摘要，减少重复计算。
- 保持兼容空数据场景，提供降级文案与安全默认值。

## Implementation Notes

- 已有执行页存在任务中心和采购协同跳转（`onNavigateTasks`、`#/procurement`），新增 Agent 动作需沿用同类跳转模式。
- PRD 明确 Agent 不可绕过状态机，所有自动建议必须可转人工并留痕。
- `projectDigitalEmployee.*` 当前已存在但未接入详情页，可优先复用，避免重复建模。
- 控制影响范围在项目详情域，不改 `ProjectItem` 主模型与 `App.tsx` 路由解析逻辑。

## Architecture Design

- **展示层**：`ProjectDetailPage.tsx` 内为六标签增加 Agent 区块（建议、自动动作、人工接管、审计摘要）。
- **数据层**：从 `projectDigitalEmployee.data.ts` 提供 Agent 能力、任务、日志 mock；映射为项目经理视角摘要。
- **交互层**：通过既有按钮动作和日志回调，串联“建议 -> 执行/转人工 -> 留痕”。
- **约束层**：状态流转仍由既有状态机与守卫控制，Agent 仅提供辅助输入。

## 设计方案

采用高密度深色管理后台风格，将六标签统一为“摘要条 + 主工作区 + Agent辅助区 + 操作审计区”。
Agent 区强调可执行建议、可信度和人工接管按钮；状态颜色区分建议、预警、阻断。
交互上通过悬浮高亮、按钮微动效和时间线反馈，突出项目经理快速决策路径。
