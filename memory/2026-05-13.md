## 关键决策

| #   | 决策                                                                                                       | 理由                                                             |
| --- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| D1  | 三层文档架构：L1 产品规划 / L2 V1 产品定义 / L3 模块 PRD                                                   | 用户要求分层，L1 简化（147行），L2 细化项目定义，L3 保持原模块级 |
| D2  | 任务驱动模型（B方案）：项目无独立状态机，状态从任务聚合推导                                                | 业务变化在任务层吸收，项目模型不动                               |
| D3  | 五条技能调用链：需求定义→规划→执行→验收上线→文档维护。产出物即下一链的触发信号                             | 解决 skills 调用率低的问题，让 Agent 自动推进                    |
| D4  | intent-orchestrator 引导技能：独立 SKILL.md + 产品 manager prompt 引用，分类用户意图后触发对应技能链       | 解决 Flash 模型跳过流程的问题，用外部结构约束行为                |
| D5  | MCP Memory 修复：sqlite_vec 引擎 + all-MiniLM-L6-v2 embedding（MPS 加速），2026-05-13 首次激活             | 之前有空配置无数据，跨会话记忆从未生效                           |
| D6  | "阅读后再精炼"不保证进入产品经理的推理回路。技能文件是"参考信息"，不是"行为协议"。"遵守流程"需要外部约束   | Flash 模型的天性是以速度优化，不会主动检查"我走完阶段2没有"      |
| D7  | 模型升级（Flash → Pro）是更本质的解法，但在无条件下引导技能是 Flash 模型下的最优解                         | Flash 倾向跳过流程，Pro 天生维持推理链                           |
| D8  | 使用 Google Agent Skills（addyosmani/agent-skills）作为引导技能试运行。暂停自定义 intent-orchestrator 开发 | 两种框架本质相同，Agent Skills 更成熟完备，直接拿来测试成本最低  |
