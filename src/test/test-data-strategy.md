## 单元测试数据策略 (pure function + Mock external deps)

- 目标：确保单元测试仅测试纯函数逻辑，尽量不依赖外部状态，必要时通过 mock 替换外部依赖。
- 做法：对纯函数使用直接输入输出断言；对外部依赖（如数据库、网络请求、文件系统）使用 mock/fake 实现，避免真实 I/O。
- 约束：避免在单元测试中引入真实数据库连接、网络请求或真实文件操作。
- 验证方式：通过独立的单元测试用例覆盖输入的边界情况，确保无副作用。

## 仓库测试数据策略 (Mock Prisma client, in-memory data)

- 目标：对数据访问层/仓库层进行隔离测试，模拟 Prisma 客户端行为，确保逻辑正确性。
- 做法：实现一个 Prisma 客户端的 mock（基于 in-memory 数据），暴露 project.findMany、project.create、project.update、project.delete 等方法。
- 约束：不连接实际数据库，只在内存中维护数据快照。
- 验证方式：通过对仓库接口的测试，断言返回数据符合预期，且对增、改、删操作数据状态正确更新。

## 集成测试数据策略 (Express test instance, Mock Prisma)

- 目标：在服务层/路由层进行集成测试，验证 Express 实例、路由、控制器协同工作。
- 做法：启动一个 Express 测试实例；在需要的场景中使用上述 Prisma mock 实例，确保路由和控制器交互如预期。
- 约束：不要连接真实后端服务，使用 mock 数据和中间件仿真真实依赖。
- 验证方式：对 API 路由发起请求，断言状态码和响应数据符合预期。

## 数据隔离原则 (per-test create/cleanup, no shared state)

- 目标：测试之间互不影响，确保每个用例在独立的数据上下文中执行。
- 做法：在每个测试用例开始前创建独立的 mock 数据快照，测试结束后清理，确保没有全局共享的状态。
- 验证方式：使用独立的数据副本进行操作，结束后断言内存数据未被跨测试污染。
