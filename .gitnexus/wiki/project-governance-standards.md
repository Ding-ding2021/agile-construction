# 项目治理与标准模块

## 概述

项目治理与标准模块定义了连锁店施工管理系统的规则、约定和质量保证流程。它是代码编写方式、组件开发方式、文档管理方式以及MVP阶段质量维护方式的权威依据。

本模块并非可执行代码——它是一系列约束和指导所有开发活动的治理文档。参与本项目的每位开发者和AI代理都必须遵守这些标准。

---


## 关键组件

### 1. 设计规范（`design-specification.md`）

所有视觉设计决策的单一权威来源。定义了：

- **色彩系统**：主色、背景色、文字色、功能色和边框色，以CSS自定义属性（`--pm-*`）表示
- **排版**：字号（10px–24px）、字重（400–700）、行高
- **间距系统**：内边距、外边距、间距（4px–32px）
- **圆角**：8px（小）到999px（全圆角）
- **阴影系统**：4级盒阴影
- **布局规则**：页面网格、卡片结构、响应式断点（768px、1024px、1360px、1920px）
- **组件SOP**：从Figma到代码审查的分步开发工作流程

**执行要求**：所有CSS值必须引用`--pm-*`变量。禁止使用硬编码的颜色、间距或圆角。

### 2. 编码标准（`coding-standards.md`）

定义了代码生成和审查的可执行规则：

- **ESLint/Prettier配置**：包含规则的确切配置文件
- **TypeScript约定**：接口与类型、枚举使用、类型守卫、属性模式
- **React模式**：组件结构（hooks → computed → effects → handlers → render）、自定义hooks、组合
- **文件组织**：目录结构、命名约定（组件使用PascalCase、函数使用camelCase、文件使用kebab-case）
- **注释规则**：仅对边界条件、复杂流程和关键业务逻辑使用中文注释——非逐行注释
- **性能**：`useMemo`/`useCallback`、懒加载、虚拟滚动
- **Git提交格式**：`type(scope): subject`，定义的类型包括feat、fix、docs、style、refactor、perf、test、chore

### 3. 组件开发契约（`component-development-contract.md`）

连接设计与实现的标准化交接模板。每个共享组件都需要包含以下内容的契约：

- **属性接口**：包含默认值和约束的类型定义
- **设计令牌映射**：每个视觉属性映射到CSS变量
- **状态与交互契约**：悬停、点击、加载、空状态、错误状态
- **依赖与组合规则**：组件可以导入的内容及其组合方式
- **测试与验收标准**：视觉、功能和代码质量检查
- **交付检查清单**：源文件、测试、示例、设计截图

**执行要求**：新共享组件"无契约，不开发"。

### 4. 设计检查清单（`design-checklist.md`）

确保视觉和功能一致性的开发前/后验证检查清单：

- **开发前**：设计文件审查、设计系统对齐、业务逻辑确认
- **开发中**：布局、样式、排版、响应式验证
- **开发后**：视觉回归、功能测试、性能、代码质量
- **常见问题诊断**：布局破坏、样式不一致、响应式失败
- **代码审查要点**：视觉、布局、性能、可访问性

### 5. MVP代码质量计划V2（`mvp-code-quality-plan-v2.md`）

针对一名产品经理+AI代理团队（无人类开发者）的质量保证策略。实施三层防御模型：

```
第1层：自动化防御（零人力成本，必须100%通过）
  - TypeScript严格模式
  - 带阻断规则的ESLint
  - 构建失败 = 拒绝

第2层：AI编码过程控制（低人力成本，规则驱动）
  - 可解释的交付物：单元测试 + 流程图 + 边界条件表
  - 双AI交叉验证（模型A编写，模型B审查）
  - 核心模块稳定化（草稿 → 稳定 → 冻结，附带变更请求）

第3层：产品经理验收（每次迭代10分钟）
  - 测试覆盖率审查（阅读测试用例列表）
  - 流程图验证（与需求对比）
  - 真实场景演练（浏览器操作）
```

**红线**（触发立即拒绝）：

- 缺少可解释的交付物
- 测试覆盖率不足
- 流程图与需求矛盾
- 真实场景操作失败
- 未经授权修改稳定模块

### 6. 文档治理（`document-governance.md`）

管理文档系统本身：

- **目录结构**：`00-governance`（规则）、`01-product`（需求）、`02-architecture`（设计）、`03-engineering`（交付）、`04-operations`（指标）、`99-archive`（历史）
- **状态模型**：`draft`（不可执行）、`active`（单一权威来源）、`deprecated`（保留历史）、`archived`（仅历史参考）
- **前置元数据要求**：每个活跃文档必须包含`id`、`title`、`owner`、`status`、`last_updated`、`source_of_truth`、`related_code`、`related_docs`
- **变更流程**：修改 → 更新前置元数据 → 更新索引 → 运行检查清单 → 合并 → 记录审计日志

### 7. PR文档检查清单（`pr-doc-checklist.md`）

文档变更的合并关卡：

- [ ] 变更仅限`docs/`目录
- [ ] 活跃文档具有完整的前置元数据
- [ ] `docs/README.md`索引已更新
- [ ] 所有链接使用`docs/...`路径
- [ ] 同一主题无重复的活跃文档
- [ ] 历史文档已归档或标记为废弃
- [ ] 如果受影响，设计规范和检查清单已同步

### 8. 推理语言（`reasoning-language.md`）

配置要求所有AI推理（意图声明、思维链、分析）以中文输出。这确保了AI生成的解释一致性，并使中文产品团队能够理解。
