## 核心文档

### 1. 架构决策记录 (ADR)

#### ADR-002: FeatureRegistry

**文件:** `adr-002-feature-registry.md`

引入集中式页面注册模式，以消除跨多个文件的分散配置。`FeatureConfig` 类型和 `FEATURE_REGISTRY` 数组将页面注册整合到单一位置。

**关键组件:**

- `FeatureConfig` 类型 — 定义页面元数据（页面标识符、路径、组件、标签、分类）
- `FEATURE_REGISTRY` 数组 — 页面注册的单一事实来源
- 页面分类数组的自动推导（`SIMPLE_PAGES`、`PARAM_PAGES` 等）

**影响:** 新页面设置从 6 个步骤减少到 2 个步骤（注册表条目 + 路由解析分支）。

#### ADR-001: 数据层架构

**文件:** `data-layer-decision-record.md`

定义了 MVP 阶段的双写持久化策略。采用**方案 C（分层写入）**，其中：

- **Zustand persist 中间件** 处理常规状态变更的自动本地持久化
- **projectRepository** 处理关键业务操作的手动持久化（项目创建、状态转换、里程碑同步）
- **auditRepository** 独立记录操作轨迹

**架构流程:**

```
用户操作 → Zustand 存储（内存）
                    ↓
        ┌───────────┴───────────┐
        ↓                       ↓
Zustand persist          projectRepository
（自动，所有变更）      （手动，关键操作）
        ↓                       ↓
  localStorage            localStorage + 远程
```

### 2. 技术设计文档

#### 多智能体平台 V1

**文件:** `multi-agent-v1-technical-design.md`

多智能体门店施工管理平台的蓝图。定义了：

- **7 种智能体类型：** 客服、品牌需求、项目经理、资源调度、资源执行、质检、结算对账
- **智能体运行时：** 注册表、提示词模板中心、工具路由器、上下文构建器、决策日志记录器、降级处理器
- **架构层：** 前端 → API 网关 → 业务服务 → 智能体编排 → 数据层 → 外部能力
- **关键原则：** 智能体是受控的节点能力，而非自治系统。所有智能体输出必须是结构化的 JSON。

#### 状态机设计

**文件:** `state-machine-design.md`

所有核心业务对象的全面状态机规范：

| 对象     | 状态数    | 关键转换                                                         |
| -------- | --------- | ---------------------------------------------------------------- |
| 项目     | 9 个状态  | `待立项 → 待确认 → 待拆解 → 执行中 → 待验收 → 待结算 → 已归档`   |
| 任务     | 10 个状态 | `草稿 → 待分配 → 待执行 → 执行中 → 待提交 → 待验收 → 已完成`     |
| 采购     | 8 个状态  | `待申请 → 待审批 → 待下单 → 待到货 → 已到货`                     |
| 验收     | 8 个状态  | `待申请 → 待初验 → 初验通过/不通过 → 整改中 → 待复验 → 复验通过` |
| 结算     | 5 个状态  | `待生成建议 → 待审核 → 待确认 → 已完成`                          |
| 资产归档 | 5 个状态  | `待登记 → 待验收归档 → 已归档`                                   |
| 工单     | 5 个状态  | `待分发 → 处理中 → 待回告 → 已关闭`                              |

每个状态机定义了：

- 转换的守卫条件
- 跨对象联动规则
- 审计要求
- 智能体参与边界

#### 任务树建模

**文件:** `task-tree-modeling.md`

定义了具有 4 层层次结构的任务执行模型：

```
项目（WBS 根节点）
  └── 工作包（成本聚合）
        └── 任务（原子执行单元）
              └── 子任务（可选协调）
```

**关键概念：**

- 任务是最小的可执行单元
- 工作包映射到成本账户
- 服务是带有合同的外包工作包
- 标准绑定和快照确保可追溯性
- 结果对象（采购订单、验收报告等）与任务分离

#### 标准库结构

**文件:** `structured-standard-library.md`

将标准建模为结构化的、可计算的对象，而非文档：

```
标准来源 → 标准文档 → 标准条款 → 规则项 → 执行/验收标准 → 标准包 → 任务绑定 → 标准快照
```

**V1 范围：** 专注于将执行标准和验收标准作为任务模板可直接引用的对象。完整的来源到条款管道保留给 V2。

### 3. 项目规则与契约

#### 项目规则

**文件:** `project-rules.md`

在所有开发领域建立强制性规则：

- **业务建模：** 对象分层（项目 → 任务 → 标准 → 结果对象）
- **状态机：** 统一的状态定义和守卫规则
- **UI 实现：** 共享侧边栏/页眉组件，一致的设计系统
- **编码标准：** TypeScript 严格模式、工程注释、目录约定
- **智能体治理：** 受控的节点能力、审计要求
- **安全：** RBAC、数据隔离、审计追踪

#### 模板数据契约

**文件:** `template-data-contract.md`

定义了模板管理与任务实例化之间的结构化契约：

- **项目模板：** 定义项目容器初始化（阶段、里程碑、标准上下文）
- **任务模板：** 定义任务节点结构、依赖关系、角色分配、标准绑定
- **实例字段与模板字段：** 清晰分离 — 运行时字段（状态、实际日期、进度）不存储在模板中
- **版本控制：** 语义化版本控制，附带兼容性策略（`x.0.0` 破坏性变更，`1.x.0` 向后兼容）
