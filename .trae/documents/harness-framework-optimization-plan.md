# Harness 工程框架优化建议

## 一、当前框架优势（先肯定）

1. **七阶段流水线设计清晰** — 定义→规划→构建→测试→评审→交付→进化，覆盖完整软件生命周期
2. **四专业角色分工明确** — 产品、设计、开发、测试各有专属技能和指标
3. **三组评审把关机制** — 评审小组、计划评审、验收小组形成质量防线
4. **51项指标量化体系** — 覆盖5个考核节奏（每任务/每日/每周/每月/每发布）
5. **L2.5自适应治理** — 基于指标表现自动调节Agent行为，形成测量→反馈→矫正闭环
6. **72技能体系** — 按专业分配，支持按需加载

---

## 二、发现的痛点与优化建议

### 2.1 流程执行层面

#### 痛点 1：评审小组（阶段1）可能流于形式

**现状**：评审小组要求"四人并行独立输出"，但框架未规定评审的**最低时间投入**或**输出质量标准**。
**风险**：复杂任务可能因评审仓促导致后期返工。

**优化建议**：

- 增加评审**时间盒**约束：L1任务≥15分钟，L2任务≥30分钟，L3任务≥60分钟
- 评审输出必须包含：**风险清单**（至少3条）+ **替代方案**（至少1个）
- 引入"评审质量"指标：由人类在阶段5反向评估阶段1评审的预见性

#### 痛点 2：阶段3（构建）与阶段4（测试）的反馈循环过长

**现状**：构建完成后才进入测试阶段，发现问题需回到构建阶段修复。
**风险**：小问题累积成大返工。

**优化建议**：

- 在阶段3内嵌**微型测试循环**：每完成一个子任务立即运行相关测试
- 开发工程师的"自检"步骤明确包含：**单元测试先行**（TDD）+ **lint检查**
- 阶段3出口条件增加：**子任务级测试覆盖率≥70%**

#### 痛点 3：进化阶段（阶段7）的触发条件模糊

**现状**："模式已记录 + 知识库有更新"作为出口条件，但缺乏量化标准。
**风险**：进化阶段容易被跳过，框架无法真正自进化。

**优化建议**：

- 明确进化触发阈值：
  - L1：每任务必须产出 ≥1 条模式记录
  - L2：每周累积 ≥5 条模式才触发知识库更新
  - L3：每月累积 ≥3 个文档变更才触发框架评估
- 增加"进化贡献度"指标：记录每个任务对知识库的实际贡献

### 2.2 角色协作层面

#### 痛点 4：产品经理（林墨）职责过重

**现状**：林墨参与6个阶段（定义、规划、测试、评审、交付、进化），是3个评审组的组长，还要负责日志、统计、提交、PR。
**风险**：成为瓶颈，且可能因过度参与而削弱其他角色的主动性。

**优化建议**：

- **交付阶段（阶段6）拆分**：日志写入由开发工程师负责，统计由测试工程师汇总，产品负责最终审核
- **进化阶段（阶段7）引入轮换制**：每月由不同角色主导（产品→设计→开发→测试循环）
- **评审组长增加副职**：复杂任务（L3）设"副组长"，由测试工程师担任，分担裁定压力

#### 痛点 5：开发工程师（陈锋）在评审阶段（阶段5）参与度不足

**现状**：角色矩阵显示开发工程师不参与评审阶段（—）。
**风险**：代码质量问题可能在评审阶段才发现，增加返工成本。

**优化建议**：

- 开发工程师在阶段5承担**"代码解释者"**角色：向验收组讲解架构决策和技术难点
- 增加"代码自审"前置步骤：开发在提交前必须完成自审清单（参考 `code-review-checklist.md`）
- 开发工程师的指标增加"代码一次通过率"，与评审阶段挂钩

#### 痛点 6：测试工程师（周严）在构建阶段（阶段3）缺位

**现状**：测试工程师不参与构建阶段。
**风险**：测试策略可能在规划阶段设计后，在构建阶段被遗忘或变形。

**优化建议**：

- 构建阶段增加**"测试左移"**机制：测试工程师每2天检查一次构建进度，确保测试策略落地
- 引入"测试契约"：开发工程师在构建阶段必须产出与测试计划对应的测试用例草稿

### 2.3 指标与考核层面

#### 痛点 7：44项指标中"自报"指标占比过高（14项），存在主观性风险

**现状**：自报指标包括需求偏差度、返工轮次、计划准确率等，依赖Agent自我评估。
**风险**：Agent可能倾向于低估问题（乐观偏差）。

**优化建议**：

- **交叉验证机制**：自报指标必须与客观数据交叉验证
  - 例："需求偏差度" ↔ 阶段5验收时人类评估的实际偏差
  - 例："返工轮次" ↔ git commit 历史中的回退/修复记录
- **引入"指标可信度"评分**：人类定期抽查自报指标，标记可信度等级
- **减少自报指标数量**：将可自动采集的指标（如返工次数）改为自动采集

#### 痛点 8：通用考核指标中的"反对意见率"和"盲从率"难以量化

**现状**：要求"反对意见率 ≥ 0.3/任务"和"盲从率 = 0"。
**风险**：Agent可能为了达标而故意提出无意义的反对意见。

**优化建议**：

- 将"反对意见率"改为**"有价值的反对意见率"**：由人类在评审阶段评估每条反对意见的质量
- 增加"反对意见采纳率"指标：被采纳的反对意见 / 总反对意见，目标 ≥ 30%
- 引入"建设性对话深度"指标：衡量讨论中观点交锋的质量（而非数量）

#### 痛点 9：缺乏"团队协作流畅度"指标

**现状**：指标聚焦个人产出，未衡量角色间协作效率。
**风险**：角色间信息传递损耗未被监控。

**优化建议**：

- 新增指标：
  - **交接清晰度**：下游角色对上游交付物的理解准确度（由下游在阶段开始时评分）
  - **阻塞等待时间**：每个阶段因等待其他角色而产生的平均等待时长
  - **信息回传率**：下游发现上游问题后，成功回传并解决的比例

### 2.4 技能与工具层面

#### 痛点 10：72技能缺乏"熟练度"评估

**现状**：技能只有"有/无"状态，没有使用质量评估。
**风险**：Agent可能加载了技能但未正确应用。

**优化建议**：

- 引入**技能熟练度等级**：
  - L1 了解：知道何时加载
  - L2 掌握：能正确应用产出合格结果
  - L3 精通：能灵活运用并优化产出
- 熟练度通过**技能应用质量**评估：由验收组在阶段5评估技能产出的质量
- 框架根据熟练度动态调整：L1技能强制加载指导文档，L3技能简化加载

#### 痛点 11：MCP工具利用率指标（≥5/周）过低

**现状**：要求每周使用MCP工具≥5次。
**风险**：对于高频开发任务，这个阈值可能过低；对于低频任务，又可能过高。

**优化建议**：

- 改为**"工具利用率 = 实际使用次数 / 应使用次数"**，按任务类型动态计算
  - 例：UI任务应使用 chrome-devtools，未使用则扣分
  - 例：代码重构任务应使用 GitNexus，未使用则扣分
- 引入"工具使用质量"评估：使用工具后是否产出了有效结果

### 2.5 自适应治理层面

#### 痛点 12：L2.5自适应治理的调节动作缺乏"渐进性"

**现状**：调节动作从L1的"注入提示"直接跳到L2的"强制技能"，再到L3的"冻结自主决策"。
**风险**：调节过于突兀，可能影响Agent的正常工作流。

**优化建议**：

- 增加**渐进式调节阶梯**：
  - L1：提示（inject_prompt）
  - L1.5：警告（warning_banner）— 在Agent输出中显示黄色警告条
  - L2：强制（force_skill）
  - L2.5：限制（restrict_autonomy）— 限制某些自主决策权限
  - L3：升级（escalate_depth）
  - L3.5：监督（supervised_mode）— 所有动作需人类确认
  - L4：冻结（freeze_autopilot）
- 每个调节动作生效前，Agent必须**确认已理解**调节原因

#### 痛点 13：调节的"7天过期"机制可能过于僵化

**现状**：所有调节7天后自动过期。
**风险**：对于持续性问题（如长期代码质量差），7天可能不足以形成习惯。

**优化建议**：

- 引入**调节续期评估**：过期前自动评估指标是否恢复，未恢复则续期
- 区分**临时调节**和**持久调节**：
  - 临时调节（如inject_prompt）：7天过期
  - 持久调节（如force_skill）：需连续14天指标正常才解除
- 人类可手动标记某调节为"长期生效"

### 2.6 知识管理层面

#### 痛点 14：知识库更新依赖人工判断，缺乏自动化触发

**现状**：L3知识更新由产品"按需"触发。
**风险**：知识库可能长期不更新，导致Agent使用过时信息。

**优化建议**：

- 引入**知识库健康度指标**：
  - 文档新鲜度：最近更新时间与当前时间的差值
  - 引用准确率：Agent引用文档时，文档内容与实际代码的一致性
  - 知识缺口：新出现的代码模式未在知识库中记录的比例
- 自动化触发：当健康度指标低于阈值时，自动生成知识库更新任务

#### 痛点 15：MEMORY.md 的写入规范不够具体

**现状**：MEMORY.md用于记录长期决策、偏好、模式，但缺乏写入格式和更新频率要求。
**风险**：MEMORY.md可能变得混乱，难以检索。

**优化建议**：

- 制定**MEMORY.md写入模板**：
  ```markdown
  ## 模式记录 #{n}

  | 发现时间 | YYYY-MM-DD |
  | 发现来源 | 任务ID / 评审反馈 |
  | 模式描述 | 一句话描述 |
  | 适用场景 | 何时应用 |
  | 反模式 | 何时不适用 |
  | 验证状态 | 已验证 / 待验证 |
  ```
- 引入**MEMORY.md自动索引**：每次写入时自动更新目录，确保可检索
- 每月进行一次**MEMORY.md清理**：归档过期模式，合并重复记录

### 2.7 复制与扩展层面

#### 痛点 16：复制指南（08-replication-guide.md）对"技能重写"的要求过于笼统

**现状**：要求"03-skills.md 全量重写"，但未提供重写的方法论。
**风险**：新项目可能遗漏关键技能或错误分配技能。

**优化建议**：

- 提供**技能映射工作表**：
  | 新项目技术栈 | 原框架技能 | 保留/替换/新增 | 替换为 |
  |------------|----------|-------------|-------|
  | React → Vue | frontend-ui-engineering | 替换 | vue-ui-engineering |
- 提供**技能优先级评估矩阵**：按"使用频率 × 影响程度"排序，确保高优先级技能不遗漏
- 增加"技能验证清单"：复制后运行一次全技能加载测试，确保无缺失

## 三、优化优先级排序

| 优先级 | 优化项                        | 影响范围   | 实施难度 |
| ------ | ----------------------------- | ---------- | -------- |
| P0     | 评审时间盒约束 + 输出质量标准 | 阶段1      | 低       |
| P0     | 自报指标交叉验证机制          | 考核体系   | 中       |
| P1     | 构建阶段内嵌微型测试循环      | 阶段3      | 中       |
| P1     | 产品经理职责拆分（交付/进化） | 角色体系   | 低       |
| P1     | 技能熟练度评估                | 技能体系   | 中       |
| P2     | 渐进式调节阶梯                | 自适应治理 | 中       |
| P2     | 团队协作流畅度指标            | 考核体系   | 中       |
| P2     | MEMORY.md写入模板             | 知识管理   | 低       |
| P3     | 知识库健康度自动化            | 知识管理   | 高       |
| P3     | 复制指南技能映射工作表        | 扩展性     | 低       |

## 四、实施建议

1. **短期（1-2周）**：实施P0和P1中的低难度项，快速见效
2. **中期（1个月）**：实施P1和P2项，完善框架核心机制
3. **长期（季度）**：实施P3项，提升框架智能化水平

---

## 五、总结

Harness框架整体设计优秀，七阶段流水线、四角色分工、三评审机制、51指标量化、L2.5自适应治理构成了完整的AI驱动开发体系。

主要优化方向：

1. **强化评审质量** — 时间盒 + 输出标准
2. **平衡角色负载** — 产品职责拆分 + 开发/测试深度参与
3. **提升指标客观性** — 交叉验证 + 自动采集
4. **细化技能管理** — 熟练度评估 + 质量追踪
5. **优化自适应治理** — 渐进调节 + 灵活过期
6. **完善知识管理** — 自动化触发 + 规范写入

这些优化将使框架从"可用"走向"好用"，从"规范"走向"智能"。
