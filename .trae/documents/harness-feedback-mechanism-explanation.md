# Harness 反馈纠偏机制 — 讲解计划

## 目标

用产品经理能理解的方式，讲解 harness 框架的反馈纠偏机制全链路，并在理解基础上讨论改进方向。

## 受众

产品经理（不懂代码），关注产品逻辑和设计合理性。

## 讲解结构（5 个环节 + 改进讨论）

### 第一环节：为什么需要这个机制？（问题背景）

- 多 Agent 协作天然会"跑偏"：跳过规定技能、用英文思考、需求理解偏差
- 人工监控不可持续，需要自动化"自我纠偏"
- 用一个生活化类比帮助理解：自动驾驶的偏离预警 + 车道保持

### 第二环节：全景图 — 五段闭环

- 图解：检测 → 生成 → 送达 → 验证 → 自进化 → 回到检测
- 每段一句话说清楚干什么
- 关键角色：Agent（数据生产者）、定时任务（扫描者）、AGENTS.md（送达通道）

### 第三环节：逐段拆解

#### ① 检测 — Agent 的"工作日志"

- Agent 每天在工作日志中填写"避坑"自检块
- 记录了：跳过了多少次技能、用了多少次英文思考、遗漏了几次编码规范等
- 类比：医生写的查房记录

#### ② 生成 — 定时"体检" + 自动"开处方"

- 每周五自动运行评估引擎
- 扫描最近 14 天的所有自检记录
- 用 8 条规则匹配违规模式（如"技能遗漏率 ≥3次/日 → 触发强制技能注入"）
- 严重度分 L1-L4 四级，L3 及以上需要人类审批
- 产出三样东西：调节档案、矫正指令、人机待办清单

#### ③ 送达 — 矫正指令如何到达 Agent

- 矫正指令写入 AGENTS.md（每个 Agent 会话启动时自动加载的文件）
- L1-L2：自动生效，Agent 必须遵守
- L3-L4：仅显示提醒，等待人类决策后执行

#### ④ 验证 — "吃了药有没有好转"

- 对比矫正前 7 天 vs 矫正后 7 天的同一指标数据
- 三种结论：改善（自动移除调节）、未改善（保持观察）、恶化（升级预警）
- 恶化时不自动做更高等级修改，而是提交给人类审批

#### ⑤ 自进化 — 框架自我修复

- 还有一个独立的"真相源校验器"，检查框架配置文件是否一致
- 发现不一致时 CI 报警
- 确保框架本身的配置不腐化

### 第四环节：当前状态评估

- 8 个修复项状态（7 完成 / 1 待确认）
- 8 条当前活跃调节（4 个角色 × 2 条）
- 首次验证报告：全部改善
- 已知待办：AGENTS.md 矫正块待注入（已就位，需运行一次脚本）

### 第五环节：改进讨论

基于你的"想法"展开讨论，可能的改进方向：

- 方向 A：增加指标覆盖面（当前的 8 条规则是否够？）
- 方向 B：缩短反馈周期（从周级缩短到日级？）
- 方向 C：增加人类介入的可视化界面
- 方向 D：防止"过度纠正"的熔断机制
- 方向 E：你的具体想法

## 假设

- 对话形式，交互式推进
- 每个环节讲完后停下来确认理解，再进入下一环
- 改进讨论放在最后，不打断讲解节奏
