# Agent 自我进化精准度提升方案

> 写给产品经理的学习型方案，带你理解"自适应调节系统"为什么不够准、怎么改、改完怎么看效果。

---

## 一、背景：什么是"自适应调节"？

先理解当前的进化系统是怎么工作的：

```
┌─────────────────────────────────────────────────────┐
│              当前的"自适应调节"流程                      │
│                                                     │
│  ① Agent 会话中自我检查                                │
│     └→ 记录违规到 memory/YYYY-MM-DD.md               │
│        （例如："跳skill 3次"）                         │
│                                                     │
│  ② 每周五，evaluate-adjustments.py 扫描日志             │
│     └→ 提取违规数据                                   │
│        （例如：本周累计 跳skill=5次）                    │
│                                                     │
│  ③ 匹配 8 条硬编码触发规则                              │
│     └→ 如果 跳skill ≥ 3 → 生成 force_skill 调节         │
│                                                     │
│  ④ 写入 AGENTS.md 矫正块                               │
│     └→ 下次 Agent 会话时读取，行为受约束                   │
│                                                     │
│  ⑤ 7天后自动过期                                      │
└─────────────────────────────────────────────────────┘
```

**核心问题**：这个流程像什么？像一台只有"温度计"的空调——它只知道温度到了30度就开制冷，但不知道：

- 温度是真的热还是温度计坏了？（**数据不可靠**）
- 温度在上升还是下降？（**无趋势**）
- 开了制冷后温度降了吗？（**无效果验证**）
- 有人觉得热吗？（**无用户反馈**）

## 二、精准度不足的根因分析

我发现了 **4 个链路层**的问题，每个链路解决一个最关键的短板。

### 2.1 数据层：唯一信号源是 Agent 自己

| 问题     | 当前状态                         | 后果                       |
| -------- | -------------------------------- | -------------------------- | ------------------------ |
| 数据来源 | 仅 Agent 自检块（`               | 避坑 \| 跳skill 3次 \|`）  | Agent 既是运动员又是裁判 |
| 虚假信号 | 无法区分"真违规"和"误报"         | 误调节，Agent 被不必要约束 |
| 缺失信号 | 没有用户满意度、没有产品经理反馈 | 调节方向与真实需求脱节     |

**类比**：一个 App 只靠"用户自己的使用日志"来判断体验好坏，不看评分、不看出错率、不看用户反馈。

### 2.2 分析层：二进制决策 = 0 或 1，没有灰度

| 问题   | 当前代码               | 问题所在                                         |
| ------ | ---------------------- | ------------------------------------------------ |
| 硬阈值 | `if val >= threshold:` | 2次不触发，3次触发——区别在哪？                   |
| 无权重 | 3次跳skill = 3次反模式 | 不同违规的严重程度一样？                         |
| 无趋势 | 只算累计值             | "上周5次这周2次"和"上周0次这周3次"触发相同的调节 |

当前匹配代码（来自 evaluate-adjustments.py 第 221-232 行）：

```python
def match_rules(metrics, role_hint=None):
    for rule in TRIGGER_RULES:
        if rule["role"] != "全部" and rule["role"] != role_hint:
            continue
        for metric_name, metric_val in metrics.items():
            if metric_name == rule["metric"]:
                val = metric_val["cumulative"]
                if val >= rule["threshold"]:
                    matched.append(rule)  # 只看阈值，不看趋势、不看置信度
```

**类比**：体温 37.4°C 和 39.5°C 的处理方式完全一样——都吃退烧药。

### 2.3 执行层：调了之后不检查效果

| 问题       | 当前状态           | 后果                                    |
| ---------- | ------------------ | --------------------------------------- |
| 无效果追踪 | 调节生成后就结束了 | 不知道调节是否改善了问题                |
| 固定过期   | 所有调节都是7天    | L1 的"提醒一下"和 L3 的"升级流程"一样长 |
| 无增量调节 | 不满阈值就不管     | "差一点触发"的预警信号丢失              |

### 2.4 反馈层：产品经理无法参与进化

| 问题          | 当前状态                           | 后果                                   |
| ------------- | ---------------------------------- | -------------------------------------- |
| 只有 override | 人类只能改 status:overridden       | 要么接受、要么全关，没有中间状态       |
| 无评分机制    | 系统不知道自己调得好不好           | 每次评估都是从零开始，没有经验累积     |
| 无建议输入    | 产品经理不能告诉系统"重点关注什么" | 系统调节方向固定，不能适应项目阶段变化 |

## 三、改进方案设计（全链路 × 小步快跑）

核心设计原则：**每个链路只改最关键的 1 件事**，确保 1-2 天可完成。

### 3.1 数据层改进：引入"反馈信号"

#### 改动内容

在 daily log 末尾增加**反馈标记行**，产品经理或人类审查者可以在日志里加一行标记：

```markdown
<!-- feedback: accurate -->    <!-- 这次自检是准的 -->
<!-- feedback: false_alarm -->  <!-- 这次是误报 -->
<!-- feedback: missed -->       <!-- 漏掉了真正的违规 -->
```

#### 为什么有效

- 给 Agent 自报数据加了一个"质检"环节
- 即使10条里只有2条有反馈，也能大幅提升数据可信度
- 反馈数据进入分析引擎，用于计算**信号置信度**

#### 产品设计思考

- **为什么用 HTML 注释**？不影响 markdown 渲染，产品经理可以不关心它的存在
- **为什么要简化为3种**？越多选项越没人用。3种覆盖了"准/误报/漏报"三种核心场景
- **为什么不是必填**？降低使用门槛。可填可不填，填了系统更好，不填也照常运行

#### 改动范围

- 修改 `evaluate-adjustments.py`：增加 `parse_feedback_markers()` 函数
- 输出结构增加 `signal_confidence` 字段

---

### 3.2 分析层改进：引入"置信度评分"

#### 改动内容

把二进制阈值匹配改为**加权评分模型**：

```
评分 = 违规值 × 严重度权重 × 趋势系数 × 置信度
```

| 因子       | 含义                 | 计算方式                             |
| ---------- | -------------------- | ------------------------------------ |
| 违规值     | 原始违规次数         | 直接取自自检块                       |
| 严重度权重 | 不同违规的相对严重性 | 配置表中增加 weight 字段             |
| 趋势系数   | 恶化/好转/持平       | 对比前一周的数据                     |
| 置信度     | 信号可靠程度         | 反馈标记越多、准确率越高、置信度越高 |

#### 触发逻辑变化

**之前**（二进制决策）：

```python
# 跳skill ≥ 3 → 触发
if violations["跳skill"] >= 3:
    trigger_rule("force_skill")
```

**之后**（评分决策）：

```python
# 评分 = 3次 × 权重0.8 × 趋势1.2 × 置信度0.7 = 2.02
# 评分 ≥ 触发阈值 → 触发
score = violations["跳skill"] * rule.weight * trend_factor * confidence
if score >= rule.trigger_score:
    trigger_rule("force_skill", severity=calculate_level(score))
```

#### 产品设计思考

- **为什么加分比加规则好**？加规则让系统变复杂，加分让系统变聪明。规则是"如果...就..."，分是"多维度综合判断"
- **为什么保留阈值**？因为需要"触发门槛"。但阈值不再是 0/1 门，而是"评分基准线"
- **为什么趋势系数是核心**？因为它区分了"老问题在改善"和"新问题在恶化"，这两种情况需要完全不同的调节策略

#### 改动范围

- 修改 TRIGGER_RULES：增加 `weight`、`trend_sensitivity` 字段
- 新增 `compute_score()` 函数
- 新增 `calculate_trend()` 函数（对比连续两周数据）
- 新增 `calculate_confidence()` 函数（基于反馈标记）

---

### 3.3 执行层改进：调节效果追踪 + 弹性过期

#### 改动内容 A：效果追踪

在每次生成调节时，记录**调节前的指标值**：

```yaml
active_adjustments:
  - id: adj-20260512-dev-TR-001
    trigger: '技能遗漏率高'
    applied_value: 5 # 调节前的违规值（新增）
    current_value: 3 # 当前值（新增，每次评估更新）
    trend: 'improving' # improving/worsening/stable（新增）
    expires: '2026-05-19'
```

下次评估时自动对比：

- `applied_value → current_value`：是好转还是恶化了？
- 好转 → 降低调节级别或提前过期
- 恶化 → 升级调节（如 L1→L2）
- 持平 → 延长调节周期

#### 改动内容 B：弹性过期

当前是死板的"所有调节7天过期"，改为：

| 级别 | 基础过期 | 自动续期条件                  |
| ---- | -------- | ----------------------------- |
| L1   | 3 天     | 指标恶化 → 延长至 7 天        |
| L2   | 7 天     | 指标恶化 → 延长至 14 天       |
| L3   | 14 天    | 指标恶化 → 转为 L4 + 人工审批 |
| L4   | 人类解除 | 永不自动过期                  |

#### 产品设计思考

- **为什么效果追踪是核心功能**？因为没有效果反馈的调节就像没有测试的代码——你不知道它是不是在起作用
- **为什么弹性过期比固定过期好**？因为不同问题的"消化时间"不同。L1 的"提醒一下"两天就够了，L3 的"升级流程"需要更长时间验证
- **为什么"自动续期"需要条件**？防止无限循环。只有指标恶化才续期，好转或稳定就正常过期

#### 改动范围

- 修改 `adjustments.yaml` 结构：增加 `applied_value`、`current_value`、`trend` 字段
- 修改 `evaluate-adjustments.py` 的 `evaluate_effectiveness()` 新函数
- 修改 `expire_old_adjustments()`：基于趋势的动态过期

### 3.4 反馈层改进：人工评分机制

#### 改动内容

在 `.harness/` 下增加一个**人工反馈通道**：

```yaml
# .harness/feedback.yaml
feedbacks:
  - adjustment_id: adj-20260512-dev-TR-001
    rating: -1 # +1: 认同、0: 中立、-1: 不认同
    comment: '跳skill主要是因为新项目不熟悉，不是态度问题' # 可选
    created_at: '2026-05-13'
```

产品经理可以在每个调节生效后，对调节行为打分。评分进入下次评估：

- `rating: +1` → 该调节类型的置信度提高，未来类似情况优先触发
- `rating: -1` → 该调节类型的效果存疑，未来触发阈值提高
- `rating: 0` → 无影响

#### 为什么这是关键一环

产品经理的反馈是**人类监督 AI 进化的最高效方式**。不需要改代码、不需要理解算法，只需要打分。系统会自动学习：

```
你打了3次 -1 给 force_skill 调节 → 系统降低 force_skill 的触发权重
你打了2次 +1 给 inject_prompt → 系统优先使用 inject_prompt
```

这就是**人类反馈强化学习（RLHF）**的最简实践版。

#### 产品设计思考

- **为什么是 +1/0/-1 三个值**？太多值让人犹豫，太少没有区分度。三个值是心理学上的最优选择
- **为什么 feedback 不直接放在 adjustments.yaml 里**？分离关注点。adjustments 是系统自动生成的，feedback 是人类输入的，放在不同文件降低误编辑风险
- **为什么 comment 是可选的**？强制写评论会降低参与率。允许纯评分，愿意的话再写原因

#### 改动范围

- 新建 `read_feedback()` 函数
- 新建 `write_feedback()` 函数
- 修改评分计算：加入 feedback 影响因子
- 在周评估中新增反馈收集步骤

### 数据流对比

【改进前】

【改进后】

## 六、验证方案

### 怎么判断改成功了？

| 验收标准               | 测量方式                            | 目标值           |
| ---------------------- | ----------------------------------- | ---------------- |
| 调节触发的误报率下降   | 统计 feedback -1 的比例             | 误报率 < 20%     |
| 调节过期前的指标改善率 | 对比 applied_value vs current_value | > 60% 的调节有效 |
| 产品经理参与率         | 每周至少 3 条 feedback              | 连续 4 周达标    |
| 人类 override 次数下降 | 统计 overridden 次数                | 比原来减少 50%   |

### 灰度策略

1. **第一周**：新老逻辑并行运行，只记录不调节（dry-run + log）
2. **第二周**：新逻辑生效，保留老逻辑的 override 通道
3. **第三周**：完全切换到新逻辑，老代码保留作为回退方案

## 七、产品经理学习要点

通过这个改方案例，你可以学到：

### 7.1 系统设计思维

- **闭环思维**：任何系统都应该是闭环的——有输入、有处理、有输出、有反馈
- **渐进式复杂**：不要一步到位，先做最简可行版本，让系统运行起来再优化
- **信号 vs 噪声**：区分真实信号和噪声是数据分析的核心能力

### 7.2 AI 产品设计原则

- **人在回路中（Human-in-the-Loop）**：纯自动系统最终会偏离目标，需要人类的校准信号
- **反馈粒度**：+1/0/-1 比 5 星评分更有效，因为决策成本低
- **置信度优先**：不是所有数据都值得信任。给数据打上"置信度标签"比"不用它"更好

### 7.3 工程师思维入门

- **阈值不是拍脑袋的**：为什么是 3 不是 2？为什么是 7 天不是 5 天？每个数字都应该有"为什么"
- **保持向后兼容**：改结构时保证旧数据还能读，旧格式还能用
- **灰度发布**：新功能不要全量上线，先小范围验证
