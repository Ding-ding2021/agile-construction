## 4. 审查方法

### 4.1 文档一致性审查

对 10 份 harness 文档 + 4 份角色文档进行交叉引用检查：

1. **编号一致性**：检查 frontmatter 中的 id、number 是否唯一、连续
2. **引用有效性**：检查所有内部链接（`[text](path)`）是否指向存在的文件
3. **状态一致性**：检查文档 status 与内容是否匹配（active/draft/archived）
4. **条款一致性**：检查相同概念在不同文档中的定义是否一致

### 4.2 配置同步审查

对比 H00~H09 与 C01~C03：

1. **角色同步**：检查文档中的角色定义与 registry.yaml 中的角色配置是否一致
2. **技能同步**：检查文档中的技能列表与 registry.yaml 中的技能列表是否一致
3. **指标同步**：检查文档中的指标定义与 registry.yaml 中的指标配置是否一致
4. **钩子同步**：检查文档中的钩子定义与实际 .husky/、.github/workflows/ 配置是否一致

### 4.3 运行时证据审查

分析 memory/ 日志和实际配置：

1. **技能调用率**：统计各技能的实际调用次数与应调用次数的比例
2. **流程合规率**：统计各任务是否按 Harness 工作流执行
3. **门禁通过率**：统计 lint/build/test 的通过率
4. **评审参与率**：统计评估组/验收组的实际参与情况

## 5. 评分标准

每个维度按以下标准评分：

| 得分   | 等级   | 说明                           |
| ------ | ------ | ------------------------------ |
| 90-100 | 优秀   | 规范执行良好，偶有小偏差       |
| 70-89  | 良好   | 规范基本执行，有部分偏差需改进 |
| 50-69  | 及格   | 规范执行不足，有明显问题需关注 |
| 30-49  | 不及格 | 规范执行差，大量问题需整改     |
| 0-29   | 严重   | 规范基本未执行，需重新设计     |

总分 = Σ(维度得分 × 维度权重)

## 6. 输出物

1. **问题清单**：按维度分类列出所有发现的问题，含风险等级和建议修复方案
2. **评分报告**：各维度得分、总分、等级评定
3. **改进建议**：针对发现的问题提出具体改进措施

## 7. 执行步骤

1. **步骤 1**：文档一致性审查（10 份 harness 文档 + 4 份角色文档）
2. **步骤 2**：配置同步审查（registry.yaml、manifest.yaml、adjustments.yaml 与文档对比）
3. **步骤 3**：运行时证据审查（memory/ 日志、.husky/、.github/workflows/）
4. **步骤 4**：技能文件审查（62 个 SKILL.md 与文档映射对比）
5. **步骤 5**：汇总评分、生成问题清单与改进建议
