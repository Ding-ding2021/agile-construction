# 技能：前端 UI 集成

## 目的

在我们的主要 Web 应用程序中实现或扩展面向用户的工作流，集成**现有后端 API**并遵循我们的**设计系统、路由和测试约定**。

## 何时使用此技能

- 该功能主要是基于一个或多个现有 API 的 **UI/UX 变更**。
- 后端契约、认证模型和核心业务规则**已存在**。
- 变更**仅**影响 Web 前端（无 schema 或服务所有权变更）。

## 输入

- **功能描述**：用户流程和结果的简短说明。
- **相关 API**：端点、请求/响应类型以及源定义链接。
- **目标路由/组件**：路径、组件名称或功能模块。
- **设计参考**：Figma 链接或需参考的现有界面。
- **约束条件**：性能限制、可访问性要求和安全约束。

## 不包含范围

- 创建新的后端服务或更改持久数据模型。
- 修改认证/授权流程。
- 引入新的前端框架或设计系统。

## 约定

- **框架**：React + TypeScript。
- **路由**：使用现有路由器和路由布局模式。
- **样式**：使用内部设计系统组件（按钮、输入框、模态框、提示条等）。
- **状态管理**：优先使用现有状态库（如 React Query、Redux、Zustand），遵循已有模式。

## 必要行为

1. 实现 UI 变更时，所有 props 和 API 响应**必须强类型化**。
2. 使用现有原语处理加载、空数据、错误和成功状态。
3. 确保 UI **键盘可访问**且屏幕阅读器友好。
4. 在适用处尊重功能开关和发布机制。

## 必要产出

- 在相应功能模块中更新的组件和 hooks。
- 核心展示逻辑的**单元测试**。
- 新流程的**集成或组件测试**（如 React Testing Library、Cypress、Playwright），基于仓库已有工具。
- 简要的**CHANGELOG 或 PR 描述**，总结行为变更（放在 PR 中，而非此文件）。

## 实现检查清单

1. 定位相关功能模块和现有组件。
2. 确认后端 API 和类型，必要时更新共享 TypeScript 类型。
3. 实现 UI，通过现有数据层接入 API 调用。
4. 添加或更新测试以覆盖新行为和边界情况。
5. 运行必要的验证命令（见下文）。

## 验证

运行以下命令（根据项目实际情况调整）：

- `pnpm lint`
- `pnpm test -- --runInBand --watch=false`
- `pnpm typecheck`（如单独配置）

技能完成标准：

- 所有测试、linter 和类型检查通过。
- 新 UI 在正常、错误和边界情况下行为符合预期。
- 未修改无关文件或模块。

## 安全与升级

- 如果请求的变更需要修改后端契约，请**停止**并请求创建后端任务。
- 如果设计参考与现有可访问性标准冲突，优先保证可访问性，并在 PR 描述中说明差异。
