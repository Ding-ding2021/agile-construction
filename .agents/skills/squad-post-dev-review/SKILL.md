# Squad Post-Dev Review

## 概述

开发交付者标记完成后，派发验收小组从功能、代码质量、UI 三个视角并行验收，保障交付质量。

**硬性门禁**：验收组未全票通过，不得进入交付收尾阶段。

## 触发条件

- 开发交付者标记"完成待验收"
- 用户主动要求验收
- 例外（可跳过验收）：L1 低风险任务——组长单人验收即可

## 流程

### Step 1: 确定任务风险等级

| 等级 | 判断标准                        | 验收组配置        |
| ---- | ------------------------------- | ----------------- |
| L1   | 单文件/纯样式/文档/简单 bug fix | 组长单人验收      |
| L2   | 新增组件/单页面/小功能          | 组长 + 1 名验收员 |
| L3   | 跨多文件/架构/新数据模型        | 全量 4 人         |

### Step 2: 并行派发子代理

根据风险等级派发：

```
L3 全量：
  ├── agent-1: 功能验收员（v4-flash）
  ├── agent-2: 代码质量验收员（v4-pro）
  ├── agent-3: UI 验收员（v4-flash）
  └── agent-0: 组长（v4-pro）—— 同步等待，后续汇总

L2 精简：
  ├── agent-1: 验收员（v4-flash，任选一个视角）
  └── agent-0: 组长（v4-pro）

L1 单人：
  └── agent-0: 组长（v4-flash/go）
```

### Step 3: 各子代理输出独立验收报告

每个子代理必须输出自己的独立验收报告。

**功能验收员 checklist：**

- [ ] 需求逐条覆盖——评估报告中的每条需求都已实现
- [ ] 边界条件——空数据、极限值、异常输入已处理
- [ ] 错误场景——网络异常、权限不足、非法操作有反馈
- [ ] 测试覆盖——单元/组件测试覆盖了核心路径
- [ ] **E2E 覆盖——前端功能有 Playwright E2E 测试覆盖核心用户流程**
- [ ] E2E 通过——`npm run test:e2e` 零失败
- [ ] 回归验证——现有功能未被破坏

**代码质量验收员 checklist：**

- [ ] 编码规范——无 `any` 类型、命名规范、无多余注释
- [ ] 架构合规——未绕过状态机守卫、未直接操作 localStorage
- [ ] 无死代码——定义了未被使用的函数/变量/组件
- [ ] 无过度抽象——不为"未来可能用"创建抽象
- [ ] TDD 遵循度——测试先于实现编写
- [ ] 无 `@ts-ignore` / `@ts-expect-error`（除非有 Issue 引用）

**UI 验收员 checklist（纯代码审查，无需多模态）：**

- [ ] 色值/Token——所有颜色来自 `@theme` 而非硬编码
- [ ] 状态覆盖——loading / empty / error / success 各态齐全
- [ ] 暗色模式——`dark:` 前缀覆盖所有表层
- [ ] 响应式——sm/md/lg/xl 各断点 class 齐全
- [ ] 可访问性——`aria-label`、`role`、键盘导航已实现
- [ ] 交互反馈——hover / focus / active / transition 到位
- [ ] 空状态——无数据时显示占位 UI

### Step 4: 组长汇总

1. 收集全部独立报告
2. 判断结论：

| 结论                  | 条件              | 后续动作                             |
| --------------------- | ----------------- | ------------------------------------ |
| **APPROVED**          | 全员通过          | 进入交付收尾                         |
| **CHANGES REQUESTED** | 存在可修复问题    | 打回开发交付者修复，仅派问题角色重审 |
| **BLOCKED**           | 架构性/原则性问题 | 组长仲裁，升级给用户决策             |

3. 输出汇总验收报告（使用 agent-squad-protocol.md 中的模板）

### Step 5: 打回后增量重审

打回时**只派有问题的角色**重审，避免重复调用：

```
第一次验收：功能✅ 代码❌（死函数） UI✅
   ↓ 打回修复
第二次验收：仅代码质量验收员重审
   ↓ 通过
最终结论：APPROVED
```

### Step 6: 通过后通知收尾

验收通过后，通知开发交付者执行 `finishing-a-development-branch`。

## 铁律

- **验收组未全票通过，不交付**
- **子代理各自输出独立报告，不共享状态**
- **打回后增量重审，只派问题角色**
- **UI 验收通过代码审查 + checklist 完成，无需多模态**

## 常见错误

- 功能验收只看"实现了没"不看"边界条件处理好了没"
- 代码质量验收只检查格式不检查架构合规
- UI 验收员说"看不到渲染效果所以不做检查"
- 打回后全量重审导致成本翻倍

## 输出物清单

- [ ] 功能验收报告
- [ ] 代码质量验收报告
- [ ] UI 验收报告
- [ ] 组长汇总验收报告
- [ ] （如打回）修复后增量重审报告
