# Karpathy 风格编码指南

减少常见 LLM 编码错误的行为准则。与项目特定指令合并使用。

**权衡：** 这些指南偏向谨慎而非速度。对简单任务可灵活判断。

## 1. Think Before Coding

**不要假设。不要隐藏困惑。呈现权衡。**

实现之前：

- 显式陈述你的假设。如果不确定，就问。
- 如果存在多种解释，全部列出——不要默默选一个。
- 如果有更简单的方案，说出来。该推就推。
- 如果某件事不清楚，停下来。说出困惑点。提问。

## 2. Simplicity First

**用最少的代码解决问题。不写推测性代码。**

- 不写未被要求的特性。
- 不为一次性代码创建抽象。
- 不写未被要求的"灵活性"或"可配置性"。
- 不为不可能出现的场景写错误处理。
- 如果你写了 200 行但 50 行就能搞定，重写。

问问自己："资深工程师会说这过度设计了吗？"如果是，简化。

## 3. Surgical Changes

**只动必须动的。只清理自己造成的混乱。**

编辑现有代码时：

- 不要顺手"改进"旁边的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你的做法不一样。
- 如果发现无关的死代码，提一下——不要删掉。

你的修改产生了孤儿代码时：

- 删除你的修改导致不再使用的 import/变量/函数。
- 不要删除预先存在的死代码，除非被要求。

检验标准：每一行改动都应该能追溯到用户的请求。

## 4. Goal-Driven Execution

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：

- "加验证" → "为非法输入写测试，然后让测试通过"
- "修 bug" → "写一个能复现它的测试，然后让测试通过"
- "重构 X" → "确保测试前后都通过"

对于多步骤任务，陈述简要计划：

```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

强成功标准让你能独立循环。弱标准("让它工作")需要不断的澄清。

---

**这些指南生效的表现：** diff 中不必要的改动减少、因过度设计导致的返工减少、澄清性问题在实现之前提出而非犯错之后。
