## 三、技术债务与风险识别

### 3.1 高优先级技术债务

#### 🔴 债务1: 组件重复实现严重

根据 `component-refactoring-plan.md` 统计：

| 组件类型         | 当前实现数 | 目标 | 风险等级 |
| ---------------- | ---------- | ---- | -------- |
| 侧边栏           | 5套        | 1套  | 🔴 高    |
| 统计卡片         | 7+套       | 1套  | 🔴 高    |
| 页头             | 3套        | 1套  | 🟡 中    |
| `navigateByHash` | 重复5次    | 1处  | 🟡 中    |
| `isActive`逻辑   | 5个副本    | 1处  | 🟡 中    |

**影响**: 新增路由需修改5-6个文件，维护成本高，已出现路由跳转错误。

#### 🔴 债务2: CSS硬编码色值~200+处

- `digital-employee-page.css`: 17KB硬编码
- `procurement-management.css`: 10KB硬编码
- `project-detail.css`: 43KB硬编码

**影响**: 设计规范变更时无法统一调整，视觉一致性差。

#### 🟡 债务3: 状态机联动为mock

```typescript
// projectStatusMachine.ts
const enterStatusHooks: Partial<Record<ProjectStatus, string[]>> = {
  待拆解: ['联动：任务树初始化已触发（mock）'],
  执行中: ['联动：风险重算已触发（mock）'],
  待验收: ['联动：验收摘要生成已触发（mock）'],
}
```

**影响**: 状态流转后无真实业务联动，系统停留在"页面演示"层面。

#### 🟡 债务4: 任务前置依赖逻辑简化

```typescript
// taskStateMachine.guards.ts
function getPredecessorTasks(taskCode: string, allTasks: TaskItem[]): TaskItem[] {
  // 当前实现基于parentPath推断，实际应该使用task_relation表
  // 这里简化处理，返回同parentPath下的前序任务
```

**影响**: 依赖关系校验不准确，关键路径识别能力弱。

#### 🟡 债务5: 项目详情"执行"标签为占位

`ProjectDetailPage`中执行标签渲染的是计划标签内容，非真实执行视图。

**影响**: 6标签生命周期结构不完整，执行阶段缺少任务-WBS合并视图。

### 3.2 架构风险

| 风险                     | 等级  | 说明                                           |
| ------------------------ | ----- | ---------------------------------------------- |
| localStorage作为数据后端 | 🟡 中 | MVP阶段可用，但无法支撑多用户/多品牌隔离       |
| 无API层抽象              | 🟡 中 | 直接操作localStorage，后续接入后端需大面积改造 |
| 无权限控制实现           | 🔴 高 | PRD定义了品牌/平台/资源方权限，代码中无体现    |

| 无真实闭环验证           | 🔴 高 | 立项→任务→派单→执行→验收→结算链路未跑通        |

### 4.3 具体实施建议

#### 建议1: 先完成组件重构（参考已有计划）

`component-refactoring-plan.md` 已制定详细计划，建议立即执行：

```
阶段一（1-2天）: CSS变量补齐 + 路由集中化 + 共享组件目录
阶段二（3-4天）: AppSidebar + PageHeader + StatCard + TabNav开发
阶段三（3-4天）: 逐个模块替换旧组件
阶段四（2天）: 清理废弃代码 + 性能优化
```

**注意**: 共享组件目录 `src/components/shared/` 已部分创建（`AppSidebar`、`StatCard`、`PageHeader`已有基础实现），但尚未被业务模块采用。

#### 建议2: 项目详情页补齐为当前最高业务优先级

PRD明确要求：

> "产品验收以'生命周期闭环可执行'替代'页面数量覆盖'"

当前缺口：

- 执行标签 = 计划标签（占位）
- 监控标签缺少风险看板
- 收尾标签缺少结算建议联动

建议先补齐这3个标签的真实内容，使单项目生命周期可演示。

#### 建议3: 状态机联动从mock改为真实

当前状态流转后的hook都是mock字符串：

```typescript
// 当前
待拆解: ['联动：任务树初始化已触发（mock）']

// 目标
待拆解: ['任务树初始化', '里程碑生成', '标准包绑定']
```

建议为每个状态定义真实的联动动作列表，并在`App.tsx`的`transitionProjectStatus`中执行。

#### 建议4: 建立最小可演示闭环（MVP验证）

V1核心目标是验证：

> "跑通单品牌、单区域、标准店型的建店标准化闭环"

建议按以下顺序建立可演示链路：

```
1. 新建项目（空白/模板）
2. 项目详情-启动标签（范围确认+基准冻结）
3. 项目详情-计划标签（阶段+里程碑+WBS）
4. 项目详情-执行标签（任务推进+派单）
5. 任务中心（任务列表+详情+状态流转）
6. 项目详情-监控标签（验收+风险）
7. 项目详情-收尾标签（结算建议+归档）
```

#### 建议5: Agent能力分阶段引入

PRD定义了7个Agent，建议按优先级分阶段：

| 阶段     | Agent         | 触发节点             | 复杂度 |
| -------- | ------------- | -------------------- | ------ |
| 第一阶段 | 项目经理Agent | 任务树生成建议、催办 | 中     |
| 第一阶段 | 验收质检Agent | 资料完整性检查       | 低     |
| 第二阶段 | 品牌需求Agent | 立项信息补全         | 中     |
| 第二阶段 | 调度Agent     | 资源推荐             | 中     |
| 第三阶段 | 结算建议Agent | 差异分析             | 高     |
| 第三阶段 | 客服Agent     | 工单分发             | 高     |

第一阶段可先采用"规则引擎+模板匹配"模拟Agent输出，不依赖真实LLM。

## 五、结论

### 5.1 当前状态总结

**已完成（优势）**:

- ✅ 完整的需求文档体系（10+PRD，状态机，任务树建模）
- ✅ 前端工程基础（React+TS+Tailwind，Hash路由，懒加载）
- ✅ 项目+任务状态机框架（守卫校验基础版）
- ✅ 核心页面骨架（项目列表/详情，任务中心，人员管理）
- ✅ 组件重构计划与共享组件目录已启动

**待完成（缺口）**:

- ❌ 组件重复实现严重（5套侧边栏/7+套统计卡片）
- ❌ 状态机联动为mock，无真实业务联动
- ❌ Agent能力完全缺失（PRD核心卖点）
- ❌ 结果对象模型未建立（采购/验收/整改/资产）
- ❌ 权限控制未实现
- ❌ 工作台/数据中心缺失
- ❌ 后端API未对接（localStorage为数据层）

### 5.2 关键成功指标（下阶段目标）

建议下阶段（2-3周）聚焦以下可验证目标：

1. **组件重构完成**: 侧边栏/统计卡片/页头统一为1套，新增路由只需改1处
2. **项目生命周期可演示**: 6标签结构完整，执行标签有真实内容
3. **任务闭环可演示**: 任务从待分配→已完成的状态流转+守卫校验完整
4. **整改任务可演示**: 验收不通过→生成整改任务→复验通过的链路

### 5.3 最终评估

| 维度       | 评分       | 说明                               |
| ---------- | ---------- | ---------------------------------- |
| 需求完整性 | ⭐⭐⭐⭐⭐ | 文档体系完善，字段级定义到位       |
| 架构设计   | ⭐⭐⭐⭐   | 状态机+任务树设计合理，有扩展性    |
| 代码实现   | ⭐⭐⭐     | 骨架完成，但技术债务多，联动为mock |
| 可演示性   | ⭐⭐⭐     | 单页面可看，跨页面闭环未跑通       |
| 可维护性   | ⭐⭐       | 组件重复严重，CSS硬编码多          |

**总体评估**: 这是一个**文档驱动、设计先行、实现跟进**的项目。需求与架构层面成熟度高于代码实现层面。当前最紧迫的不是加功能，而是**还技术债务+补齐核心闭环**。建议先花1-2周完成组件重构和项目详情补齐，再进入功能扩展阶段。
