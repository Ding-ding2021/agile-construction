# 标准库结构化说明

> **文档版本**：V1.0
> **文档状态**：草稿中
> **适用阶段**：V1 / MVP
> **适用范围**：品牌方营建标准、国家/行业/地方标准、企业内部标准、供应商执行标准
> **关联文档**：`产品规划文档.md`、`任务树建模说明.md`、`多Agent建店管理平台_V1_PRD.md`、`多Agent建店管理平台_V1_技术方案.md`

---

## 1. 文档目标

本文档用于统一产品、设计、研发、测试对`标准库`的理解，明确标准如何被采集、结构化、版本化、绑定到任务，并最终驱动执行、验收、整改和资产沉淀。

标准库在本项目中不是资料仓库，也不是只读知识中心，而是系统的`专业能力底座`。

本项目必须统一以下口径：

- `标准来源必须可追溯`
- `标准条款必须可结构化`
- `结构化条款必须可转规则项`
- `规则项必须可绑定任务、模板和检查项`
- `执行中的任务必须固化标准快照`
- `标准库服务于任务树，而不是脱离任务树独立存在`

---

## 2. 建模定位

### 2.1 为什么要做标准库结构化

品牌方营建部门长期面临的核心问题，不是“没有标准”，而是：

- 标准分散在国家规范、行业文件、企业制度、品牌手册、供应商要求中
- 标准多以 PDF、Word、图片、表格存在，难以直接被系统使用
- 执行人员理解口径不一致
- 验收口径依赖个人经验
- 项目、采购、验收、整改之间缺少统一标准依据

因此，系统必须把标准从“文档”转为“可计算、可引用、可绑定、可追溯的结构化对象”。

### 2.2 标准库在系统中的角色

标准库承担 6 个职责：

1. 统一沉淀品牌方营建专业能力
2. 为任务树生成提供规则来源
3. 为执行动作提供操作依据
4. 为验收动作提供判断口径
5. 为 Agent 提供受控知识和模板基础
6. 为审计、争议处理、整改闭环提供依据链

### 2.3 建模原则

- `来源可追溯`：任何规则项都必须能追溯到原始标准来源
- `结构优先`：优先沉淀字段化内容，不只保存原文附件
- `规则可执行`：结构化结果必须可用于任务、校验、检查项生成
- `版本可治理`：标准修改必须有版本管理与生效边界
- `快照可回放`：执行中任务使用的标准必须能回放当时口径
- `人工可校对`：AI 可辅助抽取，但标准生效前必须可人工审核
- `高频优先`：V1 优先覆盖高频、标准化程度高的标准域

## 3. 适用范围与 V1 边界

### 3.1 适用范围

标准库用于承接以下类型标准：

- 国家标准
- 行业标准
- 地方标准
- 企业内部制度
- 品牌营建标准
- 供应商执行标准
- 项目级补充标准

标准库重点服务以下业务环节：

- 项目立项与模板匹配
- 任务树生成
- 执行指导
- 资料要求生成
- 验收检查项生成
- 整改要求生成
- 资产归档要求
- Agent 检索与结构化输出

### 3.2 V1 优先覆盖范围

V1 建议先覆盖以下高频标准域：

- 店型基础建设标准
- 施工执行标准
- 安全文明施工标准
- 材料与设备基础要求
- 验收标准
- 资料提交标准
- 资产归档标准

### 3.3 V1 建模边界

V1 阶段采取保守策略：

- 先支持结构化`条款`与`规则项`，不追求全部原文自动语义解析
- 先覆盖高频标准店型，不优先处理复杂非标项目
- 先支持规则配置和检查项生成，不做复杂法规推理引擎
- 允许原文附件与结构化对象并存
- AI 抽取结果默认需人工校对后生效
- 不支持自动判断跨标准冲突后的完全自主决策

## 4. 标准库总体分层

标准库建议采用如下分层模型：

1. `标准来源层`
2. `标准文档层`
3. `标准条款层`
4. `规则项层`
5. `执行标准 / 验收标准层`
6. `标准包 / 模板层`
7. `任务绑定 / 检查项层`
8. `标准快照层`

统一链路如下：

`标准来源 -> 标准文档 -> 标准条款 -> 规则项 -> 执行标准/验收标准 -> 标准包/模板 -> 任务绑定 -> 检查项 -> 标准快照`

这样设计的目的：

- 保证来源可追溯
- 保证原文与结构化对象有映射关系
- 保证业务端用的是规则与标准对象，而不是直接解析原文
- 保证执行过程可固化版本

## 5. 核心对象定义

### 5.1 标准来源 Standard Source

用于定义标准来自哪里。

典型来源：

- 国家标准
- 行业标准
- 地方标准
- 企业制度
- 品牌营建手册
- 供应商施工手册
- 项目补充说明

### 5.2 标准文档 Standard Document

用于保存具体标准文件及其元信息。

标准文档既可以是正式标准文件，也可以是企业内部操作手册、作业指引、验收规范。

### 5.3 标准条款 Standard Clause

用于表达从原始文档中拆出的最小语义条款。

条款是标准结构化的基本单元，但仍然保留强文本属性和原文映射关系。

### 5.4 规则项 Rule Item

规则项是把标准条款进一步转成系统可计算、可筛选、可绑定的业务规则对象。

例如：

- 必传资料规则
- 材料限制规则
- 施工动作要求规则
- 验收阈值规则
- 整改触发规则

### 5.5 执行标准 Execution Standard

执行标准回答的是：`这个任务应该怎么做`。

它更关注：

- 步骤要求
- 作业约束
- 资料要求
- 安全要求
- 用料要求
- 责任口径

### 5.6 验收标准 Acceptance Standard

验收标准回答的是：`这个任务如何判定通过`。

它更关注：

- 必检项
- 阈值条件
- 合格判定
- 不合格等级
- 整改要求
- 复验条件

### 5.7 标准包 Standard Package

标准包用于把一组执行标准、验收标准、规则项、检查项模板按业务场景打包。

常见场景：

- 标准店施工标准包
- 灯具安装标准包
- 竣工资料标准包
- 门店验收标准包

### 5.8 模板 Template

模板是标准库面向业务的落地载体，可绑定到：

- 项目模板
- 任务模板
- 流程模板
- 服务模板
- 检查项模板

### 5.9 标准快照 Standard Snapshot

标准快照用于在任务进入执行、验收或整改阶段时固化当时生效的标准版本。

它是审计与争议处理的关键依据。

## 6. 标准来源模型

### 6.1 来源分类

标准来源建议使用统一枚举：

- `national`
- `industry`
- `local`
- `enterprise`
- `brand`
- `supplier`
- `project`

### 6.2 来源核心字段

每个标准来源至少包含：

- `source_id`
- `source_type`
- `source_name`
- `source_owner`
- `source_org`
- `region_scope`
- `industry_scope`
- `trust_level`
- `status`
- `remark`

### 6.3 来源优先级建议

当多个来源同时适用于同一业务场景时，默认优先级建议为：

1. 国家标准
2. 行业标准
3. 地方标准
4. 企业内部制度
5. 品牌标准
6. 供应商标准
7. 项目补充标准

说明：

- 项目补充标准可作为特批覆盖，但必须留痕
- 低层级来源不应直接违反高层级来源
- 冲突判断结果必须可追溯到来源对象

## 7. 标准文档模型

### 7.1 文档作用

标准文档用于承载原始标准资料及元信息，是结构化的上游对象。

系统不能只保存结构化结果而丢失原文依据。

### 7.2 文档核心字段

每个标准文档建议包含：

- `document_id`
- `source_id`
- `document_code`
- `document_name`
- `document_version`
- `publish_org`
- `effective_date`
- `expire_date`
- `region_scope`
- `store_type_scope`
- `business_domain`
- `attachment_id`
- `parse_status`
- `review_status`
- `status`

### 7.3 文档状态建议

- `草稿中`
- `待解析`
- `解析中`
- `待校对`
- `待生效`
- `已生效`
- `已停用`
- `已归档`

### 7.4 文档与附件关系

- 一个标准文档可关联多个附件版本
- 解析结果必须绑定到具体附件版本
- 重新上传附件后，不应直接覆盖历史解析结果

---

## 8. 标准条款结构化模型

### 8.1 条款定位

标准条款是从原文中拆解出的最小语义片段，是连接“原文”和“规则”的桥梁。

### 8.2 条款拆解原则

条款拆解建议遵循：

- 一条条款表达一个核心要求
- 复杂长句可拆成多个条款
- 保留原始条款编号和原文片段
- 保留条款所在章节、页码、附件定位信息
- 拆解后仍需可回到原文定位

### 8.3 条款核心字段

每条标准条款建议包含：

- `clause_id`
- `document_id`
- `clause_code`
- `chapter_path`
- `original_text`
- `normalized_text`
- `clause_type`
- `domain_tags`
- `scene_tags`
- `profession_tags`
- `material_tags`
- `device_tags`
- `risk_level`
- `page_ref`
- `status`

### 8.4 条款类型建议

V1 建议先支持以下条款类型：

- `定义类`
- `强制要求类`
- `禁止类`
- `流程类`
- `资料类`
- `验收类`
- `整改类`
- `安全类`
- `材料类`
- `设备类`

### 8.5 条款结构化维度

为便于后续生成规则项，建议从条款中提取以下结构化维度：

- 适用对象
- 适用场景
- 执行动作
- 限制条件
- 阈值参数
- 计量单位
- 资料要求
- 验收方式
- 不合格后果
- 风险等级

### 8.6 条款结构化示例

原文：

`隐蔽工程施工完成后，必须上传现场照片并经确认后方可进入下一工序。`

结构化后可表达为：

- 适用场景：隐蔽工程
- 执行动作：上传现场照片
- 条款类型：资料类 + 流程类
- 是否强制：是
- 前置条件：施工完成
- 后置限制：未确认前不得进入下一工序
- 风险等级：高

## 9. 规则项模型

### 9.1 规则项定位

规则项是标准条款向业务执行层转换后的核心对象。

系统真正绑定到任务、模板、检查项上的，不应该是原始长文本，而应该是规则项。

### 9.2 规则项来源

一个规则项可以来源于：

- 单条标准条款
- 多条标准条款合并
- 条款与企业口径叠加
- 品牌内部裁剪后的规则集合

### 9.3 规则项分类建议

V1 建议先支持以下分类：

- `适用性规则`
- `前置条件规则`
- `执行动作规则`
- `资料提交规则`
- `验收判定规则`
- `整改触发规则`
- `版本覆盖规则`
- `风险预警规则`

### 9.4 规则项核心字段

每个规则项建议包含：

- `rule_id`
- `rule_name`
- `rule_type`
- `source_clause_ids`
- `applicable_scope`
- `condition_expression`
- `action_expression`
- `threshold_value`
- `threshold_unit`
- `required_flag`
- `severity_level`
- `exception_policy`
- `version`
- `status`

### 9.5 规则项表达方式

V1 不建议直接引入复杂 DSL，可采用`结构化字段 + 简单表达式`方式。

例如：

- `if task_type = 隐蔽工程 then require attachment_type = 现场照片`
- `if acceptance_type = 初验 then check mandatory_items >= 100%`
- `if store_type = 标准店 and domain = 灯具安装 then apply standard_package = 灯具安装标准包`

### 9.6 规则项治理要求

- 每个规则项必须可回溯到来源条款
- 每次变更必须产生版本
- 已生效任务使用的规则项不能被静默覆盖
- 规则项删除应改为停用，不建议物理删除

## 10. 执行标准与验收标准模型

### 10.1 二者区别

#### 执行标准

关注`怎么做`。

#### 验收标准

关注`怎么算通过`。

二者不能混为一个字段，也不应只靠一段说明文本表达。

### 10.2 执行标准核心组成

执行标准建议包含：

- 适用任务类型
- 执行步骤
- 必备材料/设备
- 安全要求
- 作业约束
- 过程留痕要求
- 必传资料
- 异常上报要求

### 10.3 验收标准核心组成

验收标准建议包含：

- 验收对象
- 检查项列表
- 合格阈值
- 判定方式
- 缺陷等级
- 整改时限
- 复验要求
- 验收结论模板

### 10.4 标准对象字段建议

执行标准 / 验收标准可共用部分字段：

- `standard_id`
- `standard_type`
- `standard_name`
- `business_domain`
- `task_type_scope`
- `store_type_scope`
- `region_scope`
- `rule_ids`
- `template_ids`
- `version`
- `status`

### 10.5 绑定原则

- 执行标准与验收标准可以分别绑定到同一个任务
- 一个任务可绑定多个标准对象
- 同类标准并存时必须有优先级与覆盖说明
- 关键任务必须至少绑定一个执行标准和一个验收标准

### 11.1 标准包作用

标准包用于把多个标准对象按业务场景打包，降低项目创建和任务配置成本。

### 11.2 标准包核心字段

- `package_id`
- `package_name`
- `package_type`
- `applicable_brand`
- `applicable_store_type`
- `applicable_region`
- `execution_standard_ids`
- `acceptance_standard_ids`
- `rule_ids`
- `priority`
- `version`
- `status`

### 11.3 模板绑定关系

建议采用以下绑定方式：

- 标准包绑定到项目模板
- 标准包绑定到任务模板
- 验收标准绑定到检查项模板
- 资料要求规则绑定到资料模板

### 11.4 模板选择逻辑

模板匹配至少考虑以下维度：

- 品牌
- 店型
- 区域
- 项目类型
- 任务类型
- 专业域
- 服务包类型

## 12. 标准绑定到任务树的方式

### 12.1 绑定链路

标准库与任务树的绑定链路建议为：

`标准来源 -> 条款 -> 规则项 -> 标准对象 -> 标准包 -> 任务模板 -> 具体任务 -> 标准快照`

### 12.2 绑定层级

标准可在以下层级发生绑定：

1. 项目模板层
2. 任务模板层
3. 具体任务层
4. 检查项层
5. 整改任务层

### 12.3 绑定规则

- 父任务可绑定上层标准上下文
- 子任务继承父任务标准上下文
- 子任务可增加更细规则项
- 子任务覆盖标准时必须记录原因
- 进入执行中的任务必须生成标准快照

### 12.4 检查项生成关系

建议采用：

`验收标准 -> 检查项模板 -> 验收单检查项`

`执行标准 -> 执行清单模板 -> 任务执行清单`

这样标准对象才能真正驱动页面和流程。

## 13. 标准快照机制

### 13.1 为什么需要快照

如果任务执行过程中标准版本发生变化，而系统没有快照机制，就会出现：

- 执行人与验收人口径不一致
- 历史项目无法复盘
- 争议无法判断到底按哪个版本执行
- Agent 可能基于新版本误判旧任务

### 13.2 快照触发点

V1 建议在以下节点生成快照：

- 任务进入`待执行`
- 验收单创建
- 整改任务生成
- 结算建议生成前的依据固化

### 13.3 快照内容

标准快照至少包含：

- 任务或结果对象 ID
- 标准对象 ID
- 标准版本
- 规则项清单
- 检查项清单
- 来源条款引用
- 快照生成时间
- 生成人 / Agent

### 13.4 快照原则

- 快照创建后默认不可静默修改
- 如因重大错误需修正，必须产生修订记录
- 快照是争议处理与审计优先依据

## 14. 版本治理与生效机制

### 14.1 版本层级

建议同时管理以下版本：

- 标准文档版本
- 条款结构化版本
- 规则项版本
- 标准对象版本
- 标准包版本
- 快照版本

### 14.2 生效状态建议

- `草稿中`
- `评审中`
- `待生效`
- `已生效`
- `已停用`
- `已废弃`

### 14.3 生效原则

- 草稿版本不能直接被任务调用
- 只有已生效版本可参与模板匹配与任务生成
- 停用不影响历史快照
- 废弃需要保留历史引用关系

### 14.4 覆盖原则

当多个标准同时命中一个任务时，建议采用：

- 先按来源优先级判断
- 再按业务适用范围判断
- 再按品牌/区域/店型细分范围判断
- 最后按人工确认结果固化

---

## 15. AI 与人工协同机制

### 15.1 AI 可承担的工作

AI 可辅助完成：

- 文档切分
- 条款抽取
- 结构化字段建议
- 条款归类
- 规则项草案生成
- 相似标准检索
- 冲突提示

### 15.2 AI 不应直接决定的事项

V1 阶段不应让 AI 直接无审查完成：

- 正式标准生效
- 冲突裁决
- 高风险强制条款删除
- 关键验收规则覆盖
- 项目执行口径变更

### 15.3 人工审核节点

以下动作必须有人审：

- 新文档首次入库后的解析结果确认
- 强制条款的结构化确认
- 规则项生效
- 标准包发布
- 高优先级覆盖规则确认

---

## 16. 审计与治理要求

### 16.1 必须留痕的动作

- 标准文档上传
- 条款解析
- 条款修改
- 规则项创建/变更/停用
- 执行标准变更
- 验收标准变更
- 标准包发布
- 模板绑定变更
- 快照生成
- AI 建议与人工确认结果

### 16.2 审计记录字段建议

每条关键日志至少包含：

- 对象 ID
- 对象类型
- 动作类型
- 动作前后值
- 操作人 / Agent
- 来源文档版本
- 原因说明
- 时间

### 16.3 风险控制要求

- 强制条款禁止无审批删除
- 已生效标准禁止直接覆盖历史版本
- 高风险规则项变更需触发影响评估
- 与任务树已绑定的标准变更需提示影响范围

---

## 17. V1 实施建议

### 17.1 先做什么

V1 建议优先落地以下能力：

1. 标准来源管理
2. 标准文档入库与附件管理
3. 条款结构化编辑
4. 规则项管理
5. 执行标准 / 验收标准管理
6. 标准包管理
7. 任务绑定与标准快照
8. 基础检索与 Agent 受控调用

### 17.2 暂不做什么

V1 不建议一开始就做：

- 全量法规自动解析
- 复杂法规推理引擎
- 全自动冲突消解
- 跨品牌共享标准市场
- 无人工审核的自动生效机制

### 17.3 成功标准

标准库在 V1 达到可用，至少应满足：

- 能沉淀高频标准域的结构化条款
- 能把条款转成规则项
- 能形成执行标准与验收标准
- 能驱动任务模板和检查项生成
- 能在执行环节生成标准快照
- 能被 Agent 受控引用并输出结构化结果

## 18. 示例

### 18.1 示例结构化链路

原始标准：

`隐蔽工程施工完成后，必须上传现场照片并经确认后方可进入下一工序。`

结构化链路：

- 标准来源：品牌营建标准
- 标准文档：`门店施工验收与资料规范 V1`
- 标准条款：隐蔽工程完工后必须上传现场照片
- 规则项 1：隐蔽工程任务必须上传现场照片
- 规则项 2：资料未确认前不得进入下一工序
- 执行标准：隐蔽工程执行标准
- 验收标准：隐蔽工程资料验收标准
- 绑定任务：`上传隐蔽工程照片`
- 生成检查项：现场照片数量、照片角度、时间要求、确认状态
- 快照：任务进入待执行时固化版本 `V1.0`

### 18.2 示例对象映射

以`灯具安装`为例：

- 标准来源：品牌标准 + 供应商安装标准
- 条款结构化：安装位置、固定要求、通电测试要求
- 规则项：必须固定牢靠、必须完成通电测试、必须上传安装完成照片
- 执行标准：灯具安装执行标准
- 验收标准：灯具安装验收标准
- 标准包：主材安装标准包
- 任务模板：灯具安装任务模板
- 具体任务：北京朝阳店 / 主材安装 / 灯具安装
- 结果对象：验收单检查项、整改任务、资产归档资料

## 19. 本文档结论

标准库是本系统最核心的专业能力底座。

V1 必须坚持以下建模纪律：

- 标准必须从来源开始建模，不只存结果
- 条款必须结构化，规则项必须可执行
- 执行标准和验收标准必须分离建模
- 标准包和模板必须成为业务入口
- 标准必须绑定任务树并生成检查项
- 执行中的业务对象必须生成标准快照
- AI 只做辅助，生效与高风险变更必须可人工治理
