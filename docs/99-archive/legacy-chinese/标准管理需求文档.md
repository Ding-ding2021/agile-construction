## 1. 模块概述

### 1.1 模块定位

标准管理模块是系统的专业能力底座，用于统一沉淀品牌方营建标准、行业规范、供应商执行要求和项目补充规则，并将其转化为任务、模板、执行清单、验收检查项和标准快照可直接使用的结构化对象。

在本系统中：

- `标准管理`负责标准来源、条款、规则项、标准对象、标准包与模板中心
- `项目管理`负责承接项目容器和标准上下文
- `任务中心`负责把标准绑定到任务并在执行阶段落地
- `验收整改`负责依据验收标准和检查项完成结果判断

因此，标准管理不是资料归档模块，而是驱动`任务生成、执行指导、验收判断、整改闭环`的核心能力中心。

标准管理模块的核心作用：

- 统一沉淀可追溯的标准来源
- 将原始标准文档切分为结构化条款
- 将条款进一步转为可执行规则项
- 管理执行标准、验收标准和标准包
- 将标准绑定到项目模板、任务模板、检查项模板
- 为运行中的任务和结果对象提供标准快照依据

### 1.2 模块目标

V1 阶段，标准管理模块重点解决以下问题：

- 标准分散在多个来源，难以统一查询和治理
- 原始标准文本难以直接被系统执行
- 执行标准和验收标准口径不一致
- 模板生成任务时缺少稳定的标准底座
- 标准变更对在途任务影响不可控
- AI 可辅助理解标准，但缺少人工审核和生效机制

### 1.3 V1 范围

V1 纳入范围：

- 标准来源管理
- 标准文档上传与元信息管理
- 条款切分与结构化编辑
- 规则项管理
- 执行标准管理
- 验收标准管理
- 标准包管理
- 模板中心绑定
- 标准版本治理与生效机制
- 标准快照查询与影响追溯
- 品牌需求 Agent / 验收质检 Agent 辅助能力

V1 不纳入范围：

- 全量法规自动语义解析引擎
- 自动裁决复杂跨标准冲突
- 无人工审核的自动生效
- 复杂知识图谱治理体系
- 面向外部供应商的独立标准开放平台
- 基于图像/视频的深度 AI 自动判定引擎

## 2. 角色与场景

### 2.1 核心角色

#### 品牌标准管理员

负责品牌标准录入、条款校对、规则项维护、标准发布和版本治理。

#### 平台产品 / 运营配置人员

负责标准包配置、模板绑定、适用范围管理和跨模块口径统一。

#### 项目经理 / 任务配置人员

关注任务模板是否命中正确标准包，标准是否能驱动任务生成和执行。

#### 验收 / 质检角色

关注验收标准是否清晰、检查项是否可生成、整改规则是否明确。

#### 资源执行角色

关注执行标准是否可理解、是否能转成执行清单和资料要求。

#### 品牌需求 Agent

辅助匹配项目模板、标准包和结构化标准建议。

#### 验收质检 Agent

辅助条款抽取、规则项草案生成、检查项建议和标准冲突提示。

### 2.2 核心业务场景

#### 场景 1：录入新的品牌营建标准

标准管理员上传标准文档，系统提取元信息并切分条款，管理员完成结构化校对后形成可治理对象。

#### 场景 2：把标准条款转成规则项

管理员基于条款内容生成资料规则、动作要求、验收阈值、整改触发规则，并补充适用范围和优先级。

#### 场景 3：配置执行标准和验收标准

运营人员将规则项组合成执行标准和验收标准，分别用于指导任务执行和任务验收。

#### 场景 4：将标准包绑定到模板中心

配置人员把执行标准、验收标准和检查项模板打包，绑定到项目模板、任务模板和检查项模板。

#### 场景 5：任务执行前固化标准快照

任务进入`待执行`时，系统基于当时生效版本生成快照，保证执行与验收口径一致。

#### 场景 6：标准变更影响评估

某条规则项需要调整时，标准管理员查看当前已引用模板、在途任务和历史快照影响范围，再决定是否发布新版本。

#### 场景 7：验收不通过后追溯依据

验收人员查看任务所使用的标准快照、规则项来源条款和检查项生成依据，支撑整改和争议处理。

## 3. 业务对象定义

### 3.1 标准来源 Standard Source

标准来源用于回答`标准来自哪里`。

V1 典型来源包括：

- 国家标准
- 行业标准
- 地方标准
- 企业内部制度
- 品牌营建标准
- 供应商执行标准
- 项目补充标准

核心字段建议：

- `source_id`
- `source_type`
- `source_name`
- `source_owner`
- `source_org`
- `region_scope`
- `industry_scope`
- `effective_from`
- `effective_to`
- `status`

### 3.2 标准文档 Standard Document

标准文档用于保存具体文件及其元信息，是条款切分和规则抽取的原始载体。

核心字段建议：

- `document_id`
- `source_id`
- `document_code`
- `document_name`
- `document_type`
- `document_version`
- `publish_date`
- `effective_date`
- `file_url`
- `parse_status`
- `status`

### 3.3 标准条款 Standard Clause

标准条款是从原始文档中拆出的最小语义单元，既保留原文映射关系，又具备结构化编辑能力。

核心字段建议：

- `clause_id`
- `document_id`
- `chapter_path`
- `clause_code`
- `clause_title`
- `clause_text`
- `clause_type`
- `mandatory_level`
- `applicable_scope`
- `structured_status`
- `version`

### 3.4 规则项 Rule Item

规则项是标准条款向业务执行层转换后的核心对象，系统真正绑定的应是规则项，而不是原始长文本。

规则项分类建议：

- 动作要求规则
- 资料要求规则
- 材料限制规则
- 安全约束规则
- 验收阈值规则
- 整改触发规则
- 复验条件规则

核心字段建议：

- `rule_id`
- `rule_name`
- `rule_type`
- `source_clause_ids`
- `expression_type`
- `expression_content`
- `priority`
- `applicable_brand`
- `applicable_region`
- `applicable_store_type`
- `applicable_task_type`
- `status`
- `version`

### 3.5 执行标准 Execution Standard

执行标准用于回答`任务应该怎么做`，偏向作业动作、资料要求、安全约束和过程要求。

核心字段建议：

- `execution_standard_id`
- `standard_name`
- `rule_ids`
- `task_type_scope`
- `service_package_scope`
- `required_attachment_types`
- `version`
- `status`

### 3.6 验收标准 Acceptance Standard

验收标准用于回答`任务如何判定通过`，偏向必检项、阈值、问题等级、整改要求和复验条件。

核心字段建议：

- `acceptance_standard_id`
- `standard_name`
- `rule_ids`
- `checklist_template_ids`
- `task_type_scope`
- `defect_level_rule`
- `rectification_rule`
- `version`
- `status`

### 3.7 标准包 Standard Package

标准包用于把执行标准、验收标准、规则项和检查项模板按业务场景打包，降低项目和任务配置成本。

核心字段建议：

- `package_id`
- `package_name`
- `package_type`
- `execution_standard_ids`
- `acceptance_standard_ids`
- `rule_ids`
- `applicable_brand`
- `applicable_store_type`
- `applicable_region`
- `priority`
- `version`
- `status`

### 3.8 模板中心 Template Binding

模板是标准管理面向业务落地的主要出口，可绑定到：

- 项目模板
- 任务模板
- 流程模板
- 服务模板
- 检查项模板
- 资料模板

模板绑定核心字段建议：

- `binding_id`
- `template_type`
- `template_id`
- `package_id`
- `rule_ids`
- `binding_scope`
- `priority`
- `status`

### 3.9 标准快照 Standard Snapshot

标准快照用于在任务进入执行、验收或整改阶段时固化当时生效的标准版本，是审计和争议处理优先依据。

核心字段建议：

- `snapshot_id`
- `object_type`
- `object_id`
- `execution_standard_ids`
- `acceptance_standard_ids`
- `rule_ids`
- `checklist_items`
- `source_clause_refs`
- `snapshot_version`
- `generated_at`
- `generated_by`

## 4. 任务结构设计

### 4.1 模块承接的结构定位

标准管理模块虽然不直接承接业务任务执行，但必须服务于任务树和模板体系，因此其内部结构设计应围绕`来源 -> 文档 -> 条款 -> 规则项 -> 标准对象 -> 标准包 -> 模板绑定 -> 任务绑定 -> 标准快照`展开。

该结构必须满足：

- 向上可追溯到原始来源
- 向下可落到任务、清单和检查项
- 中间可做版本治理、审核和生效控制

### 4.2 总体分层结构

V1 建议采用以下 8 层：

1. 标准来源层
2. 标准文档层
3. 标准条款层
4. 规则项层
5. 执行标准 / 验收标准层
6. 标准包 / 模板层
7. 任务绑定 / 检查项层
8. 标准快照层

这样设计的目的：

- 原文和结构化对象保持映射关系
- 业务端使用规则和标准对象，而不是直接解析原文
- 模板与任务绑定逻辑稳定可复用
- 在途任务可以固化版本，避免新旧口径混乱

### 4.3 条款结构化设计

条款结构化是标准管理 V1 的关键动作，建议从条款中提取以下维度：

- 适用对象
- 适用阶段
- 适用任务类型
- 动作要求
- 资料要求
- 阈值条件
- 不合格定义
- 整改要求
- 复验条件
- 来源章节路径

结构化原则：

- 保留原文内容与章节路径
- 支持字段化编辑和人工校正
- 支持一条条款拆成多个规则项
- 不追求一次性全自动准确解析

### 4.4 规则项生成与治理结构

规则项生成采用`AI 草案 + 人工确认`模式。

V1 建议流程：

1. 上传文档并完成条款切分
2. 系统生成规则项草案
3. 标准管理员校对规则类型、表达式和适用范围
4. 通过审核后进入标准对象组合阶段

治理要求：

- 每个规则项必须可回溯到来源条款
- 规则项删除改为停用，不做物理删除
- 高风险规则项变更需触发影响评估
- 已引用任务快照的规则项不可被静默覆盖

### 4.5 标准对象与标准包组合结构

#### 组合原则

- 执行标准关注过程约束
- 验收标准关注结果判断
- 标准包关注业务场景组合
- 模板关注标准落地位置

#### 组合关系

- 多条规则项可组成一个执行标准
- 多条规则项可组成一个验收标准
- 一个标准包可同时包含多个执行标准和验收标准
- 一个模板可绑定多个标准包，但必须有优先级和覆盖说明

### 4.6 任务与模板输出结构

标准管理模块对外至少输出以下结构化能力：

- 项目模板所需标准包
- 任务模板所需执行标准与验收标准
- 执行清单模板
- 检查项模板
- 快照生成依据
- 冲突提示与覆盖说明

### 4.7 版本与适用范围结构

标准管理必须支持以下适用维度：

- 品牌
- 店型
- 区域
- 项目类型
- 任务类型
- 专业域
- 服务包类型

匹配顺序建议：

1. 先判断业务适用范围
2. 再判断品牌 / 区域 / 店型细分范围
3. 再判断优先级
4. 最后由人工确认固化

## 5. 标准绑定设计

### 5.1 绑定目标

标准管理模块必须回答以下问题：

- 标准从哪里来
- 标准如何变成系统可执行对象
- 标准如何绑定到模板和任务
- 标准变更后如何控制影响范围
- 在途任务如何保留执行依据

### 5.2 标准绑定链路

建议采用统一链路：

`标准来源 -> 标准文档 -> 标准条款 -> 规则项 -> 执行标准/验收标准 -> 标准包 -> 模板 -> 具体任务 -> 标准快照`

### 5.3 绑定层级

V1 支持以下层级绑定：

1. 项目模板层
2. 任务模板层
3. 具体任务层
4. 执行清单层
5. 验收检查项层
6. 派生整改任务层

### 5.4 模板中心绑定规则

- 标准包可绑定到项目模板
- 执行标准和验收标准可绑定到任务模板
- 验收标准可绑定到检查项模板
- 资料要求规则可绑定到资料模板
- 模板发布前必须校验标准对象是否已生效

### 5.5 任务绑定规则

- 父任务可承接标准上下文
- 子任务默认继承父任务标准上下文
- 子任务可增加更细规则项
- 子任务覆盖标准时必须记录原因
- 关键任务至少绑定一个执行标准和一个验收标准
- 进入执行态前必须生成标准快照

### 5.6 执行清单与检查项生成关系

建议采用：

- `执行标准 -> 执行清单模板 -> 任务执行清单`
- `验收标准 -> 检查项模板 -> 验收检查项`

标准管理模块负责：

- 维护模板来源关系
- 提供生成字段与规则依据
- 输出检查项模板和资料模板
- 支撑任务中心和验收整改模块在正确节点调用

### 5.7 标准快照机制

V1 建议在以下节点生成快照：

- 任务进入`待执行`
- 验收单创建
- 整改任务生成
- 结算建议生成前的依据固化

快照至少包含：

- 任务或结果对象 ID
- 标准对象 ID
- 标准版本
- 规则项清单
- 检查项清单
- 来源条款引用
- 快照生成时间
- 生成人 / Agent

### 5.8 标准变更影响控制

标准变更发布前，系统需支持查看：

- 被哪些模板引用
- 被哪些任务类型使用
- 是否存在在途项目引用
- 是否存在已生成快照历史对象

V1 原则：

- 草稿版本不能被任务调用
- 只有已生效版本可参与模板匹配
- 停用不影响历史快照
- 已生效任务使用的快照不可静默替换

## 6. 页面结构与功能点

### 6.1 页面总体结构

标准管理建议包含以下核心页面：

1. 标准中心首页
2. 标准来源管理页
3. 标准文档 / 条款结构化页
4. 规则项管理页
5. 执行标准 / 验收标准管理页
6. 标准包与模板中心页
7. 标准快照查询页

V1 页面原则：

- 以标准中心首页承接查询、筛选和统计
- 以条款结构化页承接标准加工流程
- 以规则项和标准对象页承接发布治理
- 以模板中心承接对业务落地出口
- 以快照查询页承接审计和争议回溯

### 6.2 标准中心首页

#### 页面定位

用于统一查看标准对象、版本状态、解析进度、待审核项和高影响变更。

#### 视图模式

建议支持以下 3 种视图：

- 列表视图（默认）
- 卡片视图（适合按类型浏览）
- 看板视图（适合按状态管理）

#### 顶部统计区

建议展示：

- 全部标准数
- 待结构化数
- 待审核规则项数
- 已生效标准数
- 待发布标准包数
- 高风险变更数

#### 筛选与查询

支持以下维度：

- 来源类型
- 标准对象类型
- 品牌
- 店型
- 区域
- 状态
- 版本
- 最近更新时间
- 是否被模板引用
- 是否存在在途任务引用

### 6.3 标准来源管理页

#### 页面定位

维护标准来源、来源范围、失效时间和文档归属。

#### 核心功能

- 新增 / 编辑标准来源
- 维护来源分类与适用范围
- 查看来源下文档列表
- 查看来源是否仍有效
- 停用来源并保留历史引用

### 6.4 标准文档 / 条款结构化页

#### 页面定位

承接文档上传、解析、条款切分、字段结构化和人工校对。

#### 页面结构建议

1. 文档元信息区
2. 原文预览区
3. 条款列表区
4. 结构化编辑区
5. AI 建议区
6. 审核记录区

#### 核心功能

- 上传标准文档
- 维护文档版本和发布日期
- 自动切分条款
- 编辑条款结构化字段
- 生成规则项草案
- 标记强制条款和高风险条款
- 提交审核

### 6.5 规则项管理页

#### 页面定位

统一管理规则项分类、表达式、适用范围、优先级和版本变更。

#### 核心功能

- 查看规则项列表
- 按规则类型筛选
- 编辑表达式和适用范围
- 标记优先级
- 进行审核、生效、停用
- 查看来源条款与被引用情况
- 查看冲突提示

### 6.6 执行标准 / 验收标准管理页

#### 页面定位

将规则项组合成面向业务执行的标准对象。

#### 核心功能

- 创建执行标准
- 创建验收标准
- 批量关联规则项
- 配置任务类型和服务包范围
- 维护资料要求和检查项模板
- 查看版本差异
- 发起发布审批

### 6.7 标准包与模板中心页

#### 页面定位

用于完成标准对象打包及对项目模板、任务模板、检查项模板的绑定。

#### 核心功能

- 创建标准包
- 绑定执行标准与验收标准
- 配置适用品牌 / 区域 / 店型
- 绑定项目模板、任务模板、检查项模板
- 查看命中逻辑和优先级
- 查看标准变更影响范围

### 6.8 标准快照查询页

#### 页面定位

用于审计、复盘和争议处理，查看某个任务、验收单或整改任务当时使用的标准依据。

#### 核心功能

- 按项目 / 任务 / 验收单查询快照
- 查看快照中的规则项与来源条款
- 对比快照版本与当前版本差异
- 查看生成时间、生成人、触发节点
- 导出快照依据摘要

### 6.9 关键交互规则

- 草稿标准不可被模板直接调用
- 高风险规则项变更必须显示影响提示
- 已生效对象修改应生成新版本而不是覆盖原版本
- 已有快照引用的版本默认只读
- AI 建议结果必须与人工结论并存留痕

## 7. 状态机设计

### 7.1 状态对象范围

标准管理模块涉及以下状态对象：

- 标准文档解析状态
- 条款结构化状态
- 规则项生效状态
- 执行标准 / 验收标准生效状态
- 标准包发布状态

### 7.2 文档解析状态

建议状态：

- `待上传`
- `待解析`
- `解析中`
- `待校对`
- `已确认`
- `解析失败`

主路径：

`待上传 -> 待解析 -> 解析中 -> 待校对 -> 已确认`

异常路径：

- `解析中 -> 解析失败`
- `待校对 -> 待解析`

### 7.3 条款与规则项状态

建议统一采用：

- `草稿中`
- `评审中`
- `待生效`
- `已生效`
- `已停用`
- `已废弃`

主路径：

`草稿中 -> 评审中 -> 待生效 -> 已生效`

异常路径：

- `评审中 -> 草稿中`
- `已生效 -> 已停用`
- `已停用 -> 已废弃`

### 7.4 守卫条件

#### 草稿中 -> 评审中

- 来源条款完整
- 结构化字段完整
- 适用范围明确
- 责任人明确

#### 评审中 -> 待生效

- 人工审核通过
- 冲突检查通过或已确认兜底口径
- 影响范围已评估

#### 待生效 -> 已生效

- 生效时间明确
- 版本号已冻结
- 模板绑定校验通过

#### 已生效 -> 已停用

- 已有替代版本或确认不再使用
- 已通知相关配置责任人
- 历史引用关系完整保留

### 7.5 联动规则

- 文档确认后才允许生成可生效规则项
- 规则项生效后才允许加入执行标准和验收标准
- 标准对象生效后才允许加入标准包
- 标准包生效后才允许参与模板命中
- 任务进入执行态时基于已生效版本生成标准快照

### 7.6 状态治理原则

- 生效对象默认不可直接覆盖编辑
- 关键变更必须生成新版本
- 停用不影响历史快照和历史任务回放
- 废弃仅代表不再使用，不代表删除历史依据

## 8. 权限与角色控制

### 8.1 品牌标准管理员权限

- 可管理本品牌标准来源、文档、条款、规则项和标准包
- 可提交审核和发起发布
- 不可查看其他品牌私有标准数据
- 不可绕过审核直接生效高风险规则

### 8.2 平台管理员 / 配置运营权限

- 可管理平台级通用标准和模板绑定
- 可维护适用范围、优先级和覆盖规则
- 可查看跨模块引用关系与影响分析
- 可停用或废弃平台标准对象

### 8.3 项目经理 / 业务配置人员权限

- 可查看已生效标准和标准包
- 可在模板层选择绑定对象
- 不可直接修改已生效标准正文和规则表达式

### 8.4 验收 / 质检角色权限

- 可查看验收标准、检查项模板和整改规则
- 可提出规则修订建议
- 不可直接发布新的品牌标准

### 8.5 Agent 权限

- 可辅助文档切分、条款抽取、规则项草案生成、冲突提示
- 不可直接让标准对象正式生效
- 不可删除强制条款或覆盖高风险验收规则
- 所有建议必须可回溯到输入和输出摘要

### 8.6 数据权限原则

- 品牌私有标准隔离
- 平台级标准可按授权共享
- 最小权限访问
- 关键操作全程留痕
- 历史版本与快照只读保存

## 9. 异常与人工介入

### 9.1 典型异常类型

- 文档解析失败或条款切分错误
- 条款结构化字段缺失
- 规则项表达式冲突或不完整
- 同一任务命中多个冲突标准
- 高风险规则项误停用
- 模板绑定引用未生效版本
- 快照生成失败
- AI 建议与人工判断差异过大

### 9.2 人工介入触发条件

以下情况必须支持人工介入：

- 强制条款结构化结果不确定
- 同一场景存在多条冲突规则
- 高风险验收规则需要覆盖
- 规则项影响范围过大
- 在途任务引用的标准需要修订
- 快照依据与当前生效版本差异异常
- AI 多轮抽取仍低置信度

### 9.3 人工介入动作

V1 允许以下动作：

- 手工重切条款
- 手工修订结构化字段
- 手工编写规则表达式
- 手工指定优先级和覆盖逻辑
- 手工回退到草稿状态
- 手工停用或废弃版本
- 手工确认模板绑定口径
- 手工确认快照争议依据

### 9.4 异常治理要求

- 所有异常需记录来源、影响范围、处理人和处理结论
- 高风险异常需关联通知相关模块责任人
- 人工覆盖 AI 建议必须说明原因
- 快照异常需优先保障历史口径可回放

## 10. 埋点与验收标准

### 10.1 关键埋点事件

建议埋点以下事件：

- `standard.source_created`
- `standard.document_uploaded`
- `standard.document_parsed`
- `standard.clause_structured`
- `standard.rule_created`
- `standard.rule_reviewed`
- `standard.rule_published`
- `standard.execution_published`
- `standard.acceptance_published`
- `standard.package_published`
- `standard.template_bound`
- `standard.snapshot_generated`
- `standard.version_stopped`
- `standard.conflict_detected`

### 10.2 核心指标

#### 效率指标

- 文档解析平均耗时
- 条款结构化平均耗时
- 规则项审核平均耗时
- 标准发布平均耗时
- 模板绑定完成耗时

#### 质量指标

- 条款结构化完成率
- 规则项可执行率
- 标准绑定成功率
- 快照生成成功率
- 首次验收规则命中准确率

#### 治理指标

- 生效标准数量
- 停用 / 废弃标准数量
- 高风险变更次数
- 冲突规则发现次数
- 在途任务受影响标准变更次数

### 10.3 功能验收标准

- 可录入标准来源并上传标准文档
- 可把文档切分为条款并进行结构化编辑
- 可将条款转成规则项并完成审核生效
- 可管理执行标准、验收标准和标准包
- 可把标准包绑定到项目模板、任务模板和检查项模板
- 可在业务对象运行时生成标准快照
- 可追溯快照对应的来源条款和规则项
- 可查看标准变更的影响范围

### 10.4 业务验收标准

- 标准必须映射为规则项并绑定任务与模板
- 关键任务可获得明确执行标准和验收标准
- 检查项可由验收标准驱动生成
- 已执行任务的标准依据可回放
- 争议场景可通过快照追溯历史口径
- AI 生成结果可被人工审核并受控生效

### 10.5 模块结论

标准管理模块是系统从`文档经验驱动`走向`结构化规则驱动`的关键模块。

它的核心价值不在于存更多标准文件，而在于把标准转成任务、模板、清单、检查项和快照真正可调用的业务对象。

V1 的重点不是一次性把全部专业知识自动化，而是先稳定跑通以下链路：

- 标准来源可追溯
- 条款可结构化
- 规则项可执行
- 标准对象可组合
- 模板绑定可落地
- 在途任务可快照、可回放、可审计

当这条链路稳定后，任务中心、验收整改、采购管理、资产管理都会建立在同一套标准底座上持续扩展。
