# 任务树建模说明

> **文档版本**：V1.0
> **文档状态**：草稿中
> **适用阶段**：V1 / MVP
> **适用范围**：品牌方营建部门建店项目、流程任务、服务任务
> **关联文档**：`产品规划文档.md`、`多Agent建店管理平台_V1_PRD.md`、`多Agent建店管理平台_V1_技术方案.md`

---

## 1. 文档目标

本文档用于统一产品、设计、研发、测试对`任务树`的理解，明确系统中任务的定义、层级关系、状态规则、标准绑定、自动生成逻辑与边界约束。

任务树不是页面概念，而是系统底层执行模型。

本项目必须统一以下口径：

- `任务是最小执行单元`
- `项目 / 流程 / 服务是任务的不同组织形态`
- `标准库为任务生成、执行指导、验收判断提供底座能力`
- `任务负责推进，结果对象负责沉淀`

---

## 2. 建模定位

### 2.1 为什么采用任务树

品牌方营建部门真正管理的不是单一“项目表单”，而是一系列持续推进、互相约束、可交付、可验收的任务集合。

采用任务树建模的原因：

- 可以把复杂项目拆成可执行、可分配、可追踪的最小动作
- 可以让标准库直接作用到执行动作和验收动作
- 可以统一承接项目、流程、服务三类业务形态
- 可以让 Agent 只在明确任务节点内工作，避免无边界发挥
- 可以天然支持催办、预警、整改、回溯和审计

### 2.2 任务树的角色

任务树在系统中承担 5 个职责：

1. 承接项目目标的分解
2. 承接标准的落地执行
3. 承接资源与责任分配
4. 承接状态流转和异常升级
5. 承接结果对象的生成入口

### 2.3 建模原则

- `原子化`：一个任务只表达一个明确可交付动作
- `层级化`：复杂工作通过父子任务表达，不靠长文本描述
- `标准化`：任务必须能绑定执行标准和验收标准
- `状态化`：任务推进依赖状态机，不依赖口头协同
- `结果分离`：采购、验收、整改、资产、结算不直接等同于任务本身
- `审计化`：任务创建、拆解、改派、关闭、回退都必须留痕
- `可控复杂度`：V1 先限制层级、关系类型和跨项目能力

## 3. 适用范围与边界

### 3.1 适用范围

任务树用于承接以下业务：

- 建店项目拆解
- 阶段任务推进
- 施工执行协同
- 采购协同任务
- 验收任务
- 整改任务
- 资产归档任务
- 结算前置整理任务
- Agent 节点触发任务

### 3.2 V1 建模边界

V1 阶段采用保守边界，避免任务模型过早失控：

- 一个任务只能归属一个项目
- 一个任务只能有一个直接父任务
- 一个任务可以有多个子任务
- 任务依赖默认只允许在同一项目内建立
- 不支持跨项目共享同一个执行任务
- 不支持循环依赖
- 不支持无约束的多父任务图结构
- 不支持让 Agent 直接绕过状态机关闭任务

### 3.3 层级控制建议

理论上任务树可支持更深层级，但 V1 建议：

- `推荐层级`：4 层以内
- `系统硬上限`：6 层以内

推荐层级示例：

1. 项目根任务
2. 阶段任务
3. 工作包任务
4. 执行任务

如果还需要更细粒度控制，优先通过`执行清单`或`标准检查项`表达，而不是无限继续拆子任务。

## 4. 核心对象定义

### 4.1 任务 Task

任务是系统最小执行单元，必须满足以下条件：

- 有明确目标
- 有责任主体
- 有状态
- 有时间要求
- 有执行依据
- 有验收口径
- 有输出结果或结果引用

### 4.2 父任务 Parent Task

父任务用于承载阶段性目标、责任范围和汇总视图，不一定直接执行，但必须能描述其子任务完成后形成的业务结果。

父任务的主要作用：

- 聚合子任务进度
- 承载阶段目标
- 作为统计、预警、汇报入口
- 约束子任务顺序与边界

### 4.3 子任务 Sub Task

子任务是父任务的拆解执行项，通常具备更明确的责任人、时间、标准和交付动作。

### 4.4 任务关系 Task Relation

除父子关系外，任务之间还可以存在执行关系和业务关系。V1 建议支持以下关系类型：

- `parent_child`：父子层级关系
- `depends_on`：前置依赖关系
- `derived_from`：由其他任务结果派生而来
- `relates_to`：业务相关但不形成阻塞

说明：

- 父子关系用于表达结构
- 依赖关系用于表达执行先后
- 派生关系常用于验收不通过后生成整改任务
- 关联关系用于表达资料、采购、资产等弱关联

### 4.5 任务模板 Task Template

任务模板用于基于店型、场景、标准包快速生成任务树，通常包含：

- 适用条件
- 标准任务结构
- 默认责任角色
- 默认 SLA
- 默认前置关系
- 标准绑定规则
- 必要结果对象规则

### 4.6 标准快照 Standard Snapshot

任务在进入执行阶段后，需要固化当时使用的标准版本，形成标准快照。

这样即使标准库后续更新，也不会影响已经执行中的任务口径。

## 6. 任务层级模型

### 6.1 层级组织方式

任务树采用`单父节点树结构 + 任务关系扩展`模式。

即：

- 主结构靠 `parent_task_id` 形成树
- 非层级关系通过 `task_relation` 单独表达

这样可以同时满足：

- UI 易理解
- 数据结构稳定
- 依赖关系可扩展
- 不把树结构和网状关系混在一起

### 6.2 层级角色建议

#### 第 1 层：项目根任务

用于表达某个门店建设目标的顶层任务容器。

例如：

- 北京朝阳店建店项目

#### 第 2 层：阶段任务

用于表达一个项目中的大阶段。

例如：

- 设计准备
- 施工实施
- 验收交付

#### 第 3 层：工作包任务

用于表达某个阶段中的专业域或交付包。

例如：

- 水电工作包
- 主材安装工作包
- 竣工资料工作包

#### 第 4 层：执行任务

用于表达可分配、可执行、可验收的最小动作。

例如：

- 完成电路布线
- 上传隐蔽工程照片
- 提交初验申请

### 6.3 父子任务继承规则

子任务默认可继承父任务的以下属性：

- 项目归属
- 品牌与门店归属
- 标签
- 阶段上下文
- 默认标准包
- 默认责任角色

子任务可以覆盖以下属性：

- 执行人
- 时间计划
- 优先级
- 标准绑定
- 附件要求
- 是否里程碑

### 6.4 父任务汇总规则

父任务本身不是简单求和对象，建议遵循以下规则：

- 只要存在未完成必做子任务，父任务不得完成
- 任何关键子任务阻塞时，父任务显示风险或阻塞提示
- 父任务进度按必做子任务完成率计算
- 父任务完成时间取最后一个必做子任务完成时间
- 父任务验收结论由关键子任务验收结果汇总得到

## 7. 任务关系模型

### 7.1 关系类型

V1 任务关系采用轻量化设计，优先支持以下 4 类：

1. `parent_child`
2. `depends_on`
3. `derived_from`
4. `relates_to`

### 7.2 depends_on 规则

`depends_on` 表示任务 B 依赖任务 A 的完成或达成条件后才能继续。

V1 先采用最简单的`完成后开始`口径，不引入复杂的 FS/SS/FF/SF 专业调度模型。

也就是说：

- 前置任务未完成，后置任务默认不可进入执行中
- 前置任务关闭或取消时，后置任务需要人工确认是否继续
- 前置任务延期时，后置任务应触发风险重算

### 7.3 derived_from 规则

以下场景适合创建派生任务：

- 验收不通过生成整改任务
- 资料缺失生成补传任务
- 采购异常生成补货任务
- 资产校验异常生成补登记任务

派生任务需要记录：

- 来源任务 ID
- 派生原因
- 触发对象
- 派生时间
- 处理人

### 7.4 循环校验规则

建立任务关系时必须校验：

- 不能形成父子循环
- 不能形成依赖循环
- 不能让父任务依赖自己的子任务完成后再创建上层死锁

## 8. 任务状态模型

### 8.1 标准状态口径

任务状态统一采用以下口径：

- `待创建`
- `待分配`
- `待执行`
- `执行中`
- `待提交`
- `待验收`
- `不通过`
- `已完成`
- `已关闭`

### 8.2 状态含义

#### 待创建

任务尚未正式落库或由模板预生成但未确认。

#### 待分配

任务已创建，但尚未明确到执行主体。

#### 待执行

任务已明确执行主体和条件，可开始执行。

#### 执行中

任务已经开始，允许记录进度、阻塞、资料上传等动作。

#### 待提交

执行动作已结束，等待提交结果资料或结果确认。

#### 待验收

任务结果已提交，等待按验收标准判断。

#### 不通过

任务经校验或验收未通过，需要整改、补传或重新处理。

#### 已完成

任务完成，且满足执行与验收要求。

#### 已关闭

任务因取消、替代、合并、终止等原因结束，不再继续推进。

### 8.3 状态流转规则

基础流转建议：

`待创建 → 待分配 → 待执行 → 执行中 → 待提交 → 待验收 → 已完成`

异常流转建议：

- `待验收 → 不通过`
- `不通过 → 待执行`
- `执行中 → 已关闭`
- `待分配 → 已关闭`

### 8.4 状态控制原则

- 关键状态变更必须留痕
- Agent 只能在授权节点推进状态
- 不允许跳过`待验收`直接把需验收任务改为`已完成`
- `已完成`和`已关闭`默认不可直接编辑关键字段
- 重新打开任务必须记录原因与责任人

## 9. 标准绑定模型

### 9.1 绑定目标

每个任务都应明确：

- `怎么做`：执行标准
- `怎么判定做完`：验收标准
- `需要提交什么`：资料要求
- `出现异常怎么办`：整改规则

### 9.2 标准绑定层级

标准可在以下层级绑定：

1. 标准模板绑定到任务模板
2. 任务模板生成任务时下发到具体任务
3. 任务创建后按实际场景补充或覆盖
4. 任务进入执行时生成标准快照

### 9.3 标准继承规则

- 子任务默认继承父任务的标准上下文
- 子任务可增加更细的执行标准
- 子任务可覆盖验收标准，但必须说明原因
- 已生成快照的任务不应被新标准静默覆盖

### 9.4 标准与检查项关系

建议采用三层映射：

- 标准来源条款
- 结构化规则项
- 任务执行/验收检查项

即：

`标准来源 -> 规则项 -> 任务标准绑定 -> 检查项`

这样标准才能真正驱动执行，而不只是做文档存档。

## 10. 自动生成逻辑

### 10.1 自动生成输入

任务树生成至少需要以下输入：

- 品牌
- 店型
- 城市 / 区域
- 建店类型
- 面积 / 预算 / 工期
- 开业目标时间
- 适用标准包
- 适用服务范围

### 10.2 自动生成输出

系统生成任务树时，应至少输出：

- 项目根任务
- 阶段任务
- 执行任务清单
- 任务依赖关系
- 默认责任角色
- 计划时间
- 标准绑定结果
- 必要结果对象入口
- 风险提示

### 10.3 生成来源

V1 采用`模板 + 规则 + Agent 建议 + 人工确认`模式：

- 模板决定基础结构
- 规则决定适配条件与标准包
- Agent 负责补全建议与异常提示
- 人工负责最终确认

### 10.4 生成触发点

任务树自动生成或补生成的典型触发点：

- 项目立项确认后
- 店型/范围发生重大变更后
- 验收不通过生成整改后
- 缺资料判断成立后
- 采购异常触发补充任务后

### 10.5 生成后的人工处理

系统生成的任务树默认不是不可修改的，允许：

- 删除非必做任务
- 新增本项目特殊任务
- 调整责任角色
- 调整计划时间
- 调整依赖关系

但以下内容必须受控：

- 删除关键必做任务
- 移除必绑标准
- 关闭关键验收任务
- 破坏关键路径关系

## 11. 项目 / 流程 / 服务统一视图

### 11.1 项目视图

项目是围绕某个门店建设目标组织起来的任务集合与管理容器。

项目视图关注：

- 目标门店
- 阶段进度
- 里程碑
- 关键路径
- 风险汇总
- 结果对象总览

### 11.2 流程视图

流程是围绕某类业务动作形成的任务链路。

流程视图关注：

- 任务状态流转
- 前置后置关系
- 异常回退
- 节点触发条件
- Agent 参与节点

例如：

- 采购流程任务链
- 验收整改任务链
- 资产归档任务链

### 11.3 服务视图

服务是围绕某类交付结果组织起来的任务包。

服务视图关注：

- 服务包范围
- 服务交付物
- 服务责任主体
- 服务成本与结果

例如：

- 设计服务包
- 施工服务包
- 验收服务包

### 11.4 统一原则

项目、流程、服务不是三套独立底层模型，而是同一任务底座上的不同查询、编排和展示方式。

V1 不单独建设三套相互独立的数据主模型。

## 12. 结果对象关联规则

### 12.1 结果对象定义

任务负责推进动作，结果对象负责沉淀业务成果。

V1 重点结果对象包括：

- 采购申请单
- 采购单
- 验收单
- 整改单
- 资产台账
- 一店一档
- 结算建议单
- Agent 执行记录
- 人工干预记录

### 12.2 关联原则

- 一个任务可以关联多个结果对象
- 一个结果对象也可以回溯到一个或多个任务
- 结果对象状态独立维护，不直接等于任务状态
- 任务完成不代表结果对象一定全部完成

### 12.3 常见映射关系

- 采购类任务 -> 采购申请单 / 采购单
- 验收类任务 -> 验收单
- 整改类任务 -> 整改单
- 资产类任务 -> 资产台账 / 一店一档
- 结算整理任务 -> 结算建议单

---

## 13. Agent 与人工协同规则

### 13.1 Agent 在任务树中的职责

Agent 在任务树中只做受控动作，不做无限制决策。

可承担的典型能力：

- 任务拆解建议
- 标准匹配建议
- 风险识别
- 催办摘要
- 资料完整性检查
- 整改任务建议

### 13.2 Agent 约束

- Agent 不能绕过状态机
- Agent 不能跳过人工审批节点
- Agent 不能删除关键任务
- Agent 不能修改已固化的关键标准快照
- Agent 输出必须结构化并可审计

### 13.3 人工兜底场景

以下场景必须支持人工接管：

- 模板无法覆盖的非标项目
- 多任务互相阻塞
- 验收连续不通过
- 成本或工期异常
- 标准冲突
- 任务关系调整引发关键路径变化

## 14. 审计与治理要求

### 14.1 必须留痕的动作

- 任务创建
- 任务拆解
- 任务删除
- 任务关闭
- 状态变更
- 责任人变更
- 标准绑定变更
- 依赖关系变更
- Agent 决策结果
- 人工接管动作

### 14.2 审计记录字段建议

每条关键日志至少包含：

- 业务对象 ID
- 动作类型
- 动作前后值
- 操作人 / Agent
- 操作来源
- 触发原因
- 时间
- 关联标准版本

### 14.3 风险控制要求

- 对关键任务启用变更审批
- 对关键路径任务启用风险预警
- 对删除/关闭动作要求填写原因
- 对自动生成任务保留生成依据

## 15. V1 实施建议

### 15.1 先做什么

V1 建议优先落地以下能力：

1. 单项目任务树
2. 父子任务管理
3. 基础依赖关系
4. 执行标准 / 验收标准绑定
5. 模板生成任务树
6. 验收失败生成整改任务
7. 审计日志与人工兜底

### 15.2 暂不做什么

V1 不建议一开始就做：

- 超复杂甘特排程
- 多父节点任务图
- 跨项目共享子任务
- 自动重排全局任务网络
- 无边界 AI 自动改任务树

### 15.3 成功标准

任务树建模在 V1 达到可用，至少应满足：

- 能稳定生成标准店型任务树
- 能绑定执行标准和验收标准
- 能支撑项目、流程、服务三类视图
- 能产生整改派生任务
- 能支撑催办、预警、审计和人工接管

## 17. 本文档结论

任务树是本系统最核心的执行底座。

V1 必须坚持以下建模纪律：

- 任务是最小执行单元
- 树结构与关系结构分离
- 标准绑定与标准快照必须存在
- 项目 / 流程 / 服务共用同一任务底座
- 结果对象独立沉淀
- Agent 只能在受控节点参与
- 复杂度优先受控，不盲目追求全功能图模型
