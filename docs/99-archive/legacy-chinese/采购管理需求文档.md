## 1. 模块概述

### 1.1 模块定位

采购管理模块用于承接建店过程中的材料、设备、服务采购协同，统一管理采购申请、审批、下单、到货、异常处理和结果沉淀。

在本系统中：

- `项目管理`负责从项目视角汇总采购摘要、关键物料到位和采购风险
- `任务中心`负责承接采购前置任务、补货任务和与采购结果对象的关联
- `采购管理`负责采购申请、采购单、到货记录、补货与异常处理
- `验收整改`负责使用到货结果和物料资料完成检查与整改判断
- `资产管理`负责把符合归档条件的设备和材料沉淀为资产台账
- `合同结算`负责基于采购结果、验收结果和差异项形成结算依据

因此，采购管理不是孤立的下单工具，而是连接`任务执行、物料协同、验收交付、资产沉淀、结算依据`的结果对象模块。

采购管理模块的核心作用：

- 承接任务或项目产生的采购需求
- 管理采购申请、审批、下单、到货与异常闭环
- 与供应商 / 资源方协同采购进度和到货结果
- 为任务执行提供物料到位依据
- 为验收、资产、结算提供真实采购结果和留痕数据
- 为采购异常、补货、替代和人工介入提供统一入口

### 1.2 模块目标

V1 阶段，采购管理模块重点解决以下问题：

- 项目和任务产生的采购需求缺少统一承接对象
- 材料、设备、服务采购进度无法和任务状态联动
- 到货信息分散，关键物料是否到位不透明
- 采购异常、补货和替代处理缺少可追踪链路
- 采购结果无法稳定沉淀到验收、资产和结算环节
- 项目经理对采购风险和影响范围缺少统一视图

### 1.3 V1 范围

V1 纳入范围：

- 采购申请单管理
- 采购审批基础能力
- 采购单管理
- 到货记录与收货确认
- 部分到货、异常到货、补货处理
- 采购与项目、任务、资产、结算的关联
- 采购结果留痕与附件管理
- 调度 Agent / 结算建议 Agent 辅助能力

V1 不纳入范围：

- 复杂 SRM / ERP 一体化采购平台
- 多供应商自动比价优化引擎
- 自动付款与完整财务核算
- 仓储库存系统
- 复杂框架合同和长期采购计划体系
- 全自动替代料推荐与自主裁决

## 2. 角色与场景

### 2.1 核心角色

#### 品牌方审批人

关注采购申请是否合理、预算是否可接受、关键物料是否按期开工所需。

#### 平台项目经理

关注采购是否影响关键任务、是否存在到货延期、是否需要补货或替代处理。

#### 采购运营 / 协同人员

关注采购申请受理、订单下发、供应商协同、到货回传和异常处理。

#### 供应商 / 资源方协同角色

关注采购单内容、交付时间、发货回执、到货差异和补货要求。

#### 验收 / 资产角色

关注到货物料是否满足验收与归档要求，设备是否需要转入资产台账。

#### 调度 Agent

辅助识别采购对任务排期的影响、提示关键物料风险和补货建议。

#### 结算建议 Agent

辅助汇总采购结果、到货差异、验收结论与后续结算依据。

### 2.2 核心业务场景

#### 场景 1：任务执行前发起采购申请

施工或安装任务识别出所需材料、设备或服务后，系统生成采购需求草案，项目经理补充并提交采购申请。

#### 场景 2：审批通过后生成采购单

品牌方或平台审批通过后，系统生成采购单，明确供应商、交付时间、数量和收货要求。

#### 场景 3：关键物料到货驱动任务解阻

采购单进入到货阶段后，收货人确认到货数量和资料，系统解除相关任务阻塞并更新项目采购摘要。

#### 场景 4：部分到货触发补货或改派

关键物料仅部分到货，系统识别影响任务范围，触发补货申请、替代确认或任务重排建议。

#### 场景 5：采购异常升级处理

出现供应商失约、到货延期、数量不符、质量异常时，系统记录异常，通知项目经理并支持人工介入。

#### 场景 6：设备采购结果进入资产沉淀

设备到货并验收通过后，采购结果可映射为资产台账的来源依据。

#### 场景 7：采购结果参与结算建议

系统在结算前汇总采购单、到货记录、验收差异和补货情况，为结算建议提供基础依据。

## 3. 业务对象定义

### 3.1 采购申请单 Purchase Request

采购申请单用于承接项目或任务产生的采购需求，是采购流程的起点对象。

核心字段建议：

- `purchase_request_id`
- `request_code`
- `project_id`
- `task_id`
- `request_type`：材料 / 设备 / 服务
- `request_source_type`：任务触发 / 项目补充 / 异常补货 / 人工创建
- `request_reason`
- `requester_id`
- `brand_id`
- `store_id`
- `planned_need_date`
- `budget_estimate`
- `request_status`
- `priority`

### 3.2 采购明细 Purchase Request Item

采购明细用于表达单个申请单中的具体采购项。

核心字段建议：

- `request_item_id`
- `purchase_request_id`
- `item_name`
- `item_category`
- `item_spec`
- `unit`
- `quantity`
- `estimated_unit_price`
- `estimated_total_price`
- `required_flag`
- `asset_candidate_flag`
- `standard_rule_refs`

### 3.3 采购单 Purchase Order

采购单用于承接审批通过后的正式采购执行对象。

核心字段建议：

- `purchase_order_id`
- `order_code`
- `purchase_request_id`
- `project_id`
- `task_id`
- `supplier_id`
- `supplier_name`
- `order_type`
- `order_amount`
- `planned_delivery_at`
- `actual_delivery_at`
- `receiver_id`
- `delivery_address`
- `purchase_status`
- `approval_result`
- `close_reason`

### 3.4 到货记录 Delivery Record

到货记录用于记录每次发货、签收、差异和收货确认，是采购状态推进和任务解阻的重要依据。

核心字段建议：

- `delivery_record_id`
- `purchase_order_id`
- `delivery_batch_no`
- `delivered_at`
- `received_at`
- `received_by`
- `delivery_status`
- `received_quantity`
- `difference_quantity`
- `difference_reason`
- `attachment_ids`
- `receipt_confirm_flag`

### 3.5 补货 / 替代申请 Replenishment Request

补货 / 替代申请用于处理部分到货、质量异常、缺件等场景。

核心字段建议：

- `replenishment_id`
- `purchase_order_id`
- `source_delivery_record_id`
- `replenishment_type`：补货 / 替代 / 重发
- `reason_type`
- `affected_task_ids`
- `required_arrival_at`
- `status`
- `created_by`

### 3.6 采购异常 Purchase Exception

采购异常用于沉淀对项目执行有影响的采购偏差和处置记录。

核心字段建议：

- `exception_id`
- `project_id`
- `task_id`
- `purchase_order_id`
- `exception_type`
- `exception_level`
- `exception_desc`
- `impact_scope`
- `owner_id`
- `due_at`
- `exception_status`
- `resolution`

### 3.7 供应商协同对象 Supplier Collaboration

V1 不建设完整供应商门户，但采购管理需要保留基础供应商协同信息。

核心字段建议：

- `supplier_id`
- `supplier_name`
- `supplier_type`
- `contact_person`
- `contact_phone`
- `city_scope`
- `qualification_status`
- `cooperation_status`

### 3.8 采购结果与下游对象映射

采购结果需要与以下对象建立映射：

- 任务：采购前置任务、安装任务、补货任务
- 项目：采购摘要、采购风险、关键物料到位里程碑
- 资产：设备类采购可映射为资产台账来源
- 结算：采购金额、到货差异、补货记录可作为结算依据

建议关联字段：

- `linked_task_ids`
- `linked_project_id`
- `linked_asset_ids`
- `linked_settlement_ids`
- `source_snapshot_id`

## 4. 任务结构设计

### 4.1 采购与项目、任务、资产、结算的关系

采购管理模块必须重点描述以下关系：

#### 与项目的关系

- 一个项目可关联多个采购申请单和采购单
- 项目维度需要汇总采购金额、到货进度、采购风险和关键物料到位情况
- 采购异常会回写项目风险摘要

#### 与任务的关系

- 采购需求通常由任务触发或为任务服务
- 采购未到货可阻塞相关执行任务
- 到货确认可解除相关任务阻塞
- 采购异常可触发补货任务或任务计划调整

#### 与资产的关系

- 设备类采购结果可作为资产台账来源
- 资产建档应可追溯到采购单、到货记录和验收结果
- 非资产类耗材不进入长期资产台账

#### 与结算的关系

- 采购单和到货记录可形成结算依据的一部分
- 补货、差异、替代和异常处理结果需进入结算说明
- 未闭环采购异常会影响结算建议可信度

### 4.2 采购流程结构

V1 采购流程建议采用以下轻量结构：

1. 采购需求识别
2. 采购申请提交
3. 审批通过 / 驳回
4. 采购单创建
5. 发货与到货回传
6. 收货确认与差异处理
7. 异常闭环 / 补货处理
8. 结果沉淀到验收、资产、结算

该结构重点不是做复杂供应链系统，而是确保建店项目的关键物料、设备、服务采购可以被任务执行有效消费。

### 4.3 采购需求来源设计

V1 采购需求主要来自以下来源：

- 项目模板或任务模板预置需求
- 执行任务中的物料 / 设备 / 服务申请
- 验收或整改触发的补货需求
- 人工新增的项目补充采购需求

生成规则建议：

- 模板可预置常规采购清单
- 标准规则可限定必备材料 / 设备
- 关键任务可要求采购前置校验
- 异常补货应保留来源任务和来源采购单关系

### 4.4 采购与任务前置关系设计

采购与任务关系建议采用以下口径：

- 关键安装任务可依赖采购到货
- 采购单不等于任务本身，而是任务的结果对象或前置条件对象
- 一个任务可关联多个采购单
- 一个采购单也可服务多个关联任务
- 部分到货时，应根据明细判断哪些任务可继续、哪些仍需阻塞

### 4.5 补货与派生任务设计

出现以下情况时允许派生补货任务或异常处理任务：

- 到货数量不足
- 到货规格不符
- 关键物料损坏
- 到货延期影响施工
- 验收不通过需要替换材料或设备

派生任务必须记录：

- 来源采购单
- 来源到货记录
- 影响任务
- 补货原因
- 紧急程度

### 4.6 关键里程碑与风险设计

采购管理需要支撑以下项目级识别能力：

- 关键物料到位
- 关键设备到位
- 采购审批完成
- 补货闭环完成

风险识别建议包括：

- 到货延期风险
- 部分到货风险
- 价格偏差风险
- 质量异常风险
- 关键任务阻塞风险

### 4.7 V1 结构边界

V1 不做：

- 仓储库存深度管理
- 多级供应链协同图谱
- 长周期采购计划管理
- 复杂成本拆账与财务记账

V1 先把以下链路跑通：

- 申请
- 审批
- 下单
- 到货
- 差异处理
- 补货
- 任务联动
- 结果沉淀

## 5. 标准绑定设计

### 5.1 绑定目标

采购管理模块中的标准绑定，重点回答：

- 该采购项可以买什么
- 规格、品牌、材质、数量上有什么限制
- 到货后需要提交什么资料
- 哪些设备需要进入资产台账
- 哪些异常会触发补货或替代审批

### 5.2 绑定链路

采购管理与标准库的绑定链路建议为：

`标准来源 -> 条款 -> 规则项 -> 执行标准 / 验收标准 -> 标准包 -> 任务模板 / 采购模板 -> 采购申请 / 采购单 -> 标准快照`

### 5.3 绑定层级

V1 建议支持以下层级：

1. 项目模板层
2. 任务模板层
3. 采购模板层
4. 采购申请层
5. 采购单层
6. 到货验收层

### 5.4 采购标准对象建议

采购管理重点使用以下标准内容：

- 材料 / 设备规格规则
- 品牌与型号限制规则
- 数量与单位规则
- 必传回执与照片规则
- 到货验收与签收规则
- 资产归档规则
- 替代与补货触发规则

### 5.5 标准绑定规则

- 采购申请可继承项目或任务的标准上下文
- 关键设备采购必须绑定规格与验收规则
- 需要归档的设备应绑定资产归档规则
- 已下单对象进入执行阶段后应固化标准快照
- 已生成快照的采购对象不允许被新标准静默覆盖

### 5.6 资料与检查要求

采购管理 V1 至少应支持以下资料要求：

- 采购申请依据附件
- 审批记录
- 采购单回执
- 发货 / 到货凭证
- 收货照片或签收证明
- 质量异常说明
- 设备铭牌或序列号资料

### 5.7 快照与争议处理

采购对象建议在以下节点生成快照：

- 采购单创建后
- 关键设备到货确认前
- 补货或替代申请确认时

快照至少包含：

- 采购对象 ID
- 适用标准对象 ID
- 规格与资料要求
- 规则项清单
- 来源条款引用
- 生成时间与生成来源

## 6. 页面结构与功能点

### 6.1 页面总体结构

采购管理建议包含以下核心页面：

1. 采购中心首页
2. 采购申请单列表页
3. 采购单列表页
4. 到货与收货页
5. 采购详情页
6. 采购异常与补货页

V1 页面原则：

- 以采购中心首页承接统计和筛选
- 以申请单和采购单作为主业务列表
- 以详情页承接完整协同动作
- 以异常与补货页承接异常闭环
- 以到货页承接任务解阻与收货确认

### 6.2 采购中心首页

#### 页面定位

统一查看采购申请、采购执行进度、关键物料风险和到货状态。

#### 视图模式

建议支持以下 3 种视图：

- 列表视图（默认）
- 卡片视图（适合按采购类型浏览）
- 看板视图（适合按状态查看）

#### 顶部统计区

建议展示：

- 全部采购申请数
- 待审批数
- 待下单数
- 待到货数
- 部分到货数
- 异常中数
- 关键物料未到位数

#### 筛选与查询

支持以下维度：

- 项目
- 门店
- 采购类型
- 状态
- 供应商
- 计划到货时间
- 是否关键物料
- 是否资产类
- 是否异常

### 6.3 采购申请单列表页

#### 页面定位

承接采购需求提交、审批前校验和申请状态跟踪。

#### 核心功能

- 新建采购申请单
- 查看申请明细
- 关联项目与任务
- 查看预算估算
- 发起审批 / 撤回申请
- 查看审批结果与意见

### 6.4 采购单列表页

#### 页面定位

用于管理正式采购执行对象和供应商协同状态。

#### 核心功能

- 查看采购单列表
- 查看供应商与计划到货时间
- 维护采购金额与订单信息
- 查看订单回执与发货状态
- 发起催货或异常上报
- 查看关联任务阻塞范围

### 6.5 到货与收货页

#### 页面定位

用于承接发货回传、到货签收、差异确认和任务解阻。

#### 核心功能

- 录入到货批次
- 查看明细到货数量
- 上传收货凭证和照片
- 标记部分到货 / 已到货 / 异常到货
- 确认是否解除任务阻塞
- 对设备类对象采集资产建档必要信息

### 6.6 采购详情页

#### 页面定位

采购详情页是采购对象的主操作面板，用于承接申请、审批、订单、到货、异常、补货和下游引用关系。

#### 页面结构建议

1. 基本信息区
2. 申请与审批区
3. 明细与规格区
4. 到货记录区
5. 异常与补货区
6. 任务与项目关联区
7. 资产 / 结算关联区
8. 审计记录区

#### 关键展示信息

- 申请单 / 采购单编号
- 项目 / 任务路径
- 采购类型
- 供应商信息
- 当前状态
- 计划到货 / 实际到货时间
- 金额与数量摘要
- 关键风险标签
- 关联资产与结算对象

### 6.7 采购异常与补货页

#### 页面定位

统一承接采购异常、补货申请、替代确认和升级处理。

#### 核心功能

- 查看异常列表
- 查看影响任务与项目范围
- 发起补货 / 替代 / 重发
- 查看处理时限和责任人
- 查看历史处理记录
- 手工关闭异常

### 6.8 关键交互规则

- 未审批通过的采购申请不得生成正式采购单
- 关键采购单未到货时需提示受影响任务
- 部分到货必须明细化到具体采购项
- 异常中采购对象不得静默标记为已完成
- 关键设备收货前需校验必要资料是否齐全

## 7. 状态机设计

### 7.1 采购状态口径

V1 采购主状态统一为：

- `待申请`
- `待审批`
- `待下单`
- `待到货`
- `部分到货`
- `已到货`
- `异常中`
- `已关闭`

### 7.2 主流转路径

`待申请 -> 待审批 -> 待下单 -> 待到货 -> 已到货`

### 7.3 异常流转路径

- `待审批 -> 已关闭`
- `待到货 -> 部分到货`
- `待到货 -> 异常中`
- `部分到货 -> 异常中`
- `异常中 -> 待到货`
- `异常中 -> 已关闭`

### 7.4 守卫条件

#### 待申请 -> 待审批

- 采购内容、数量、关联任务明确
- 申请依据完整
- 必填预算信息完整

#### 待审批 -> 待下单

- 审批通过
- 预算校验通过
- 供应商范围已确认或可进入下单选择

#### 待下单 -> 待到货

- 采购单已生成
- 供应商信息明确
- 计划到货时间明确

#### 待到货 -> 已到货

- 到货数量满足要求
- 收货确认完成
- 关键资料已提交或确认不适用

#### 待到货 -> 部分到货

- 仅部分采购项完成交付
- 差异已记录
- 受影响范围已识别

### 7.5 联动规则

- 采购状态异常可阻塞相关执行任务
- 到货完成可解除相关任务阻塞
- 部分到货可触发补货或改派动作
- 关键设备到货可触发资产建档前置动作
- 采购异常未关闭时可提升项目采购风险等级

### 7.6 关闭与回退规则

- 已关闭采购对象默认只读
- 从异常中回到待到货必须记录处理结论
- 关键采购单关闭需提示受影响任务和项目范围
- 已到货对象如需回退，必须记录原因并由人工确认

## 8. 权限与角色控制

### 8.1 品牌方权限

- 可查看本品牌项目下采购申请、采购单和到货结果
- 可审批品牌侧采购申请
- 可查看采购对项目进度的影响
- 不可查看其他品牌数据

### 8.2 平台项目经理权限

- 可创建和管理采购申请
- 可查看采购异常、补货和到货影响范围
- 可关联任务、调整优先级、发起催货和升级
- 不可绕过审批直接完成关键采购单

### 8.3 采购运营权限

- 可管理采购单、供应商信息和到货记录
- 可处理异常、补货、替代和关闭动作
- 可维护收货状态与资料完整性

### 8.4 供应商 / 资源方协同权限

- 仅可查看分配给本方的采购协同信息
- 可回传订单回执、发货信息、到货说明
- 不可查看其他供应商数据
- 不可修改审批结果和项目级关键信息

### 8.5 验收 / 资产 / 结算相关权限

- 验收角色可查看采购结果和资料证明
- 资产角色可读取设备类采购到货信息用于建档
- 结算角色可读取采购金额、差异和异常说明
- 上述角色不可反向修改采购审批结果

### 8.6 Agent 权限

- 调度 Agent 可提示采购影响和补货建议
- 结算建议 Agent 可读取采购结果用于差异分析
- Agent 不可跳过审批节点
- 所有 Agent 动作必须保留依据摘要和日志

## 9. 异常与人工介入

### 9.1 典型异常类型

- 采购申请信息缺失
- 审批未通过
- 供应商无法履约
- 到货延期
- 部分到货
- 到货数量不符
- 到货规格不符
- 到货质量异常
- 关键物料导致任务阻塞
- 补货后仍未闭环

### 9.2 人工介入触发条件

以下情况必须支持人工介入：

- 关键采购对象即将影响开工或里程碑
- 连续补货仍未满足需求
- 供应商失联或拒绝履约
- 替代方案是否允许存在争议
- 到货差异金额较大
- 关键设备资料缺失影响资产建档
- 采购异常影响结算口径

### 9.3 人工介入动作

V1 允许以下动作：

- 手工修正采购申请内容
- 手工改派供应商或协同对象
- 手工发起补货 / 重发 / 替代
- 手工标记受影响任务
- 手工关闭或重开异常
- 手工确认关键设备建档口径
- 手工确认采购差异结论

### 9.4 异常治理要求

- 异常需记录来源、影响范围、责任人、处理结论
- 关键异常需同步项目风险
- 人工覆盖系统建议需记录原因
- 补货与替代必须保留来源链路

## 10. 埋点与验收标准

### 10.1 关键埋点事件

建议埋点以下事件：

- `purchase.request_created`
- `purchase.request_submitted`
- `purchase.request_approved`
- `purchase.order_created`
- `purchase.delivery_reported`
- `purchase.receipt_confirmed`
- `purchase.partial_received`
- `purchase.exception_created`
- `purchase.replenishment_created`
- `purchase.status_changed`
- `purchase.block_task`
- `purchase.unblock_task`
- `purchase.asset_mapped`
- `purchase.settlement_linked`

### 10.2 核心指标

#### 效率指标

- 采购申请平均处理耗时
- 审批平均耗时
- 下单平均耗时
- 到货平均耗时
- 补货闭环平均耗时

#### 质量指标

- 关键物料按时到位率
- 到货一次满足率
- 补货发生率
- 到货资料完整率
- 设备建档映射成功率

#### 风险指标

- 到货延期数
- 部分到货数
- 异常中采购单数
- 因采购导致的任务阻塞数
- 因采购差异导致的结算争议数

### 10.3 功能验收标准

- 可创建采购申请单并关联项目与任务
- 可完成审批、下单、到货基本闭环
- 可记录部分到货、异常到货和补货处理
- 可把采购结果回写到任务阻塞和项目采购摘要
- 可将设备类采购结果映射到资产建档来源
- 可把采购差异和异常记录输出给结算环节
- 可查看采购全流程操作日志和附件资料

### 10.4 业务验收标准

- 至少跑通 1 条真实建店项目的采购协同闭环
- 采购状态流转与状态机口径一致
- 到货完成可正确解除相关任务阻塞
- 采购异常可正确触发补货或人工介入
- 项目、任务、资产、结算四类关联关系可追溯
- 关键物料是否到位可在项目视图中清晰展示

### 10.5 模块结论

采购管理模块是 V1 建店闭环里承上启下的结果对象模块。

它既承接任务执行对物料、设备、服务的真实需求，也把采购结果继续传递给验收、资产和结算环节。

V1 的重点不是做完整供应链平台，而是稳定跑通以下链路：

- 采购需求有对象承接
- 审批与下单有状态留痕
- 到货能影响任务推进
- 异常可补货、可升级、可追溯
- 设备结果可沉淀资产
- 采购差异可进入结算依据

当这条链路稳定后，后续可以继续增强供应商协同、库存能力、成本控制和合同结算一体化能力。
