# 状态机设计说明

> **文档版本**：V1.0
> **文档状态**：草稿中
> **适用阶段**：V1 / MVP
> **适用范围**：项目、任务、采购、验收、资产归档、结算、工单、Agent 节点
> **关联文档**：`产品规划文档.md`、`任务树建模说明.md`、`标准库结构化说明.md`、`多Agent建店管理平台_V1_PRD.md`、`多Agent建店管理平台_V1_技术方案.md`

---

## 1. 文档目标

本文档用于统一系统中各核心业务对象的状态定义、状态流转、触发条件、权限边界、异常处理和审计要求。

本系统必须明确：

- `流程由状态机驱动，不由人或 Agent 自由推动`
- `任务树负责承接执行颗粒度，状态机负责约束推进路径`
- `标准库负责提供进入下一状态的判断依据`
- `所有关键状态变化都必须可追溯`
- `异常场景必须允许人工接管，但不能破坏状态纪律`

---

## 2. 状态机设计定位

### 2.1 为什么必须做状态机

本项目涉及项目、任务、采购、验收、整改、资产、结算多个对象，如果没有统一状态机，会出现：

- 同一业务对象被不同角色用不同口径描述
- Agent 输出结果无法稳定写回系统
- 任务推进依赖口头协同和人工判断
- 验收、整改、结算无法形成标准闭环
- 历史项目无法审计和回放

因此，状态机不是页面上的标签枚举，而是业务系统的推进规则。

### 2.2 状态机在系统中的角色

状态机承担 6 个职责：

1. 定义业务对象当前所处阶段
2. 定义哪些动作可以发生
3. 定义进入下一状态所需条件
4. 定义异常回退与升级路径
5. 定义 Agent 可参与的节点边界
6. 定义通知、审计、统计的统一触发点

### 2.3 设计原则

- `单一真实状态`：一个对象同一时刻只应有一个主状态
- `动作触发流转`：状态变化必须由明确事件触发
- `条件校验先行`：未满足条件不得进入下一状态
- `关键节点留痕`：状态变化必须记录操作者、原因、时间
- `人工兜底受控`：人工可干预，但必须走受控动作
- `对象状态分离`：项目、任务、验收、结算等对象各自维护状态，不互相替代
- `兼容 Agent`：Agent 只能在授权节点建议或执行受限动作

## 3. 适用范围与 V1 边界

### 3.1 纳入状态机治理的对象

V1 纳入统一状态机治理的对象包括：

- 项目
- 任务
- 采购
- 验收
- 资产归档
- 结算
- 工单
- Agent 执行记录

### 3.2 V1 边界

V1 先采用`轻量主状态机 + 异常分支 + 审计日志`方式，不追求超复杂 BPMN 编排。

V1 阶段建议：

- 每类对象先只维护一个主状态字段
- 子状态、标签、风险等级作为补充维度，不替代主状态
- 跨对象联动通过事件驱动实现，不做强耦合同步写死
- 暂不引入超复杂并行子流程状态网络
- 暂不支持用户任意自定义状态

## 4. 状态机总体结构

### 4.1 分层设计

建议按以下分层理解状态机：

1. `业务对象主状态`
2. `状态触发事件`
3. `前置校验条件`
4. `流转动作`
5. `联动副作用`
6. `异常回退机制`
7. `审计与通知`

### 4.2 状态机统一结构

每个状态机至少要回答以下问题：

- 当前对象有哪些状态
- 从哪个状态可以流转到哪个状态
- 谁可以触发这次流转
- 触发前要校验什么
- 流转后要联动什么对象
- 是否要通知、生成日志、触发 Agent、生成任务或结果对象

### 4.3 通用元模型建议

每个状态流转动作建议至少包含：

- `object_type`
- `object_id`
- `from_status`
- `to_status`
- `event_code`
- `operator_type`
- `operator_id`
- `trigger_source`
- `reason`
- `guard_result`
- `executed_at`
- `trace_id`

## 5. 核心通用规则

### 5.1 主状态与补充状态分离

- 主状态表达业务阶段
- 风险等级表达风险程度
- 标签表达分类属性
- 阻塞原因表达异常说明
- 审批结果表达管理动作

以上补充字段都不能替代主状态。

### 5.2 状态触发来源

状态变化的触发来源统一分为：

- `人工操作`
- `系统自动`
- `规则触发`
- `Agent 节点输出`
- `外部事件回传`

### 5.3 流转校验规则

任意状态流转前，至少校验：

- 当前状态是否合法
- 操作角色是否有权限
- 前置对象是否已满足条件
- 必传字段是否完整
- 关键标准或快照是否已绑定
- 是否命中异常拦截规则

### 5.4 回退与关闭原则

- 回退必须有明确原因
- 关闭必须区分`完成关闭`与`终止关闭`
- 关键对象不允许无痕回退
- 已归档对象默认不可直接回退到执行态

### 5.5 审计原则

以下变化必须留痕：

- 主状态变化
- 异常升级
- 人工接管
- Agent 推进状态
- 拒绝、驳回、关闭、重开
- 关键依赖对象联动触发

## 6. 项目状态机

### 6.1 项目状态口径

V1 项目主状态统一为：

- `待立项`
- `待确认`
- `待拆解`
- `执行中`
- `待验收`
- `整改中`
- `待结算`
- `已归档`
- `已中止`

### 6.2 状态含义

#### 待立项

品牌需求尚未形成有效项目草案。

#### 待确认

项目基础信息已形成，等待品牌或管理角色确认立项。

#### 待拆解

项目已确认，但任务树、阶段、标准绑定尚未准备完成。

#### 执行中

任务树已经生成并开始推进。

#### 待验收

项目核心执行任务已完成，进入整体验收阶段。

#### 整改中

项目存在未通过验收项，正在整改闭环。

#### 待结算

项目验收闭环完成，等待结算建议与审核。

#### 已归档

项目交付完成，资料、资产、结算建议已归档到位。

#### 已中止

项目因取消、暂停、重大异常等原因终止推进。

### 6.3 主流转路径

`待立项 -> 待确认 -> 待拆解 -> 执行中 -> 待验收 -> 待结算 -> 已归档`

### 6.4 异常流转路径

- `待确认 -> 已中止`
- `待拆解 -> 已中止`
- `执行中 -> 已中止`
- `待验收 -> 整改中`
- `整改中 -> 待验收`
- `待结算 -> 整改中`

### 6.5 关键守卫条件

#### 待确认 -> 待拆解

- 项目基础信息完整
- 品牌、门店、店型、工期等核心字段齐全
- 审批动作完成

#### 待拆解 -> 执行中

- 任务树已生成
- 关键任务已绑定责任角色
- 关键标准已绑定
- 至少一个执行任务进入可执行状态

#### 执行中 -> 待验收

- 关键执行任务已完成
- 必要资料已提交
- 验收申请已发起

#### 待验收 -> 待结算

- 验收状态达到通过口径
- 不存在未关闭整改任务

#### 待结算 -> 已归档

- 结算建议生成完成
- 资产归档完成
- 一店一档资料齐全

### 6.6 联动规则

- 项目进入`待拆解`后触发任务树生成
- 项目进入`执行中`后启动任务催办与预警
- 项目进入`待验收`后触发验收流程
- 项目进入`整改中`后汇总未通过任务
- 项目进入`待结算`后触发结算建议流程
- 项目进入`已归档`后冻结关键编辑入口

## 7. 任务状态机

### 7.1 任务状态口径

V1 任务主状态统一为：

- `待创建`
- `待分配`
- `待执行`
- `执行中`
- `待提交`
- `待验收`
- `不通过`
- `已完成`
- `已关闭`

### 7.2 主流转路径

`待创建 -> 待分配 -> 待执行 -> 执行中 -> 待提交 -> 待验收 -> 已完成`

### 7.3 异常流转路径

- `待验收 -> 不通过`
- `不通过 -> 待执行`
- `待分配 -> 已关闭`
- `执行中 -> 已关闭`
- `待提交 -> 执行中`

### 7.4 状态含义

#### 待创建

任务模板已命中或需求已产生，但尚未正式确认创建。

#### 待分配

任务已存在，但尚未完成执行主体分配。

#### 待执行

任务责任明确、标准明确、前置条件满足，可开始执行。

#### 执行中

任务正在执行，允许记录进度、资料、阻塞、催办结果。

#### 待提交

执行动作完成，等待提交成果资料、回执或结果说明。

#### 待验收

成果已提交，等待按验收标准进行检查与判定。

#### 不通过

任务验收未通过或校验不满足，需要整改或补充。

#### 已完成

任务按标准完成，验收口径已满足。

#### 已关闭

任务因取消、替代、合并、终止而关闭。

### 7.5 守卫条件

#### 待分配 -> 待执行

- 已分配责任角色或执行人
- 前置依赖满足
- 执行标准已绑定
- 标准快照已生成

#### 待执行 -> 执行中

- 执行主体已确认开始
- 无阻塞前置任务

#### 执行中 -> 待提交

- 执行动作完成
- 必传资料已提交或已明确不适用

#### 待提交 -> 待验收

- 提交结果完整
- 验收标准已绑定
- 检查项已生成

#### 待验收 -> 已完成

- 检查项通过
- 不存在未关闭缺陷

#### 待验收 -> 不通过

- 存在关键不合格项
- 必传资料缺失
- 关键阈值未满足

### 7.6 联动规则

- 任务进入`待执行`时生成标准快照
- 任务进入`执行中`时开启 SLA 计时
- 任务进入`待验收`时生成验收检查项
- 任务进入`不通过`时可派生整改任务
- 任务进入`已完成`时更新父任务汇总状态
- 关键任务关闭时需触发项目风险重算

### 7.7 父子任务约束

- 父任务默认不能早于关键子任务完成
- 子任务未完成时，父任务不得进入完成态
- 子任务批量不通过时，父任务可显示阻塞但不必自动关闭
- 父任务关闭时应提示影响范围并要求人工确认

## 8. 采购状态机

### 8.1 采购状态口径

V1 采购主状态统一为：

- `待申请`
- `待审批`
- `待下单`
- `待到货`
- `部分到货`
- `已到货`
- `异常中`
- `已关闭`

### 8.2 主流转路径

`待申请 -> 待审批 -> 待下单 -> 待到货 -> 已到货`

### 8.3 异常流转路径

- `待审批 -> 已关闭`
- `待到货 -> 部分到货`
- `待到货 -> 异常中`
- `部分到货 -> 异常中`
- `异常中 -> 待到货`
- `异常中 -> 已关闭`

### 8.4 守卫条件

#### 待申请 -> 待审批

- 采购内容、数量、关联任务明确
- 申请依据完整

#### 待审批 -> 待下单

- 审批通过
- 预算校验通过

#### 待下单 -> 待到货

- 采购单已生成
- 供应商信息明确

#### 待到货 -> 已到货

- 到货数量满足要求
- 收货确认完成

### 8.5 联动规则

- 采购状态异常可阻塞相关执行任务
- 到货完成可解除相关任务阻塞
- 部分到货可触发补货或改派动作

## 9. 验收状态机

### 9.1 验收状态口径

V1 验收主状态统一为：

- `待申请`
- `待初验`
- `初验通过`
- `初验不通过`
- `整改中`
- `待复验`
- `复验通过`
- `已关闭`

### 9.2 主流转路径

`待申请 -> 待初验 -> 初验通过`

或：

`待申请 -> 待初验 -> 初验不通过 -> 整改中 -> 待复验 -> 复验通过`

### 9.3 守卫条件

#### 待申请 -> 待初验

- 验收申请已提交
- 必传资料齐全
- 验收标准已绑定
- 检查项已生成

#### 待初验 -> 初验通过

- 检查项通过
- 无关键缺陷

#### 待初验 -> 初验不通过

- 存在关键缺陷
- 关键资料缺失
- 关键阈值不达标

#### 初验不通过 -> 整改中

- 整改任务已生成或整改要求已明确

#### 整改中 -> 待复验

- 整改结果已提交
- 复验申请已发起

#### 待复验 -> 复验通过

- 复验检查通过
- 整改项已全部关闭

### 9.4 联动规则

- 初验不通过应派生整改任务
- 复验通过可推动任务或项目进入完成/待结算
- 验收关闭后冻结检查项关键字段

### 11.1 结算状态口径

V1 结算主状态统一为：

- `待生成建议`
- `待审核`
- `待确认`
- `已完成`
- `争议中`

### 11.2 主流转路径

`待生成建议 -> 待审核 -> 待确认 -> 已完成`

### 11.3 异常流转路径

- `待审核 -> 争议中`
- `待确认 -> 争议中`
- `争议中 -> 待审核`
- `争议中 -> 已完成`

### 11.4 守卫条件

#### 待生成建议 -> 待审核

- 验收通过
- 合同、报价、变更、验收依据已齐
- 差异分析完成

#### 待审核 -> 待确认

- 财务或业务审核通过
- 风险项已标记

#### 待确认 -> 已完成

- 最终确认完成
- 争议项已处理

### 11.5 联动规则

- 结算进入`待审核`可触发结算建议 Agent 输出草案
- 进入`争议中`时应冻结自动推进
- 结算完成可作为项目归档条件之一

## 13. Agent 节点状态约束

### 13.1 Agent 不是独立主状态机

V1 不为 Agent 单独构建主业务流程状态机替代业务对象状态机。

Agent 的角色是：

- 在授权节点接收触发
- 输出结构化建议或受限动作
- 由规则引擎校验后影响业务对象状态

### 13.2 Agent 可参与的典型节点

- 项目待确认前的信息补全
- 项目待拆解阶段的任务树建议
- 任务待分配阶段的资源推荐
- 任务执行中的催办与预警
- 任务待验收阶段的资料完整性检查
- 结算待生成建议阶段的差异草案生成

### 13.3 Agent 约束原则

- 不能绕过状态机直接改关键状态
- 不能跳过审批节点
- 不能直接删除关键对象
- 不能直接修改已固化快照
- 只能在授权事件中执行

### 13.4 Agent 执行记录建议状态

为便于追踪，可给 Agent 执行记录使用轻量状态：

- `待执行`
- `执行中`
- `已完成`
- `已转人工`
- `已失败`

该状态仅用于 Agent 运行记录，不替代业务对象状态。

---

## 14. 跨对象联动规则

### 14.1 项目与任务

- 项目进入`待拆解`后创建任务树
- 关键任务大面积阻塞时，项目风险提升
- 项目是否进入`待验收`取决于关键任务完成情况

### 14.2 任务与采购

- 采购未到货可阻塞相关执行任务
- 到货确认可解除任务阻塞

### 14.3 任务与验收

- 任务进入`待验收`时创建验收对象或验收检查项
- 验收不通过可使任务进入`不通过`

### 14.4 验收与整改

- 初验不通过必须生成整改要求或整改任务
- 复验通过才能关闭对应整改闭环

### 14.5 项目与结算

- 项目进入`待结算`前必须没有未关闭整改项
- 结算完成是项目归档的必要条件之一

### 14.6 项目与资产归档

- 资产归档完成是一店一档完整性的关键组成
- 项目归档前应校验资产归档状态

---

## 15. 事件、通知与副作用

### 15.1 事件类型建议

关键状态变化后建议统一发布事件：

- `project.status_changed`
- `task.status_changed`
- `purchase.status_changed`
- `acceptance.status_changed`
- `asset.status_changed`
- `settlement.status_changed`
- `ticket.status_changed`

### 15.2 常见副作用

状态流转后可触发：

- 站内通知
- 待办生成
- SLA 计时开始/停止
- Agent 节点触发
- 检查项生成
- 整改任务生成
- 风险重算
- 日志写入

### 15.3 通知原则

- 通知服务于状态变化，不反过来定义状态
- 同一状态变化的通知模板应标准化
- 高风险状态变化需通知更高层级角色

---

## 16. 权限与人工干预规则

### 16.1 权限原则

- 不同角色可触发的流转动作不同
- 查看权限与流转权限必须分开设计
- Agent 权限必须显式声明，不默认继承管理员权限

### 16.2 人工干预动作建议

V1 建议显式支持以下人工动作：

- 驳回
- 关闭
- 重开
- 转人工
- 异常升级
- 解除阻塞
- 手动回退

### 16.3 干预要求

- 干预必须填写原因
- 干预必须记录操作人
- 对关键状态回退需二次确认或审批
- 已归档对象原则上不允许直接修改状态，只允许发起变更流程

---

## 17. 审计与数据治理要求

### 17.1 必须记录的日志字段

每次状态变化至少记录：

- 对象类型
- 对象 ID
- 变更前状态
- 变更后状态
- 触发事件
- 操作人 / Agent
- 触发来源
- 原因
- 发生时间
- 关联项目 / 任务 / 单据

### 17.2 数据治理要求

- 枚举值统一管理
- 状态描述统一口径
- 禁止前后端各自维护不同状态集
- 历史状态枚举废弃时要保留兼容映射

### 17.3 风险控制要求

- 对非法流转必须拦截并报错
- 对批量流转要保留批次 trace
- 对 Agent 触发的状态变化要保留依据摘要
- 对跨对象联动失败要支持补偿或人工处理

---

## 18. V1 实施建议

### 18.1 先做什么

V1 建议优先落地：

1. 项目状态机
2. 任务状态机
3. 验收状态机
4. 结算状态机
5. 状态流转日志
6. 事件通知机制
7. 人工兜底动作

### 18.2 暂不做什么

V1 不建议一开始就做：

- 超复杂并行子流程状态网络
- 用户自定义状态配置中心
- 无限制跨对象自动联动
- 多 Agent 自治协商推进流程
- 图形化全流程编排器完整版

### 18.3 成功标准

状态机在 V1 达到可用，至少应满足：

- 所有核心对象有统一主状态口径
- 关键流转有明确守卫条件
- 关键状态变化可触发通知与审计
- Agent 只能在受控节点参与推进
- 项目、任务、验收、结算能形成稳定闭环

---

## 19. 示例

### 19.1 项目主流程示例

`待立项 -> 待确认 -> 待拆解 -> 执行中 -> 待验收 -> 待结算 -> 已归档`

### 19.2 任务执行示例

`待创建 -> 待分配 -> 待执行 -> 执行中 -> 待提交 -> 待验收 -> 已完成`

异常分支：

`待验收 -> 不通过 -> 待执行`

### 19.3 验收整改示例

`待申请 -> 待初验 -> 初验不通过 -> 整改中 -> 待复验 -> 复验通过`

### 19.4 结算示例

`待生成建议 -> 待审核 -> 待确认 -> 已完成`

争议分支：

`待审核 -> 争议中 -> 待审核`

---

## 20. 本文档结论

状态机是本系统流程治理的骨架。

V1 必须坚持以下纪律：

- 状态驱动流程，不能靠自由协同推进
- 主状态、标签、风险、审批结果必须分层表达
- 任务树承接执行，状态机承接约束
- 标准库负责进入下一状态的判断依据
- Agent 只能在受控节点参与
- 所有关键状态变化必须可审计、可回放、可人工兜底
