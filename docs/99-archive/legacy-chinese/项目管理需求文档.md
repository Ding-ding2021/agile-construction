## 1. 模块概述

### 1.1 模块定位

项目管理模块是品牌方营建部门的核心工作入口，用于承接门店建设目标、立项信息、阶段计划、里程碑、风险、资料和任务总览。

在本系统中，`项目不是最小执行单元`，而是围绕某个建店目标组织起来的`任务集合容器与管理视图`。

项目管理模块的核心作用：

- 承接立项与项目容器创建
- 形成项目阶段与里程碑框架
- 汇总任务树执行进展
- 提供项目风险识别与预警
- 汇总采购、验收、资产、结算等结果对象
- 为品牌方、调度平台、管理者提供统一项目视图

### 1.2 模块目标

V1 阶段，项目管理模块重点解决以下问题：

- 立项信息零散、确认慢
- 项目创建后缺少统一容器承接任务与结果
- 项目经理人工跟单成本高
- 里程碑和关键节点不可视
- 风险识别依赖人工经验
- 品牌方与执行侧缺少统一进度口径

### 1.3 V1 范围

V1 纳入范围：

- 新建立项与立项确认
- 项目容器创建
- 项目阶段模型
- 里程碑管理
- 项目总览与详情
- 项目风险记录与预警
- 任务、采购、验收、资产、结算结果汇总
- 项目日志与资料总览
- 品牌需求 Agent / 项目经理 Agent 辅助能力

V1 不纳入范围：

- 复杂多项目组合管理
- 多品牌项目经营分析
- 高级成本控制与自动核算
- 完整 PMO 能力中心
- 复杂跨项目资源统筹优化

## 2. 角色与场景

### 2.1 核心角色

#### 品牌方营建负责人

关注立项确认、项目整体进度、关键风险、里程碑达成和交付结果。

#### 品牌项目负责人

关注项目详情、任务推进、风险记录、验收状态、结算进度。

#### 调度平台运营 / 项目经理

关注项目池、阶段分布、关键任务阻塞、派单前置条件、异常升级。

#### 采购 / 验收 / 财务角色

关注项目下的相关结果对象和当前协同状态。

#### 品牌需求 Agent

辅助立项信息补全、模板匹配、项目草案生成。

#### 项目经理 Agent

辅助项目初始化、阶段建议、里程碑生成、风险摘要、催办建议。

### 2.2 核心业务场景

#### 场景 1：品牌方发起开店立项

品牌方输入门店基础信息，系统补全缺失字段，生成立项草案，待品牌确认。

#### 场景 2：立项确认后生成项目容器

项目确认后，系统创建项目，并根据模板生成阶段、里程碑与初始任务树。

#### 场景 3：品牌方查看项目整体进度

品牌方在项目中心查看项目状态、阶段完成度、关键风险、验收进度和结算进度。

#### 场景 4：调度平台在项目池管理执行项目

调度平台按状态、区域、风险、店型筛选项目，识别高风险项目并介入。

#### 场景 5：项目经理处理延期或异常项目

项目经理查看风险原因、阻塞任务、未完成里程碑，并决定催办、升级或转人工。

#### 场景 6：交付前进行项目总核对

项目进入待验收、待结算、待归档阶段时，系统在项目维度统一展示关键结果对象和缺口。

## 4. 任务结构设计

### 4.1 项目与任务树关系

项目管理模块不直接替代任务中心，而是从项目视角承接任务树：

- 一个项目对应一棵主任务树
- 项目可汇总多个阶段任务
- 里程碑与关键任务强关联
- 项目进度来自关键任务、阶段任务和里程碑汇总
- 项目风险来源于关键任务阻塞、延期、验收失败、采购异常等事件

### 4.2 项目容器模型

项目容器至少要承接以下内容：

- 项目基础信息
- 阶段列表
- 里程碑列表
- 关键任务摘要
- 风险摘要
- 采购摘要
- 验收摘要
- 资产摘要
- 结算摘要
- 项目日志
- 附件资料总览

### 4.3 阶段模型设计

阶段是项目管理视角下的上层分组，主要作用：

- 表达项目当前所处阶段
- 汇总阶段内任务完成情况
- 对外输出更容易理解的进度视图
- 作为里程碑和风险识别的管理区间

V1 规则：

- 阶段默认由项目模板或任务模板生成
- 阶段允许人工调整顺序，但必须留痕
- 阶段完成依赖关键任务与里程碑达成
- 阶段本身不作为最小派单对象

### 4.4 里程碑模型设计

里程碑必须满足：

- 有明确业务意义
- 可被验证是否达成
- 可映射到任务或结果对象
- 可用于预警、统计和汇报

里程碑状态建议：

- `未开始`
- `进行中`
- `已达成`
- `已延期`
- `已取消`

里程碑达成来源：

- 关键任务完成
- 验收结果通过
- 采购到货完成
- 资产归档完成
- 人工确认达成

### 4.5 项目进度汇总规则

项目进度不应只按任务数量平均计算，V1 建议采用以下规则：

- 关键任务权重大于普通任务
- 已达成里程碑对项目进度有显著贡献
- 验收与整改状态应影响项目健康度
- 项目总进度与阶段进度分开展示

建议展示维度：

- 总进度
- 阶段进度
- 关键任务完成率
- 里程碑准时率
- 风险数量与等级分布

### 4.6 风险机制设计

风险识别来源：

- 关键任务延期
- 任务长时间阻塞
- 采购异常
- 验收不通过
- 连续整改
- 关键资料缺失
- 预算异常
- 客诉升级
- Agent 风险提示

风险等级建议：

- `低`
- `中`
- `高`
- `严重`

风险处理规则：

- 新风险产生时写入项目风险记录
- 高风险必须进入工作台提醒
- 严重风险必须支持升级和转人工
- 风险关闭必须记录处理动作和结论

## 5. 标准绑定设计

### 5.1 项目层标准上下文

项目管理模块主要承接`标准上下文`，而不是直接替代标准管理模块。

项目层应至少绑定：

- 店型模板
- 适用标准包
- 所属品牌标准上下文
- 区域适用标准上下文
- 项目特殊补充标准

### 5.2 项目与模板关系

项目创建时，需要根据以下维度匹配模板：

- 品牌
- 店型
- 区域 / 城市
- 项目类型
- 服务范围
- 工期要求

模板匹配输出：

- 阶段模型
- 默认里程碑
- 任务树结构
- 标准包
- 默认责任角色
- 风险提示

### 5.3 项目层标准应用规则

- 项目层标准更多用于确定标准上下文
- 具体执行标准与验收标准由任务层绑定
- 项目详情页要展示当前适用的标准包与标准版本
- 项目发生重大范围变更时，应允许重新匹配标准包，但需保留历史记录

### 5.4 快照展示要求

项目模块需要能看到以下快照摘要：

- 当前项目适用标准包版本
- 关键任务标准快照版本数量
- 存在标准覆盖或冲突提示

## 7. 状态机设计

### 7.1 项目主状态

项目状态统一采用：

- `待立项`
- `待确认`
- `待拆解`
- `执行中`
- `待验收`
- `整改中`
- `待结算`
- `已归档`
- `已中止`

### 7.2 主流转路径

`待立项 -> 待确认 -> 待拆解 -> 执行中 -> 待验收 -> 待结算 -> 已归档`

### 7.3 异常流转路径

- `待确认 -> 已中止`
- `待拆解 -> 已中止`
- `执行中 -> 已中止`
- `待验收 -> 整改中`
- `整改中 -> 待验收`
- `待结算 -> 整改中`

### 7.4 守卫条件

#### 待立项 -> 待确认

- 立项基础字段完整或达到最小确认要求
- 草案已生成

#### 待确认 -> 待拆解

- 品牌确认动作完成
- 项目基础信息完整
- 模板匹配成功或人工确认无需模板

#### 待拆解 -> 执行中

- 项目容器已创建
- 阶段与里程碑已生成
- 任务树已生成
- 关键标准上下文已绑定

#### 执行中 -> 待验收

- 关键执行任务已完成
- 验收申请已发起
- 项目无关键阻塞项

#### 待验收 -> 待结算

- 验收结果达到通过口径
- 整改项已关闭

#### 待结算 -> 已归档

- 结算建议完成
- 资产归档完成
- 一店一档资料齐全

### 7.5 联动要求

- 项目进入`待拆解`后触发任务树初始化
- 项目进入`执行中`后启用催办、预警和风险重算
- 项目进入`待验收`后展示验收摘要和问题项
- 项目进入`已归档`后默认冻结关键编辑动作

## 8. 权限与角色控制

### 8.1 品牌方权限

- 可创建本品牌项目草案
- 可确认立项草案
- 可查看本品牌项目列表和详情
- 可查看里程碑、风险、验收、结算结果
- 不可查看其他品牌项目数据

### 8.2 调度平台权限

- 可查看项目池全部项目
- 可查看风险项目
- 可处理项目异常
- 可查看 Agent 生成的项目建议与日志
- 可转人工和升级处理

### 8.3 项目经理权限

- 可维护阶段与里程碑
- 可触发任务树补生成
- 可查看项目风险详情
- 可发起催办、升级、关闭建议

### 8.4 Agent 权限

#### 品牌需求 Agent

- 可读取立项表单与模板信息
- 可输出立项草案与缺失字段建议
- 不可直接确认立项

#### 项目经理 Agent

- 可输出阶段、里程碑、风险摘要建议
- 可触发受控的任务生成建议
- 不可绕过状态机直接关闭项目

## 9. 异常与人工介入

### 9.1 异常场景

以下情况必须支持人工介入：

- 立项信息关键字段长期缺失
- 模板无法匹配非标项目
- 关键里程碑明显延期
- 关键任务长期阻塞
- 高风险项目争议
- 验收连续不通过
- 采购 / 结算异常影响项目主流程
- Agent 建议不可信或失败

### 9.2 人工动作

V1 建议支持以下动作：

- 驳回立项草案
- 手动确认立项
- 手动调整阶段与里程碑
- 手动标记风险
- 手动关闭风险
- 转人工处理
- 升级项目
- 中止项目
- 发起变更流程

### 9.3 异常处理要求

- 所有人工动作必须填写原因
- 对关键状态回退需保留审计日志
- 高风险项目需要更高角色可见
- 中止项目后需保留历史任务和结果对象引用

## 10. 埋点与验收标准

### 10.1 核心埋点建议

#### 立项链路埋点

- 进入开店立项页
- 保存立项草稿
- 调用品牌需求 Agent
- 生成立项草案
- 确认立项
- 驳回立项

#### 项目中心埋点

- 查看项目列表

- 打开项目详情
- 查看风险视图
- 切换多视图

#### 项目详情埋点

- 查看里程碑
- 查看风险记录
- 查看任务摘要
- 发起催办
- 发起升级
- 转人工处理

### 10.2 指标建议

#### 效率指标

- 立项平均耗时
- 立项确认转化率
- 项目创建平均耗时
- 人工催办次数

#### 履约指标

- 里程碑准时率
- 延期项目占比
- 高风险项目占比
- 异常处理时长

#### 自动化指标

- 立项信息自动补全率
- 自动生成项目成功率
- 风险自动识别覆盖率
- Agent 建议采纳率

### 10.3 功能验收标准

V1 至少满足：

- 可创建并确认立项草案
- 可在确认后生成项目容器
- 可查看项目列表与项目详情
- 可查看阶段、里程碑、风险、日志
- 可联动查看任务、验收、结算摘要
- 可识别并展示风险预警项目
- Agent 失败时可人工接管
- 关键状态变化具备审计日志

### 10.4 业务验收标准

- 至少支撑 1 条真实项目从立项到归档的主流程
- 项目状态流转正确
- 里程碑与任务推进口径一致
- 风险记录能支撑真实项目管理动作
- 品牌端与调度平台项目信息口径一致

## 11. 本模块结论

项目管理模块不是简单的项目列表页，而是品牌方营建部门管理建店目标的核心容器模块。

V1 必须坚持以下原则：

- 项目是任务集合容器，不替代任务原子模型
- 阶段、里程碑、风险是项目管理的三大核心结构
- 项目进度来自任务树和结果对象汇总，不靠人工口头更新
- 项目状态必须受状态机约束
- 标准上下文必须在项目创建阶段明确
- Agent 只能做辅助建议，不能替代项目治理职责
