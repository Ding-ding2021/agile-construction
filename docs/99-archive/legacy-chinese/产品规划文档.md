# 一、系统概述

## 1.1 项目定位

连锁门店营建管理系统是一套面向品牌方营建部门的`建店标准化管理系统`。
系统围绕门店建设全过程中的核心职责，统一承接`项目管理、标准管理、采购协同、资产沉淀`四大业务域，并通过 Agent 能力提升标准执行效率和流程推进效率。

系统的核心价值，不只是提供一个项目协同工具，而是将原本依赖营建专业人员经验的能力沉淀为系统能力，让品牌方把更多精力放在品牌经营、门店拓展和增长结果上。

一句话概括：

**把营建专业能力交给系统，把品牌精力留给品牌。**

## 1.2 项目目标

### 1.2.1 业务目标

- 提升品牌方营建部门对建店过程的可控性
- 将营建标准从“依赖个人经验”转为“依赖系统规则”
- 打通项目、采购、验收、资产之间的数据链路
- 降低人工跟单、催办、资料校验和验收整理成本
- 为后续多品牌复制和平台化演进建立基础

### 1.2.2 产品目标

V1 的核心目标不是建设完整平台生态，而是验证以下能力：

1. 系统是否能承接品牌方营建部门的核心管理职责
2. 标准库是否能驱动任务生成、执行指导和验收判断
3. 任务树模型是否能统一支撑项目、流程、服务三种业务形态
4. Agent 是否能在受控流程节点中提升执行效率

### 1.2.3 V1 核心闭环

V1 优先跑通以下最小业务闭环：

`需求接入 → 项目立项 → 任务生成 → 执行推进 → 采购协同 → 验收整改 → 资产归档 → 结算建议`

## 1.3 目标用户

### 1.3.1 核心用户

#### 品牌方营建部门

负责门店建设全过程管理，是系统最核心用户。

核心职责包括：

- 管项目
- 管标准
- 管采购
- 管资产

### 1.3.2 关键协同角色

- 品牌项目负责人
- 采购专员
- 验收/质量人员
- 财务审核人员
- 资源方负责人
- 现场执行人员
- 系统管理员

## 1.4 核心价值主张

### 对品牌方营建部门

- `项目可控`：项目进度、风险、责任和状态透明可追踪
- `标准可复制`：设计、施工、验收标准沉淀为系统模板和规则
- `采购可协同`：采购过程与项目、任务、验收联动
- `资产可沉淀`：门店交付后形成一店一档和资产基础台账
- `能力可复制`：跨城市、跨团队、跨门店复制能力增强

### 对品牌管理层

- 建店节奏更稳定
- 扩店效率更高
- 交付质量更可控
- 营建体系更可复制

## 1.5 V1 范围与边界

### 1.5.1 V1 纳入范围

- 标准化项目管理
- 任务树执行管理
- 标准库与模板库
- 采购基础协同
- 验收整改闭环
- 交付资产归档
- 人员、权限、通知、审计
- 节点型 Agent 辅助能力

### 1.5.2 V1 不纳入范围

- 完整双边平台撮合
- 资源方独立经营系统完整版
- Token 商业化计费体系
- 复杂自动付款和自动审批
- 高级 BI / 地图 / 大屏
- 深度图像视频 AI 质检
- 完整设施运维系统
- 多 Agent 自治协商

## 1.6 核心原则

- `标准先行`：先沉淀标准，再做自动化和智能化
- `任务原子化`：任务是最小执行单元
- `结果对象独立`：任务推进动作，结果对象沉淀成果
- `状态驱动流程`：所有业务流转以状态机为准
- `人工兜底`：关键节点必须支持人工接管
- `全程留痕`：关键动作和 Agent 决策必须可审计

# 二、功能模块规划

## 2.1 规划思路

本系统不按传统“菜单平铺式”规划，而是围绕品牌方营建部门的核心工作方式进行设计。

底层采用`任务树`建模，标准库作为专业能力底座，项目、流程、服务作为任务的不同组织形态，采购、验收、资产、结算等作为结果对象进行沉淀。

## 2.2 核心建模方式

### 2.2.1 任务树模型

任务是系统的最小执行单元，支持父子层级拆解。

- `父任务`：承载阶段目标和责任范围
- `子任务`：承载具体执行动作和结果回传

每个任务都可绑定：

- `执行标准`
- `验收标准`
- `标签`
- `责任人`
- `状态`
- `时间要求`

### 2.2.2 项目 / 流程 / 服务统一视图

#### 项目

围绕某个建店目标组织起来的任务集合与管理容器。

#### 流程

围绕某类业务动作形成的任务链路与状态流转规则。

#### 服务

围绕某类交付结果组织起来的任务包。

三者底层都由任务树和任务关系统一支撑。

### 2.2.3 结果对象模型

系统中的重要业务结果通过独立对象沉淀，包括：

- 采购申请单
- 采购单
- 验收单
- 整改单
- 结算建议单
- 资产台账
- 一店一档
- Agent 执行记录
- 人工干预记录

原则：

**任务负责推进，结果对象负责沉淀。**

## 2.3 标准化底座

标准化底座是系统最核心的能力，不只是标准文档中心，而是任务生成、执行指导、验收判断和整改闭环的驱动层。

包括：

- 标准来源管理
- 标准条款结构化
- 规则项管理
- 执行标准管理
- 验收标准管理
- 模板中心
- 标准版本管理
- 标准快照机制

标准来源包括：

- 国家标准
- 行业标准
- 地方标准
- 企业内部标准
- 品牌营建标准
- 供应商执行标准

## 2.4 核心业务域

### 2.4.1 项目管理

承载门店建设目标、计划、里程碑、风险、资料和任务总览。

### 2.4.2 标准管理

沉淀品牌方营建专业能力，管理结构化标准、规则和模板。

### 2.4.3 采购管理

承接材料、设备、服务采购，与项目和任务联动。

### 2.4.4 资产管理

沉淀门店交付后的资产台账、一店一档和交付资料。

## 2.5 支撑模块

### 工作台

统一查看待办、预警、消息和项目动态。

### 任务中心

统一管理父任务、子任务、任务关系、状态流转和标准绑定。

### 验收整改

统一处理验收申请、检查项、初验、整改和复验。

### 合同结算

承接合同台账、结算建议和付款记录。

### 人员管理

管理组织、角色、账号、资源人员和能力标签。

### 系统设置

管理权限、字典、编码规则、通知、Agent 配置和审计。

### 数据中心

提供基础统计分析，V1 以项目、任务、采购、验收、资产统计为主。

## 2.6 模块关系

系统主链路如下：

`标准管理`
→ 定义模板、规则、标准
→ `项目管理`
→ 生成项目容器
→ `任务中心`
→ 生成并推进任务树
→ `采购管理`
→ 协同物料、设备、服务采购
→ `验收整改`
→ 判断结果并闭环问题
→ `资产管理`
→ 沉淀交付成果和一店一档
→ `合同结算`
→ 形成结算建议
→ `数据中心`
→ 汇总经营与执行结果

# 三、版本迭代规划

## 3.1 迭代目标

V1 迭代目标不是“菜单做全”，而是“闭环跑通”。
整个版本规划围绕以下主线展开：

`标准沉淀 → 任务生成 → 项目执行 → 采购协同 → 验收整改 → 资产沉淀 → 结算建议`

## 3.2 迭代原则

- `标准先行`
- `任务原子化优先`
- `核心业务域优先`
- `闭环优先`
- `试点优先`
- `人工兜底优先`

## 3.3 版本路线图

### V1.0 MVP 试点版

目标：
跑通单品牌、单区域、标准店型的建店标准化闭环。

重点建设：

- 项目管理
- 任务树执行
- 标准库基础能力
- 采购基础协同
- 验收整改
- 资产归档
- 基础 Agent 接入

### V1.5 增强版

目标：
提升可复制性与协同效率。

重点建设：

- 合同结算基础版
- 标准模板增强
- 数据中心增强
- 工单协同
- 采购与资产联动增强

### V2.0 平台版

目标：
从品牌方营建管理系统升级为平台化系统。

重点建设：

- 资源方独立门户
- 双边协同与撮合
- 商业化体系
- 高级分析能力
- 深度智能化能力

# 六、Agent 开发规划

## 6.1 Agent 定位

Agent 在本项目中的角色不是自由自治的“第二套系统”，而是嵌入在业务流程中的`受控节点能力`。

它们的主要作用是：

- 承接标准化动作
- 输出结构化建议
- 推动任务执行
- 识别异常并转人工
- 提升营建部门日常工作效率

## 6.2 建设目标

V1 阶段 Agent 的目标是验证：

1. 是否能基于标准和模板输出稳定结果
2. 是否能在任务节点中辅助执行
3. 是否能在不破坏状态机的前提下提升效率
4. 是否能在异常场景中准确转人工
5. 是否能全程留痕、可审计

## 6.3 规划原则

- `标准驱动`
- `节点驱动`
- `状态约束`
- `人工兜底`
- `全程留痕`
- `可配置可迭代`

## 6.4 V1 业务 Agent

### 品牌需求 Agent

负责：

- 需求补全
- 模板匹配
- 立项草案
- 初始任务建议

### 项目经理 Agent

负责：

- 任务树生成
- 依赖梳理
- 催办与预警
- 项目推进建议

### 调度 Agent

负责：

- 资源筛选
- 派单建议
- 排期建议
- 冲突识别

### 验收质检 Agent

负责：

- 资料完整性校验
- 检查项生成
- 初验判断
- 整改建议

### 结算建议 Agent

负责：

- 汇总执行结果
- 差异识别
- 风险提示
- 结算建议草案

### 客服 / 工单 Agent

负责：

- 查询应答
- 催办承接
- 工单创建
- 工单分发

## 6.5 Agent 与模块关系

- `项目管理`：品牌需求 Agent、项目经理 Agent
- `任务中心`：项目经理 Agent、调度 Agent
- `标准管理`：品牌需求 Agent、验收质检 Agent
- `采购管理`：调度 Agent、结算建议 Agent
- `验收整改`：验收质检 Agent
- `资产管理`：验收质检 Agent
- `合同结算`：结算建议 Agent
- `工作台 / 工单`：客服 Agent

## 6.6 接入方式

### 节点触发

如立项前、任务生成时、验收申请后、结算前。

### 人工触发

如手动调用任务拆解建议、资料校验、资源推荐。

### 事件触发

如延期、冲突、资料缺失、异常差异等。

## 6.7 权限与审计

Agent 必须遵循：

- 最小权限访问
- 仅在授权节点执行
- 不跨品牌访问数据
- 不跳过审批
- 不修改关键结果

所有 Agent 动作必须记录：

- 触发时间
- 输入摘要
- 标准版本
- 输出结果
- 执行动作
- 是否转人工

## 6.8 开发优先级

### 第一优先级

- 品牌需求 Agent
- 项目经理 Agent
- 验收质检 Agent

### 第二优先级

- 调度 Agent
- 客服 / 工单 Agent

### 第三优先级

- 结算建议 Agent

## 6.9 V1 不纳入能力

- 完全自治项目 Agent
- 多 Agent 自主协商
- 自动审批
- 自动付款
- 深度视频图像质检
- 工程师 Agent 产品化能力

# 七、附录

## 7.1 核心术语

### 任务

系统最小执行单元。

### 子任务

父任务的拆解执行项。

### 项目

围绕建店目标组织的任务集合与管理容器。

### 流程

围绕业务动作形成的任务链路。

### 服务

围绕交付结果组织的任务包。

### 执行标准

定义任务“怎么做”。

### 验收标准

定义任务“怎么算通过”。

### 标准库

沉淀结构化标准的能力中心。

### 标准快照

任务运行时固化的标准版本副本。

### 结果对象

采购单、验收单、整改单、资产台账等业务沉淀对象。

### Agent 节点

明确允许 Agent 执行的流程节点。

### 人工兜底

异常和争议场景下由人工接管处理。

## 7.5 试点边界

### 试点假设

- 品牌已有相对稳定流程
- 标准店型具备一定标准化程度
- 部分标准可被结构化
- 资源协同链条相对可控

### 试点边界

- 单品牌优先
- 1-2 个城市优先
- 标准店型优先
- 高频标准域优先
- 高复杂争议项目暂不纳入

## 7.6 风险提示

### 标准结构化风险

建议先聚焦高频标准域，AI 辅助拆解，人工校对。

### 任务树复杂度风险

建议 V1 先控制层级和关系类型。

### 标准与业务脱节风险

标准必须映射规则项并绑定任务与模板。

### Agent 失控风险

必须采用节点触发、状态约束、权限限制、全程审计。

### 试点接受度风险

必须优先解决高频痛点，并用模板和自动生成降低录入负担。

## 7.7 本章结论

全项目必须统一以下口径：

- `任务是最小执行单元`
- `项目 / 流程 / 服务是任务的不同组织形态`
- `标准库是系统专业能力底座`
- `结果对象承接业务沉淀`
- `Agent 是受状态、规则和权限约束的节点能力`
