## 🚨 P0：系统性问题（3 项）

### P0-1 阶段跳跃，底座不牢

| 属性         | 内容                                                                                                                                                                                             |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **描述**     | 开发顺序严重偏离计划：P1.5（底座收官）完成度仅 40%、P2（标准库）被完全跳过就直接进入了 P3 任务中心。导致"任务-标准联动"只绑了个字段 ID，标准库本身不存在。底座没夯实就盖二楼，返工成本持续上升。 |
| **影响范围** | 全系统，所有依赖标准库的功能                                                                                                                                                                     |
| **根因**     | 阶段优先级管理失控，未严格执行"先还债再盖楼"策略                                                                                                                                                 |
| **证据**     | P1.5 完成度 40%（MUI 组件化 5%、CSS Token 治理 20%、渐变 Token 化 35%）；P2 9 个 Issue 全部 OPEN；P3 5 个 Issue 已 CLOSED                                                                        |
| **关联**     | `docs/04-operations/phase1/development-progress-assessment-2026-04-25.md`                                                                                                                        |

---

### P0-2 数据层三路存储不同步

| 属性         | 内容                                                                                                                                                                                                                                                 |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **描述**     | Zustand persist、taskRepository、API 三套存储并行但互不知晓。API 通路（Express + SQLite 实体表）设计完备但从未真正启用——catch 分支无条件降级到 localStorage，数据库初始为空导致 API 永远返回空数据。所有增删改查实际只存在浏览器的 localStorage 里。 |
| **影响范围** | 所有数据操作：换机/清缓存丢失、无法多用户、无法权限隔离、无法审计追溯                                                                                                                                                                                |
| **根因**     | P1-T4 验收标准只要求"ORM 能跑、API 能响应"，未要求前端真正使用 API 作为数据源；后续 catch 降级策略屏蔽了问题暴露                                                                                                                                     |
| **代码证据** | `taskRepository.ts:321-332` API 失败时在 localStorage 上修改并返回"成功"；`taskRepository.ts:176-189` API 返回空后返回 localStorage 数据；`controller/tasks.ts:343-363` 白名单仅 18/58 字段可写                                                      |
| **关联**     | `src/services/repositories/taskRepository.ts`, `src/services/repositories/projectRepository.ts`, `src/services/api/serverAdapter.ts`, `local-api/controllers/tasks.ts`                                                                               |

### P3-2 数据策略方向未定

| 属性         | 内容                                                                                                                                                                                               |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **描述**     | 后端数据模式选择（localStorage + 快照 vs 实体表 RESTful API）影响后续所有数据相关任务，当前无明确决策                                                                                              |
