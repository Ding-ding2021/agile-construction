## 1. 模块概述

### 1.1 模块定位

项目管理模块是品牌方营建部门的核心工作入口，用于承接门店建设目标、立项信息、阶段计划、里程碑、风险、资料和任务总览。

在本系统中，`项目不是最小执行单元`，而是围绕某个建店目标组织起来的`任务集合容器与管理视图`。

项目管理模块的核心作用：

- 承接立项与项目容器创建
- 形成项目阶段与里程碑框架
- 汇总任务树执行进展
- 提供项目风险识别与预警
- 汇总采购、验收、资产、结算等结果对象
- 为品牌方、调度平台、管理者提供统一项目视图

### 1.2 模块目标

V1 阶段，项目管理模块重点解决以下问题：

- 立项信息零散、确认慢
- 项目创建后缺少统一容器承接任务与结果
- 项目经理人工跟单成本高
- 里程碑和关键节点不可视
- 风险识别依赖人工经验
- 品牌方与执行侧缺少统一进度口径

### 1.3 V1 范围

V1 纳入范围：

- 新建立项与立项确认
- 项目容器创建
- 项目阶段模型（待立项 / 待确认 / 待拆解 / 执行中 / 待验收 / 整改中 / 待结算 / 已归档 / 已中止）
- 里程碑管理
- 项目总览与详情
- 项目详情 8 标签结构（PMBOK 领域标签：项目概览 / 范围与任务 / 进度管理 / 成本与采购 / 质量与验收 / 资源与人员 / 风险与沟通 / 设置与 Agent）
- 项目详情内「任务与工作分解（WBS）」合并入口
- 保留全局任务管理入口（`#/tasks`）与跨项目任务能力
- 项目风险记录与预警
- 任务、采购、验收、资产、结算结果汇总
- 项目日志、问题日志与资料总览
- 品牌需求 Agent / 项目经理 Agent 辅助能力

V1 不纳入范围：

- 复杂多项目组合管理
- 多品牌项目经营分析
- 高级成本控制与自动核算
- 完整 PMO 能力中心
- 复杂跨项目资源统筹优化

## 2. 角色与场景

### 2.1 核心角色

#### 品牌方营建负责人

关注立项确认、项目整体进度、关键风险、里程碑达成和交付结果。

#### 品牌项目负责人

关注项目详情、任务推进、风险记录、验收状态、结算进度。

#### 调度平台运营 / 项目经理

关注项目池、阶段分布、关键任务阻塞、派单前置条件、异常升级。

#### 采购 / 验收 / 财务角色

关注项目下的相关结果对象和当前协同状态。

#### 品牌需求 Agent

辅助立项信息补全、模板匹配、项目草案生成。

#### 项目经理 Agent

辅助项目初始化、阶段建议、里程碑生成、风险摘要、催办建议。

### 2.2 核心业务场景

#### 场景 1：品牌方发起开店立项

品牌方输入门店基础信息，系统补全缺失字段，生成立项草案，待品牌确认。

#### 场景 2：立项确认后生成项目容器

项目确认后，系统创建项目，并根据模板生成阶段、里程碑与初始任务树。

#### 场景 3：品牌方查看项目整体进度

品牌方在项目中心查看项目状态、阶段完成度、关键风险、验收进度和结算进度。

#### 场景 4：调度平台在项目池管理执行项目

调度平台按状态、区域、风险、店型筛选项目，识别高风险项目并介入。

#### 场景 5：项目经理处理延期或异常项目

项目经理查看风险原因、阻塞任务、未完成里程碑，并决定催办、升级或转人工。

#### 场景 6：交付前进行项目总核对

项目进入待验收、待结算、待归档阶段时，系统在项目维度统一展示关键结果对象和缺口。

## 4. 任务结构设计

### 4.1 项目与任务树关系

项目管理模块不直接替代任务中心，而是从项目视角承接任务树：

- 一个项目对应一棵主任务树
- 项目可汇总多个阶段任务
- 里程碑与关键任务强关联
- 项目进度来自关键任务、阶段任务和里程碑汇总
- 项目风险来源于关键任务阻塞、延期、验收失败、采购异常等事件
- 项目详情页内合并“任务 + 工作分解（WBS）”入口，减少重复导航
- 全局任务管理页面与 `#/tasks` 路由保留，继续承担跨项目任务管理能力

### 4.2 项目容器模型

项目容器至少要承接以下内容：

- 项目基础信息
- 阶段列表
- 里程碑列表
- 关键任务摘要
- 风险摘要
- 采购摘要
- 验收摘要
- 资产摘要
- 结算摘要
- 项目日志
- 附件资料总览

### 4.3 阶段模型设计

阶段是项目管理视角下的上层分组，主要作用：

- 表达项目当前所处阶段
- 汇总阶段内任务完成情况
- 对外输出更容易理解的进度视图
- 作为里程碑和风险识别的管理区间

V1 规则：

- 阶段默认由项目模板或任务模板生成
- 阶段允许人工调整顺序，但必须留痕
- 阶段完成依赖关键任务与里程碑达成
- 阶段本身不作为最小派单对象

### 4.4 里程碑模型设计

里程碑必须满足：

- 有明确业务意义
- 可被验证是否达成
- 可映射到任务或结果对象
- 可用于预警、统计和汇报

里程碑状态建议：

- `未开始`
- `进行中`
- `已达成`
- `已延期`
- `已取消`

里程碑达成来源：

- 关键任务完成
- 验收结果通过
- 采购到货完成
- 资产归档完成
- 人工确认达成

### 4.5 项目进度汇总规则

项目进度不应只按任务数量平均计算，V1 建议采用以下规则：

- 关键任务权重大于普通任务
- 已达成里程碑对项目进度有显著贡献
- 验收与整改状态应影响项目健康度
- 项目总进度与阶段进度分开展示

建议展示维度：

- 总进度
- 阶段进度
- 关键任务完成率
- 里程碑准时率
- 风险数量与等级分布

### 4.6 风险机制设计

风险识别来源：

- 关键任务延期
- 任务长时间阻塞
- 采购异常
- 验收不通过
- 连续整改
- 关键资料缺失
- 预算异常
- 客诉升级
- Agent 风险提示

风险等级建议：

- `低`
- `中`
- `高`
- `严重`

风险处理规则：

- 新风险产生时写入项目风险记录
- 高风险必须进入工作台提醒
- 严重风险必须支持升级和转人工
- 风险关闭必须记录处理动作和结论

## 5. 标准绑定设计

### 5.1 项目层标准上下文

项目管理模块主要承接`标准上下文`，而不是直接替代标准管理模块。

项目层应至少绑定：

- 店型模板
- 适用标准包
- 所属品牌标准上下文
- 区域适用标准上下文
- 项目特殊补充标准

### 5.2 项目与模板关系

项目创建时，需要根据以下维度匹配模板：

- 品牌
- 店型
- 区域 / 城市
- 项目类型
- 服务范围
- 工期要求

模板匹配输出：

- 阶段模型
- 默认里程碑
- 任务树结构
- 标准包
- 默认责任角色
- 风险提示

### 5.3 项目层标准应用规则

- 项目层标准更多用于确定标准上下文
- 具体执行标准与验收标准由任务层绑定
- 项目详情页要展示当前适用的标准包与标准版本
- 项目发生重大范围变更时，应允许重新匹配标准包，但需保留历史记录

### 5.4 快照展示要求

项目模块需要能看到以下快照摘要：

- 当前项目适用标准包版本
- 关键任务标准快照版本数量
- 存在标准覆盖或冲突提示

## 6. 页面结构与功能点

### 6.1 页面清单

V1 项目管理模块建议包含以下页面：

#### 品牌端

- 开店立项页
- 项目中心页
- 项目详情页

#### 调度平台

- 项目池页
- 项目详情页
- 项目风险视图

### 6.2 开店立项页

#### 页面目标

帮助品牌方快速完成立项输入、模板选择和立项确认。

#### 主要功能

- 新建立项
- 店型模板选择
- 对话式需求录入
- 缺失字段提示
- 立项草案预览
- 品牌确认立项

#### 关键字段

- 门店名称
- 城市
- 商圈
- 店型
- 面积
- 计划开业时间
- 预算区间
- 特殊要求

#### 页面动作

- 保存草稿
- 调用品牌需求 Agent 补全
- 生成立项草案
- 确认立项
- 驳回草案

### 6.3 项目中心 / 项目池页

#### 页面目标

以列表化、多视图方式统一查看项目集合，并支持筛选、分组、风险识别和快速进入详情。

#### 页面能力

- 统计卡片筛选
- 搜索
- 分组
- 高级筛选
- 排序
- 设置面板
- 分页

#### 建议统计卡片

- 全部项目
- 执行中项目
- 待验收项目
- 风险预警项目

#### 建议视图

- 网格视图
- 列表视图
- 看板视图
- 地图视图
- 日历视图

#### 常用筛选维度

- 项目状态
- 城市 / 区域
- 品牌
- 店型
- 风险等级
- 项目负责人
- 开业时间区间

#### 列表关键字段

- 项目名称
- 品牌 / 门店
- 项目状态
- 阶段
- 里程碑进度
- 风险等级
- 当前负责人
- 计划开业时间
- 更新时间

### 6.4 项目详情页

#### 页面目标

在单项目维度形成「总览 + PMBOK 领域管理视图」，以知识领域为组织单元，将同一业务对象聚合在同一标签内，降低跨标签查找成本：

- 用 8 个领域标签覆盖项目管理十大知识领域
- 「项目概览」提供状态总览、快捷操作、风险提示
- 状态机（待立项→已归档）与领域标签正交：状态机回答「项目现在在哪一步」，标签回答「在这个领域我需要管理什么」

#### 页面主结构（8 标签）

1. 项目概览
2. 范围与任务
3. 进度管理
4. 成本与采购
5. 质量与验收
6. 资源与人员
7. 风险与沟通
8. 设置与 Agent

#### 顶部摘要信息（全标签共用）

- 项目名称
- 项目编码
- 品牌 / 门店
- 项目状态（来自状态机：待立项 / 待确认 / 待拆解 / 执行中 / 待验收 / 整改中 / 待结算 / 已归档 / 已中止）
- 风险等级
- 项目负责人
- 计划开业时间
- 总进度
- 三大基准状态（范围 / 进度 / 成本）

#### 项目概览标签（Overview）

- 项目状态总览（状态、进度、里程碑达成、关键阻塞）
- 快捷操作（催办、升级、转人工、发起变更、进入任务中心）
- 风险提示（高/严重风险、临期节点、异常趋势）
- 关键结果对象概览（验收、结算、归档、问题日志）
- 今日待办与关注项
- 变更日志

#### 范围与任务标签（Scope & Tasks）

- 立项信息确认与范围基准（In/Out Scope）
- 任务树（WBS）与任务关系
- 标准绑定与标准快照
- 需求跟踪与范围变更申请

## 7. 状态机设计（单级状态机）

### 7.1 状态模型定义

项目状态采用「单级状态机」模型，与 PMBOK 领域标签正交：

- **状态机**：描述项目的生命周期推进路径，回答「项目现在在哪一步」
- **领域标签**：描述项目管理的知识领域视图，回答「在这个领域我需要管理什么」
- 状态变化不触发标签切换，标签切换也不影响状态

### 7.2 状态集合

项目状态固定为以下 9 个：

- `待立项`
- `待确认`
- `待拆解`
- `执行中`
- `待验收`
- `整改中`
- `待结算`
- `已归档`
- `已中止`

说明：

- `已归档` 和 `已中止` 为终态，进入后默认冻结关键编辑
- 其余状态为中间态，允许按规则向前流转或回退

### 7.3 合法流转路径

```
待立项 → 待确认 → 待拆解 → 执行中 → 待验收 → 整改中 → 待结算 → 已归档
            ↓         ↓          ↓         ↓          ↓          ↓
          已中止    已中止     已中止    已中止     已中止     已中止
```

说明：

- 默认单向前进
- 允许回退，但需具备高级权限并记录审计日志
- 进入「已中止」可从任意中间状态触发，但必须填写原因

### 7.4 守卫条件

#### 待立项 → 待确认

- 项目容器已创建
- 项目基础信息完整

#### 待确认 → 待拆解

- 立项审批完成
- 范围确认完成（In/Out Scope）

#### 待拆解 → 执行中

- 项目容器已创建
- 阶段与里程碑已生成
- 任务树已生成
- 关键标准上下文已绑定

#### 执行中 → 待验收

- 关键任务完成度达到阈值

#### 执行中 → 整改中

- 存在未通过验收项
- 需填写原因

#### 待验收 → 待结算

- 验收结果达到通过口径

#### 待验收 → 整改中

- 验收未通过
- 需填写原因

#### 整改中 → 待验收

- 整改项已关闭或完成豁免审批

#### 待结算 → 已归档

- 结算完成

#### 任意中间状态 → 已中止

- 需填写原因
- 记录审计日志

### 7.5 联动要求

- 项目进入`待拆解`后触发任务树初始化
- 项目进入`执行中`后启用催办、预警和风险重算
- 项目进入`待验收`后展示验收摘要和问题项
- 项目进入`整改中`后汇总未通过任务并跟踪整改进度
- 项目进入`待结算`后触发结算建议流程
- 项目进入终态（`已归档`/`已中止`）后默认冻结关键编辑动作

## 8. 权限与角色控制

### 8.1 品牌方权限

- 可创建本品牌项目草案
- 可确认立项草案
- 可查看本品牌项目列表和详情
- 可查看里程碑、风险、验收、结算结果
- 不可查看其他品牌项目数据

### 8.2 调度平台权限

- 可查看项目池全部项目
- 可查看风险项目
- 可处理项目异常
- 可查看 Agent 生成的项目建议与日志
- 可转人工和升级处理

### 8.3 项目经理权限

- 可维护阶段与里程碑
- 可触发任务树补生成
- 可查看项目风险详情
- 可发起催办、升级、关闭建议

### 8.4 Agent 权限

#### 品牌需求 Agent

- 可读取立项表单与模板信息
- 可输出立项草案与缺失字段建议
- 不可直接确认立项

#### 项目经理 Agent

- 可输出阶段、里程碑、风险摘要建议
- 可触发受控的任务生成建议
- 不可绕过状态机直接关闭项目

## 9. 异常与人工介入

### 9.1 异常场景

以下情况必须支持人工介入：

- 立项信息关键字段长期缺失
- 模板无法匹配非标项目
- 关键里程碑明显延期
- 关键任务长期阻塞
- 高风险项目争议
- 验收连续不通过
- 采购 / 结算异常影响项目主流程
- Agent 建议不可信或失败

### 9.2 人工动作

V1 建议支持以下动作：

- 驳回立项草案
- 手动确认立项
- 手动调整阶段与里程碑
- 手动标记风险
- 手动关闭风险
- 转人工处理
- 升级项目
- 中止项目
- 发起变更流程

### 9.3 异常处理要求

- 所有人工动作必须填写原因
- 对关键状态回退需保留审计日志
- 高风险项目需要更高角色可见
- 中止项目后需保留历史任务和结果对象引用

## 11. 本模块结论

项目管理模块不是简单的项目列表页，而是品牌方营建部门管理建店目标的核心容器模块。

V1 必须坚持以下原则：

- 项目是任务集合容器，不替代任务原子模型
- 阶段、里程碑、风险是项目管理的三大核心结构
- 项目详情采用「项目概览/范围与任务/进度管理/成本与采购/质量与验收/资源与人员/风险与沟通/设置与 Agent」8 标签，按 PMBOK 知识领域组织，同一业务对象不跨标签重复
- 范围与任务标签内合并「任务 + WBS」以降低认知与切换成本
- 全局任务中心保留，项目内与全局形成分层治理
- 项目进度来自任务树和结果对象汇总，不靠人工口头更新
- 项目状态必须受单级状态机约束（待立项→已归档），与领域标签正交
- 待立项与已归档设门禁，保障范围基准与交付闭环
- 标准上下文必须在项目创建阶段明确
- Agent 只能做辅助建议，不能替代项目治理职责

## 12. 十大知识领域映射与实施路线

### 12.1 十大知识领域映射（简版）

- 整合管理：项目概览标签统一承载，状态机 + 日志 + 门禁串联全流程
- 范围管理：范围与任务标签确认范围基准、维护 WBS 与变更
- 进度管理：进度管理标签承载里程碑、阶段计划、关键路径与偏差监控
- 成本管理：成本与采购标签承载预算基准、执行跟踪、采购结果对象
- 质量管理：质量与验收标签承载验收申请、检查项、整改追踪
- 资源管理：资源与人员标签承载成员角色、工队分配、责任边界
- 沟通管理：风险与沟通标签承载项目日志、关键通知、升级留痕
- 风险管理：风险与沟通标签承载风险识别、分级预警、关闭结论
- 采购管理：成本与采购标签承载采购申请、订单、到货跟踪、异常联动
- 相关方管理：风险与沟通标签承载品牌方、项目经理、平台运营、财务等角色权限与看板

### 12.2 分期落地建议

#### MVP（当前阶段）

- 完成8标签信息架构落地（含项目概览）
- 完成范围与任务标签内任务-WBS合并
- 完成待立项/已归档门禁与审计日志
- 保持 `#/tasks` 与全局任务管理能力稳定

#### V2（增强阶段）

- 引入更细粒度成本与绩效分析（如偏差趋势）
- 增强相关方沟通矩阵与治理报表
- 引入更完整的问题知识沉淀与复盘模板
