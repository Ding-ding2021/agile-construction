# MVP 代码质量保证方案 V2（全 AI 编码版）

> **适用阶段**：V1.0 MVP（第 1-6 月）
> **核心约束**：产品经理 1 人 + AI 辅助编码，**不会写代码**
> **目标**：在资源极其有限、全部代码由 AI 生成的情况下，守住代码质量底线
> **核心理念**：代码是 AI 写的，但"对不对"必须能用**非代码的方式**证明

---

## 一、核心思路转变

| 旧方案（不可行） | 新方案（现实）                           |
| ---------------- | ---------------------------------------- |
| 核心模块人工手写 | 核心模块 AI 写，但强制输出"可解释交付物" |
| 人读代码 Review  | 人看测试用例 + 流程图 + 真实场景验证     |
| 单人 AI 生成     | 双人 AI 交叉验证（模型 A 写，模型 B 审） |
| 代码所有权模糊   | 测试用例即文档，流程图即说明书           |

**关键认知**：不会写代码 ≠ 无法保证质量。产品经理对业务的理解，加上结构化的验收手段，完全可以作为质量门禁。

---

## 二、三层防线模型

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：自动化防线（零人力成本，必须 100% 通过）            │
│  TypeScript 严格模式 + ESLint + 构建阻断                     │
├─────────────────────────────────────────────────────────────┤
│  第二层：AI 编码流程控制（低人力成本，规则驱动）             │
│  可解释交付物 + 双人 AI 交叉验证 + Prompt 约束               │
├─────────────────────────────────────────────────────────────┤
│  第三层：产品经理验收（每轮迭代 10 分钟）                    │
│  测试用例审查 + 流程图确认 + 真实场景操作                    │
└─────────────────────────────────────────────────────────────┘
```

---

## 三、第一层：自动化防线

### 3.1 构建阻断机制

每次 AI 生成代码后，必须先通过以下检查才能进入验收：

| 检查项              | 命令               | 失败处理                 | 配置位置            |
| ------------------- | ------------------ | ------------------------ | ------------------- |
| TypeScript 类型检查 | `npx tsc --noEmit` | 让 AI 修正，不通过不验收 | `tsconfig.app.json` |
| ESLint 代码检查     | `npm run lint`     | 让 AI 修正，不通过不验收 | `eslint.config.js`  |
| 构建成功            | `npm run build`    | 让 AI 修正，不通过不验收 | `vite.config.ts`    |

**执行方式**：复制命令 → 粘贴到终端 → 看结果。报错就丢给 AI："修正这些错误"。

### 3.2 ESLint 关键规则（已配置）

`eslint.config.js` 已启用以下规则，AI 必须遵守：

| 规则                                 | 作用                      | AI 违规后果         |
| ------------------------------------ | ------------------------- | ------------------- |
| `@typescript-eslint/no-explicit-any` | 禁止 `any` 类型           | `error` —— 构建失败 |
| `@typescript-eslint/no-unused-vars`  | 禁止未使用变量            | `error` —— 构建失败 |
| `eqeqeq`                             | 强制 `===`                | `error` —— 构建失败 |
| `no-console`                         | 限制 `console.log`        | `warn` —— 提醒清理  |
| `react-hooks/exhaustive-deps`        | 强制 `useEffect` 依赖完整 | `error` —— 构建失败 |

> 这些规则的存在就是为了**自动拦截 AI 最常见的问题**，不需要人逐行看代码。

## 四、第二层：AI 编码流程控制

### 4.1 可解释交付物制度（核心机制）

**所有核心模块**（`src/domain/`、`src/data/`、`src/App.tsx`）由 AI 生成时，必须**同步输出三类可解释交付物**：

| 交付物           | 格式                | 作用                         | 产品经理怎么审                         |
| ---------------- | ------------------- | ---------------------------- | -------------------------------------- |
| **完整单元测试** | TypeScript 测试代码 | 证明代码在所有场景下行为正确 | 看测试用例列表，确认覆盖了所有业务场景 |
| **文本版流程图** | ASCII 或 Mermaid    | 展示状态流转/数据流/决策逻辑 | 对照需求文档，确认跳转关系正确         |
| **边界条件清单** | Markdown 表格       | 列出所有异常输入和预期行为   | 确认空数据、异常状态、越界输入都有处理 |

**铁律**：AI 不输出这三样，产品经理不验收代码。

### 4.2 Prompt 模板：核心模块生成

每次让 AI 生成核心模块时，使用以下 Prompt（可复制）：

```
请生成 [具体功能] 的核心代码，并严格遵守以下交付要求：

【代码要求】
1. 使用 TypeScript，禁止 any 类型
2. 代码放在 src/domain/[文件名].ts
3. 不要修改 src/domain/ 和 src/data/ 下已有的其他文件

【必须同步输出的可解释交付物】
1. 完整单元测试（覆盖所有分支和边界，放在 src/domain/__tests__/[文件名].test.ts）
2. 文本版状态流转图（用 Mermaid 格式，放在文档中）
3. 边界条件清单（表格形式：输入条件 | 预期行为 | 对应测试用例名）

【验证清单】
- 是否所有状态/分支都有测试？
- 是否有异常输入的处理？
- 状态流转图是否与实际代码一致？

请先输出可解释交付物，我确认后再输出代码。
```

### 4.3 Prompt 模板：双人 AI 交叉验证

核心模块在交付前，用**另一个 AI 会话**（或另一个模型）做独立审查：

```
以下代码由另一个 AI 生成，请作为独立审查者进行审查：

【审查要求】
1. 检查逻辑漏洞：是否有遗漏的分支或边界条件？
2. 检查测试完整性：单元测试是否覆盖了所有代码路径？
3. 检查与现有代码的兼容性：是否破坏了已有接口或类型？
4. 输出审查报告：通过/不通过，如有问题列出具体行号和修改建议

【被审查代码】
[粘贴 AI-A 生成的代码和测试]

请只输出审查报告，不要重写代码。
```

**执行方式**：

1. AI-A（当前会话）生成代码和交付物
2. 新开一个 AI 会话（AI-B），粘贴上述 Prompt + 代码
3. AI-B 输出审查报告
4. 将审查报告丢回 AI-A，让其修正
5. 循环直到审查通过

### 4.4 核心模块稳定化机制

核心模块一旦通过验收、真实场景跑通，标记为 `STABLE`：

| 状态   | 标签     | 变更规则                              |
| ------ | -------- | ------------------------------------- |
| 开发中 | `DRAFT`  | AI 自由修改，每次修改需重新输出交付物 |
| 已验收 | `STABLE` | AI 修改前必须输出**变更申请**（见下） |
| 已冻结 | `FROZEN` | 禁止 AI 修改，只允许人工触发变更      |

**变更申请模板**（AI 修改 STABLE 模块前必须输出）：

```markdown
## 变更申请

- **目标文件**：src/domain/xxx.ts
- **变更理由**：为什么需要改？（如：新增状态"暂停"）
- **影响范围**：改了什么？会影响哪些文件？
- **测试更新**：需要新增/修改哪些测试用例？
- **向后兼容**：是否破坏已有接口？如何兼容？
```

产品经理看完变更申请，确认"影响可控"后，才让 AI 执行修改。

## 五、第三层：产品经理验收检查清单

### 5.1 每轮迭代必查 5 项（10 分钟）

不需要看代码，只需确认以下交付物：

| #   | 检查项           | 验收方式                     | 通过标准                                 |
| --- | ---------------- | ---------------------------- | ---------------------------------------- |
| 1   | **测试覆盖**     | 看测试用例列表               | 所有业务场景（正常+异常）都有对应测试    |
| 2   | **流程图正确**   | 看 Mermaid 图                | 状态跳转与需求文档一致，无非法路径       |
| 3   | **边界清单**     | 看边界条件表格               | 空数据、异常状态、越界输入都有处理       |
| 4   | **真实场景跑通** | 在浏览器操作                 | 实际业务场景（如：立项→执行→验收）能走通 |
| 5   | **变更影响**     | 看 AI 输出的影响报告（如有） | 改动范围可控，不波及无关模块             |

### 5.2 红线判定（触发即回滚）

以下情况出现时，立即要求 AI 重新生成，不进入修正流程：

1. **没有可解释交付物** —— 测试/流程图/边界清单缺任一
2. **单元测试覆盖率不足** —— 产品经理通过阅读测试列表，发现明显遗漏场景
3. **流程图与需求矛盾** —— 如：需求说"执行中不能直接回待拆解"，但流程图显示可以
4. **真实场景操作失败** —— 在浏览器按业务流程操作，出现报错或行为异常
5. **AI 未经申请修改了 STABLE 模块** —— 直接回滚，要求重新走变更申请流程

## 六、核心模块清单与关注重点

以下模块属于"核心模块"，必须严格执行可解释交付物制度：

| 文件                                 | 关注重点                 | 验收关键             |
| ------------------------------------ | ------------------------ | -------------------- |
| `src/domain/projectStatusMachine.ts` | 状态流转是否闭环         | 合法/非法跳转矩阵    |
| `src/domain/projectStatusView.ts`    | 视图映射是否与状态机一致 | 每个状态对应视图正确 |
| `src/domain/workItem.ts`             | 工作项模型变更影响       | 下游组件是否兼容     |
| `src/data/projects.ts`               | 项目数据结构向后兼容     | 已有数据不丢失       |
| `src/App.tsx`                        | 路由和全局状态变更       | 现有页面不受影响     |

## 七、测试策略（最小可行）

MVP 阶段不追求高覆盖率，但核心模块必须测：

| 模块                                 | 测试类型 | 最低要求              | 验收方式                                            |
| ------------------------------------ | -------- | --------------------- | --------------------------------------------------- |
| `src/domain/projectStatusMachine.ts` | 单元测试 | 100% 状态流转路径覆盖 | 产品经理阅读测试用例，确认每个合法/非法跳转都有测试 |
| `src/domain/workItem.ts`             | 单元测试 | 所有计算分支覆盖      | 看测试用例列表                                      |
| `src/data/projects.ts`               | 集成测试 | 工厂函数关键路径      | 真实场景操作验证                                    |
| `src/App.tsx`                        | 手工验收 | 主要流程跑通          | 浏览器操作                                          |

## 八、质量指标与监控

### 8.1 每周记录（5 分钟）

| 指标                      | 目标值 | 超限处理                                         |
| ------------------------- | ------ | ------------------------------------------------ |
| AI 生成代码验收拒绝率     | < 30%  | 拒绝率过高则调整 Prompt 约束或更换模型           |
| 构建失败次数              | 0      | 每次构建失败记录根因                             |
| 缺少可解释交付物的提交    | 0      | 发现即要求补全                                   |
| STABLE 模块未经申请被修改 | 0      | 触发红线，回滚并追责（检查 Prompt 是否漏了约束） |

### 8.2 双周架构健康度检查（15 分钟）

快速扫描以下信号：

- [ ] 是否有核心模块从 STABLE 变回 DRAFT？（说明不稳定，需加固）
- [ ] AI 是否新增了 `// eslint-disable` 注释？（AI 在绕过质量门禁）
- [ ] `package.json` 依赖是否无故增加？（AI 乱加依赖）
- [ ] 是否有重复代码片段？（AI 复制粘贴，运行 `npx jscpd src/`）
- [ ] 组件平均行数是否超过 300 行？（AI 容易生成超大组件，需拆分）

## 十、Roadmap 配套更新要点

以下 Roadmap 章节在 V1.2 中需要同步更新：

### 10.1 §5.8「质量门禁」替换为：

```
| **自动化检查** | `tsc --noEmit` 零报错 + `npm run lint` 零报错 + 构建成功 | CI/本地 |
| **功能验收** | 需求文档中的验收标准逐条通过真实场景操作 | 产品经理 |
| **可解释性验证** | 核心模块必须附带：①完整单元测试 ②文本流程图 ③边界条件清单 | AI 生成要求 |
| **双人 AI 验证** | 核心模块由 AI-A 生成后，AI-B（不同模型）独立审查逻辑漏洞 | AI 交叉验证 |
| **测试即审查** | 核心逻辑单元测试覆盖率 100%，产品经理通过阅读测试用例判断场景覆盖度 | 产品经理 |
```

### 10.2 §5.8「知识传递风险」替换为：

```
- **应对**：核心模块由 AI 生成时，必须同步输出三类可解释交付物：
  1. **完整单元测试**：覆盖所有状态/分支/边界，测试即文档
  2. **文本版流程图**：用 Mermaid 或 ASCII 图展示状态流转/数据流
  3. **边界条件清单**：列出所有异常输入和预期行为
  - 产品经理通过"读测试用例 + 看流程图"即可判断逻辑正确性，无需阅读代码
  - 非核心 UI 页面可 AI 生成但需标注生成来源
```

### 10.3 §7.7 风险矩阵「AI 生成代码质量不可控」更新为：

```
| **技术风险** | AI 生成代码质量不可控，核心模块出现隐蔽缺陷 | 高 | 高 | 🔴 严重 | 核心模块 AI 生成时必须同步输出：①完整单元测试 ②状态流转图/流程图 ③边界条件清单；采用双人 AI 交叉验证（模型 A 写，模型 B 审）；所有核心变更必须附带影响分析报告；产品经理通过"测试用例+流程图"验证正确性，无需读代码 | 产品经理/AI |
```

## 十一、总结

MVP 阶段全 AI 编码的质量保证，核心策略就三句话：

1. **自动化守住底线**：TypeScript + ESLint + 构建阻断，零代码能力要求
2. **可解释交付物是质量锚点**：测试用例 + 流程图 + 边界清单，产品经理看得懂、验得了
3. **双人 AI 交叉验证防盲区**：模型 A 写、模型 B 审，降低单模型幻觉风险

> **不要追求完美，但要防止腐烂。** MVP 允许代码粗糙，但不允许核心逻辑失控。产品经理的业务判断力 + 结构化的验收手段 = 足够的质量保障。
