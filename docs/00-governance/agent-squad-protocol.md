# Agent 评审小组协议

## Clause 1. 评审小组总则

### 1.1 [强制] 小组定义

**1.1.1 [强制]** 评审小组由两个独立小组组成：

| 小组   | 英文 ID                    | 时机       |
| ------ | -------------------------- | ---------- |
| 评估组 | `squad-pre-dev-evaluation` | 开发前评估 |
| 验收组 | `squad-post-dev-review`    | 开发后验收 |

**1.1.2 [强制]** 每个评审小组由多个专业 Agent 组成，模拟真实团队的评审过程。

**1.1.3 [强制]** 评估/验收报告必须使用中的小组成员头像，使产品经理清晰分辨不同角色的独立判断。

---

## Clause 2. 评审流程

### 2.1 [强制] 流程总览

```
需求 → 评估组 → 通过 → 开发 → 验收组 → 通过 → 交付
               ↘ 驳回 → 回到澄清阶段           ↘ 驳回 → 回到修改
```

### 2.2 [强制] 触发规则

**2.2.1 [强制]** 评估组在 PM 发起指定 skill 时自动触发。

**2.2.2 [强制]** 验收组在 PM 发起指定 skill 时自动触发。

**2.2.3 [强制]** 评审结论必须由 PM 汇总后报人类确认。

### 2.3 [强制] 组间隔离

**2.3.1 [强制]** 评估组成员和验收组成员必须来自不同的专业角色，避免同一角色既评估又验收。

**2.3.2 [强制]** 每个评审小组至少包含 2 个不同专业的 Agent。

---

## Clause 3. 评估组

### 3.1 [强制] 职责

**3.1.1 [强制]** 评估输入的清晰度、完整性和可执行性。

**3.1.2 [强制]** 识别潜在风险和前置依赖。

**3.1.3 [强制]** 给出 "通过" 或 "驳回" 的明确结论。

### 3.2 [强制] 评估组成员

| 角色 | 席位     | 负责评估                       |
| ---- | -------- | ------------------------------ |
| 产品 | 评审组长 | 需求完整性、商业逻辑、用户价值 |
| 设计 | 评审员   | UI/UX 可行性                   |
| 开发 | 评审员   | 技术可行性、技术风险           |
| 测试 | 评审员   | 可测试性、验收标准清晰度       |

### 3.3 [强制] 评估输出格式

```markdown
## 评估报告

**评估对象**：[需求/功能名称]
**评估结论**：[通过 / 驳回]

### 各角色评估

**产品**：...
**设计**：...
**开发**：...
**测试**：...

### 风险项

1. [风险描述]
2. ...

### 前置条件

- [ ] ...
```

**3.3.1 [强制]** 评估结论必须明确：通过 / 驳回。

**3.3.2 [强制]** 驳回时必须说明具体原因和改进建议。

---

## Clause 4. 验收组

### 4.1 [强制] 职责

**4.1.1 [强制]** 验证交付物是否满足规格说明中的全部要求。

**4.1.2 [强制]** 检查代码质量、测试覆盖、UI 还原度。

**4.1.3 [强制]** 给出 "通过", "有条件通过" 或 "驳回" 的结论。

### 4.2 [强制] 验收组成员

| 角色 | 席位     | 负责验收               |
| ---- | -------- | ---------------------- |
| 产品 | 验收组长 | 功能完整性、需求对齐度 |
| 测试 | 验收员   | 测试覆盖、E2E 通过     |
| 设计 | 验收员   | UI 还原度、视觉一致性  |
| 开发 | 验收员   | 代码质量、架构合理性   |

### 4.3 [强制] 验收输出格式

```markdown
## 验收报告

**验收对象**：[功能名称]
**验收结论**：[通过 / 有条件通过 / 驳回]

### 各角色验收

**产品**：[测试列表验证 / 需求对齐度]
**测试**：[测试结果、覆盖率、漏测]
**设计**：[UI 还原度、Token 一致、暗色/响应式]
**开发**：[构建门禁、代码质量、架构合理]

### 驳回项

1. [驳回原因]
2. ...

### 有条件通过条件

- [ ] ...
```

### 4.4 [强制] 验收标准

**4.4.1 [强制]** 验收结论等级：

| 结论       | 条件                           | 处理方式           |
| ---------- | ------------------------------ | ------------------ |
| 通过       | 所有检查项通过                 | 直接交付           |
| 有条件通过 | 非功能性小问题（如：代码风格） | 修复后自动交付     |
| 驳回       | 功能缺失 / 严重 Bug / 红线触发 | 退回修改后重新验收 |

---

## Clause 5. 执行约定

### 5.1 [强制] 角色加载

**5.1.1 [强制]** PM 必须在发起评审前确保所有评审员角色已加载对应的角色配置文件。

**5.1.2 [强制]** 角色配置文件的查找路径：

- `docs/00-governance/harness/roles/product.md`
- `docs/00-governance/harness/roles/design.md`
- `docs/00-governance/harness/roles/development.md`
- `docs/00-governance/harness/roles/testing.md`

### 5.2 [强制] 评审顺序

**5.2.1 [强制]** 评估组评审顺序：产品 → 设计 → 开发 → 测试。

**5.2.2 [强制]** 验收组验收顺序：产品 → 测试 → 设计 → 开发。

**5.2.3 [强制]** 每个角色独立输出意见，不应受其他角色影响。

### 5.3 [强制] 结论汇总

**5.3.1 [强制]** PM 收集所有评审意见后：

| 汇总结论   | 条件                            |
| ---------- | ------------------------------- |
| 全票通过   | 所有角色通过                    |
| 有条件通过 | 非关键角色有 minor 意见且可协商 |
| 驳回       | 任一角色给出明确驳回            |

**5.3.2 [强制]** PM 给出最终裁定并输入给人类确认。

---

## Clause 6. 红线和降级

### 6.1 [强制] 红线

**6.1.1 [强制]** 以下情况直接触发驳回，不进入讨论流程：

- 任何评审员明确给出驳回意见
- 验收组发现功能缺失
- 评估组发现无法解决的技术风险
- 质量门禁不通过

### 6.2 [强制] 降级处理

**6.2.1 [强制]** L1（低风险、小修小改）可跳过评估组，由 PM 直接委派开发。

**6.2.2 [强制]** L1 任务仍需验收组确认，不可完全跳过评审。

**6.2.3 [强制]** P0 紧急修复可在评审后 24 小时内补文档。
