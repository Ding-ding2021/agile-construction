# Agent Squad 小组协作协议

> **版本**: v1.0
> **最后更新**: 2026-05-07

---

## 1. 概述

本协议定义了三组协作模式：**Pre-dev Squad（评估组）→ 开发交付者 → Post-dev Squad（验收组）**。

核心原则：

- **多视角评估**：开发前从产品/UI/技术多角度评估需求，降低返工
- **全票通过制**：评估/验收需小组全票通过方可进入下一阶段
- **分级调度**：按任务风险等级决定小组投入规模，平衡质量与成本
- **模型分级**：推理任务用强模型，检查任务用经济模型

---

## 2. 三组结构

```
用户需求
    │
    ▼
┌───────────────────────────────────────┐
│ 第 1 步：判断任务风险等级               │
│  ├─ L1（单文件/纯样式/文档/简单fix）   │ → 跳过评估组，组长单人代检 → 开发
│  ├─ L2（新增组件/单页面/小功能）       │ → ↓
│  └─ L3（跨多文件/架构/新数据模型）    │ → ↓
└───────────────────────────────────────┘
                   │ L2/L3
                   ▼
┌───────────────────────────────────────┐
│ Pre-dev Squad（评估组）               │
│  ├─ L2：组长 + 1 名评估员             │
│  └─ L3：组长 + 3 名评估员并行         │
└──────────┬────────────────────────────┘
           │ 全票通过
           ▼
┌───────────────────────────────────────┐
│ 开发交付者                             │
│  ├─ 按计划 + 测试策略实施 TDD 开发     │
│  ├─ verification-before-completion 自检│
│  └─ 日志/记忆/合并/PR 收尾            │
└──────────┬────────────────────────────┘
           │ 标记完成
           ▼
┌───────────────────────────────────────┐
│ Post-dev Squad（验收组，4角色并行）    │
│  ├─ L2：组长 + 1 名验收员             │
│  └─ L3：组长 + 3 名验收员并行         │
└──────────┬────────────────────────────┘
           │ 全票通过
           ▼
        交付完成
```

## 3. 角色职责

### 3.1 Pre-dev Squad 评估组

| 角色             | 模型     | 职责                                                 | 输出物                            |
| ---------------- | -------- | ---------------------------------------------------- | --------------------------------- |
| **组长**         | v4-pro   | 调度评估流程、汇总意见、仲裁分歧、输出设计文档       | 评估报告 + 设计文档（含测试策略） |
| **产品评估员**   | v4-flash | 评估需求合理性、用户场景、功能价值                   | 产品评估报告                      |
| **UI/UX 评估员** | v4-flash | 评估设计规范一致性、交互模式、可访问性               | UI 评估报告                       |
| **技术评估员**   | v4-pro   | 评估技术可行性、影响范围、技术债务、输出测试策略概览 | 技术评估报告（含测试策略）        |

#### 技术评估员的测试策略输出

技术评估员在评估报告中必须包含测试策略概览：

```markdown
### 测试策略概览

- 单元测试：[核心逻辑/工具函数]，预计 X 个用例
- 组件测试：[关键交互场景]，预计 Y 个用例
- **E2E 测试：[核心用户流程 Playwright 场景]，预计 Z 个用例（前端功能必填）**
- 边界条件：[空/极限/异常输入场景列表]
- 验收标准：build/lint/test/e2e 全部通过
```

### 3.2 开发交付者

| 角色           | 模型   | 职责                                                                                   | 输出物                       |
| -------------- | ------ | -------------------------------------------------------------------------------------- | ---------------------------- |
| **开发交付者** | v4-pro | 编码前必须先加载 karpathy-guidelines，按评估组计划和测试策略实施开发，完成后自检并收尾 | 代码 + 测试 + 日志 + 合并/PR |

#### 调用链

```
1. karpathy-guidelines → 加载编码行为准则（Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution）
2. writing-plans → 拆分实现计划
3. executing-plans / subagent-driven-development → 编码实现
4. TDD → 先写测试后写实现
5. 按需调辅助技能（shadcn-*, frontend-*, gitnexus-* 等）
6. verification-before-completion → 自检
7. 写日志到 .workbuddy/memory/YYYY-MM-DD.md
8. 更新 MEMORY.md（如有架构决策）
9. finishing-a-development-branch → 合并/PR
```

### 3.3 Post-dev Squad 验收组

| 角色               | 模型     | 职责                                                         | 输出物           |
| ------------------ | -------- | ------------------------------------------------------------ | ---------------- |
| **组长**           | v4-pro   | 调度验收流程、汇总意见、给出结论                             | 验收报告         |
| **功能验收员**     | v4-flash | 逐条核对需求覆盖、测试边界条件、错误场景                     | 功能验收报告     |
| **代码质量验收员** | v4-pro   | 检查编码规范、架构合规、无死代码、TDD 遵循度                 | 代码质量验收报告 |
| **UI 验收员**      | v4-flash | 代码审查 + checklist：色值/状态覆盖/暗色模式/响应式/可访问性 | UI 验收报告      |

## 4. 任务风险分级（自动触发规则）

**任何开发任务必须先评估风险等级**，L2/L3 自动触发对应规模的评估组，L1 跳过评估组直接开发。

| 等级          | 判断标准                                    | 示例                               | 评估组               | 验收组         |
| ------------- | ------------------------------------------- | ---------------------------------- | -------------------- | -------------- |
| **L1 低风险** | 单文件修改、纯样式、注释/文档、简单 bug fix | 改按钮颜色、修文案、补测试         | 免评估，组长单人代检 | 组长单人       |
| **L2 中风险** | 新增组件、单页面、小功能模块                | 加 StatCard、新增列表页            | 组长 + 1评估员       | 组长 + 1验收员 |
| **L3 高风险** | 跨多文件、架构变更、新增路由、新数据模型    | WBS 树视图、状态机变更、新增 store | 全量4人              | 全量4人        |

## 6. 评估/验收标准

### 6.1 评估结论规则

| 结论                  | 条件                             | 后续动作                   |
| --------------------- | -------------------------------- | -------------------------- |
| **APPROVED**          | 全员通过                         | 进入开发阶段               |
| **CHANGES REQUESTED** | 有角色提出修改建议但非原则性反对 | 修复后重审（仅派问题角色） |
| **BLOCKED**           | 有角色投原则性反对票             | 组长仲裁，升级给用户决策   |

### 6.2 验收结论规则

| 结论                  | 条件                  | 后续动作                     |
| --------------------- | --------------------- | ---------------------------- |
| **APPROVED**          | 全员通过              | 进入交付收尾                 |
| **CHANGES REQUESTED** | 有角色发现可修复问题  | 打回开发交付者修复后增量重审 |
| **BLOCKED**           | 存在架构性/原则性问题 | 组长仲裁，升级给用户决策     |

## 7. 核心铁律

1. **强制评估**：任何编码前必须调用 `squad-pre-dev-evaluation` skill（L1 除外）
2. **强制验收**：开发完成后必须调用 `squad-post-dev-review` skill（L1 除外）
3. **全票通过制**：评估/验收组中任一角色投反对票则流程中止（组长可仲裁）
4. **独立输出**：子代理各自输出独立报告，不得共享状态
5. **先评估后计划**：评估组通过后方可编写实现计划
6. **Karpathy 准则**：所有开发活动遵循 Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution
7. **增量重审**：打回后仅派有问题的角色重审，避免重复调用

```markdown

- 结论: APPROVED / CHANGES REQUESTED / BLOCKED
- 说明: {理由}

### 产品评估

- 需求合理性: {通过/问题}
- 用户场景: {描述}
- 建议: {如有}

### UI/UX 评估

- 设计规范一致: {通过/问题}
- 交互模式: {描述}
- 可访问性: {通过/问题}
- 建议: {如有}

### 技术评估

- 可行性: {通过/问题}
- 影响范围: {描述}
- 技术债务: {如有}
- 测试策略:
  - 单元测试: {描述}，约 {N} 用例
  - 组件测试: {描述}，约 {N} 用例
  - 边界条件: {列表}

- 建议: {如有}
```

- 问题: {如有}

## 9. 与现有流程的关系

| 现有流程              | 本协议对应     | 说明                                               |
| --------------------- | -------------- | -------------------------------------------------- |
| AGENTS.md 工作流      | 三组模式       | Squad 模式是该工作流的具体实现                     |
| Code Review Checklist | 验收组检查项   | 验收组以 checklist 为基础展开                      |
| 任务结束协议          | 开发交付者收尾 | 日志 + 记忆更新由开发交付者完成                    |
| Pre-commit 流水线     | 开发交付者自检 | 自检包含 lint/build/test                           |
| Project Board 状态    | 三组映射       | Backlog→Ready→评估→In Progress→验收→In Review→Done |
