# Harness 工程框架概述

## Clause 1. 定义与定位

**1.1 [强制]** Harness 是项目的工程框架，定义从需求到交付的完整运行范式。

**1.2 [强制]** 所有任务必须通过 Harness 框架流转。

**1.3 [参考]** 框架按以下设计原则构建：

| 原则   | 说明                                             |
| ------ | ------------------------------------------------ |
| 流程化 | 每个任务必须走完七阶段流水线                     |
| 角色化 | 每个阶段由对应角色负责                           |
| 门禁化 | 每个阶段都有质量关卡，不满足条件不能进入下一阶段 |
| 可追溯 | 所有决策和变更都记录在案                         |

---

## Clause 2. 七阶段流水线

**2.1 [强制]** 定义项目的标准交付流程为七阶段流水线：

```text
Align → Plan → Build → Test → Review → Deploy → Evolve
```

**2.2 [参考]** 各阶段详解：

| 条款  | 阶段   | 核心动作                       | 产出物   | 负责角色    |
| ----- | ------ | ------------------------------ | -------- | ----------- |
| 2.2.1 | Align  | 需求澄清、目标对齐、可行性分析 | 对齐文档 | 产品经理    |
| 2.2.2 | Plan   | 任务拆解、方案设计、WBS 制定   | 实施计划 | 产品 + 开发 |
| 2.2.3 | Build  | 编码实现、自测                 | 代码变更 | 开发        |
| 2.2.4 | Test   | 自动化测试 + 人工验证          | 测试报告 | 测试        |
| 2.2.5 | Review | 代码审查 + 验收                | 审查结论 | 开发 + 测试 |
| 2.2.6 | Deploy | 上线部署                       | 部署记录 | 开发        |
| 2.2.7 | Evolve | 复盘、反哺、沉淀               | 改进建议 | 全角色      |

---

## Clause 3. 四角色体系

**3.1 [强制]** 项目设四个核心角色，每个角色有明确的职责、权责边界和使用的技能。

**3.2 [参考]** 角色概览：

| 角色       | 人物 | 职责                                   |
| ---------- | ---- | -------------------------------------- |
| 产品经理   | 林墨 | 人类接口、需求管理、任务拆解、评审组织 |
| UI 设计师  | 苏染 | 交互设计、视觉规范、组件标准           |
| 开发工程师 | 陈锋 | 架构决策、编码实现、代码质量           |
| 测试工程师 | 周严 | 质量策略、自动化测试、问题跟踪         |

详见 [02-roles.md](02-roles.md)。

---

## Clause 4. 技能映射体系

**4.1 [强制]** 每个角色配有一套技能库，按场景自动加载。

**4.2 [参考]** 技能设计原则：

| 原则     | 说明                                        |
| -------- | ------------------------------------------- |
| 场景驱动 | 每个操作场景对应一个 Skill                  |
| 强制触发 | 关键场景必须调用对应技能                    |
| 持续扩充 | 发现新场景时，创建新 Skill 或扩展现有 Skill |
| 统一存储 | 所有技能在 `.agents/skills/` 统一管理       |

详见 [03-skills.md](03-skills.md)。

---

## Clause 5. 三组评审机制

**5.1 [强制]** 项目设三个评审组，按任务等级触发：

| 条款  | 评审组                   | 触发条件         | 通过要求         |
| ----- | ------------------------ | ---------------- | ---------------- |
| 5.1.1 | 评估组（Pre-dev Squad）  | L2/L3 任务开发前 | 全票通过         |
| 5.1.2 | 验收组（Post-dev Squad） | 开发完成后       | 全票通过         |
| 5.1.3 | 人类决策                 | 任何时候         | 可中止/打回/批准 |

详见 [agent-squad-protocol.md](../agent-squad-protocol.md)。

---

## Clause 6. 五级考核体系

**6.1 [参考]** 质量关卡分五级：

| 级别 | 名称     | 执行者     | 说明                                       |
| ---- | -------- | ---------- | ------------------------------------------ |
| L1   | 自动化   | Husky + CI | 提交时自动执行 lint/build/test/coverage    |
| L2   | AI 流程  | AI Agent   | 提交前检查边界条件、测试覆盖、可解释性     |
| L3   | 人类验收 | 人类       | In Review 阶段逐条确认验收标准             |
| L4   | 阶段评审 | 人类 + AI  | 里程碑关闭检查回归测试、文档同步、技术债务 |
| L5   | 自适应   | 系统       | 基于行为指标的自动奖惩调节（L2.5）         |

---

## Clause 7. 框架治理

**7.1 [强制]** Harness 框架变更必须经过以下流程：

| 条款  | 步骤                                                      |
| ----- | --------------------------------------------------------- |
| 7.1.1 | 修改 `docs/00-governance/harness/` 下的对应 Markdown 文档 |
| 7.1.2 | 同步更新 `.harness/registry.yaml`                         |
| 7.1.3 | 更新相关配置文件（角色 YAML、Agent 配置等）               |
| 7.1.4 | 通过评估组评审                                            |
| 7.1.5 | 更新索引文档引用                                          |

**7.2 [参考]** 本框架涉及的配置和文档对应关系见 [document-governance.md §7.3](../document-governance.md#73-强制-harness-%E5%8F%8C%E6%BA%90%E6%89%A7%E8%A1%8C%E8%A7%84%E5%88%99)。
