## 临时组织

角色是持续存在的身份，评审小组/计划评审/验收小组是角色在特定阶段的临时编组。

### 三组评审流程

```
定义 ──▶ [评审小组] ──▶ 规划 ──▶ [计划评审] ──▶ 构建 ──▶ 测试 ──▶ [验收小组] ──▶ 交付 ──▶ 进化
```

---

### 评审小组（定义阶段）

**使命**：在开工前审议需求，决定能否进入规划

| 席位         | 担任者      | 审什么                         |
| ------------ | ----------- | ------------------------------ |
| **组长**     | 林墨 (产品) | 商业价值、优先级、范围边界     |
| **设计评审** | 苏染 (设计) | UI/UX 可行性、交互模式合理性   |
| **技术评审** | 陈锋 (开发) | 技术可行性、影响范围、架构风险 |
| **测试评审** | 周严 (测试) | 可测试性、验收标准清晰度       |

---

### 计划评审（规划阶段）

**使命**：审议实现方案，决定能否进入构建

| 席位         | 担任者      | 审什么                                       |
| ------------ | ----------- | -------------------------------------------- |
| **组长**     | 林墨 (产品) | 计划是否满足 spec、任务优先级是否合理        |
| **设计评审** | 苏染 (设计) | UI/UX 实现方案是否合理、设计交付物是否清晰   |
| **技术评审** | 陈锋 (开发) | 技术方案合理性、依赖顺序、模块拆分、潜在风险 |
| **测试评审** | 周严 (测试) | 测试策略是否充分、E2E 场景是否覆盖关键路径   |

---

### 验收小组（评审阶段）

**使命**：验收交付物，决定能否上线

| 席位         | 担任者      | 验什么                                         |
| ------------ | ----------- | ---------------------------------------------- |
| **组长**     | 林墨 (产品) | 功能与规格一致性、需求覆盖                     |
| **设计验收** | 苏染 (设计) | 设计还原度、Token 一致性、暗色/响应式/可访问性 |
| **代码验收** | 陈锋 (开发) | 编码规范、架构合规、死代码、性能               |
| **测试验收** | 周严 (测试) | 测试覆盖率、边界条件、设计还原度               |

---

### 编组规则

| 规则         | 内容                                       |
| ------------ | ------------------------------------------ |
| 组长固定     | 三组组长都是林墨，负责召集、合并、裁定     |
| 并行 Fan-out | 四人同时工作，不共享状态                   |
| 全票通过制   | 任一席位反对即不通过                       |
| 增量重审     | 验收打回后只让有问题的席位重验，不全量重来 |
| L1 例外      | 低复杂度任务三组各缩减为产品单人代检       |
| 最终决策     | AI 出报告，人类做决策                      |
