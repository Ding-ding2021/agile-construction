# Harness 工作流

## Clause 1. 三大基本单元

**1.1 [参考]** Harness 框架由三大基本单元构成：

| 条款  | 单元         | 说明                                                                          |
| ----- | ------------ | ----------------------------------------------------------------------------- |
| 1.1.1 | 七阶段流水线 | 需求到交付的完整路径（见项目宪法 Clause 3 及本文档 Clause 2）                 |
| 1.1.2 | 四角色体系   | 每个阶段有明确的负责人（见 [02-roles.md](02-roles.md)）                       |
| 1.1.3 | 治理门禁     | 质量关卡和决策评审（见 [project-charter.md](../project-charter.md) Clause 5） |

**1.2 [强制]** 所有任务必须明确所属阶段、负责角色和门禁条件才能流转。

---

## Clause 2. 三组评审

**2.1 [强制]** 项目设三个评审组，非 P0/P1 在特定场景下激活：

| 条款  | 评审组                   | 成员                      | 职责                             |
| ----- | ------------------------ | ------------------------- | -------------------------------- |
| 2.1.1 | 评估组（Pre-dev Squad）  | 产品 + 设计 + 开发 + 测试 | 开发前检查需求完整性、方案可行性 |
| 2.1.2 | 验收组（Post-dev Squad） | 产品 + 设计 + 开发 + 测试 | 开发后全面验收                   |
| 2.1.3 | 人类决策                 | 人类                      | 仲裁、中止、批准                 |

**2.2 [强制]** 评估组和验收组必须全票通过方可进入下一阶段。

**2.3 [强制]** 人类决策权高于任何自动流程。

---

## Clause 3. 工作流类型

### 3.1 标准全流程

**3.1.1 [适用条件]** L2/L3 任务，需要完整评估和验收。

**3.1.2 [流程]**

```text
需求输入 → 林墨澄清 → 评估组评审 → 陈锋开发 → 验收组验收 → 林墨交付 → 人类确认
```

| 步骤    | 操作者      | 产出           | 门禁                 |
| ------- | ----------- | -------------- | -------------------- |
| 1. 澄清 | 林墨        | 对齐的需求描述 | 人类确认需求无误     |
| 2. 评估 | 评估组      | 评估报告       | 全票通过             |
| 3. 规划 | 林墨 + 陈锋 | 实施计划       | Plan 文档            |
| 4. 开发 | 陈锋        | 代码 + 测试    | lint/build/test 全绿 |
| 5. 验收 | 验收组      | 验收报告       | 全票通过             |
| 6. 交付 | 林墨        | 交付物         | 人类确认             |
| 7. 归档 | 林墨        | 归档记录       | 反哺 knowledge/      |

### 3.2 L1 轻量流程

**3.2.1 [适用条件]** L1 任务，跳过评估组和验收组。

**3.2.2 [流程]**

```text
需求输入 → 林墨澄清 → 陈锋开发 → 林墨交付 → 人类确认
```

| 步骤    | 操作者 | 产出        | 门禁                 |
| ------- | ------ | ----------- | -------------------- |
| 1. 澄清 | 林墨   | 需求确认    | /                    |
| 2. 开发 | 陈锋   | 代码 + 测试 | lint/build/test 全绿 |
| 3. 交付 | 林墨   | 交付物      | 人类确认             |

### 3.3 快速修复流程

**3.3.1 [适用条件]** P0 紧急修复（线上 Bug、安全漏洞），跳过全部评审，修复后补流程。

**3.3.2 [强制]** 快速修复后必须在 24 小时内补充评估记录和验收记录。

**3.3.3 [流程]**

```text
紧急需求 → 林墨确认 → 陈锋直接修复 → 修复后补评估 + 验收
```

| 步骤    | 操作者          | 产出          | 门禁          |
| ------- | --------------- | ------------- | ------------- |
| 1. 确认 | 林墨            | 紧急确认      | /             |
| 2. 修复 | 陈锋            | 热修复代码    | 紧急绕过      |
| 3. 补录 | 评估组 + 验收组 | 事后评估/验收 | 24 小时内完成 |

## Clause 4. 流程触发规则

### 4.1 指令触发

**4.1.1 [强制]** 由林墨发出的流程指令触发指定工作流：

| 条款  | 指令       | 触发流程   | 适用场景    |
| ----- | ---------- | ---------- | ----------- |
| 4.1.1 | 标准全流程 | Clause 3.1 | L2/L3 任务  |
| 4.1.2 | L1 轻量    | Clause 3.2 | L1 任务     |
| 4.1.3 | 快速修复   | Clause 3.3 | P0 紧急修复 |

**4.1.2 [强制]** 林墨在任务开始时必须明确触发哪种流程类型。

### 4.2 事件触发

**4.2.1 [强制]** 以下事件自动触发对应流程：

| 条款  | 事件                            | 触发                          |
| ----- | ------------------------------- | ----------------------------- |
| 4.2.1 | 人类提出新需求                  | 林墨启动澄清                  |
| 4.2.2 | 需求澄清完毕                    | 林墨分级 → 触发对应工作流类型 |
| 4.2.3 | 开发完成 → lint/build/test 通过 | 林墨启动交付                  |
| 4.2.4 | 人类验收不通过                  | 林墨记录问题 → 退回开发       |
| 4.2.5 | 文档变更                        | 调用 `document-sync` 技能     |
| 4.2.6 | 任务完成                        | 林墨归档                      |

---

## Clause 5. 流程规则

**5.1 [强制]** 同一时间只有一个 In Progress。

**5.2 [强制]** L2/L3 任务必须先走评估组，通过后才能进入开发。

**5.3 [强制]** 质量门禁不满足时，禁止进入下一阶段。

**5.4 [强制]** 人类决策权高于任何自动流程。

**5.5 [强制]** P0 快速修复后 24 小时内必须补充评估记录和验收记录。
