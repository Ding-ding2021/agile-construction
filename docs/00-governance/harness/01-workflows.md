# 工作流 — 七阶段线性流水线

## 核心原则

**无分支。所有任务走同一条管道，只有执行深度不同。**

```
定义 ────▶ 规划 ────▶ 构建 ────▶ 测试 ────▶ 评审 ────▶ 交付 ────▶ 进化
DEFINE     PLAN       BUILD      TEST       REVIEW     SHIP       EVOLVE
```

---

## 阶段全景

```
┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
```

## 阶段 1：定义（DEFINE）

**输入**：原始需求
**目标**：将模糊需求转化为可执行的规格文档

| 步骤         | 内容                     | 角色     | 技能/工具                     |
| ------------ | ------------------------ | -------- | ----------------------------- |
| 1a. 需求澄清 | 理解意图、约束、成功标准 | 林墨     | `brainstorming`               |
| 1b. 影响评估 | 评估改动范围、风险等级   | 林墨     | 风险定级标准                  |
| 1c. 小组评审 | 四专业并行审议           | 评审小组 | `squad-pre-dev-evaluation`    |
| 1d. 人类决策 | 🧑 审批评审报告          | 人类     | —                             |
| 1e. 规格定稿 | 输出设计文档             | 林墨     | 写入 `docs/01-product/specs/` |

**出口条件**：设计文档已写入 + 评审小组通过 + 人类已签署

---

## 阶段 2：规划（PLAN）

**输入**：已签署的规格文档
**目标**：将规格分解为可执行的实现计划

| 步骤         | 内容                         | 角色     | 技能/工具                          |
| ------------ | ---------------------------- | -------- | ---------------------------------- |
| 2a. 编码准则 | 加载编码行为准则             | 陈锋     | `karpathy-guidelines`              |
| 2b. 任务分解 | 拆分任务、识别依赖、排优先级 | 陈锋     | `planning-and-task-breakdown`      |
| 2c. 计划评审 | 四专业审议方案               | 计划评审 | 四专业并行                         |
| 2d. 人类决策 | 🧑 审批评审报告              | 人类     | —                                  |
| 2e. 计划定稿 | 输出实现计划                 | 陈锋     | 写入 `docs/02-architecture/plans/` |

**出口条件**：实现计划已写入 + 计划评审通过 + 人类已签署

---

## 阶段 3：构建（BUILD）

**输入**：已审批的实现计划
**目标**：按计划编码实现

| 步骤         | 内容                     | 角色      | 技能/工具                        |
| ------------ | ------------------------ | --------- | -------------------------------- |
| 3a. 增量实现 | 逐任务编码               | 陈锋      | `incremental-implementation`     |
| 3b. TDD 开发 | 测试先行                 | 陈锋      | `test-driven-development`        |
| 3c. 领域技能 | 按需加载 UI/API/架构技能 | 陈锋+苏染 | `frontend-ui-engineering` 等     |
| 3d. 自检     | 代码完成后的自我验证     | 陈锋      | `verification-before-completion` |

**出口条件**：代码实现完成 + 自检通过 + 无遗留 TODO

---

## 阶段 4：测试（TEST）

**输入**：已完成的代码
**目标**：自动化质量门禁

| 步骤          | 内容                   | 触发      | 工具               |
| ------------- | ---------------------- | --------- | ------------------ |
| 4a. Lint 检查 | 代码规范检查           | 开发/自动 | `npm run lint`     |
| 4b. 构建检查  | TypeScript + Vite 构建 | 开发/自动 | `npm run build`    |
| 4c. 单元测试  | Vitest 单元/组件       | 开发      | `npm run test:run` |
| 4d. E2E 测试  | Playwright 端到端      | 自动/测试 | `npm run test:e2e` |

**出口条件**：lint 通过 + build 通过 + 测试全绿

---

## 阶段 5：评审（REVIEW）

**输入**：通过自动化测试的代码
**目标**：人工多维度验收

| 步骤         | 内容                         | 角色 | 技能/工具                 |
| ------------ | ---------------------------- | ---- | ------------------------- |
| 5a. 功能验收 | 需求覆盖、边界条件、错误场景 | 林墨 | `code-review-and-quality` |
| 5b. 代码验收 | 编码规范、架构合规、死代码   | 陈锋 | —                         |
| 5c. UI 验收  | 色值、状态、暗色、响应式     | 苏染 | `ui-layout-rules`         |
| 5d. 测试验收 | 覆盖率、边界、设计还原度     | 周严 | —                         |
| 5e. 组长合并 | 汇总报告、裁定结论           | 林墨 | —                         |
| 5f. 人类决策 | 🧑 审批验收报告              | 人类 | —                         |
| 5g. 如有缺陷 | 打回陈锋修复 → 增量重审      | 陈锋 | —                         |

**出口条件**：验收小组全票通过 + 人类已签署

---

## 阶段 6：交付（SHIP）

**输入**：验收通过的代码
**目标**：记录、提交、上线

| 步骤         | 内容                 | 角色 | 工具                       |
| ------------ | -------------------- | ---- | -------------------------- |
| 6a. 写日志   | 任务记录写入         | 林墨 | `task-memory`              |
| 6b. 写统计   | 质量数据写入         | 林墨 | `.workbuddy/stats/`        |
| 6c. 提交代码 | Conventional Commits | 林墨 | git commit                 |
| 6d. 推送 PR  | 创建 PR / 直接合入   | 林墨 | git push / gh pr           |
| 6e. CI 门禁  | Github Actions 检查  | 自动 | `.github/workflows/ci.yml` |

**出口条件**：日志已写入 + 代码已提交 + CI 绿灯

---

## 阶段 7：进化（EVOLVE）

**输入**：已交付的任务经验
**目标**：把每次交付的经验反哺回框架本身，形成自进化闭环

### 四层学习引擎

| 层  | 名称     | 触发       | 内容                         | 产出                         |
| --- | -------- | ---------- | ---------------------------- | ---------------------------- |
| L1  | 复盘度量 | 每任务     | 记录 KPI、偏差分析           | `.workbuddy/stats/`          |
| L2  | 模式沉淀 | 积累到阈值 | 提取成功模式、常见错误       | `MEMORY.md` patterns         |
| L3  | 知识更新 | 按需       | 更新知识库、修正文档         | `docs/knowledge-base/` 更新  |
| L4  | 框架迭代 | 结构性变化 | bump 版本、更新 harness 文档 | `manifest.yaml` version bump |

**出口条件**：模式已记录 + 知识库有更新 + 框架版本如需要已 bump

---

## 执行深度调节

所有任务走同一管道，但每阶段执行深度因任务复杂度而异：

| 阶段 | L1（简单任务） | L2（中等任务）     | L3（复杂任务）          |
| ---- | -------------- | ------------------ | ----------------------- |
| 定义 | 林墨直接定稿   | 林墨 + 苏染 / 周严 | 全量 4 人评估组         |
| 规划 | 轻量计划       | 完整任务分解       | 完整计划 + 架构评审     |
| 构建 | 单文件修改     | 增量实现 + TDD     | 多文件 + TDD + 多重技能 |
| 测试 | lint + build   | + 单元/组件测试    | + E2E 测试              |
| 评审 | 林墨单人验收   | 林墨 + 苏染 / 周严 | 全量 4 人验收组         |
| 交付 | 日志 + 提交    | 日志 + 统计 + 提交 | 日志 + 统计 + PR + CI   |
| 进化 | 基础日志       | + 模式提取         | + 知识库更新 + 框架迭代 |

## 人类决策链路

```
定义 ──▶ [评审小组] ──▶ 🧑 人类 ──▶ 规划 ──▶ [计划评审] ──▶ 🧑 人类 ──▶ 构建
                                                                          │
                                                                          ▼
进化 ◀── 交付 ◀── 🧑 人类 ◀── [验收小组] ◀── 评审 ◀── 测试 ◀──────────────┘
```

三次人类决策：

1. **🧑1** — 这个需求做不做
2. **🧑2** — 这个方案对不对
3. **🧑3** — 这个交付能不能上线
