# Project Charter

## Clause 1. 概述

**1.1 [强制]** project-charter.md 是项目的治理宪法，定义项目运行的根本原则和整体框架。

**1.2 [强制]** 所有团队活动（人类 + AI）必须遵循本文件。

**1.3 [参考]** 本文件的定位与变更规则如下：

| 定位     | 说明                                                                                             |
| -------- | ------------------------------------------------------------------------------------------------ |
| 治理宪法 | 六大原则统领全部行为，不可违背                                                                   |
| 执行细则 | 由 Harness 工程框架（`docs/00-governance/harness/`）定义具体的七阶段流水线、四角色体系和技能映射 |
| 变更规则 | 本文件本身受治理——修改需经过评审小组审批                                                         |

---

## Clause 2. 六大原则

### 2.1 规范驱动

**2.1.1 [强制]** 所有行为必须对应一条成文规范，没有规范就不执行。

**2.1.2 [参考]** 规范体系由以下文档构成：

| 规范体系         | 指向文档                      | 核心内容                                                   |
| ---------------- | ----------------------------- | ---------------------------------------------------------- |
| Harness 工程框架 | `docs/00-governance/harness/` | 七阶段流水线、四角色体系、技能映射、三组评审机制、五级考核 |
| 设计规范         | `docs/02-design/`             | 视觉系统、组件标准、交互范式、Design Token                 |
| 代码规范         | `coding-standards.md`         | 铁律 8 条、TypeScript strict、双栈编码、Git 提交           |
| 测试规范         | `testing-standards.md`        | TDD 铁律、测试金字塔、Mock 策略、覆盖率门禁                |
| 文档治理         | `document-governance.md`      | 三层架构、状态模型、Frontmatter 模板、编号规则、领域分类   |

**2.1.3 [强制]** 红线：规范未定义的操作 → 先补充规范，再执行。

---

### 2.2 测试驱动

**2.2.1 [强制]** 先写测试，后写代码。测试失败是因为功能缺失，不是代码错误。

**2.2.2 [强制]** TDD 流程：RED（写失败测试）→ GREEN（最简通过）→ REFACTOR（保持全绿）。

**2.2.3 [强制]** 测试金字塔比例：

| 层级     | 占比  |
| -------- | ----- |
| 单元测试 | ≥ 70% |
| 组件测试 | ≥ 15% |
| 集成测试 | ≥ 10% |
| E2E 测试 | ≤ 5%  |

**2.2.4 [强制]** CI 阻断规则：

- 测试失败 = 阻断合并
- 覆盖率低于阈值 = 阻断合并

**2.2.5 [参考]** 覆盖率目标（2026-Q2）：

| 维度      | 目标  |
| --------- | ----- |
| lines     | ≥ 70% |
| functions | ≥ 60% |
| branches  | ≥ 55% |

**2.2.6 [推荐]** 核心模块必须附带：完整单元测试 + 流程图 + 边界条件清单。

详见 [testing-standards.md](../04-testing/testing-standards.md)。

### 2.3 角色驱动

**2.3.1 [强制]** 每个任务由特定角色承担，角色决定职责边界和验收标准。

**2.3.2 [参考]** 四角色体系：

| 角色       | 人物 | 职责                                           | 配置                                |
| ---------- | ---- | ---------------------------------------------- | ----------------------------------- |
| 产品经理   | 林墨 | 人类接口：接需求、沟通、拆解、排程、归档、进化 | `.trae/agents/product-manager.yaml` |
| UI 设计师  | 苏染 | UI/UX 设计：交互规范、视觉一致性、可访问性     | `.trae/agents/designer.yaml`        |
| 开发工程师 | 陈锋 | 软件开发：编码、架构、安全、性能、自检         | `.trae/agents/developer.yaml`       |
| 测试工程师 | 周严 | 质量保证：测试策略、E2E 验证、调试定位         | `.trae/agents/tester.yaml`          |

**2.3.3 [强制]** Squad 协作协议：

| 条款    | 内容                                                                    |
| ------- | ----------------------------------------------------------------------- |
| 2.3.3.1 | 评估组（Pre-dev Squad）：L2/L3 任务开发前，从产品/UI/技术多角度评估需求 |
| 2.3.3.2 | 开发交付者：按 TDD + Karpathy 准则实施开发                              |
| 2.3.3.3 | 验收组（Post-dev Squad）：开发完成后，从功能/代码/UI 多角度验收         |
| 2.3.3.4 | 全票通过制：评估/验收需小组全票通过方可进入下一阶段                     |
| 2.3.3.5 | 人类最终决策：三组评审给出结论，批准权永远在人类手中                    |

**2.3.4 [参考]** 自适应治理（L2.5 行为调节层）：

基于行为指标的自动奖惩调节机制，在指标采集和 Agent 行为之间建立反馈闭环。触发源为 `memory/` 中自检块的"避坑"统计数据。

| 维度 | 触发条件                           | 调节动作                | 级别  |
| ---- | ---------------------------------- | ----------------------- | ----- |
| 奖励 | 零返工、全票通过、自检合规         | 正向肯定                | L1    |
| 奖励 | 主动反哺 knowledge/ 或 patterns.md | 降低后续审批深度        | L2    |
| 惩罚 | 跳过强制技能                       | 强制加载                | L2    |
| 惩罚 | 英文推理、不问需求直接干           | 注入矫正提示            | L1    |
| 惩罚 | 同模式错误 ≥ 3 次（连续会话）      | 升级审批                | L2-L3 |
| 惩罚 | 重复红线违规                       | 冻结自动执行            | L3-L4 |
| 惩罚 | 核心模块未经申请擅自修改           | 冻结自动执行 + 人类审批 | L4    |
| 惩罚 | 覆盖率持续低于阈值                 | 注入补救指令            | L1    |

**2.3.5 [强制]** 调节安全规则：

| 条款    | 内容                                  |
| ------- | ------------------------------------- |
| 2.3.5.1 | L1-L2 自动执行，L3-L4 先审后行        |
| 2.3.5.2 | 所有调节 7 天过期，逾期自动移除       |
| 2.3.5.3 | 人类可用 `status=overridden` 手动覆盖 |
| 2.3.5.4 | 永不惩罚人类                          |

### 2.4 技能驱动

**2.4.1 [强制]** 不靠记忆靠技能——每个操作场景对应一个 Skill，Agent 按场景加载。

**2.4.2 [参考]** 技能体系概况：

| 条目         | 内容                                                                                                           |
| ------------ | -------------------------------------------------------------------------------------------------------------- |
| 技能总数     | 按四角色分配（产品 / 设计 / 开发 / 测试）                                                                      |
| 核心强制技能 | `karpathy-guidelines`（编码前）、`document-sync`（文档变更后）、`verification-before-completion`（声称完成前） |

**2.4.3 [强制]** 每次收到新用户意图，先查 [03-skills.md](harness/03-skills.md) 映射表匹配技能。

**2.4.4 [强制]** 15-call 自检强制检查核心技能是否加载。

**2.4.5 [强制]** 跳过 skill 直接行动 = 违规。

---

### 2.5 需求澄清

**2.5.1 [强制]** 收到需求后先澄清意图，再做规划，最后执行。禁止不问清楚直接动手。

**2.5.2 [强制]** 风险分级：

| 等级      | 判断标准                     | 流程                        |
| --------- | ---------------------------- | --------------------------- |
| L1 低风险 | 单文件、纯样式、简单 fix     | 免评估，直接开发            |
| L2 中风险 | 新增组件、单页面、小功能     | 精简评估（组长 + 1 评估员） |
| L3 高风险 | 跨多文件、架构变更、新增路由 | 全量评估（4 人）            |

**2.5.3 [强制]** 澄清流程：

```text
收到需求 → 判断风险等级 → L1: 直接进入规划
                          → L2/L3: 触发评估组 → 全票通过 → 进入规划
                          → 需求不清: 退回澄清 → 确认后再分级
```

**2.5.4 [强制]** 红线：未评估风险等级就编码 = 违规。

详见 [agent-squad-protocol.md](agent-squad-protocol.md)。

---

### 2.6 自我进化

**2.6.1 [强制]** 系统必须自运行——每次交付反哺规范，每次错误沉淀为规则。

**2.6.2 [参考]** 四阶段进化机制：

| 阶段           | 机制                                                            | 产出                        |
| -------------- | --------------------------------------------------------------- | --------------------------- |
| 观察（会话级） | 每 15 条用户消息执行一次自检                                    | 写入 `memory/YYYY-MM-DD.md` |
| 提炼（跨会话） | 同模式 ≥ 3 次 → 写入 patterns.md；同错误 ≥ 2 次 → 写入 rules.md | 知识积累                    |
| 精炼（定时）   | 全量 Lint → 标记过期文档 → 修复断裂引用                         | 文档健康度                  |
| 归档（每月）   | 日志聚类 → 更新 USER.md 避坑清单                                | 用户画像进化                |

详见 [document-governance.md](document-governance.md)、[knowledge/](../ai/knowledge/)。

## Clause 3. 工作流

### 3.1 任务分级与 Squad 触发

**3.1.1 [强制]** 任务必须先确认风险等级。L1 走快通道，L2/L3 自动触发对应规模的评估组。

| 步骤     | 操作者           | 条件          | 动作                   |
| -------- | ---------------- | ------------- | ---------------------- |
| 分级     | 产品经理（林墨） | 收到需求后    | 判断 L1/L2/L3          |
| 评估     | 评估组           | L2/L3         | 全票通过 → 进入开发    |
| 开发     | 开发交付者       | 评估通过 / L1 | 按 TDD + Karpathy 实施 |
| 验收     | 验收组           | L2/L3         | 全票通过 → 交付        |
| 人类决策 | 人类             | 任何时候      | 可中止/打回/批准       |

### 3.2 阶段流转

**3.2.1 [强制]** 任务生命周期状态流转：

```text
Backlog → Ready → [评估] → In Progress → [验收] → Done
                            ↓ (失败)              ↑
                       Reopened → In Progress ────┘
```

| 阶段         | 操作者 | 条件                   | 动作                         |
| ------------ | ------ | ---------------------- | ---------------------------- |
| Backlog      | 人类   | 新任务                 | 添加标签、设定优先级         |
| Ready        | 人类   | 依赖已解决、优先级已定 | 移入 Ready 列                |
| In Progress  | AI     | 人类指定任务           | 开始编码（L2/L3 先走评估）   |
| AI Completed | AI     | build/lint/test 通过   | 提交、移入 In Review         |
| In Review    | 人类   | AI 标记完成            | 验收，通过则 Close，否则回退 |
| Done         | 人类   | 验收通过               | Close                        |
| Reopened     | 人类   | 验收不通过             | 说明问题，移回 In Progress   |

### 3.3 核心原则

**3.3.1 [强制]** 同一时间只有一个 In Progress：单人 + AI 模式下，不允许并行任务。

**3.3.2 [强制]** 质量门禁：In Review → Done 必须经过人类验收。

**3.3.3 [推荐]** 每日同步：每次会话开始，查看当前状态。

## Clause 4. Label 体系

### 4.1 Type（任务类型）

| 条款  | Label           | 用途            |
| ----- | --------------- | --------------- |
| 4.1.1 | `type:feature`  | 新功能开发      |
| 4.1.2 | `type:bug`      | 缺陷修复        |
| 4.1.3 | `type:refactor` | 重构 / 技术债务 |
| 4.1.4 | `type:docs`     | 文档编写/更新   |
| 4.1.5 | `type:test`     | 测试编写        |
| 4.1.6 | `type:infra`    | CI/构建/工具链  |
| 4.1.7 | `type:release`  | 发布管理        |

### 4.2 Priority（优先级）

| 条款  | Label         | 含义                    |
| ----- | ------------- | ----------------------- |
| 4.2.1 | `priority:P0` | 阻塞/紧急，必须立即处理 |
| 4.2.2 | `priority:P1` | 高优先级，当前迭代内    |
| 4.2.3 | `priority:P2` | 中优先级，下次迭代      |
| 4.2.4 | `priority:P3` | 低优先级，排期外        |

### 4.3 Status（辅助状态）

| 条款  | Label              | 含义               |
| ----- | ------------------ | ------------------ |
| 4.3.1 | `status:blocked`   | 被阻塞，需外部条件 |
| 4.3.2 | `status:in-review` | 待验收 / Review    |

## Clause 5. 质量关卡

**5.1 [强制]** 四级质量关卡体系：

| 关卡            | 时机       | 执行者     | 检查项                           |
| --------------- | ---------- | ---------- | -------------------------------- |
| L1 Automation   | 每次提交   | Husky + CI | lint → build → test → coverage   |
| L2 AI Process   | 提交前     | AI         | 边界条件、测试覆盖、可解释交付物 |
| L3 Human Accept | In Review  | 人类       | 验收标准逐条确认                 |
| L4 Phase Review | 里程碑关闭 | 人类 + AI  | 回归测试、文档同步、技术债务     |

## Clause 6. 通信语言规范

**6.1 [强制]** 所有推理过程、分析说明、与用户的正式回答必须使用中文。

**6.2 [强制]** Agent 委托子任务的 `task` 工具 prompt 必须使用英文（系统限制）。

**6.3 [参考]** 代码、文件路径、技术术语本身不受此限制。

**6.4 [参考]** 本条款取代原 `reasoning-language.md`（GOV-009），原文件已归档。
