# 人机协作双轨策略

## Clause 1. 背景与问题

### 1.1 [参考] 核心矛盾

AI 开发面临两个核心矛盾：

**1.1.1 [参考]** **长会话与冷启动**：单次会话中 AI 能记住上下文，但新会话启动时完全失忆。

**1.1.2 [参考]** **通用知识与项目知识**：基础模型有通用编程知识，但缺乏项目上下文（目录结构、API 端点、团队规范）。

### 1.2 [参考] 现有局限

**1.2.1 [参考]** CLAUDE.md / .trae/rules 能定义规则，但无法存储动态记忆（已完成功能、决策记录）。

**1.2.2 [参考]** MCP Memory Service 管理持久记忆，但目录结构需精细规划。

---

## Clause 2. 三层记忆架构

### 2.1 [参考] 架构总览

| 层级        | 存储位置       | 时效性 | 内容类型             |
| ----------- | -------------- | ------ | -------------------- |
| L1 会话记忆 | 当前上下文     | 单会话 | 对话状态、临时推理   |
| L2 短期记忆 | `memory/` 目录 | 任务级 | 当前任务状态、待办   |
| L3 长期记忆 | MCP Memory     | 跨任务 | 决策记录、模式、知识 |

### 2.2 [强制] 记忆级别说明

**2.2.1 [强制]** L1 会话记忆：AI 原生上下文，无需显式管理，但会话结束时自动丢失。

**2.2.2 [强制]** L2 短期记忆：存储在项目 `memory/` 目录的文件，用于跨子任务的状态传递。

**2.2.3 [强制]** L3 长期记忆：通过 MCP Memory Service 管理的持久化知识图谱。

## Clause 3. 双轨协作策略

### 3.1 [强制] 人类轨（决策轨）

**3.1.1 [强制]** 人类（项目发起人）负责提出需求，不参与文档编写。

**3.1.2 [强制]** 人类通过以下方式参与协作：

| 活动      | 职责                                      | 频率       |
| --------- | ----------------------------------------- | ---------- |
| 提出需求  | 描述项目意图或功能需求                    | 按需       |
| 阅读文档  | 审阅 `docs/` 下 AI 产出的规范、计划、报告 | 每次交付时 |
| 反馈      | 对文档内容提出修改意见                    | 阅读后     |
| 批准/驳回 | 确认文档是否符合预期，决定是否合入        | 终审时     |

### 3.2 [强制] AI 轨（执行轨）

**3.2.1 [强制]** AI 负责编写 `docs/` 下的所有文档，包括需求澄清后的 Spec、Plan、设计规范、评审报告等。

**3.2.2 [强制]** 产品经理角色（林墨）负责：

| 职责     | 说明                           |
| -------- | ------------------------------ |
| 需求澄清 | 接收人类需求，通过对话明确意图 |
| 产出规范 | 编写 Spec、Plan，拆解任务      |
| 角色委派 | 按需加载开发/设计/测试角色     |
| 组织评审 | 发起计划评审和验收小组         |
| 交付汇报 | 汇总完成情况，提交人类确认     |

**3.2.3 [强制]** AI 自主管理 `memory/` 目录，存储任务进度、中间状态。

**3.2.4 [强制]** AI 通过 MCP Memory Service 写入跨会话知识：

| 知识类型 | 何时写入 | 示例                       |
| -------- | -------- | -------------------------- |
| 项目结构 | 首次探索 | `src/components/` 目录目的 |
| 完成功能 | 交付时   | "用户登录功能已完成"       |
| 问题模式 | 解决时   | "API 超时问题根因为 X"     |
| 决策记录 | 决策时   | "选择 Zustand 原因：..."   |

### 3.3 [强制] 双轨同步规则

**3.3.1 [强制]** 协作流水线：

```text
人类 需求 → AI 写 docs/ → 人类 阅读 → 人类 反馈 → AI 修改 → 人类 终审 → AI document-sync
```

**3.3.2 [强制]** AI 发现的重要模式或决策，必须先写入 `memory/` 或 `docs/ai/knowledge/`，再通过交付汇报告知人类。由人类决定是否需要升级为正式规范写入 `docs/`。

**3.3.3 [强制]** `docs/` 是 SSOT（单一可信源），`memory/` 和 MCP Memory 是缓存和索引。

## Clause 4. 实施方法

### 4.1 [强制] 目录结构

**4.1.1 [强制]** `memory/` 目录结构：

```
memory/
  current-task.md      # 当前任务状态（由 AI 自动维护）
  decisions.md         # 近期决策速记（由 AI 维护，人类确认后迁入 docs/adr/）
  patterns.md          # 模式识别（由 AI 维护）
  progress-log.md      # 进度日志（每次交付后更新）
```

### 4.2 [强制] 各文件职责

**4.2.1 [强制]** `memory/current-task.md`：

- 当前任务名称与目标
- 已完成步骤清单 ✓
- 待完成步骤清单 ☐
- 当前阻塞项

**4.2.2 [推荐]** `memory/decisions.md`：

- 临时决策
- 等待人类确认的决策

**4.2.3 [推荐]** `memory/patterns.md`：

- 反复出现的代码模式
- 常见问题与解决方案

**4.2.4 [强制]** `memory/progress-log.md`：

- 每次完整交付后追加一条记录
- 格式：`[日期] 功能名 — 状态（完成/部分完成）— 要点`
