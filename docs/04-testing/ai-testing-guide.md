# AI 测试覆盖操作指南

> **目标**：让不会写代码的人，也能驱动 AI 产出完整的测试覆盖
> **前提**：项目已配置 Vitest 测试框架（`npm run test:run` 即可执行）

---

## 一、核心原则

### 1.1 测试即文档

测试用例的名字应该直接描述业务场景。你不需要读懂代码，只需要读懂测试用例的名字，就能判断"这个场景有没有被测到"。

**好的测试用例名字**：

- `应阻止从"待立项"流转到"待确认"（缺少容器）`
- `应阻止从"待拆解"流转到"执行中"（任务树未生成）`

**不好的测试用例名字**：

- `test case 1`
- `canTransition returns false`

### 1.2 让 AI 输出"测试覆盖报告"

不要直接让 AI 写测试。先让 AI 输出一份**测试覆盖报告**（纯文本表格），你确认场景齐全后，再让它写代码。

这是最关键的一步——**先审报告，再审代码**。

---

## 二、操作流程（三步走）

### 第一步：输出测试覆盖报告

把下面这段 Prompt 贴给 AI：

```
请为文件 src/domain/projectStatusMachine.ts 输出一份【测试覆盖报告】，格式如下：

## 测试覆盖报告

### 1. 被测函数清单
| 函数名 | 功能描述 | 当前是否有测试 |

### 2. 合法路径测试矩阵
| 起始状态 | 目标状态 | 前提条件 | 预期结果 | 是否已覆盖 |

### 3. 非法路径测试矩阵
| 起始状态 | 目标状态 | 违反的约束 | 预期错误信息 | 是否已覆盖 |

### 4. 边界条件清单
| 边界场景 | 输入条件 | 预期行为 | 是否已覆盖 |

### 5. 缺失测试汇总
列出所有【未覆盖】的测试场景

要求：
- 每个状态流转路径都必须有对应测试
- 每个守卫条件（guard）的 true/false 分支都必须有测试
- 异常输入（空值、非法状态组合）必须有测试
```

**你做什么**：看第 5 节"缺失测试汇总"，确认这些场景是否真的需要测。

### 第二步：确认后生成测试代码

如果确认覆盖报告齐全，再发第二条 Prompt：

```
请根据上面的测试覆盖报告，生成完整的单元测试代码。

要求：
1. 测试文件路径：src/domain/__tests__/projectStatusMachine.test.ts
2. 使用 vitest（import { describe, it, expect } from 'vitest'）
3. 测试用例名字必须用中文，格式：应[允许/阻止]从"X"[流转/操作]到"Y"（[条件]）
4. 每个测试用例只测一个场景，不要一个 it 里面塞多个 expect
5. 如果有已存在的测试，保留并补充缺失的，不要删除已有测试
6. 最后输出【测试统计】：共 X 个测试用例，覆盖 Y 个函数，Z 个分支
```

### 第三步：运行测试并验收

复制以下命令到终端执行：

```bash
# 运行测试
npx vitest run src/domain/__tests__/projectStatusMachine.test.ts

# 或者运行全部测试
npm run test:run
```

**验收标准**：

- ✅ 测试全部通过（PASS）
- ✅ 测试用例数量 ≥ 覆盖报告中的场景数
- ✅ 没有测试报错（FAIL）

如果有测试失败，把报错信息贴给 AI："修正这些失败的测试"。

## 三、Prompt 模板合集

### 模板 A：新模块首次生成测试

```
请为 src/domain/xxx.ts 生成完整单元测试。

【步骤】
1. 先输出测试覆盖报告（格式见上方）
2. 等我确认后，再输出测试代码

【代码要求】
- 使用 vitest
- 测试文件路径：src/domain/__tests__/xxx.test.ts
- 用例名用中文，描述业务场景
- 覆盖所有分支和边界
- 复杂场景使用工厂函数创建 mock 数据

【禁止】
- 不要生成没有断言的空测试
- 不要一个 it 里塞多个不相关的场景
- 不要使用 any 类型
```

### 模板 B：已有模块补充测试

```
请检查 src/domain/__tests__/xxx.test.ts 的测试覆盖度，并补充缺失的测试。

【步骤】
1. 先阅读 src/domain/xxx.ts 的源码，列出所有函数和分支
2. 对比现有测试文件，输出缺失场景清单
3. 补充缺失的测试用例，不要删除已有测试

【输出要求】
- 先输出【补充清单】：列出要补充的测试场景
- 再输出【完整测试代码】：包含已有测试 + 新增测试
- 最后输出【测试统计】
```

### 模板 C：修改代码后同步更新测试

```
我刚修改了 src/domain/xxx.ts，请同步更新对应的测试文件。

【变更说明】
[描述你改了什么，比如：新增了"暂停"状态，从"执行中"可以流转到"暂停中"]

【要求】
1. 先输出【影响分析】：变更影响了哪些函数、哪些测试需要更新
2. 更新测试文件，确保：
   - 新增的状态/分支有对应测试
   - 受影响的旧测试仍然通过（如果行为变了，更新预期结果）
   - 输出【变更后的测试覆盖报告】
3. 不要删除与新逻辑无关的已有测试
```

### 模板 D：非核心模块快速测试

```
请为 src/components/xxx.tsx 生成基础单元测试。

【要求】
- 只测核心交互逻辑（用户点击、表单提交、状态变化）
- 不测纯 UI 渲染（颜色、布局等）
- 使用 vitest + @testing-library/react
- 测试文件路径：src/components/__tests__/xxx.test.tsx
- 输出【测试场景清单】后再输出代码
```

## 四、产品经理验收检查清单（不看代码版）

### 4.1 看测试覆盖报告（2 分钟）

| 检查项               | 通过标准                          | 你的动作              |
| -------------------- | --------------------------------- | --------------------- |
| 所有函数都有测试     | 函数清单中"当前是否有测试"全为 ✅ | 如有 ❌，要求 AI 补充 |
| 所有合法路径都有测试 | 合法路径矩阵中无 ❌               | 如有 ❌，要求 AI 补充 |
| 所有非法路径都有测试 | 非法路径矩阵中无 ❌               | 如有 ❌，要求 AI 补充 |
| 边界条件都有测试     | 边界条件清单中无 ❌               | 如有 ❌，要求 AI 补充 |

### 4.2 看测试用例名字（3 分钟）

打开测试文件，扫一遍 `it('...')` 里的名字：

| 好的信号 ✅                        | 坏的信号 ❌                       |
| ---------------------------------- | --------------------------------- |
| 名字包含完整业务场景               | 名字是英文或缩写                  |
| 有"应允许..."和"应阻止..."成对出现 | 只有正向测试，没有异常测试        |
| 有具体的条件描述（如"缺少容器"）   | 条件描述模糊（如"失败情况"）      |
| 数量 ≥ 业务场景数                  | 明显偏少（如状态机只有 3 个测试） |

### 4.3 运行测试并看结果（1 分钟）

```bash
npm run test:run
```

| 结果        | 含义         | 处理                       |
| ----------- | ------------ | -------------------------- |
| `X passed`  | 全部通过     | ✅ 验收通过                |
| `X failed`  | 有测试失败   | 把报错贴给 AI 修正         |
| `X skipped` | 有测试被跳过 | 问 AI 为什么跳过，要求补全 |

## 五、示例：projectStatusMachine 完整测试覆盖

### 5.1 当前测试的问题

现有 `src/domain/__tests__/projectStatusMachine.test.ts` 只有 12 个测试用例，**大量场景缺失**：

- ❌ "待拆解"→"执行中"的 4 个守卫条件只测了 1 个路径
- ❌ "执行中"→"待验收"缺少"关键任务未完成"的测试
- ❌ "待验收"→"整改中"缺少"缺少验收反馈"的测试
- ❌ "整改中"→"待验收"缺少"整改未闭环"的测试
- ❌ "待结算"→"整改中"完全没有测试
- ❌ 所有"已中止"和"已归档"的边界测试缺失
- ❌ `getEnterStatusHooks` 完全没有测试

### 5.2 让 AI 补充的 Prompt

```
请检查 src/domain/__tests__/projectStatusMachine.test.ts，补充缺失的测试场景。

【已知缺失场景】
1. "待拆解"→"执行中"：只测了 hasContainer，缺少 hasMilestones/hasTaskTree/hasStandardBinding 的 false 分支
2. "执行中"→"待验收"：缺少 keyTasksCompleted=false 的阻止测试
3. "待验收"→"整改中"：缺少 hasAcceptanceFeedback=false 的阻止测试
4. "整改中"→"待验收"：缺少 rectificationClosed=false 的阻止测试
5. "待结算"→"整改中"：完全缺失
6. "待结算"→"已归档"：缺少 settlementCompleted=false 的阻止测试
7. getEnterStatusHooks：完全缺失

【要求】
1. 先输出【补充清单】确认上述缺失
2. 补充测试，保留已有测试
3. 输出【测试统计】
```

### 5.3 完整的测试覆盖报告（让 AI 输出这个格式）

```markdown
## 测试覆盖报告：projectStatusMachine

### 被测函数清单

| 函数名                     | 当前是否有测试 |
| -------------------------- | -------------- |
| canTransition              | ✅（部分）     |
| getAvailableTransitions    | ✅             |
| resolveTransitionGuardCode | ✅             |
| getEnterStatusHooks        | ❌             |

### 合法路径矩阵

| 起始状态 | 目标状态 | 前提条件           | 是否已覆盖 |
| -------- | -------- | ------------------ | ---------- |
| 待立项   | 待确认   | hasContainer=true  | ✅         |
| 待立项   | 待确认   | hasContainer=false | ✅         |
| 待立项   | 已中止   | 填写原因           | ❌         |
| ...      | ...      | ...                | ...        |

### 缺失测试汇总

1. getEnterStatusHooks 完全缺失测试
2. "待拆解"→"执行中" 的守卫条件只覆盖 1/4
3. "已中止"相关路径缺少原因验证
4. ...
```

## 六、维护测试的黄金法则

### 6.1 代码变更时必须同步更新测试

每次让 AI 修改代码时，必须在同一条 Prompt 里加一句：

```
【必须同步更新测试】
- 修改了 src/domain/xxx.ts，请同步更新 src/domain/__tests__/xxx.test.ts
- 新增的逻辑必须有新测试
- 行为变更的测试必须更新预期结果
- 输出【测试变更清单】
```

### 6.2 每周运行一次全量测试

```bash
npm run test:run
```

如果测试失败，说明之前的修改破坏了已有逻辑。立即回滚或让 AI 修正。

### 6.3 测试即文档，不要删旧测试

即使某个功能"暂时不用了"，也不要删测试。测试是理解代码行为的最好文档。

## 七、常见问题

### Q1：AI 生成的测试运行失败怎么办？

把终端的报错信息完整复制给 AI：

```
测试运行报错，请修正：
[粘贴报错]
```

### Q2：AI 说"这个函数不需要测试"怎么办？

如果函数在 `src/domain/` 或 `src/data/` 里，**一律要求测试**。只有纯 UI 组件（如一个只显示图片的按钮）可以豁免。

### Q3：测试文件太大了怎么办？

一个测试文件超过 300 行时，让 AI 按功能拆分成多个文件：

```
请将 src/domain/__tests__/xxx.test.ts 按功能拆分为：
- xxx.status.test.ts（状态流转测试）
- xxx.guard.test.ts（守卫条件测试）
- xxx.util.test.ts（工具函数测试）
```

### Q4：怎么看测试覆盖率？

```bash
npx vitest run --coverage
```

（需要安装 `@vitest/coverage-v8`：npm install -D @vitest/coverage-v8）

覆盖率报告中关注：

- `Statements`（语句覆盖）> 80% 为合格
- `Branches`（分支覆盖）> 70% 为合格
- 核心模块（domain/）要求 `Branches` > 90%

| 场景               | 命令/Prompt                                       |
| ------------------ | ------------------------------------------------- |
| 运行全部测试       | `npm run test:run`                                |
| 运行单个测试文件   | `npx vitest run src/domain/__tests__/xxx.test.ts` |
| 看覆盖率           | `npx vitest run --coverage`                       |

| 补充已有测试       | 用【模板 B】                                      |

| 快速验收           | 看用例名 + 运行测试                               |
