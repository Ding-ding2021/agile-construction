# AI 合约：任务驱动状态架构

## 模块定位

将项目从"手动9状态"重构为"状态由任务聚合推导"的多维正交模型

## 核心实体

| 实体               | 字段                                                                                                                                                                                                                                                               | 状态机                                                |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------- |
| Project            | code, name, brand, parentStatus, executionStatus, acceptanceStatus, settlementStatus, dispatchStatus, healthStatus, progress, pendingDispatchCount, pendingExecutionCount, pendingAcceptanceCount, pendingSettlementCount, plannedOpenDate, actualOpenDate, budget | 无手动状态；5维度由聚合引擎自动计算                   |
| 聚合引擎（服务层） | aggregateProjectStatus(projectId) → ProjectAggregation                                                                                                                                                                                                             | 触发点：任务/检查项/采购单变更时自动触发，每5分钟心跳 |

## 五维度定义

| 维度     | 字段             | 允许值                             | 数据来源                                        |
| -------- | ---------------- | ---------------------------------- | ----------------------------------------------- |
| 生命周期 | parentStatus     | 启动→计划→执行→监控→收尾（不可逆） | PMBOK 五过程组，从WBS/任务数据推导              |
| 执行     | executionStatus  | 未开始→进行中→已完成               | 任务树执行进度聚合                              |
| 验收     | acceptanceStatus | 待验收→验收中→已通过→整改中        | 验收检查项完成度聚合                            |
| 结算     | settlementStatus | 未结算→结算中→已结算               | 采购订单结算状态聚合（依附采购，无独立结算单）  |
| 派单     | dispatchStatus   | 未派单→派单中→已派完               | 任务分配状态聚合                                |
| 健康度   | healthStatus     | 正常→关注→预警→严重                | 进度偏差、SLA超时、风险项、未分配任务数复合计算 |

## 聚合规则

1. **executionStatus**: 全部任务为"草稿"/"待分配" → 未开始；存在"执行中"/"待提交"/"待验收"/"不通过" → 进行中；全部"已完成"/"已关闭" → 已完成
2. **acceptanceStatus**: 无验收请求 → 待验收；有"不通过"检查项 → 整改中；有"待验收"任务 → 验收中；全部通过 → 已通过
3. **settlementStatus**: 无采购单 → 未结算；全部已结算 → 已结算；存在"待结算"/"结算中" → 结算中
4. **dispatchStatus**: 全部任务未分配 → 未派单；全部已分配 → 已派完；部分已分配 → 派单中
5. **parentStatus**: 创建→"启动"；首次WBS拆解完成→"计划"；首个任务执行中→"执行"；首个检查项创建→"监控"；全部验收通过→"收尾"（不可逆）
6. **health**（简单规则第一版）：进度偏差<5%且无风险→正常；偏差5-15%或低风险→关注；偏差>15%或高风险或SLA超时→预警；关键路径超时>7天或阻断性风险→严重

## API 骨架

| 方法 | 路径                        | 说明         | 变更                                     |
| ---- | --------------------------- | ------------ | ---------------------------------------- |
| GET  | /projects/:code             | 项目详情     | 返回聚合后的状态数据                     |
| PUT  | /projects/:code             | 更新项目     | **移除 status 字段支持**，禁止手动设状态 |
| POST | /projects                   | 创建项目     | 默认 parentStatus='启动'，维度全为起始值 |
| GET  | /projects/:code/health      | 健康度详情   | 新增                                     |
| POST | /projects/:code/reaggregate | 手动触发聚合 | 新增（调试/修复用）                      |

## 业务规则

1. 项目不设手动"状态按钮"——所有维度状态由聚合引擎自动计算
2. 聚合引擎在任务/检查项/采购单状态变更时自动触发
3. health 计算用简单规则第一版，不设权重/衰减因子
4. 维度状态无需额外守卫——聚合算法本身就是守卫
5. parentStatus 不可逆（只能前进不能后退）
6. 前端维度展示只依赖聚合结果，不独立计算

## 迁移策略

1. **阶段一（数据迁移）**：旧 status 映射到 parentStatus；从现有任务数据计算各维度初始值
2. **阶段二（代码迁移）**：先建聚合引擎（不影响现有）→ 添加 health 端点 → 前端加面板（共存）→ 迁移列表过滤 → 移除旧 status → 移除旧UI

## 测试要求

| 层级     | 覆盖目标                                     | 框架               |
| -------- | -------------------------------------------- | ------------------ |
| 单元测试 | 5个维度聚合规则 + 健康度计算 → 100% 分支覆盖 | Vitest             |
| API 测试 | GET /:code 返回聚合数据 + 任务变更触发聚合   | Vitest + supertest |
| 组件测试 | HealthCard, DimensionStatus 渲染             | RTL                |
| E2E      | 创建项目→添加任务→任务推进→聚合状态刷新      | Playwright         |

## 质量门禁

- 聚合引擎每个计算函数必须有单元测试（100% 分支覆盖）
- 任务状态变更后必须触发项目聚合
- 删除旧 status 字段前确认无任何代码引用
- 迁移过程中不得同时维护两套状态系统
