# AI 合约：三组协作协议

## 模块定位

定义 Harness 七阶段中三组评审的协作规则

## 四专业角色

| 角色       | 人物 | 模型               | 阶段参与                      |
| ---------- | ---- | ------------------ | ----------------------------- |
| 产品经理   | 林墨 | deepseek-v4-flash  | 定义/规划/测试/评审/交付/进化 |
| UI设计师   | 苏染 | kimi-k2.6 (多模态) | 定义/规划/构建/测试/评审      |
| 开发工程师 | 陈锋 | deepseek-v4-flash  | 规划/构建/测试                |
| 测试工程师 | 周严 | deepseek-v4-pro    | 定义/规划/测试/评审           |

## 三组评审

| 组名     | 阶段      | 成员 | 产出         | 人类决策 |
| -------- | --------- | ---- | ------------ | -------- |
| 评审小组 | 定义→规划 | 四人 | 评估报告     | ✅       |
| 计划评审 | 规划→构建 | 四人 | 计划评审报告 | ✅       |
| 验收小组 | 评审→交付 | 四人 | 验收报告     | ✅       |

## 风险分级

| 等级 | 条件                         | 评估组规模 | 验收组规模 |
| ---- | ---------------------------- | ---------- | ---------- |
| L1   | 单文件、纯样式/文档/fix      | 林墨单人   | 林墨单人   |
| L2   | 新增组件、单页面、小功能     | 林墨 + 1   | 林墨 + 1   |
| L3   | 跨文件、架构、路由、数据模型 | 全量4人    | 全量4人    |

## 评审结论规则

| 结论              | 条件               | 后续动作            |
| ----------------- | ------------------ | ------------------- |
| APPROVED          | 全票通过           | 进入下一阶段        |
| CHANGES REQUESTED | 修改建议非原则反对 | 修复后问题角色重审  |
| BLOCKED           | 原则性反对         | 林墨仲裁 → 人类决策 |

## 核心铁律

1. 编码前必须评审（L1 除外）
2. 编码后必须验收（L1 除外）
3. 全票通过制
4. 独立输出不共享状态
5. 先评审后构建
6. 陈锋编码前加载 karpathy-guidelines
7. 增量重审（仅派问题角色）
8. 人类最终决策

## 七阶段技能映射

| 阶段 | 负责角色            | 关键技能                                      |
| ---- | ------------------- | --------------------------------------------- |
| 定义 | 林墨→苏染→陈锋→周严 | brainstorming → spec-driven-development       |
| 规划 | 林墨→苏染→陈锋→周严 | planning-and-task-breakdown                   |
| 构建 | 陈锋 + 苏染         | karpathy-guidelines → frontend-ui-engineering |
| 测试 | 四人                | verification-before-completion                |
| 评审 | 四人                | squad-post-dev-review                         |
| 交付 | 林墨                | shipping-and-launch                           |
| 进化 | 林墨                | task-memory → skill-creator                   |
