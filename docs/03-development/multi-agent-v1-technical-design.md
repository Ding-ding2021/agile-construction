# 多Agent建店管理平台 V1 技术方案草案

## 1. 技术方案目标

V1 技术方案的目标不是一次性做全自治 Agent 平台，而是支撑以下能力落地：

1. 支撑品牌端、资源调度平台、资源方端三套系统
2. 支撑 7 个 Agent 的基础运行
3. 支撑 项目 -> 任务 -> 派单 -> 执行 -> 验收 -> 结算 主流程闭环
4. 支撑规则配置、状态流转、审计追踪、人工兜底
5. 保证系统具备后续扩展到 V2/V3 的架构基础

---

## 2. 总体技术架构

建议采用分层架构 + Agent 编排中台。

### 2.1 架构分层

#### 前端应用层

- 品牌端 Web
- 调度平台 Web
- 资源方端 Web / H5

#### 业务接入层

- API 网关
- 认证鉴权
- BFF 层

#### 业务服务层

- 项目中心
- 任务中心
- 调度中心
- 资源中心
- 验收中心
- 结算中心
- 工单中心
- 消息通知中心

#### Agent 编排层

- Agent 网关
- 工作流引擎
- 规则引擎
- Agent 运行时
- Prompt/模板中心
- 工具调用层
- 审计与记忆层

#### 数据层

- MySQL / PostgreSQL
- Redis
- Elasticsearch
- 对象存储
- 日志与监控系统

#### 外部能力层

- 大模型服务
- OCR / 图片识别
- 消息推送
- 第三方系统接口

## 3. 技术架构原则

### 3.1 核心原则

- 业务状态和 Agent 状态分离
- 流程驱动，不靠 Agent 自由发挥
- 所有 Agent 动作都可审计
- 关键动作必须可转人工
- 规则可配置，Prompt 可配置，流程可追踪

### 3.2 设计理由

这个项目本质是强业务流程 + 弱开放式对话，因此：

- 不能只靠大模型直接决定业务动作
- 必须由工作流 + 规则引擎 + 状态机控制主流程
- Agent 更适合做理解、补全、推荐、摘要、判断、生成建议
- Agent 不应直接无约束修改业务状态

---

## 4. 系统拆分建议

V1 不建议一开始做很重的微服务，可以采用模块化单体 + 核心能力服务化。

### 4.1 方案 A：V1 推荐

模块化单体。

建议模块：

- auth-module
- project-module
- task-module
- dispatch-module
- resource-module
- acceptance-module
- settlement-module
- ticket-module
- agent-module

**优点**：

- 开发快
- 联调简单
- 成本低
- 适合 MVP

### 4.2 方案 B：V2 以后

逐步拆成独立服务：

- 用户权限服务
- 项目任务服务
- 调度服务
- 工单服务
- Agent 编排服务
- 通知服务

---

## 5. 前端技术方案

### 5.1 前端形态建议

- 品牌端：Web 后台
- 调度平台：Web 后台
- 资源方端：优先 H5 + Web，便于现场使用

### 5.2 推荐技术栈

- React
- TypeScript
- Next.js 或 Vite + React
- Ant Design 或 TDesign
- React Query
- Zustand 或 Redux Toolkit

### 5.3 页面层建议

- 品牌端：偏流程和查看
- 调度平台：偏表格、筛选、监控
- 资源方端：偏移动端表单、上传、打点

### 5.4 项目详情信息架构（V1 当前迭代）

- 项目详情统一为 6 标签：`dashboard / start / plan / execute / monitor / close`
- `execute` 标签内聚合“任务 + WBS”视图，避免项目内重复导航
- 继续保留全局 `#/tasks` 路由处理跨项目任务能力
- 旧路由标签（如 `overview/gantt/wbs/acceptance`）通过前端别名映射到新标签，保证链接兼容

## 7. Agent 技术架构设计

### 7.1 Agent 不应直接写死在业务代码里

建议设计独立的 Agent Runtime，每个 Agent 按统一协议运行。

### 7.2 Agent Runtime 组成

- Agent Registry：注册 7 个 Agent
- Prompt Template Center：管理不同 Agent 的提示模板
- Tool Router：给 Agent 提供可调用的业务工具
- Context Builder：组装上下文
- Decision Logger：记录输入、输出、决策依据
- Fallback Handler：异常转人工

### 7.3 Agent 调用模式

业务事件触发 → 工作流判断是否需要 Agent → 组装上下文 → 调用 Agent → Agent 输出结构化结果 → 规则引擎校验 → 写入业务系统 / 转人工

### 7.4 Agent 输出要求

V1 所有 Agent 输出都必须是结构化 JSON，不要直接输出自然语言作为业务结果。

例如：

- 是否缺失字段
- 风险等级
- 推荐资源方列表
- 是否建议转人工
- 建议工单类型

这样便于：

- 程序消费
- 日志审计
- 规则校验

## 9. 工作流引擎设计

### 9.1 为什么需要工作流引擎

主流程是固定的：

- 立项
- 拆解
- 派单
- 执行
- 验收
- 结算

Agent 只是每一步里的执行者，不是流程主宰。

### 9.2 V1 建议编排方式

采用事件驱动 + 流程节点编排。

例如：

- 项目立项确认 → 触发项目经理 Agent → 生成任务 → 任务状态变更为待调度
- 任务进入待调度 → 触发调度 Agent → 派单成功后 → 任务状态变更为待执行

### 9.3 流程节点类型

- 自动节点
- Agent 节点
- 人工审批节点
- 系统通知节点
- 异常分支节点

---

## 10. 规则引擎设计

V1 最少要把下面规则做成可配置。

### 10.1 派单规则

- 城市优先
- 品类匹配
- 资质要求
- 评分阈值
- 当前负载
- SLA 风险

### 10.2 升级规则

- 超预算
- 超时未响应
- 连续派单失败
- 验收连续不通过
- 工单超时未关闭

### 10.3 验收规则

- 不同店型不同模板
- 必传资料清单
- 必检项清单
- 整改流转规则

### 10.4 结算规则

- 是否允许增补
- 差异阈值
- 无依据费用拦截
- 审核流转条件

## 12. 搜索与知识库方案

### 12.1 客服 Agent 需要知识库

知识内容建议包括：

- 建店流程 FAQ
- 平台规则说明
- 验收标准说明
- 结算规则说明
- 常见投诉处理话术

### 12.2 知识库技术建议

- 文档入库到 Elasticsearch 或向量库
- 客服 Agent 采用 RAG
- 只回答知识库中已有内容
- 业务结果必须实时查业务数据库，不能只靠知识库回答

## 15. 权限与安全设计

### 15.1 权限模型

建议采用：

- RBAC 角色权限
- 数据范围权限
- 组织/品牌/资源方隔离

### 15.2 Agent 权限

Agent 不是超级管理员，必须限制：

- 只能在指定流程节点执行
- 只能调用授权工具
- 不能直接越权改关键状态
- 关键动作必须经过规则校验

### 15.3 安全重点

- 敏感字段脱敏
- 审计日志留存
- 文件访问鉴权
- 操作回放能力

## 17. 部署方案建议

### 17.1 V1 推荐

- 一套主应用服务
- 一套数据库
- 一套 Redis
- 一套对象存储
- 一套日志监控

### 17.2 部署形态

- Web 前端独立部署
- API 服务单独部署
- Agent 服务可独立进程部署
- 异步任务 Worker 独立部署

### 17.3 设计原因

Agent 调用和业务接口的负载特征不同，建议拆开：

- API 保持稳定响应
- Agent 异步执行可容错、可重试

---

## 18. 异步任务设计建议

以下场景建议异步：

- 生成项目草案
- 批量拆任务
- 派单计算
- 资料完整性扫描
- 验收初审
- 结算建议生成
- 工单分类与回告

这样可以避免前端长时间等待。

## 20. 技术风险

### 20.1 规则不清晰

如果业务规则没定清楚，Agent 输出很难稳定。

### 20.2 数据质量不足

资源池、模板、验收标准不完善，会直接影响自动化率。

### 20.3 Agent 幻觉风险

因此必须做到：

- 结构化输出
- 规则校验
- 高风险转人工

### 20.4 流程与 Agent 耦合过深

因此建议：

- 流程由工作流控制
- Agent 只做节点能力

---

## 21. V1 技术结论

V1 最合适的技术路线是：

> 用模块化业务系统 + Agent 编排层 + 工作流引擎 + 规则引擎 + 审计中心的方式，先完成建店项目的最小闭环自动化，而不是直接做一个完全开放式的 AI 自治系统。

---

## 22. 建议同步给技术团队的 5 个问题

1. V1 采用模块化单体还是微服务？
2. 工作流引擎用现成产品还是自研轻量版？
3. Agent Runtime 是否单独成服务？
4. 规则引擎和 Prompt 模板如何配置化？
5. Agent 审计日志存储到什么粒度？
