# 连锁门店营建管理系统 - 详细开发计划 V1.2

> **文档版本**：V1.1
> **适用场景**：单人 + AI 编码模式
> **计划性质**：工程任务分解（不考虑日历时间，按依赖关系排序）
> **当前基线**：Phase 1 已完成，Phase 1.5 进行中（底座收官）
> **核心原则**：先还技术债务，再补功能；状态机驱动；标准先行

---

## 1. Phase 1：底座搭建

**目标**：消除技术债务，建立可维护的工程底座，让前端从「页面演示」进入「数据可持久化」阶段。

**前置条件**：无（当前基线即为起点）

### P1-T1 共享组件补齐与存量替换

- **目标**：完成 `component-refactoring-plan.md` 中定义的全部共享组件开发，并将所有业务模块迁移到新组件。
- **执行要点**：
  1. 补齐 `AppSidebar` 收起/展开、嵌套菜单支持
  2. 补齐 `PageHeader` 面包屑、`dark` 变体
  3. 补齐 `StatCard` delta 趋势、点击筛选、active 状态
  4. 新增 `TabNav` 组件（项目详情 8 标签共用）
  5. 按「personnel → task → digital → procurement → project → 其他」顺序逐个替换
  6. 删除废弃的 `layout/Sidebar.tsx`、`personnel/Sidebar.tsx`、`task/TaskSidebar.tsx` 等
- **前置依赖**：无
- **验收标准**：
  - `npm run build` 零报错
  - `npm run lint` 零报错
  - 所有页面侧边栏、页头、统计卡片功能正常
  - 新增路由只需修改 `src/config/routes.ts` 一处
- **技术要点**：
  - 保留旧代码 1 个迭代周期，确认稳定后删除
  - 每次替换后立即验证对应模块路由跳转
  - 禁止在替换过程中修改业务逻辑，仅做组件替换

- **验收标准**：
  - 全局搜索 `#`、`rgba(`、`rgb(` 在 `.css` 文件中仅出现在 `index.css` 的变量定义区

### P1-T4 本地后端 Schema 实体化

- **目标**：将 local-api 从「大快照 JSON」模式重构为「结构化实体表 + RESTful API」模式，支撑细粒度 CRUD 和跨表关联。
- **执行要点**：
  1. 重新设计 `schema.sql`，按业务实体建表：
     - `projects`（项目实体）
     - `project_stages`（项目阶段）
     - `project_milestones`（里程碑）
     - `tasks`（任务实体，含 `parent_task_id`）
     - `task_relations`（任务前置依赖）
     - `standards`（标准来源）
     - `standard_clauses`（标准条款）
     - `standard_snapshots`（标准快照）
     - `procurements`（采购申请/订单）
     - `acceptance_nodes`（验收节点）
     - `acceptance_checkitems`（验收检查项）
     - `assets`（资产台账）
     - `audit_logs`（已有，增强字段）
     - `users` / `roles` / `permissions`（基础权限）
  2. 使用 Prisma 作为 ORM，替换手写 SQL
  3. 创建 `prisma/schema.prisma`，定义模型和关系
  4. 生成迁移脚本
  5. 更新 `server.ts`，按 RESTful 风格重构路由（`/api/projects`、`/api/tasks/:id` 等）
  6. 保留幂等机制，扩展到所有写接口
- **前置依赖**：无（可与 P1-T1~T3 并行）
- **验收标准**：
  - Prisma 可正常 migrate 和 generate
  - 新 API 通过 `test-api.sh` 或 Postman 集合验证
  - 各实体表可独立 CRUD，非快照覆盖
  - 外键关系正确（如 `task.project_id` → `project.id`）
- **技术要点**：
  - Prisma 的 SQLite provider 后期可平滑切到 PostgreSQL
  - 审计日志表继续手写 SQL（绕开 ORM，保证灵活性）
  - 初期保留旧快照接口作为 fallback，新旧并存 1 个迭代

---

### P1-T5 前端数据层抽象（API 适配器）

- **目标**：建立 `services/` 层，统一封装对 local-api 的调用，使前端从直接操作 localStorage 迁移到调用后端 API。
- **执行要点**：
  1. 创建 `src/services/api/client.ts`：基于 `fetch` 的 HTTP 客户端，统一处理 CORS、错误、幂等键注入
  2. 创建 `src/services/api/projects.ts`、`tasks.ts`、`standards.ts`、`procurements.ts` 等模块 API
  3. 创建 `src/services/repositories/` 下的仓库模式实现（已有骨架，需扩展）
  4. 在 `App.tsx` 中替换 `localStorage` 读写为 API 调用
  5. 实现离线降级：API 不可用时 fallback 到 localStorage
  6. 统一错误处理：网络错误 → Toast 提示；业务错误 → 表单反馈
- **前置依赖**：P1-T4（需要结构化 API 才能对接）
- **验收标准**：
  - 项目列表、任务列表数据来自 API 而非 localStorage
  - 刷新页面后数据持久存在（SQLite）
  - local-api 未启动时，系统仍能降级运行（localStorage fallback）
- **技术要点**：
  - API 模块类型定义与 `local-api/contracts.ts` 对齐
  - 引入 React Query 或 SWR（可选，V1 可用 useEffect + useState 简化）
  - 所有写操作携带 `X-Idempotency-Key`

### P1-T6 状态机联动真实化

- **目标**：将 `projectStatusMachine.ts` 中的 mock 联动字符串替换为真实业务动作调用。
- **执行要点**：
  1. 修改 `enterStatusHooks` 定义：从字符串数组改为动作函数数组
  2. 定义联动动作接口：
     - `待拆解` → 调用任务树初始化 API
     - `执行中` → 启动催办/预警定时检查
     - `待验收` → 生成验收摘要和检查项
     - `整改中` → 汇总未通过任务
     - `待结算` → 触发结算建议生成
  3. 在 `App.tsx` 的 `transitionProjectStatus` 中，状态流转成功后顺序执行联动动作
  4. 联动动作失败不阻断状态流转，但需记录警告日志
  5. 为项目状态机编写单元测试（覆盖所有合法流转和非法拦截）
- **前置依赖**：P1-T5（联动动作需要调用真实 API）
- **验收标准**：
  - 项目从「待拆解」流转到「执行中」后，任务树真实生成（可查 API）
  - 项目进入「待验收」后，验收检查项真实生成
  - 状态流转日志记录联动动作执行结果
  - 单元测试覆盖全部守卫条件
- **技术要点**：
  - 状态机定义文件（`domain/` 层）**禁止 AI 直接修改**，人工维护
  - 联动动作封装为独立 `hooks/` 函数，便于测试和复用
  - 每个联动动作需支持「人工跳过」标记

### P1.5-T2 CSS Token 治理

- **目标**：在 P1-T2 硬编码清零基础上，进一步治理 Token 命名混乱和循环引用问题。
- **执行要点**：
  1. 统一 Token 命名规范：`--pm-{范畴}-{属性}-{修饰}`（如 `--pm-color-primary`）
  2. 清理 `index.css` 中 189 个一次性变量的重复/近义项
  3. 建立 Token 分层：基础 Token（色值/字号）→ 语义 Token（背景/边框）→ 组件 Token（卡片/按钮）
  4. 修复循环引用（如 `--pm-text-white: var(--pm-text-white)`）
  5. 建立 Token 使用规范文档
- **前置依赖**：P1-T2
- **验收标准**：
  - `index.css` 变量数量从 189+ 压缩至 80 个以内
  - 无循环引用、无重复定义
  - 全局搜索 `#`、`rgba(` 在业务 CSS 中返回 0 结果
- **技术要点**：
  - 使用 CSS 自定义属性 `@property` 支持类型提示（可选）
  - Token 变更需通过查找替换批量更新，禁止手动逐行修改

### P1.5-T4 渐变色 Token 化迁移

- **目标**：将 73+ 处 `linear-gradient` 硬编码提取为语义化 Token，消除重复模式和内联 style 逃逸。
- **执行要点**：
  1. 提取 6 大渐变模式为 Token：

     - `--pm-gradient-btn-primary`：主按钮
     - `--pm-gradient-bar`：进度条
     - `--pm-gradient-kpi`：KPI 半透明背景
     - `--pm-gradient-modal`：模态框表面
  2. 统一角度标准化：`--pm-gradient-angle-diagonal: 135deg`、`vertical: 180deg`、`horizontal: 90deg`
  3. 建立语义化工具类：`.stat-card-bg-*`、`.kpi-bg-*`、`.bar-*`、`.accent-bar-*`、`.btn-*`

  5. 删除 `.pm-stat-blue` / `.tm-stat-blue` / `.om-stat-blue` 等页面前缀命名，统一为语义类名
- **前置依赖**：P1.5-T2
- **验收标准**：
  - 全局搜索 `linear-gradient` 仅出现在 `index.css` Token 定义区
  - 无内联 style 写渐变
  - 同一渐变模式全站只定义一次
- **技术要点**：
  - 禁止业务 CSS 直接使用 `linear-gradient`，一律通过 `var()` 引用
  - 渐变透明度跟随 Token，不单独硬编码

  4. 支持人工增补/删除检查项
- **前置依赖**：P2-T2、P3-T1
- **验收标准**：
  - 从模板生成的任务自动携带标准快照
  - 任务进入待验收后，检查项列表自动生成
  - 标准源文件修改后，已生成的快照不受影响
  - 检查项可逐条判定并记录结果
- **技术要点**：
  - 标准快照是深拷贝，包含条款内容和规则项
  - 检查项生成逻辑封装为独立服务函数
  - 快照表需记录生成时间和源标准版本

---

### P3-T4 任务关系管理（前置依赖 + 阻塞派生）

- **目标**：实现任务间的前置依赖关系（FS/SS/FF/SF），并在状态流转时校验依赖满足情况。
- **执行要点**：
  1. 新增 `task_relations` 表：`from_task_id`、`to_task_id`、`relation_type`（finish-start 等）、`lag_days`
  2. 前端：任务详情页增加「前置任务」和「后续任务」面板
  3. 修改任务状态机守卫：
     - `待分配 → 待执行`：校验前置依赖任务是否已完成（或满足关系类型要求）
     - 前置未满足时，任务显示「阻塞」标记和原因
  4. 支持关键路径识别（简化版：从阻塞任务反向追溯）
  5. 采购未到货也可作为阻塞原因（与 Phase 4 联动）
- **前置依赖**：P3-T1
- **验收标准**：
  - 可为任务设置前置依赖关系
  - 前置任务未完成时，后续任务无法进入「待执行」
  - 阻塞原因可查看（哪个前置任务未完成 / 哪个采购未到货）
  - 关键路径可识别（至少能高亮显示阻塞链）
- **技术要点**：
  - V1 只实现 `finish-start` 类型，其他类型预留字段
  - 循环依赖检测：添加关系时校验是否形成环
  - 阻塞状态作为「补充维度」展示，不替代主状态

---

### P3-T5 任务状态流转守卫真实化

- **目标**：补齐任务状态机的全部守卫条件，让任务从「待分配」到「已完成」的流转可真实执行。
- **执行要点**：
  1. 对照 `state-machine-design.md` 补齐守卫：
     - `待分配 → 待执行`：责任人已分配 + 前置依赖满足 + 执行标准已绑定 + 标准快照已生成
     - `待执行 → 执行中`：执行主体确认开始
     - `执行中 → 待提交`：执行动作完成 + 必传资料已提交
     - `待提交 → 待验收`：提交结果完整 + 验收标准已绑定 + 检查项已生成
     - `待验收 → 已完成`：检查项全部通过
     - `待验收 → 不通过`：存在关键不合格项
  2. 每个守卫条件失败时，返回明确的错误信息和缺口清单
  3. 前端任务详情页：状态流转按钮根据守卫条件动态显示/置灰
  4. 为任务状态机编写单元测试（覆盖率 100%）
- **前置依赖**：P3-T3、P3-T4
- **验收标准**：
  - 任务状态流转必须满足全部守卫条件
  - 守卫失败时用户可看到具体原因（如「缺少前置任务 X 完成」）
  - 单元测试覆盖全部合法和非法流转路径
- **技术要点**：
  - 守卫条件涉及跨表查询（任务 + 依赖 + 快照 + 检查项），封装为 `GuardContext` 构建函数
  - 守卫校验只做读取，不写数据

---

### P3-T6 整改任务自动派生

- **目标**：任务验收不通过时，自动生成整改任务并关联原任务，形成整改闭环。
- **执行要点**：
  1. 任务进入「不通过」状态时：
     - 解析检查项中的不通过项
     - 自动生成整改子任务（`task_type = 'rectification'`），`parent_task_id` 指向原任务
     - 整改任务默认状态：待分配
     - 整改任务预填充：名称（原任务名 + 整改）、关联不通过检查项、建议完成时间
  2. 整改任务完成后，原任务自动回到「待验收」
  3. 支持复验：整改任务完成 → 触发复验检查项 → 复验通过 → 原任务进入「已完成」
  4. 前端：任务详情页展示「整改历史」时间线
- **前置依赖**：P3-T5
- **验收标准**：
  - 验收不通过时，整改任务自动生成
  - 整改任务完成后可触发复验
  - 复验通过后原任务状态正确流转
  - 整改历史可查看（时间线形式）
- **技术要点**：
  - 整改任务生成是系统自动化动作，需记录审计日志
  - 支持人工取消自动生成的整改任务（需填写原因）
  - 复验检查项可从原验收检查项中复制未通过项

---

## 4. Phase 4：采购、资源与资产沉淀

**目标**：建立采购基础闭环、验收整改完整流程、工队管理和资产归档，使「任务推进 → 采购协同 → 验收整改 → 资产沉淀」链路可执行。

**前置条件**：Phase 3 完成

---

### P4-T1 采购管理基础闭环

- **目标**：实现采购申请 → 审批 → 下单 → 到货跟踪的完整流程。
- **执行要点**：
  1. 后端实体：
     - `procurement_requests`（采购申请）：关联项目/任务、物料清单、申请原因
     - `procurement_orders`（采购订单）：关联申请、供应商、金额、下单日期
     - `procurement_arrivals`（到货记录）：关联订单、到货数量、验收结果
  2. 采购状态机：`待申请 → 待审批 → 待下单 → 待到货 → 已到货`（见 state-machine-design.md §8）
  3. 前端：`ProcurementManagementPage` 重构为 MUI DataGrid
  4. 采购申请可从任务详情页发起（关联任务）
  5. 到货确认后，解除对相关任务的采购阻塞
- **前置依赖**：P1-T4、P3-T4
- **验收标准**：
  - 可创建采购申请并关联任务
  - 采购状态可按状态机正常流转
  - 到货确认后，关联任务的采购阻塞解除
  - 采购列表可按项目、状态、供应商筛选
- **技术要点**：
  - 采购与任务通过 `task_id` 外键关联
  - 到货数量支持「部分到货」，触发补货逻辑（V1.5 完善）
  - 采购审批 V1 先支持单级审批，多级审批 V1.5 扩展

---

### P4-T2 验收流程完整实现

- **目标**：实现验收申请 → 初验 → 整改 → 复验的完整闭环。
- **执行要点**：
  1. 后端实体：
     - `acceptance_nodes`（验收节点）：关联项目、验收类型（初验/复验）、状态
     - `acceptance_checkitems`（检查项）：关联验收节点、标准快照条款、判定结果
  2. 验收状态机：`待申请 → 待初验 → 初验通过/不通过 → 整改中 → 待复验 → 复验通过`（见 state-machine-design.md §9）
  3. 初验不通过时：
     - 生成整改任务（调用 P3-T6 逻辑）
     - 项目状态可进入「整改中」（如为整体验收）
  4. 前端：项目详情「质量与验收」标签内完成全部验收操作
  5. 验收资料上传支持（V1 先支持文本记录，文件上传 V1.5）
- **前置依赖**：P3-T3、P3-T6
- **验收标准**：
  - 可从项目或任务发起验收申请
  - 检查项逐条判定，支持通过/不通过/待检
  - 初验不通过自动生成整改任务
  - 整改完成后可发起复验
  - 复验通过后验收节点关闭
- **技术要点**：
  - 验收节点与任务状态联动：任务进入「待验收」时自动创建验收节点
  - 项目级验收汇总所有任务验收结果

---

### P4-T3 工队管理基础版

- **目标**：建立工队档案、班组结构、资质证照管理，支持在任务派单时选择工队。
- **执行要点**：
  1. 后端实体：
     - `workteams`（工队）：名称、服务城市、服务区域、评级、状态
     - `workteam_members`（班组成员）：姓名、工种、资质、联系方式
     - `workteam_qualifications`（资质证照）：证照类型、编号、有效期、附件
  2. 前端：`WorkteamManagementPage` 基础版（列表 + 详情 + 新建/编辑）
  3. 任务派单时：
     - 执行方选择增加「工队」选项
     - 根据项目城市自动推荐可服务工队
     - 显示工队当前负载（已派任务数 / 容量）
  4. 工队评级：V1 先支持手动评级（优秀/良好/一般），自动评级 V1.5
- **前置依赖**：P1-T4
- **验收标准**：
  - 可创建工队档案和班组信息
  - 任务派单时可选择工队作为执行方
  - 工队资质到期前 30 天预警（简单实现）
- **技术要点**：
  - 工队服务区域 V1 先支持城市级匹配，区域级 V1.5
  - 工队负载计算基于已分配未完成任务数

---

### P4-T4 资产归档基础

- **目标**：项目交付后，将门店资产沉淀为「一店一档」。
- **执行要点**：
  1. 后端实体：
     - `assets`（资产台账）：资产名称、类别、品牌、型号、数量、关联项目/任务、存放位置
     - `asset_categories`（资产类别）：装修类、设备类、道具类、IT 类
     - `project_deliverables`（交付资料）：资料类型、名称、存储路径、归档状态
  2. 前端：项目详情「质量与验收」或「项目概览」中增加「资产摘要」
  3. 资产可从采购到货记录和验收通过任务中自动归集
  4. 一店一档：按项目汇总资产清单 + 交付资料清单 + 验收报告 + 结算建议
  5. 资产状态机：`待登记 → 待验收归档 → 已归档`（见 state-machine-design.md §10）
- **前置依赖**：P4-T1、P4-T2
- **验收标准**：
  - 项目验收通过后，资产可自动归集
  - 可查看「一店一档」汇总页面
  - 资产台账支持按项目、类别、状态筛选
- **技术要点**：
  - 资产归集是系统自动动作（从采购和验收结果中提取）
  - 支持人工增补资产和资料
  - 一店一档数据可作为项目归档的前置条件

---

### P4-T5 结算建议基础

- **目标**：项目进入「待结算」后，系统可基于合同、采购、验收数据生成结算建议草案。
- **执行要点**：
  1. 后端实体：
     - `settlement_suggestions`（结算建议）：关联项目、建议金额、差异分析、状态
     - `settlement_items`（结算明细）：类别、合同金额、实际金额、差异、原因
  2. 结算状态机：`待生成建议 → 待审核 → 待确认 → 已完成`（见 state-machine-design.md §11）
  3. 结算建议自动生成逻辑（规则引擎，非 AI）：
     - 汇总项目关联的全部采购订单金额
     - 汇总合同金额
     - 计算差异（合同 vs 采购 vs 变更）
     - 标记异常项（超预算、无合同依据采购等）
  4. 前端：项目详情「项目概览」中展示结算摘要；独立结算页展示明细
  5. 结算建议需人工审核后方可进入「待确认」
- **前置依赖**：P4-T1、P4-T2
- **验收标准**：
  - 项目进入「待结算」后，结算建议可自动生成
  - 结算建议包含金额汇总和差异分析
  - 结算可按状态机流转（审核 → 确认 → 完成）
- **技术要点**：
  - 结算建议是「结果对象」，独立存储，不直接修改项目金额字段
  - 差异分析规则 V1 先硬编码，V2 支持配置化

---

## 5. Phase 5：Agent 与工作台

**目标**：建立 Agent 节点框架，落地 3 个核心 Agent，并完成工作台首页，让系统从「被动工具」进化为「主动辅助」。

**前置条件**：Phase 4 完成

---

### P5-T1 Agent 节点框架与执行记录

- **目标**：建立 Agent 运行的基础设施：触发节点、输入上下文、输出结构化、人工确认、执行记录。
- **执行要点**：
  1. 后端实体：
     - `agent_configs`（Agent 配置）：Agent 名称、触发节点、输入模板、输出模板、启用状态
     - `agent_executions`（执行记录）：触发对象、输入上下文、输出结果、执行状态、人工确认状态
  2. Agent 执行状态：`待执行 → 执行中 → 已完成 → 已转人工 / 已失败`
  3. Agent 输出必须经过人工确认才可影响业务对象（强制规则）
  4. 前端：「设置与 Agent」标签中展示 Agent 配置和执行记录
  5. 封装 `src/services/agent/` 层：统一调用 LLM API，支持模型路由（简单任务用低价模型，复杂任务用强模型）
- **前置依赖**：P1-T4、P1-T5
- **验收标准**：
  - Agent 可在指定节点被触发
  - Agent 输出展示在 UI 中，需人工点击「采纳」后才生效
  - 所有 Agent 执行记录可查（输入、输出、耗时、确认人）
  - Agent 失败时可一键转人工
- **技术要点**：
  - Agent 不能绕过状态机直接修改关键状态（红线）
  - LLM API Key 配置在环境变量
  - Agent 调用需设置超时（如 30s）和重试机制

---

### P5-T2 品牌需求 Agent

- **目标**：辅助品牌方完成立项信息补全、模板匹配、立项草案生成。
- **执行要点**：
  1. 触发节点：立项表单填写过程中 / 提交立项草案前
  2. 输入上下文：已填写的立项字段 + 品牌历史项目数据 + 可用模板列表
  3. Agent 能力：
     - 缺失字段识别与补全建议
     - 店型模板智能推荐（基于城市、面积、预算）
     - 立项草案自动生成（基于模板 + 输入信息）
  4. 输出形式：结构化 JSON（字段补全建议 + 模板推荐 ID + 草案文本）
  5. 前端：立项页嵌入「AI 辅助」面板，展示建议并支持一键采纳
- **前置依赖**：P5-T1、P2-T4
- **验收标准**：
  - 输入部分立项信息后，Agent 可给出缺失字段提示
  - Agent 推荐的模板与人工选择一致性 > 60%（MVP 标准）
  - 生成的立项草案可读，经人工确认后可生成为正式项目
- **技术要点**：
  - V1 先使用规则引擎 + 模板匹配模拟 Agent 输出，降低 LLM 依赖
  - 规则引擎无法覆盖的场景再调用 LLM
  - 记录 Agent 建议采纳率，用于后续优化

---

### P5-T3 项目经理 Agent

- **目标**：辅助项目经理进行任务树生成建议、催办预警、风险摘要。
- **执行要点**：
  1. 触发节点：
     - 项目进入「待拆解」→ 任务树生成建议
     - 任务即将到期（< 3 天）→ 催办建议
     - 关键任务阻塞 > 2 天 → 风险预警
  2. 输入上下文：项目数据 + 任务树 + 依赖关系 + 历史执行数据
  3. Agent 能力：
     - 任务树生成建议：基于模板和项目特征推荐任务拆分
     - 催办预警：识别即将到期和已延期任务，生成催办文案
     - 风险摘要：基于阻塞任务、延期任务、采购异常生成风险摘要
  4. 输出形式：结构化 JSON（建议任务列表 / 催办任务列表 / 风险摘要文本）
  5. 前端：工作台展示 Agent 预警卡片；项目详情展示任务树生成建议
- **前置依赖**：P5-T1、P3-T2
- **验收标准**：
  - 项目待拆解时，Agent 可输出任务树建议（用户可一键采纳生成）
  - 任务到期前 3 天，工作台显示催办预警
  - 风险摘要准确率 > 50%（MVP 标准，允许误报）
- **技术要点**：
  - 催办预警使用规则引擎（基于 SLA 和 deadline），不依赖 LLM
  - 任务树生成建议调用 LLM（输入模板 + 项目特征）
  - 风险摘要使用规则 + LLM 增强

---

### P5-T4 验收质检 Agent

- **目标**：辅助验收人员进行资料完整性检查和检查项生成。
- **执行要点**：
  1. 触发节点：
     - 任务进入「待验收」→ 检查项生成建议
     - 验收申请提交前 → 资料完整性检查
     - 初验完成后 → 缺陷分级建议
  2. 输入上下文：任务数据 + 标准快照 + 已提交资料列表 + 历史验收数据
  3. Agent 能力：
     - 检查项生成：基于标准快照自动生成验收检查项清单
     - 资料完整性检查：比对必传资料清单，识别缺失项
     - 缺陷分级：基于缺陷描述自动建议严重程度（关键/一般/轻微）
  4. 输出形式：检查项列表 / 缺失资料清单 / 缺陷分级建议
  5. 前端：验收页面展示 Agent 生成的检查项和资料检查结果
- **前置依赖**：P5-T1、P3-T3
- **验收标准**：
  - 任务待验收时，检查项可自动/半自动生成
  - 资料完整性检查可识别 80% 以上的常见缺失（MVP 标准）
  - 缺陷分级建议可供参考，最终由人工确认
- **技术要点**：
  - 检查项生成优先使用规则引擎（标准快照 → 检查项映射）
  - 资料完整性检查使用规则匹配（文件名关键词）
  - 缺陷分级使用 LLM（输入描述 → 分级建议）

### P5-T5 工作台首页

- **目标**：建立统一工作台，让项目经理一屏掌握全局：待办、预警、消息、项目动态。
- **执行要点**：
  1. 新建 `WorkbenchPage`，挂载到 `#/workbench`
  2. 工作台模块：
     - **待办清单**：我的任务（待执行、待验收）、待审批（采购、结算）、待确认（Agent 建议）
     - **预警看板**：延期任务、高风险项目、资质到期、采购异常
     - **项目动态**：我参与的项目最近更新（状态变更、里程碑达成、风险产生）
     - **快捷入口**：新建项目、进入项目中心、进入任务中心
     - **统计卡片**：我的项目数、待办任务数、高风险项目数、本周到期里程碑数
  3. 数据聚合：从 projects、tasks、procurements、acceptance_nodes、agent_executions 等表聚合
  4. 后端新增 `GET /api/workbench/summary` 聚合接口
  5. 工作台为各角色的默认首页（路由 `#/` 重定向到 `#/workbench`）
- **前置依赖**：P5-T2、P5-T3、P5-T4（Agent 产生的待办和预警）
- **验收标准**：
  - 登录后默认进入工作台
  - 待办、预警、动态数据来自真实 API
  - 点击待办可直接进入对应处理页面
  - 预警项目显示风险等级和快捷处理按钮
- **技术要点**：
  - 工作台数据聚合查询较复杂，后端需优化（SQLite 下注意索引）
  - 使用 MUI `Card` / `Paper` / `List` 组件构建工作台布局
  - 预警规则 V1 先硬编码，V1.5 支持配置化

## 6. Phase 6：联调与闭环验证

### P6-T1 端到端状态机联调

- **目标**：验证项目、任务、采购、验收、结算、资产全链路状态流转的正确性。
- **执行要点**：
  1. 设计端到端测试场景：
     - 场景 A：标准店型完整建店（立项 → 拆解 → 执行 → 验收 → 结算 → 归档）
     - 场景 B：验收不通过整改闭环（初验不通过 → 整改 → 复验 → 通过）
     - 场景 C：采购异常阻塞任务（采购异常 → 任务阻塞 → 到货解除 → 任务恢复）
  2. 每个场景手动走通，记录状态流转日志
  3. 检查跨对象联动：
     - 项目状态变更是否正确触发任务/采购/验收动作
     - 任务状态变更是否正确更新项目进度
     - 验收结果是否正确影响任务和项目状态
  4. 修复状态机漏洞和联动缺失
  5. 编写端到端测试文档（供后续自动化测试参考）
- **前置依赖**：Phase 1~5 全部完成
- **验收标准**：
  - 场景 A/B/C 均可完整走通，无状态非法跳转
  - 状态流转日志完整记录（操作人、时间、原因、联动结果）
  - 所有发现的漏洞修复后回归通过
- **技术要点**：
  - 联调过程就是「找漏洞」的过程，不追求速度
  - 每发现一个漏洞，补充对应的单元测试
  - 重点检查「边界状态」（如整改中 → 已中止）

---

### P6-T2 结果对象模型验证

- **目标**：验证采购单、验收单、整改单、资产台账、结算建议单等结果对象的独立性和完整性。
- **执行要点**：
  1. 检查每个结果对象是否满足：
     - 有独立的数据表和 API
     - 有明确的状态机或生命周期
     - 可追溯来源（由哪个项目/任务/操作生成）
     - 支持查看详情和审计日志
  2. 检查结果对象间的关联：
     - 采购单 → 到货记录 → 资产台账
     - 验收单 → 整改单 → 复验记录
     - 结算建议 → 合同 → 采购单 → 验收单
  3. 检查一店一档完整性：
     - 项目归档时，校验资产、资料、结算是否齐全
     - 缺失项明确提示
  4. 补充缺失的关联字段和查询接口
- **前置依赖**：Phase 4 完成
- **验收标准**：
  - 每个结果对象可独立 CRUD
  - 结果对象间关联可查（正向 + 反向）
  - 一店一档归档校验通过
- **技术要点**：
  - 结果对象表需记录 `created_by`、`created_from`（来源对象类型和 ID）
  - 归档校验逻辑封装为独立函数，便于复用

### P6-T3 单品牌建店闭环演示

- **目标**：基于真实或高保真 mock 数据，完成一次完整的单品牌建店闭环演示。
- **执行要点**：
  1. 准备演示数据：
     - 1 个品牌（如「测试咖啡」）
     - 1 个标准店型（如「标准店 80㎡」）
     - 1 套标准包（装修验收标准 + 设备采购标准）
     - 1 个项目模板（含阶段、里程碑、任务树）
     - 2-3 个工队档案
     - 3-5 个供应商档案
  2. 按场景 A 完整走一遍系统：
     - 立项 → 确认 → 生成项目 → 任务分配 → 采购 → 执行 → 验收 → 结算 → 归档
  3. 录制演示脚本或截图文档
  4. 收集演示过程中的体验问题和功能缺口
  5. 形成「演示问题清单」，按优先级修复
- **前置依赖**：P6-T1、P6-T2
- **验收标准**：
  - 演示全程无阻断性 bug
  - 每个环节有数据可查（非空页面）
  - 状态流转正确，结果对象完整
  - 演示问题清单记录完整，P0/P1 问题修复后二次演示通过
- **技术要点**：
  - 演示数据使用 SQL 初始化脚本，便于重置和复现
  - 演示环境独立（可基于 SQLite 文件复制）

### P6-T4 数据初始化与 Mock 数据治理

- **目标**：建立可复用的数据初始化体系，区分「演示数据」、「测试数据」、「开发数据」。
- **执行要点**：

     - `seed_tasks.sql`
     - `seed_standards.sql`

- **前置依赖**：P6-T3
- **验收标准**：

  - 演示数据、测试数据、开发数据可独立切换
  - 种子数据脚本可重复执行（幂等）
- **技术要点**：
  - 种子数据使用 SQL `INSERT OR REPLACE` 保证幂等
  - 敏感信息（如真实地址）使用虚构数据

## 9. 风险与应对（工程视角）

| 风险项                                       | 影响                 | 应对策略                                                                           |
| -------------------------------------------- | -------------------- | ---------------------------------------------------------------------------------- |
| **AI 生成代码理解偏差**                      | 功能不符合预期，返工 | 每个任务输出后人工 Review；核心模块（状态机/domain/data）人工设计，AI 只做 UI 实现 |
| **Prisma + SQLite 迁移到 PostgreSQL 不兼容** | V2 迁移困难          | 严格使用 Prisma 标准语法，避免 SQLite 特有函数；迁移前做兼容性检查                 |
| **MUI v9 暗色玻璃态主题覆盖不全**            | 视觉不一致           | Phase 1 完成关键页面视觉验证；不通过则回退到自研组件                               |
| **Agent LLM 调用成本高/不稳定**              | 功能不可用           | V1 Agent 先用规则引擎兜底，LLM 作为增强；设置超时和降级策略                        |
| **单人开发上下文丢失**                       | 跨 Phase 需求遗忘    | 每 Phase 结束更新本文档和 MEMORY.md；关键决策写入代码注释                          |
| **技术债务清偿与功能开发冲突**               | 进度拖延             | 严格执行「先还债务再补功能」；Phase 1 不新增业务功能                               |
