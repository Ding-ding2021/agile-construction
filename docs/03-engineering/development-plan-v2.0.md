# 连锁门店营建管理系统 - 详细开发计划 V2.0

> **文档版本**: V2.0
> **适用场景**: 单人 + AI 编码模式（基于 shadcn/ui + `src-next/` 新栈）
> **计划性质**: 工程任务分解（按依赖关系排序）
> **当前基线**: Phase 1 + 1.5 已完成，Phase 2 待启动
> **核心原则**: 状态机驱动；标准先行；任务原子化；模块级测试覆盖

## 1. Phase 2：新功能开发

**目标**：基于 shadcn 新栈，按照"视图 → 详情 → 标准 → 数据"顺序交付 V1.0 MVP 核心功能。

**前置条件**：Phase 1 + 1.5 全部完成 ✅

**核心原则**：
- WBS 视图为先（甘特/网络图跨项目+标准复用）
- 项目详情 8 标签做功能容器
- 标准库为任务提供验收依据
- 任务数据硬化与标准绑定同步进行

---

### P2-T1 WBS 阶段 2 — 甘特图 + 网络图

- **目标**：在阶段 1 树视图基础上，实现甘特图和网络图视图，完成 WBS 四视图中的三个。
- **执行要点**：
  1. **甘特图实现**：基于 `@svar-ui/react-gantt` 开源核心版，支持拖拽调整日期、依赖连线、进度条、今天标记线
  2. **PRO 功能自建**：关键路径（CPM 算法） / 工作日日历（dayjs business-days） / 基线对比 / 撤销重做（Zustand temporal）
  3. **网络图实现**：基于 `@xyflow/react`（React Flow）+ dagre 自动布局，PERT 依赖图
  4. **三视图共享数据**：共用 WBS Store（Zustand），切换无同步成本
  5. **跨模块复用设计**：网络图同时为审批流程设计器、质检流程编排预留节点图基础设施
- **前置依赖**：WBS 阶段 1（已完成）
- **验收标准**：
  - 甘特图：日期拖拽 / 依赖连线 / 进度条 / 今天线 / 日周月粒度切换
  - 网络图：dagre 自动布局 / 节点状态色标 / 依赖线高亮
  - 三视图数据一致，切换无闪烁
  - 单元测试覆盖 CPM 算法、进度汇总
  - E2E 测试覆盖核心交互（拖拽、展开、编辑）
- **技术要点**：
  - 网络图节点基础设施预留审批流程/质检流程复用点
  - 甘特图 PRO 功能不自研替代，用开源核心 + 自建算法补齐
  - 网络图与树视图双向高亮联动

---

### P2-T2 项目详情 8 标签内容填充

- **目标**：将项目详情页从壳子变为可用，按 PMBOK 8 领域标签填充真实数据。
- **执行要点**：
  1. **项目概览**：状态总览、关键指标、快捷操作、变更日志
  2. **范围与任务**：WBS 树视图嵌入（复用阶段 1/2 产出）、范围基准、标准绑定展示
  3. **进度管理**：甘特图嵌入（复用 P2-T1）、里程碑列表、计划 vs 实际对比
  4. **成本与采购**：采购申请摘要、订单列表、预算 vs 实际
  5. **质量与验收**：验收申请状态、检查项完成度、整改任务摘要
  6. **资源与人员**：项目成员列表、工队分配入口
  7. **风险与沟通**：风险登记册、问题日志
  8. **设置与 Agent**：项目配置、编码规则、Agent 开关（Agent 填充归入后续 Phase）
- **前置依赖**：P2-T1（WBS 标签占据范围+进度两个核心标签）
- **验收标准**：
  - 8 个标签可正常切换，URL 同步更新
  - 每个标签有独立加载状态和空态
  - 范围与进度标签嵌入真实 WBS 视图
  - 数据来自 API 非 mock
  - `npm run build` 零报错
- **技术要点**：
  - 各标签独立数据加载，按需请求
  - 标签切换不触发状态变化（状态机与标签正交）
  - 权限控制预留：不同角色可见不同标签

---

### P2-T3 标准管理基础

- **目标**：建立标准库完整体系 — 标准来源、条款结构化、规则项引擎。
- **执行要点**：
  1. **后端实体**：`standards` 表 + `standard_files` 表 + RESTful API
  2. **条款结构化**：新增 `standard_clauses` 表，条款编号、标题、内容、类型（执行/验收/通用）
  3. **规则项引擎**：新增 `standard_rules` 表，关联条款，判定方式（是/否、数值范围、枚举）
  4. **前端页面**：`StandardManagementPage` 重构为 shadcn，标准列表 + 详情 + 条款管理
  5. **标准来源类型**：品牌标准、行业规范、供应商要求、项目补充
  6. **标准绑定**：任务模板可绑定标准条款，生成快照
- **前置依赖**：P2-T2（或并行）
- **验收标准**：
  - 可创建/编辑/删除标准来源
  - 标准文件可拆分为结构化条款
  - 条款可细化为规则项
  - 标准列表可按品牌、店型、来源类型筛选
  - 规则项可与任务模板绑定
- **技术要点**：
  - 规则引擎 V1 用文本描述 + 判定类型枚举，不做表达式解析
  - 标准表预留 `standard_package_id` 供模板中心引用
  - AI 辅助条款提取（调用 LLM 切分标准文本，人工确认入库）

---

### P2-T4 任务数据硬化

- **目标**：将任务模块从"演示级 mock"升级为有真实数据结构、状态机守卫、父子关系的可独立工作单元。
- **执行要点**：
  1. **字段精简与硬化**：任务字段精简至 ≤28 个核心字段，日期/金额/数量字段改为数值类型
  2. **父子关系真实化**：`parent_task_id` 自引用外键 + `tree_level`/`tree_path`（Materialized Path）
  3. **标准绑定真实化**：`executionStandards`/`acceptanceStandards` 从 `string[]` → `standardClauseIds[]`
  4. **状态机守卫真实化**：守卫条件从 mock → 数据库查询
  5. **任务关系建模**：`task_relations` 表（前置依赖）+ 整改派生 + 派单流程
  6. **检查项闭环**：检查项独立实体化，录入/判定/整改自动派生
  7. **API 完善**：CRUD + 子树查询 + 依赖查询 + 批量创建
- **前置依赖**：P2-T1、P2-T2（或并行）
- **验收标准**：
  - 任务父子关系通过 `parent_task_id` 真实关联
  - 6 状态流转 + 守卫条件全部真实（非 mock）
  - 前置依赖可创建/查询/影响流转
  - 验收不通过 → 整改任务自动派生
  - 检查项独立实体化，录入/判定/整改闭环
  - 单元测试覆盖全部守卫和流转路径
- **技术要点**：
  - 任务数据硬化需与标准管理配合（标准 ID 引用）
  - Materialized Path 格式 `/root_id/level1_id/level2_id/`
  - 派单流程（待派单/已派单/已接单/已拒绝）独立状态维度

## 2. Phase 3+：后续阶段框架

以下各阶段保留框架定义，具体任务分解在 Phase 2 推进过程中细化。

### Phase 3：模板中心

**目标**：建立项目模板和任务模板体系。模板 = 预定义的任务树 + 默认标准绑定 + 默认工期。

**前置条件**：Phase 2 任务数据硬化完成 + 标准管理可用

| 任务 | 说明 |
|------|------|
| P3-T1 | 项目模板 CRUD（名称、品牌、店型、阶段、里程碑） |
| P3-T2 | 任务模板树 + 标准绑定 + 快照生成 |
| P3-T3 | 模板实例化（选择模板 → 批量生成任务树 → 绑定标准快照） |

### Phase 4：采购管理

**目标**：实现采购申请 → 审批 → 下单 → 到货跟踪的完整流程。

**前置条件**：Phase 2 任务数据可用

| 任务 | 说明 |
|------|------|
| P4-T1 | 采购实体 + 状态机（待申请→待审批→待下单→待到货→已到货） |
| P4-T2 | 采购申请与任务联动（从任务详情页发起） |
| P4-T3 | 到货确认后解除任务采购阻塞 |

### Phase 5：工队管理

**目标**：建立工队档案、班组结构、资质证照管理，支持任务派单时选择工队。

**前置条件**：Phase 2 任务派单流程可用

| 任务 | 说明 |
|------|------|
| P5-T1 | 工队档案 + 班组结构 + 资质证照 |
| P5-T2 | 任务派单联动（可选择工队 + 负载展示） |
| P5-T3 | 工队评级手动录入（优秀/良好/一般） |

### Phase 6：Agent 与工作台

**目标**：建立 Agent 节点框架 + 工作台首页。

**前置条件**：Phase 2~5 核心数据就绪

| 任务 | 说明 |
|------|------|
| P6-T1 | Agent 节点框架（触发节点、输入输出、人工确认、执行记录） |
| P6-T2 | 品牌需求 Agent（立项补全 + 模板匹配） |
| P6-T3 | 项目经理 Agent（催办预警 + 风险摘要） |
| P6-T4 | 验收质检 Agent（检查项生成 + 资料完整性检查） |
| P6-T5 | 工作台首页（待办、预警、项目动态、快捷入口） |

### Phase 7：联调与闭环验证

**目标**：验证端到端流程可执行，修复跨模块问题。

**前置条件**：Phase 2~6 完成

| 任务 | 说明 |
|------|------|
| P7-T1 | 端到端状态机联调（立项→拆解→执行→验收→结算→归档） |
| P7-T2 | 结果对象模型验证（采购单/验收单/整改单/资产独立性） |
| P7-T3 | 单品牌建店闭环演示 |
| P7-T4 | 数据初始化治理（种子数据体系） |

## 4. 质量门禁

每 Phase 结束必须执行：

1. `npm run build` 零报错
2. `npm run lint` 零报错（0 errors）
3. 当前 Phase 新增功能可手动走通
4. 迁移模块 E2E 测试覆盖核心用户流程
