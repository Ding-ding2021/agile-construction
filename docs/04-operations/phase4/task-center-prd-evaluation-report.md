## 二、逐章评估

### 第 1 章：模块概述 ✅ 良好

**优点**：

- 模块定位清晰，明确区分了项目管理、任务中心、标准管理、验收整改的边界
- V1 范围界定合理，排除了复杂排程引擎、跨品牌任务市场等超纲内容

**建议**：

- 第 1.3 节 V1 范围中提到"项目经理 Agent / 调度 Agent 辅助能力"，但后续章节缺少 Agent 能力的详细设计（仅在 8.5 节有权限说明），建议补充 Agent 介入的具体触发点和能力边界

---

### 第 2 章：角色与场景 ✅ 良好

**优点**：

- 6 个核心角色定义清晰，职责边界明确
- 7 个业务场景覆盖了任务中心的主要闭环

**建议**：

- 场景 7（管理者多视图查看）中提到的"任务关系图"在后续页面章节中没有详细设计，V1 是否实现需要明确

### 第 4 章：任务结构设计 ✅ 良好

**优点**：

## 五、文档改进建议

### 5.1 立即修改（影响理解与实现）

1. **统一层级口径**：删除 `task_level` 字段或将其取值限制为 1~4，与 `node_level_type` 保持一致
2. **统一状态字段名**：全文统一使用 `status` 或 `task_status`，不要混用
3. **统一来源字段**：`source_type` 与 `origin_type` 合并或明确区分语义
4. **补充"待验收 → 退回补充"流转**：在 7.4 异常路径和 `TASK_STATUS_TRANSITION_MAP` 中补充此流转

### 5.2 建议补充（提升完整性）

1. **Agent 能力设计**：补充项目经理 Agent 和调度 Agent 的具体能力清单和触发条件
2. **整改任务循环限制**：明确最大整改次数或升级规则
3. **标准重新匹配流程**：补充"未执行任务可重新匹配最新标准"的触发条件和操作步骤
4. **关键路径风险计算规则**：定义延期风险的具体计算逻辑

### 5.3 建议拆分（控制 V1 范围）

1. **字段配置与自定义字段**（附表 9~10）建议移到 V1.5 或 V2
2. **多项目集/项目组合字段**（`program_id`, `portfolio_id`）如 V1 不支持，可从字段表中移除或标注为 V2

## 六、实施优先级建议

基于 PRD 与代码的差距，建议按以下优先级推进：

### 第一阶段（本周）：补齐 P0 字段 + 修正口径

- 补齐 `priority`, `executionStandardIds`, `acceptanceStandardIds`, `is_rectification`, `derived_from_task_id` 等 P0 字段
- 修正 `executionStandards` / `acceptanceStandards` 存 ID 而非文本的口径
- 补充"待验收 → 退回补充"状态流转
- 完善守卫条件中的资料完整性和检查项校验

### 第二阶段（2 周内）：核心闭环能力

- 实现标准绑定与快照生成逻辑
- 实现派单/接单/改派流程（待派单池）
- 实现整改任务派生闭环
- 补充基础权限控制（品牌隔离、资源方隔离）

### 第三阶段（1 个月内）：体验与监控

- 执行清单与检查项模板生成
- 催办记录与阻塞处理闭环
- SLA 实时计时与预警
- 任务关系视图与日历视图完善
- 埋点事件与核心指标统计
