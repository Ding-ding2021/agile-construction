# 任务管理模块专项评估报告

> 评估日期：2026-04-23
> 评估范围：任务中心（Task Center）全部前端实现
> 对照基准：`docs/01-product/task-center-prd.md` V1.01

---

## 一、总体结论

**任务管理模块完成度：约 55-60%**

核心闭环（列表 → 详情 → 状态流转 → 派单 → 整改派生）已跑通，但处于**"前端演示态"**向**"业务真实态"**过渡阶段。状态机框架、筛选管道、多视图切换等基础能力扎实，但任务树层级模型、标准库联动、父子任务汇总、真实依赖关系等关键底座尚未落地。

### 2.2 状态机与守卫条件（PRD §7）

| 检查项               | 状态    | 说明                                                          |
| -------------------- | ------- | ------------------------------------------------------------- |
| 9 个任务状态定义     | ✅ 完成 | `TaskStatus` 联合类型 + `TASK_STATUS_OPTIONS` 完整            |
| 状态流转映射表       | ✅ 完成 | `TASK_STATUS_TRANSITION_MAP` 覆盖全部合法流转                 |
| 主流程下一状态推导   | ✅ 完成 | `TASK_PRIMARY_FLOW_NEXT_STATUS` 已定义                        |
| 待分配 → 待执行 守卫 | ✅ 实现 | 校验：已分配 owner、前置依赖、标准绑定、快照生成              |
| 待执行 → 执行中 守卫 | ✅ 实现 | 校验：执行主体确认、无阻塞前置、任务非阻塞                    |
| 执行中 → 待提交 守卫 | ⚠️ 部分 | 仅校验进度 ≥80%，**必传资料检查 TODO**                        |
| 待提交 → 待验收 守卫 | ⚠️ 部分 | 校验进度 ≥90%、标准绑定、快照；**资料完整性在 UI 层额外校验** |
| 待验收 → 已完成 守卫 | ✅ 实现 | 校验进度 =100%、非阻塞、标准绑定                              |
| 待验收 → 不通过 守卫 | ❌ 缺失 | PRD 要求"关键不合格项/必传资料缺失/阈值未满足"，当前无守卫    |
| 批量守卫校验         | ✅ 实现 | `validateBatchStatusTransition` 已提供                        |
| 可用下一状态列表     | ✅ 实现 | `getAvailableNextStatuses` 结合守卫返回允许/禁止及原因        |
| 守卫单元测试         | ✅ 完成 | 覆盖 5 个主要流转路径，共 14 个断言                           |

**关键问题**：

1. **前置任务推断为简化逻辑**：`getPredecessorTasks` 基于 `parentPath` 和数组索引推断前置关系，而非真实的 `task_relation` 表查询。这导致守卫中的前置依赖校验**在复杂场景下不可靠**。
2. **资料完整性校验分裂**：守卫层和 UI 层各自实现了部分校验，存在重复和口径不一致风险。

## 三、代码质量评估

### 3.1 优势

1. **状态管理清晰**：`TaskManagementPage.tsx` 作为单一数据源，通过 `useState` + `useCallback` 管理任务状态、日志、附件，职责集中。
2. **筛选管道可测试**：`taskManagement.selectors.ts` 纯函数设计，`filter → search → advancedFilter → sort → paginate` 管道清晰，已配单元测试。
3. **守卫与业务逻辑分离**：`taskStateMachine.guards.ts` 独立模块，纯函数设计，易于测试和扩展。
4. **数据层抽象**：`taskRepository.ts` 提供 localStorage + serverAdapter 双写能力，带 schemaVersion 兼容。
5. **模板实例化链路打通**：从项目模板 → 任务种子 → 任务实例的链路已对接 `template-instantiation.ts`。

### 3.2 技术债务

| 债务项                                     | 位置                                | 严重程度 | 说明                                                                    |
| ------------------------------------------ | ----------------------------------- | -------- | ----------------------------------------------------------------------- |
| `TaskSidebar` 为独立实现                   | `TaskSidebar.tsx`                   | 中       | 未使用共享 `AppSidebar`，导航逻辑重复，高耦合 hash 匹配                 |
| 父子任务为平级数据                         | `taskManagement.data.ts`            | **高**   | `parentTaskId` 始终 null，`parentPath` 为展示字符串，无法支撑真实树操作 |
| 前置任务推断简化                           | `taskStateMachine.guards.ts`        | **高**   | `getPredecessorTasks` 基于数组索引推断，复杂场景不可靠                  |
| 标准内容硬编码                             | `taskManagement.data.ts`            | **高**   | `buildTaskDetail` 中执行/验收标准、检查项、关系均为 mock                |
| 详情页 SVG 内联                            | `TaskDetailPage.tsx`                | 低       | 15 个 SVG 组件内联，应提取为图标组件                                    |
| 风险联动规则硬编码                         | `rejectWithRectification`           | 中       | 整改时同项目低风险→中风险的联动写死在组件内                             |

## 四、后续开发建议

### 4.1 优先级矩阵

```
            高业务价值
                 ↑
    ┌────────────┼────────────┐
    │  P0: 子任务│  P0: 标准库 │
    │  层级模型  │  联动      │
    ├────────────┼────────────┤
    │  P1: 依赖  │  P1: 待派单│
    │  关系视图  │  池独立页  │
    ├────────────┼────────────┤
    │  P2: 日历  │  P2: 自定义│
    │  视图增强  │  字段      │
    └────────────┴────────────┘
                 ↓
            低业务价值
    低技术难度 ←────→ 高技术难度
```

### 4.2 具体建议（按优先级排序）

#### P0 — 必须在 V1 完成（2-3 周）

**1. 建立真实的父子任务层级模型（5-7 天）**

- 修改 `TaskItem` 类型，强制 `parent_task_id` 字段（根节点可为 null）
- 重构 `mockTasks` 数据，建立真正的父子关系（至少 1 个项目含 2-3 个工作包，每个工作包含 2-4 个任务）
- 实现子任务进度回写父任务进度的汇总逻辑（PRD §4.3）
- 重构 `buildTaskTreeViewModel`，从 `parent_task_id` 构建树而非 `parentPath` 字符串拆分
- **阻塞点**：此改动影响 `TaskListView`（列表需支持树形展开）、`taskStateMachine.guards`（前置任务查询需真实关系）、`TaskTreeView`（已基于树模型，影响较小）

**2. 标准库联动与快照真实化（5-7 天）**

- 对接 `standard-template.data.ts` 中的真实标准模板数据
- 修改 `buildTaskDetail`：根据 `execution_standard_ids`/`acceptance_standard_ids` 查询标准内容，替代硬编码
- 实现标准快照生成逻辑：在状态流转到"待执行"时，将当前绑定的标准 ID + 版本 + 规则项固化到快照对象
- 修改检查项：从标准清单模板生成真实检查项列表，替代硬编码 3 项
- **阻塞点**：需要标准管理模块提供标准查询 API（当前 `standard-template.data.ts` 有基础数据，可先用）

**3. 守卫层前置任务真实化（2-3 天）**

- 在 `taskStateMachine.guards.ts` 中实现基于 `task_relation`（或至少 `parent_task_id` + 排序）的真实前置任务查询
- 移除 `getPredecessorTasks` 的数组索引推断逻辑
- 补充"待验收 → 不通过"的守卫条件

#### P1 — V1 建议完成（1-2 周）

**4. 待派单任务池独立页面（3-4 天）**

- 新建 `DispatchPoolPage.tsx`，专门面向调度平台运营
- 展示待派单任务列表 + 资源推荐结果 + 派单/改派/撤回操作
- 接入 `buildDispatchRecommendations` 已有算法

**5. 任务关系/依赖视图（3-4 天）**

- 基于现有 `TaskRelation` 类型，扩展只读依赖链路展示
- 在任务详情页增加"上下游依赖链"折叠面板
- 展示前置未完成项、整改链路、关联结果对象

**6. 执行清单交互化（2-3 天）**

- 支持在任务详情页勾选/取消勾选检查项
- 勾选状态回写到任务数据，联动进度计算
- 进入"待验收"前校验必传资料（检查项完成度 + 附件完整性）

**7. 父子任务汇总联动（2-3 天）**

- 子任务状态变更时，自动重算父任务进度和风险等级
- 父任务关闭时提示影响范围

#### P2 — V1 可选或 V2（1 周）

**8. 日历视图增强（2-3 天）**

- 月/周网格视图、冲突排期检测、关键节点高亮

**9. 批量操作（2-3 天）**

- 列表多选 + 批量分配/催办/调整时间/导出

**10. 技术债务清理（持续）**

- 删除 `TaskSidebar.tsx`，统一使用 `AppSidebar`
- 提取 SVG 图标组件

## 六、总结

任务管理模块的**骨架扎实**——状态机、筛选管道、多视图、派单闭环、整改派生等核心能力已具备。当前最大的三个缺口是：

1. **父子任务层级为假**：`parentPath` 字符串展示 ≠ 真实树形数据，导致任务树、进度汇总、依赖分析都建立在沙土之上。
2. **标准联动为假**：标准内容和检查项全部硬编码，"标准快照"只是字符串 ID。
3. **前置依赖为假**：守卫中的前置任务检查基于数组索引推断，复杂场景不可靠。

**建议接下来的开发节奏**：

- **第 1 周**：重构任务数据为真实父子层级（影响面大，越早做越好）
- **第 2 周**：对接标准库真实数据 + 快照生成
- **第 3 周**：待派单池独立页 + 依赖关系视图 + 执行清单交互化

这样可以在 3 周内让任务管理模块从"演示态"进入"业务可用态"。
