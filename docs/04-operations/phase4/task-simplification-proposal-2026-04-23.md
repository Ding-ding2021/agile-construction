# 任务管理模块简化建议书

> 日期：2026-04-23
> 背景：任务字段、状态、页面设计均显冗余，需一次系统性瘦身

---

## 一、当前问题诊断

### 1.1 字段层面：50+ 字段，层级混乱

| 问题                   | 现状                                                              | 影响                                     |
| ---------------------- | ----------------------------------------------------------------- | ---------------------------------------- |
| 字段总量过大           | PRD 定义 50+ 字段，代码中 TaskItem 约 40 个字段                   | 前端对象臃肿，mock 数据难以维护          |
| 必选 vs 可选边界不清   | 大量字段标记为 `?` 可选                                           | 类型安全弱，使用时需反复判空             |
| 展示字段与数据字段混用 | `parentPath` 是展示字符串，`parentTaskId` 始终为 null             | 数据结构不真实，父子层级靠字符串拆分聚合 |
| 投影字段与主字段并存   | `slaStatus`、`standardBindingStatus` 等是计算投影，却与主数据平级 | 职责不清，易引发数据同步问题             |
| 财务/合同字段过早引入  | `costAccountId`、`contractId`、`serviceId` 等已在接口中           | V1 尚未跑通主流程，财务口径不应前置      |

### 1.2 状态层面：9 个状态 + 多条分支，过于细碎

当前 9 个状态流转图：

```
待创建 → 待分配 → 待执行 → 执行中 → 待提交 → 待验收 → 已完成
                           ↓           ↓
                         已关闭      不通过 → 待执行（整改回路）
```

问题：

- **"待创建"状态存疑**：任务如果不存在，不应出现在任务列表；如果存在，就不该是"待创建"
- **"待提交"过于细分**：执行中到待验收之间增加了一个中间态，实际业务中执行完成即提交
- **"不通过"是结果，不是状态**：不通过应当产生一个结果标记（验收结论），任务本身回到"执行中"或进入整改
- **状态与派单状态并行**：`dispatchStatus`（unassigned/assigned/accepted/rejected）与主状态交织，职责重叠

### 1.3 页面层面：信息过载，层次不清

#### 任务列表（13 列）

当前列表展示：任务名称 / 父任务路径 / 项目 / 来源 / 任务类型 / 状态 / 风险等级 / 负责人 / 计划时间 / SLA状态 / 前置任务状态 / 催办次数 / 标准绑定

问题：

- 13 列在常规屏幕（1920px）下单列平均仅 ~140px，文字 truncation 严重
- "来源"、"任务类型"、"标准绑定"对日常操作优先级低
- "催办次数"、"前置任务状态"更适合以图标/徽标形式呈现，而非独立列

#### 任务详情页（8 个折叠区 + 2 个 Tab）

当前详情页区块：

1. 执行与派单（含派单选择、行内编辑、资源推荐）
2. 执行与验收标准
3. 待办检查项
4. 资料附件（含提交记录）
5. 任务关联
6. 任务概览
7. 催办与风险
8. 审计记录
9. Tab 切换：任务详情 / 流转时间轴

问题：

- 区块过多，用户进入详情页后需要大量点击展开才能看到全貌
- "任务概览"与"催办与风险"中的信息在列表已部分展示，重复呈现
- "审计记录"与"流转时间轴"内容高度重叠（都是按时间排序的操作记录）
- 行内编辑表单嵌入详情页，与"执行与派单"混在一起，职责不清

#### 工具栏（7 个操作入口）

视图切换（4 种）+ 搜索 + 分组 + 筛选（6 个维度）+ 排序 + 新建任务 + 更多

问题：筛选维度过多，实际高频使用的可能只有"状态"和"负责人"；视图切换 4 种在 MVP 阶段可能过度设计。

## 二、简化方案

### 2.1 字段精简：从 40+ 减到 18 个核心字段

#### 核心字段（任务列表 + 详情页头必须展示）

| 字段                 | 说明                        | 必填 |
| -------------------- | --------------------------- | ---- |
| `taskId`             | 任务唯一 ID                 | 是   |
| `taskCode`           | 任务编码                    | 是   |
| `taskName`           | 任务名称                    | 是   |
| `projectId`          | 所属项目 ID                 | 是   |
| `parentTaskId`       | 父任务 ID（null 表示顶层）  | 是   |
| `taskStatus`         | 任务状态（见 2.2 简化后）   | 是   |
| `assigneeId`         | 执行人 ID                   | 否   |
| `assigneeName`       | 执行人姓名（展示用）        | 否   |
| `plannedStartAt`     | 计划开始时间                | 是   |
| `plannedEndAt`       | 计划结束时间                | 是   |
| `actualStartAt`      | 实际开始时间                | 否   |
| `actualEndAt`        | 实际结束时间                | 否   |
| `priority`           | 优先级：低/中/高/紧急       | 是   |
| `progress`           | 进度 0-100                  | 是   |
| `taskType`           | 任务类型                    | 是   |
| `sourceType`         | 来源方式                    | 是   |
| `standardSnapshotId` | 标准快照 ID                 | 否   |
| `blockedReason`      | 阻塞原因（null 表示未阻塞） | 否   |

#### 移除或推迟到 V2 的字段

| 移除字段                                    | 理由                                               |
| ------------------------------------------- | -------------------------------------------------- |
| `parentPath`                                | 用 `parentTaskId` + 树形查询替代                   |
| `programId`、`portfolioId`                  | V1 无项目集/组合管理                               |
| `nodeLevelType`                             | 由 `parentTaskId` 是否为 null + 是否有子任务可推导 |
| `workPackageId`、`costAccountId`            | 财务口径 V2 再引入                                 |
| `contractId`、`serviceId`、`vendorId`       | 外包流程 V2 再引入                                 |
| `slaRuleId`                                 | SLA 规则由系统配置，任务层只存计算结果             |
| `dispatchStatus`                            | 与主状态合并（见 2.2）                             |
| `ownerRole`                                 | V1 简化为"负责人"单一概念                          |
| `closeReason`、`reopenCount`                | V1 暂不支持重新打开                                |
| `riskLevel`                                 | 与 `priority` 合并，保留优先级                     |
| `predecessorStatus`                         | 实时计算，不作为存储字段                           |
| `standardBindingStatus`                     | 由 `standardSnapshotId` 是否为空推导               |
| `snapshotStatus`                            | 由 `standardSnapshotId` 推导                       |
| `templateCode`、`templateVersion`           | 追溯字段放扩展表                                   |
| `remindCount`                               | 实时统计，不存主表                                 |
| `isBlocked`                                 | 由 `blockedReason` 是否为空推导                    |
| `assigneeType`                              | V1 简化为内部/外部标签，不单独字段                 |
| `executionStandards`、`acceptanceStandards` | 详情页从标准库实时查询，不冗余存储                 |
| `checklist`                                 | 独立 check_list 表，不嵌套                         |
| `relations`                                 | 独立 task_relation 表                              |
| `flowLogs`、`auditLogs`、`reminders`        | 独立日志表，不嵌套                                 |
| `submissions`                               | 独立提交记录表                                     |

#### 子实体独立建表（不嵌套在 Task 中）

```
Task（精简后 18 字段）
├── TaskChecklist（独立表，1:N）
├── TaskRelation（独立表，N:M）
├── TaskAttachment（独立表，1:N）
├── TaskFlowLog（独立表，1:N，按时间排序）
└── TaskSubmission（独立表，1:N）
```

### 2.2 状态精简：从 9 个减到 6 个

#### 新状态定义

| 状态       | 含义                       | 可操作动作                                     |
| ---------- | -------------------------- | ---------------------------------------------- |
| **待分配** | 任务已创建，尚未指派执行人 | 指派负责人 / 编辑任务 / 删除                   |
| **待执行** | 已指派，等待执行方开始     | 开始执行 / 改派 / 编辑计划时间                 |
| **执行中** | 执行方已开始工作           | 更新进度 / 上传附件 / 标记完成 / 上报阻塞      |
| **待验收** | 执行完成，等待验收         | 通过（→已完成）/ 不通过（→执行中，带整改标记） |
| **已完成** | 验收通过，任务结束         | 仅查看 / 下载归档资料                          |
| **已关闭** | 异常终止或取消             | 仅查看                                         |

#### 变化说明

| 原状态                   | 处理方式                                                                            |
| ------------------------ | ----------------------------------------------------------------------------------- |
| **待创建**               | ❌ 移除。任务一旦进入系统，至少为"待分配"                                           |
| **待提交**               | ❌ 移除。执行完成即自动进入"待验收"，中间无需手动提交                               |
| **不通过**               | ❌ 移除作为独立状态。改为"执行中"状态 + `rectificationFlag` 标记                    |
| **待执行**               | 保留，但合并原"待分配→待执行"的接单确认流程                                         |
| **执行中→待验收**        | 执行方标记完成即可，无需单独的"提交"动作                                            |
| **待验收→不通过→执行中** | 验收不通过时，任务回到"执行中"，并自动打上 `rectificationFlag=true`，生成整改子任务 |

#### 流转图（简化后）

```
待分配 → 待执行 → 执行中 → 待验收 → 已完成
   ↓         ↓        ↓          ↓
 已关闭   已关闭   已关闭     不通过 → 执行中（整改）
```

#### 派单状态合并到主状态

原 `dispatchStatus`（unassigned/assigned/accepted/rejected）不再独立存在：

- `待分配` = unassigned
- `待执行` = assigned（等待接单确认）
- 执行中/待验收/已完成 = accepted
- 拒单 = 任务回到 `待分配` + 记录拒单原因

### 2.3 页面简化：列表 6 列 + 详情 4 区块

#### 任务列表：从 13 列 → 6 列

| 列              | 宽度 | 说明                                 |
| --------------- | ---- | ------------------------------------ |
| 任务名称 + 编码 | 25%  | 名称为主，编码小号灰色副标题         |
| 项目            | 12%  | 所属项目名称                         |
| 状态            | 10%  | 彩色状态徽标（文字精简）             |
| 负责人          | 10%  | 头像 + 姓名                          |
| 计划时间        | 15%  | 04/20 ~ 04/25 格式                   |
| 优先级          | 8%   | 高/中/低 彩色圆点 + 文字             |
| 操作            | 8%   | 催办 / 查看 / 状态推进（主操作外露） |

**移除的列**：父任务路径（移到详情页）、来源（移到筛选条件）、任务类型（移到详情页）、风险等级（与优先级合并）、SLA状态（用时间条+颜色表达）、前置任务状态（用阻塞图标表达）、催办次数（用红点徽标表达）、标准绑定（用图标表达）

#### 列表展示增强（非列化信息）

#### 工具栏简化

- **筛选**：只保留 状态 / 负责人 / 项目 三个高频维度，其他收起
- **排序**：保留，但默认"按优先级+截止时间"
- **新建任务**：保留
- **批量操作**：V1 暂不显示，有需求时再展开

**详情页头部操作按钮简化**：

简化后：根据状态动态显示唯一主操作 + 次要操作：

- `待分配` → 主操作：[指派负责人] 次要：[编辑] [删除]
- `待执行` → 主操作：[开始执行] 次要：[改派] [催办]
- `执行中` → 主操作：[标记完成] 次要：[更新进度] [上报阻塞] [上传附件]
- `待验收` → 主操作：[验收通过] 次要：[不通过并整改]
- `已完成/已关闭` → 仅 [查看归档]

## 三、实施建议

### 3.1 实施顺序

建议按以下顺序推进，避免一次性改动过大：

**第一阶段：字段与类型简化（1-2 天）**

1. 重写 `taskManagement.types.ts`，定义精简后的 `TaskItem`（18 字段）
2. 将子实体（checklist、attachment、relation、log）独立为接口，不嵌套
3. 同步更新 `taskManagement.data.ts` 中的 mock 数据

**第二阶段：状态机简化（1-2 天）**

1. 修改 `TaskStatus` 为 6 状态
2. 简化 `TASK_STATUS_TRANSITION_MAP`
3. 更新 `taskStateMachine.guards.ts`，移除与移除状态相关的守卫
4. 更新 `TaskManagementPage.tsx` 中的状态流转逻辑

**第三阶段：列表页简化（2-3 天）**

1. 重写 `TaskListView.tsx`，6 列布局 + 卡片化信息展示
2. 简化 `TaskToolbar.tsx`，减少筛选维度
3. 更新 `TaskStatsCards.tsx`，统计口径对齐 6 状态

**第四阶段：详情页重构（3-4 天）**

1. 重写 `TaskDetailPage.tsx`，4 区块 + 操作记录侧边栏
2. 移除 Tab 切换，改为单页滚动
3. 实现状态驱动的头部操作按钮
4. 将标准展示改为从标准库动态查询

### 3.2 回滚保障

- 所有改动在独立分支进行
- 保留原 `taskManagement.types.ts` 为 `taskManagement.types.v1.ts` 作为参考
- mock 数据同步维护新旧两套，直到新页面稳定

### 3.3 验收标准

- [ ] 任务列表在 1920px 屏幕下无需横向滚动
- [ ] 任务详情页首屏（不滚动）可见全部关键信息
- [ ] 从列表进入详情到完成一次状态推进，点击次数 ≤ 3 次
- [ ] 新用户无需培训即可理解状态含义
