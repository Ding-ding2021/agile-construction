# Spec: 项目经理 Agent "Harness + MCP + Skills" 架构设计

## Objective

**我们在构建什么？**

将 AI Agent 从"PRD 上的概念"落地为 **"基于 Harness 框架的项目交付中枢"**——Agent 自动分解项目、输出结构化产物（WBS 导图/甘特图/紧前关系图等）、人类直接操作修改、Agent 持续跟踪进度/成本/质量/风险。

**为什么？**

V1 产品定义（§2.3）的核心假设：

> 在受控流程节点中引入 AI Agent 辅助，可以降低营建管理的专业门槛，让一个营建主管能管更多项目。

当前 multi-agent PRD 设计了 7 个 Agent 的宏大蓝图，但实际代码为零。本架构采用务实策略：**先跑通一个 Agent（项目经理 Agent），验证 Harness + MCP + Skills 三层架构可行，其余 Agent 换角色配置和 Skills 即可复制**。

**成功标准：**

1. 项目经理 Agent 能根据项目模板自动展开任务树（WBS）
2. Agent 能构建任务紧前关系并生成甘特图
3. Agent 产物以可操作的前端界面呈现，人类可直接修改
4. Agent 持续跟踪进度/成本/质量/风险，异常时主动预警
5. Agent 架构可复制——换 yaml 配置 + Skills 列表即可生成新 Agent

**用户是谁？**

- 项目经理：触发项目分解、审阅 Agent 产物、修改后确认
- 平台运营：查看 Agent 生成的进度/成本/风险报告
- 资源方：接收 Agent 派发的材料和道具采购单

## New Architecture Design

### 总体架构：三层分离

```
┌──────────────────────────────────────────────────────────┐
│                    前端界面层（人类操作）                   │
│                                                          │
│  WBS导图  WBS表格  紧前图  甘特图  进度  成本  质量  风险  │
│  团队负载  沟通报告                                        │
│                                                          │
│     ▲ 人类通过 REST API 读写          ▲ Agent 通过 MCP 读写 │
│     │                                 │                   │
├─────┼─────────────────────────────────┼───────────────────┤
│     │        local-api 接口层          │                   │
│     │    /tasks/*  /standards/*        │                   │
│     │    /projects/*  /personnel/*     │                   │
│     │    /procurement/*                │                   │
├─────┼─────────────────────────────────┼───────────────────┤
│     │                                 │                   │
│  ┌──▼──────┐              ┌───────────▼──────────────┐    │
│  │ SQLite  │              │      Agent 运行时层        │    │
│  │         │              │                           │    │
│  │Project  │              │  ┌─────────────────────┐  │    │
│  │Task     │◄── 共享DB ──►│  │   Harness 7阶段引擎  │  │    │
│  │Relation │              │  └──────────┬──────────┘  │    │
│  │Template │              │             │              │    │
│  │Procure..│              │  ┌──────────▼──────────┐  │    │
│  └─────────┘              │  │    MCP 工具层 (12个)  │  │    │
│                           │  └──────────┬──────────┘  │    │
│                           │             │              │    │
│                           │  ┌──────────▼──────────┐  │    │
│                           │  │   Skills 技能库 (11个)│  │    │
│                           │  └─────────────────────┘  │    │
│                           │                           │    │
│                           │  角色配置: .yaml 文件      │    │
│                           │  (定义角色名/IPA/技能列表)  │    │
│                           └───────────────────────────┘    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**核心设计原则**：Agent 和人类共享同一套 API 和数据库。Agent 产出物落库后，人在界面上看到的就是可操作的数据；人修改后，Agent 通过 MCP 重新读取能感知变化。Agent 不是替代人，是帮人把草稿先写好。

### 1. 三层组件详解

| 层       | 组件             | 职责                                                                                                                 | 换 Agent 变什么 |
| -------- | ---------------- | -------------------------------------------------------------------------------------------------------------------- | :-------------: |
| 执行引擎 | Harness 7 阶段   | Align→Plan→Build→Test→Review→Deploy→Evolve，所有 Agent 共享                                                          |      不变       |
| 工具层   | MCP 工具（12个） | Agent 的手：读任务/标准/项目/人员，写任务/依赖/采购                                                                  | 工具集可能增减  |
| 能力层   | Skills（11个）   | Agent 的脑：项目分解、依赖构建、排程、进度分析、成本跟踪、质量控制、风险识别、团队协调、材料拆解、采购同步、报告撰写 |      全换       |

### 2. MCP 工具层

Agent 通过 MCP 协议统一调用各模块 API。每个工具封装一个业务意图，输入输出严格类型化。

#### 任务工具（task-tools.ts）

| 工具名                  | 对应 API                         | 说明                                           |
| ----------------------- | -------------------------------- | ---------------------------------------------- |
| `list_project_tasks`    | GET /projects/:code/tasks        | 查项目下所有任务（支持状态/优先级/到期日筛选） |
| `get_task_detail`       | GET /projects/:code/tasks/:id    | 查单个任务详情                                 |
| `create_task`           | POST /projects/:code/tasks       | 创建新任务                                     |
| `update_task`           | PUT /projects/:code/tasks/:id    | 更新任务字段                                   |
| `get_task_tree`         | GET /projects/:code/tasks/tree   | 获取项目任务树（含进度汇总）                   |
| `batch_create_tasks`    | POST /projects/:code/tasks/batch | 批量创建任务（模板实例化）                     |
| `create_task_relations` | POST /tasks/:id/relations        | 批量创建任务依赖关系                           |

#### 标准工具（standard-tools.ts）

| 工具名                 | 对应 API                   | 说明                    |
| ---------------------- | -------------------------- | ----------------------- |
| `search_standards`     | GET /standards             | 按关键词/业务域搜索标准 |
| `get_standard_clauses` | GET /standards/:id/clauses | 查某标准下的条款列表    |

#### 项目工具（project-tools.ts）

| 工具名             | 对应 API            | 说明           |
| ------------------ | ------------------- | -------------- |
| `get_project_info` | GET /projects/:code | 查项目基本信息 |

#### 人员工具（personnel-tools.ts）

| 工具名                   | 对应 API               | 说明               |
| ------------------------ | ---------------------- | ------------------ |
| `list_available_workers` | GET /personnel/workers | 查可用执行人员列表 |

#### 采购工具（procurement-tools.ts）

| 工具名                     | 对应 API                 | 说明                   |
| -------------------------- | ------------------------ | ---------------------- |
| `create_procurement_order` | POST /procurement/orders | 创建采购单（关联任务） |

### 3. Skills 技能库

11 个 Skill 覆盖项目管理 8 大知识领域，通过 MCP 调用 API 实现读写。

```
src-next/agent/skills/
│
├── 范围管理 ──────────────────────────
├── project-decomposer.ts       ← 项目分解：模板展开→任务树（WBS导图+表格）
├── dependency-builder.ts       ← 依赖构建：模板蓝图→TaskRelation（紧前关系图）
│
├── 时间管理 ──────────────────────────
├── schedule-planner.ts         ← 排程计划：紧前关系+工期→推算时间（甘特图）
│
├── 成本管理 ──────────────────────────
├── cost-tracker.ts             ← 成本跟踪：工时偏差 + 采购金额汇总
│
├── 质量管理 ──────────────────────────
├── quality-monitor.ts          ← 质量监控：验收通过率 + 整改趋势 + 标准合规
│
├── 资源/团队管理 ──────────────────────
├── team-coordinator.ts         ← 团队协调：工作量分布 + 超载预警 + 空闲识别
│
├── 风险管理 ──────────────────────────
├── risk-identifier.ts          ← 风险识别：进度/资源/采购/质量 四维风险扫描
│
├── 采购管理 ──────────────────────────
├── material-planner.ts         ← 材料拆解：模板预设 + AI 补充→材料清单
├── procurement-syncer.ts       ← 采购同步：生成采购单 + 物流/生产状态跟踪
│
├── 沟通管理 ──────────────────────────
├── report-writer.ts            ← 沟通报告：数据聚合→周报/阶段摘要
│
├── 监控 ──────────────────────────────
├── progress-analyzer.ts        ← 进度分析：任务完成率 + 趋势 + 异常高亮
```

#### 3.1 项目分解 `project-decomposer.ts`

| 步骤 | 动作                                           | MCP 调用                        |
| ---- | ---------------------------------------------- | ------------------------------- |
| 1    | 读取项目信息                                   | `get_project_info`              |
| 2    | 读取关联的模板（项目模板→任务模板列表）        | `get_task_template`（模板 API） |
| 3    | 递归展开模板树：根节点→工作包→任务→子任务      | —（纯内存计算）                 |
| 4    | AI 补充动态材料：根据品牌/店型参数调整任务清单 | —（LLM 推理）                   |
| 5    | 批量写入 ProjectTask 树                        | `batch_create_tasks`            |

**输出**：WBS 思维导图 + WBS 树形表格（共享同一份 ProjectTask 数据，两种视图渲染）。

#### 3.2 依赖构建 `dependency-builder.ts`

| 步骤 | 动作                                                                |
| ---- | ------------------------------------------------------------------- |
| 1    | 读取模板的 `dependencyBlueprint`（JSON：前置任务→后置任务关系列表） |
| 2    | 将模板中的模板编码映射为落库后的实际任务编码                        |
| 3    | 批量写入 TaskRelation                                               |

**输出**：紧前关系图（TaskRelation 数据的可视化有向图）。

#### 3.3 排程计划 `schedule-planner.ts`

| 步骤 | 动作                                                                         |
| ---- | ---------------------------------------------------------------------------- |
| 1    | 读取任务树 + 紧前关系                                                        |
| 2    | 正向推算：从项目开始日期出发，根据紧前关系+工期推算每个任务的最早开始/结束日 |
| 3    | 反向校验：从截止日期倒推，检查是否可行                                       |
| 4    | 更新各任务的 `startDate` / `dueDate`                                         |

**输出**：甘特图（ProjectTask 时间字段的横道图渲染）。

#### 3.4 进度分析 `progress-analyzer.ts`

| 能力            | 实现                                             |
| --------------- | ------------------------------------------------ |
| 任务完成率      | 按工作包/项目汇总 `status='已完成'` 占比         |
| 趋势对比        | 对比计划进度 vs 实际进度（计划工期 vs 实际工期） |
| 异常高亮        | 超期任务、阻塞任务、无负责人任务                 |
| 服务协议（SLA） | 距截止日 1 天标临期，超截止日标超期              |

#### 3.5 成本跟踪 `cost-tracker.ts`

| 能力         | 实现                                                              |
| ------------ | ----------------------------------------------------------------- |
| 工时偏差     | 按工作包汇总 `actualWorkHours - plannedWorkHours`，偏差 >30% 标红 |
| 采购金额汇总 | 按项目汇总 ProcurementOrder 的 budgetAmount vs actualAmount       |

#### 3.6 质量监控 `quality-monitor.ts`

| 能力       | 实现                                              |
| ---------- | ------------------------------------------------- |
| 验收通过率 | 统计 TaskSubmission 的首次通过率（按工作包/项目） |
| 整改趋势   | 统计同一任务被退回 ≥2 次的清单，常见不通过原因    |
| 标准合规   | 对照绑定标准，标记从未被验证的条款                |
| 质量预警   | 同类型任务连续不通过超过阈值 → 标记               |

#### 3.7 团队协调 `team-coordinator.ts`

| 能力       | 实现                                                       |
| ---------- | ---------------------------------------------------------- |
| 工作量分布 | 按人统计当前执行中的任务数 + 总工时                        |
| 超载预警   | 任务数超过阈值 / 总工时超过容量 → 在甘特图上高亮           |
| 空闲识别   | 标记当前无任务、可分配的人员                               |
| 技能匹配   | 根据任务类型（taskType）推荐合适的执行人（ownerRole 匹配） |

#### 3.8 风险识别 `risk-identifier.ts`

| 风险维度 | 触发条件                                           | 严重度 |
| -------- | -------------------------------------------------- | :----: |
| 进度风险 | 一个任务阻塞导致 ≥3 个后续任务无法开始             |   🔴   |
| 进度风险 | 超过截止日 3 天未完成                              |   🟡   |
| 资源风险 | 必做任务处于待分配状态                             |   🔴   |
| 采购风险 | 采购状态 ≠ delivered 且关联任务的计划开始日 < 今天 |   🔴   |
| 质量风险 | 同一任务被退回 ≥2 次                               |   🟡   |
| 成本风险 | 实际工时超过计划工时 30%                           |   🟡   |

#### 3.9 材料拆解 `material-planner.ts`

| 步骤 | 动作                                                     |
| ---- | -------------------------------------------------------- |
| 1    | 从模板读取预设材料清单（`standardBinding` 中的材料引用） |
| 2    | AI 动态补充：根据品牌/店型参数推理额外材料               |
| 3    | 合并去重，生成材料清单草案                               |
| 4    | 项目经理审阅确认                                         |

**注意**：材料 = 采购管理模块（PRD-005）的 ProcurementOrder 实体。Agent 不重建采购系统，只做拆解和桥接。

#### 3.10 采购同步 `procurement-syncer.ts`

| 步骤 | 动作                                                               |
| ---- | ------------------------------------------------------------------ |
| 1    | 读取材料清单（审核通过的）                                         |
| 2    | 调用 `create_procurement_order` 创建采购单（填入 `taskCode` 关联） |
| 3    | 跟踪采购单状态（pending→ordered→delivered）                        |
| 4    | 对比采购预计交付日 vs 关联施工任务的计划开始日                     |
| 5    | 若不匹配 → 触发风险预警                                            |

#### 3.11 报告撰写 `report-writer.ts`

| 报告类型 | 内容来源                                      |
| -------- | --------------------------------------------- |
| 项目周报 | 进度分析 + 成本跟踪 + 质量监控 + 风险看板摘要 |
| 阶段摘要 | 按项目生命周期阶段提取关键指标                |
| 异常通报 | 风险识别输出 → 格式化为通知文本               |

### 5. Agent 与人类交互界面（7 个前端视图）

| #   | 界面         | 数据来源                       | Agent 产出                       | 人类操作                  |
| --- | ------------ | ------------------------------ | -------------------------------- | ------------------------- |
| 1   | WBS 思维导图 | ProjectTask.parentId           | 模板展开的完整任务树             | 拖拽调整层级、增删改名    |
| 2   | WBS 树形表格 | ProjectTask（全字段）          | 同一棵树，表格含编码/负责人/工期 | 内联编辑字段、批量调整    |
| 3   | 紧前关系图   | TaskRelation                   | 依赖蓝图自动连线                 | 拖拽连线/断线、改依赖类型 |
| 4   | 甘特图       | ProjectTask（时间字段）        | 推算的计划起止日期               | 拖拽调日期、拉长/缩短     |
| 5   | 进度跟踪     | ProjectTask + 服务协议计算     | 完成率统计 + SLA 标记            | 查看详情、催办            |
| 6   | 成本控制     | ProjectTask + ProcurementOrder | 工时偏差 + 采购金额汇总          | 查看偏差、调整预算        |
| 7   | 相关方沟通   | 上述全部数据聚合               | 周报/阶段摘要/异常通报           | 编辑后发送                |

**WBS 导图和树形表格是同一份 ProjectTask 数据的两种视图**，切换视图不产生数据冗余。

### 6. Agent 可复制性设计

换一个 Agent（如验收质检 Agent），只需：

```
1. 新建角色配置 yaml 文件（定义角色名、职责、输入输出）
2. 选择需要的 MCP 工具子集（验收 Agent 可能不需要采购工具）
3. 替换 Skills 列表（验收 Agent 用 quality-monitor + report-writer）
4. Harness 框架不变
```

**项目经理 Agent 的角色配置示例**：

```yaml
# .trae/agents/agent-project-manager.yaml
name: 项目经理 Agent
role: project_manager_agent
description: 辅助项目经理完成项目分解、排程、跟踪和风险预警
harness_phases:
  - Align
  - Plan
  - Build
  - Test
  - Review
  - Deploy
  - Evolve
mcp_tools:
  - task-tools
  - standard-tools
  - project-tools
  - personnel-tools
  - procurement-tools
skills:
  - project-decomposer
  - dependency-builder
  - schedule-planner
  - progress-analyzer
  - cost-tracker
  - quality-monitor
  - team-coordinator
  - risk-identifier
  - material-planner
  - procurement-syncer
  - report-writer
```

## Boundaries

### Always

- Agent 只能读写 API，永远不直接操作数据库
- Agent 通过 MCP 统一调用，不走直连 HTTP
- Agent 产物落库后状态为"草稿"，人类确认后才生效
- Agent 所有决策和输出留痕（TaskEventLog）
- MCP 工具输入输出严格类型化
- Agent 不替代人类决策——只出草稿，人类审阅

### Ask First

- 新增 MCP 工具
- 新增 Skill（尤其是涉及外部 API 调用的）
- 修改 Harness 阶段流程
- Agent 自动修改已确认的任务数据

### Never

- Agent 跳过人类审阅直接生效变更
- Agent 直接操作数据库
- Agent 绕过状态机守卫修改任务状态
- Agent 销毁或删除人类创建的数据
