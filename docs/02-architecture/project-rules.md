## 1. 规则目标与执行原则

### 1.1 目标

本规则用于统一本项目在**业务建模、页面实现、代码质量、交互体验、状态流转、Agent 治理、权限审计**等方面的执行口径，确保“需求可实现、实现可验收、问题可追溯”。

### 1.2 核心原则（必须遵循）

1. **标准先行**：先明确标准与规则，再做自动化与智能化。
2. **任务原子化**：任务是最小执行单元，项目是任务容器。
3. **状态驱动**：所有关键业务流转必须受状态机约束。
4. **结果对象独立**：任务负责推进，结果对象负责沉淀。
5. **人工兜底**：关键风险场景必须支持人工接管。
6. **全程留痕**：关键操作、Agent 动作、配置变更必须可审计。

## 2. 项目范围与边界规则

### 2.1 V1 必做（P0）

- 项目管理、任务中心、标准管理基础能力
- 采购基础协同、验收整改、资产归档基础能力
- 人员管理、系统设置
- 节点型 Agent 辅助能力

### 2.2 V1 非目标（禁止在 V1 作为主交付承诺）

- 多 Agent 自治协商与自动审批闭环
- 完整双边平台撮合、商业化计费体系
- 深度图像/视频 AI 质检
- 重型 BI 大屏与复杂排程引擎

## 3. 架构与技术栈规则

### 3.1 当前仓库实现口径

- 前端：`React + TypeScript + Vite`
- 路由：Hash 路由（`#/xxx`）
- 当前数据主承载：前端本地状态 + `localStorage`
- 统一入口：`src/main.tsx` -> `src/App.tsx`

### 3.2 技术方向口径（文档规划）

- 前端：`React + TypeScript + Tailwind`
- 后端：`Express + Prisma`
- 数据库：`PostgreSQL（主库） + SQLite（本地开发）`

> 规则：涉及“已落地实现”与“规划目标”冲突时，**以仓库真实代码为准**，并在文档中注明升级路径。

## 4. 业务建模规则（强制）

### 4.1 对象分层

- **项目**：管理容器（不作为最小执行单元）
- **任务**：最小执行单元（父子层级 + 关系扩展）
- **标准**：执行与验收依据（可快照）
- **结果对象**：采购单、验收单、整改单、资产台账、结算建议单等

### 4.2 建模约束

1. 项目/流程/服务必须基于统一任务底座表达。
2. 关键任务必须绑定执行标准与验收标准。
3. 进入执行态前必须具备可追溯标准上下文（或快照）。
4. 派生任务（整改/补传/补货）必须保留来源关系。
5. 禁止将“状态判断逻辑”散落在多个页面临时实现。

---

## 5. 状态机与流程守卫规则（强制）

### 5.1 项目状态（统一口径）

`待立项 -> 待确认 -> 待拆解 -> 执行中 -> 待验收 -> 待结算 -> 已归档`  
异常路径：`整改中`、`已中止` 等按状态机规则执行。

### 5.2 任务状态（统一口径）

`待创建 -> 待分配 -> 待执行 -> 执行中 -> 待提交 -> 待验收 -> 已完成/不通过 -> 已关闭`

### 5.3 守卫规则

- 前置依赖未满足，任务不得进入执行中。
- 验收前置不满足，任务不得从待提交进入待验收。
- 关键子任务未完成，父任务不得完成。
- 项目关键条件不满足，不得推进到下一个主状态。

---

## 6. 页面与交互实现规则

### 6.1 统一框架规则

1. 页面侧边栏必须复用统一侧栏组件（当前为 `PersonnelSidebar` 或后续统一替代）。
2. 页头必须复用统一页头组件（当前为 `UnifiedHeader`）。
3. 禁止在业务页面复制独立 `sidebar/header` 实现。
4. 新页面先接入统一骨架，再开发业务内容。

### 6.2 一级管理页规范

- 默认应具备：统计卡片、工具栏、主内容区、分页区
- 视图模式至少 3 种（网格/列表/看板；按模块可扩展地图/日历）
- 工具栏优先包含：视图切换、搜索、筛选、排序、设置
- 分页规则：看板/地图/日历可不显示分页

### 6.3 设计一致性规则

- 设计基线：Liquid Glass + Material Design 3
- 品牌主色：`#154DD9`
- 常用圆角：`rounded-xl`、`rounded-2xl`
- 玻璃态基线：`bg-white/[0.04] + border-white/8`
- 交互动效基线：`transition-all duration-200`

---

## 7. 编码规范规则

### 7.1 通用要求

- 全量代码使用 TypeScript 严格类型表达，避免 `any`
- 命名遵循：变量/函数 `camelCase`，组件/类型 `PascalCase`
- 复杂逻辑必须有中文注释说明（边界、守卫、业务原因）

### 7.2 注释规则（本项目执行口径）

- 采用工程化注释策略：**关键逻辑、复杂状态流、边界条件必须注释**
- 不强制“每一行代码都注释”
- 注释应解释“为什么”，而不仅是“做了什么”

### 7.3 目录与复用规则

- 通用能力放在公共层（`components/layout`、`domain`、`data`、`lib`）
- 页面代码不得内嵌大段重复样式与重复状态机逻辑
- 业务模型字段新增需同步更新类型定义与数据工厂

---

## 8. Agent 开发与治理规则

### 8.1 定位

Agent 是**受控节点能力**，不是无限制自治系统。

### 8.2 强制约束

1. 仅在授权触发点执行（手动/事件/定时）。
2. 写操作默认最小权限，超权限必须拦截。
3. 高风险动作必须进入审核流程。
4. 失败重试有上限，超限必须转人工。
5. 每次执行必须记录输入摘要、输出摘要、工具链路和结果。

---

## 9. 权限、安全与审计规则

1. 组织与品牌数据必须隔离。
2. 权限策略遵循最小权限，`deny` 优先于 `allow`。
3. 关键配置（权限/安全/通知）变更必须可审计与可回滚。
4. 关键业务动作（状态变更、派单、验收结论、替补切换）必须留痕。
5. 敏感配置（Token/密钥）不得明文暴露。

---

## 10. 开发流程与质量门禁

### 10.1 开发流程

1. 先读需求与相关模块文档，确认边界。
2. 先对齐状态机与数据模型，再写页面。
3. 优先复用统一组件与样式规范。
4. 完成功能后执行构建与必要 lint 校验。

### 10.2 命令基线

- 安装依赖：`npm install`
- 本地开发：`npm run dev`
- 生产构建：`npm run build`
- 代码检查：`npm run lint`

### 10.3 提交门禁

- 提交信息遵循 Conventional Commits（`feat/fix/docs/refactor/...`）
- 禁止提交明显未完成状态（占位逻辑、核心 TODO 未处理）
- 涉及路由、状态机、权限、Agent 的改动必须附带影响说明

---

## 11. 文档治理规则

1. 本文档作为项目规则基线；需求文档是模块细则。
2. 当实现与规则冲突时，需先更新规则再修改实现（或同时更新）。
3. 每次重大迭代需更新：版本号、变更摘要、影响模块。
4. 新增模块必须补充：状态口径、权限边界、验收标准。
