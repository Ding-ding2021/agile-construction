# Roadmap V1.2 对其他文档的影响评估

> **影响等级**：🔴 严重（必须立即修正）/ 🟡 中等（需要修正）/ 🟢 轻微（建议优化）/ ⚪ 无影响

## 二、文档影响矩阵

### 🔴 严重冲突（必须立即修正）

#### 1. `docs/01-product/project-management-prd.md`

| 冲突点           | Roadmap V1.2 口径                                                                                           | PRD 当前口径                                        | 影响章节             |
| ---------------- | ----------------------------------------------------------------------------------------------------------- | --------------------------------------------------- | -------------------- |
| **项目详情标签** | 8 标签（PMBOK 领域）：项目概览/范围与任务/进度管理/成本与采购/质量与验收/资源与人员/风险与沟通/设置与 Agent | 6 标签（生命周期）：仪表盘/启动/计划/执行/监控/收尾 | §5.2、§6.4、§7.2-7.7 |
| **项目状态机**   | 待立项→待确认→待拆解→执行中→待验收→整改中→待结算→已归档/已中止                                              | 主状态：启动→计划→执行→监控→收尾；二级状态自定义    | §7.1-7.7             |
| **阶段模型**     | 无「启动/计划/执行/监控/收尾」主状态概念，状态机与标签正交                                                  | 明确使用 PMBOK 五大过程组作为主状态                 | §1.3、§3.2、§7       |

**分析**：

- PRD 当前设计将「生命周期阶段」同时作为**状态机**和**页面标签**，这是根本性错误
- Roadmap V1.2 明确：**状态机**描述项目推进步骤（待立项→已归档），**领域标签**描述管理视图（8 个 PMBOK 领域），两者正交
- 代码实现（`src/domain/projectStatusMachine.ts`）已与 Roadmap 一致，PRD 严重滞后
- **该 PRD 当前 source_of_truth = true，但与 Roadmap 冲突，必须立即更新或降级**

**建议处理方案**：

1. 将 project-management-prd.md 的 source_of_truth 改为 `false`，标记为「待更新」
2. 重写 §5-§7（项目详情页、状态机、标签体系）
3. 或：将 PRD 拆分为两部分——项目管理 PRD（保留立项/容器/字段定义）+ 项目详情标签设计文档（引用 Roadmap §4.4）

#### 2. `docs/03-engineering/development-guide.md`

| 冲突点         | Roadmap V1.2 口径                                                     | 文档当前口径                                               | 影响章节             |
| -------------- | --------------------------------------------------------------------- | ---------------------------------------------------------- | -------------------- |
| **数据库主库** | V1 使用 **SQLite** 单文件数据库，零配置快速验证；V2 再迁移 PostgreSQL | 「PostgreSQL（主库）+ SQLite（本地开发）双 schema 策略」   | 技术栈声明（第17行） |
| **后端定位**   | V1 后端为「本地后端最小闭环」，仅支撑前端联调                         | 文档按完整后端开发流程编写（含 Prisma Studio、迁移、Seed） | §2-§3                |

**分析**：

- 文档写于 2026-03-16，当时后端计划使用 PostgreSQL，现已变更
- 文档中大量 Prisma + PostgreSQL 相关内容需要调整
- 「db:init」「db:migrate」等脚本假设了 Prisma 迁移流程，但 SQLite 可能不需要这么重的流程

**建议处理方案**：

1. 更新技术栈声明为：V1「Node.js + Express + SQLite + Prisma」，V2「PostgreSQL + Redis」
2. 数据库开发章节（§3）需要重写，SQLite 的开发和 PostgreSQL 差异较大
3. 移除或弱化「Prisma Studio」「数据库迁移」等重流程内容，改为 SQLite 文件管理

#### 6. `docs/01-product/personnel-management-prd.md`

| 冲突点          | Roadmap V1.2 口径                                                             | 文档当前口径                                               | 影响 |
| --------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------- | ---- |
| **V1 角色命名** | 品牌方总部、品牌项目负责人、采购专员、验收/质量人员、财务审核人员、系统管理员 | 平台管理员、品牌方管理员、调度运营、资源方负责人、项目经理 | §2.1 |

**分析**：

- Roadmap 从「业务职责」角度定义角色，PRD 从「系统功能」角度定义角色
- 不是严格冲突，但口径不一致会导致权限设计混乱
- 例如：Roadmap 的「系统管理员」对应 PRD 的「平台管理员」；Roadmap 的「品牌项目负责人」对应 PRD 的「项目经理」
- PRD 中的「调度运营」「资源方负责人」属于 V2 角色，不应在 V1 PRD 中作为核心角色

**建议**：按 Roadmap V1 的 6 角色重新梳理 PRD 的角色章节，V2 角色标记为「预留/平台化阶段」

#### 7. `docs/02-architecture/state-machine-design.md`

| 冲突点             | Roadmap V1.2 口径      | 文档当前口径                                                                                                            | 影响      |
| ------------------ | ---------------------- | ----------------------------------------------------------------------------------------------------------------------- | --------- |
| **验收对象状态机** | 验收整改不作为独立模块 | 将「验收」列为独立的状态机治理对象（§3.1），并定义了完整的验收状态流（待申请→待初验→初验不通过→整改中→待复验→复验通过） | §3.1、§14 |

**分析**：

- Roadmap 明确「验收整改不作为独立模块」，但状态机文档仍保留了验收作为独立对象的状态机设计
- 需要澄清：是不再需要「验收单」这个业务对象，还是只是不再设「验收整改」独立模块？
- 从代码实现看，项目状态机有「待验收→整改中→待结算」流转，任务状态机有「待验收→不通过→已完成」流转，验收逻辑已融入项目和任务

**建议**：

1. 在状态机文档中增加说明：「验收不再作为独立业务对象维护状态机，验收逻辑通过项目状态（待验收/整改中）和任务状态（待验收/不通过/已完成）承载」
2. 或：将§14「验收状态机」标记为「已归档/不采用」

## 六、总结

Roadmap V1.2 的变更对文档体系产生了**3 处严重冲突、7 处中等影响**：

1. **最紧急**：`project-management-prd.md` 与 Roadmap 在项目标签和状态机上存在根本性冲突，且两者都是 source_of_truth，必须立即降级 PRD 并启动重写
2. **次紧急**：`development-guide.md` 的数据库技术路线已变更，需要更新以避免误导开发
3. **其他**：角色体系、验收整改定位、测试框架描述等需要同步校准

建议按「P0 → P1 → P2」优先级逐批修正，每批修正后做一次文档一致性审查。
