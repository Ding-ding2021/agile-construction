# MVP 代码质量计划 V2

> **目标**：非工程师产品经理在不读代码的前提下，用结构化验收手段把控 AI 编码质量。

## Clause 1. 原则与策略

### 1.1 [参考] 三层质量防线

```
第一层：自动化防线（零代码能力要求）
第二层：AI 编码流程控制（可解释交付物）
第三层：产品经理验收检查（读测试用例 + 看流程图）
```

**1.1.1 [强制]** 每一层独立运作，上层失效时下层兜底。

### 1.2 [参考] 核心策略

**1.2.1 [参考]** 自动化守住底线：TypeScript + ESLint + 构建阻断，零代码能力要求。

**1.2.2 [参考]** 可解释交付物是质量锚点：测试用例 + 流程图 + 边界清单，产品经理看得懂、验得了。

**1.2.3 [参考]** 双人 AI 交叉验证防盲区：模型 A 写、模型 B 审，降低单模型幻觉风险。

---

## Clause 2. 第三层：产品经理验收检查清单

### 2.1 [强制] 每轮迭代必查 5 项

| #   | 检查项       | 验收方式             | 通过标准                              |
| --- | ------------ | -------------------- | ------------------------------------- |
| 1   | 测试覆盖     | 看测试用例列表       | 所有业务场景（正常+异常）都有对应测试 |
| 2   | 流程图正确   | 看 Mermaid 图        | 状态跳转与需求文档一致，无非法路径    |
| 3   | 边界清单     | 看边界条件表格       | 空数据、异常状态、越界输入都有处理    |
| 4   | 真实场景跑通 | 在浏览器操作         | 实际业务场景能走通                    |
| 5   | 变更影响     | 看 AI 输出的影响报告 | 改动范围可控，不波及无关模块          |

### 2.2 [强制] 红线判定（触发即回滚）

**2.2.1 [强制]** 没有可解释交付物 —— 测试/流程图/边界清单缺任一。

**2.2.2 [强制]** 单元测试覆盖率不足 —— 产品经理通过阅读测试列表发现明显遗漏场景。

**2.2.3 [强制]** 流程图与需求矛盾。

**2.2.4 [强制]** 真实场景操作失败。

**2.2.5 [强制]** AI 未经申请修改了 STABLE 模块。

## Clause 3. 第一层：自动化防线

### 3.1 [强制] 构建阻断机制

每次 AI 生成代码后，必须先通过以下检查才能进入验收：

| 检查项              | 命令               | 失败处理                 |
| ------------------- | ------------------ | ------------------------ |
| TypeScript 类型检查 | `npx tsc --noEmit` | 让 AI 修正，不通过不验收 |
| ESLint 代码检查     | `npm run lint`     | 让 AI 修正，不通过不验收 |
| 构建成功            | `npm run build`    | 让 AI 修正，不通过不验收 |

### 3.2 [强制] ESLint 关键规则

| 规则                                 | 作用                    | 违规后果          |
| ------------------------------------ | ----------------------- | ----------------- |
| `@typescript-eslint/no-explicit-any` | 禁止 any 类型           | error —— 构建失败 |
| `@typescript-eslint/no-unused-vars`  | 禁止未使用变量          | error —— 构建失败 |
| `eqeqeq`                             | 强制 ===                | error —— 构建失败 |
| `no-console`                         | 限制 console.log        | warn —— 提醒清理  |
| `react-hooks/exhaustive-deps`        | 强制 useEffect 依赖完整 | error —— 构建失败 |

## Clause 4. 第二层：AI 编码流程控制

### 4.1 [强制] 可解释交付物制度

所有核心模块（`src/domain/`、`src/data/`、`src/App.tsx`）由 AI 生成时，必须同步输出三类可解释交付物：

| 交付物       | 格式                | 作用                         | 产品经理怎么审                         |
| ------------ | ------------------- | ---------------------------- | -------------------------------------- |
| 完整单元测试 | TypeScript 测试代码 | 证明代码在所有场景下行为正确 | 看测试用例列表，确认覆盖了所有业务场景 |
| 文本版流程图 | ASCII 或 Mermaid    | 展示状态流转/数据流/决策逻辑 | 对照需求文档，确认跳转关系正确         |
| 边界条件清单 | Markdown 表格       | 列出所有异常输入和预期行为   | 确认空数据、异常状态、越界输入都有处理 |

**4.1.1 [强制]** AI 不输出这三样，产品经理不验收代码。

### 4.2 [强制] 核心模块稳定化机制

| 状态   | 标签     | 变更规则                              |
| ------ | -------- | ------------------------------------- |
| 开发中 | `DRAFT`  | AI 自由修改，每次修改需重新输出交付物 |
| 已验收 | `STABLE` | AI 修改前必须输出变更申请             |
| 已冻结 | `FROZEN` | 禁止 AI 修改，只允许人工触发变更      |

**4.2.1 [强制]** AI 修改 STABLE 模块前必须输出变更申请，内容包括：目标文件、变更理由、影响范围、测试更新方案、向后兼容性。

## Clause 5. 核心模块清单

### 5.1 [参考] 核心模块与验收关键

| 文件                                 | 关注重点                 | 验收关键             |
| ------------------------------------ | ------------------------ | -------------------- |
| `src/domain/projectStatusMachine.ts` | 状态流转是否闭环         | 合法/非法跳转矩阵    |
| `src/domain/projectStatusView.ts`    | 视图映射是否与状态机一致 | 每个状态对应视图正确 |
| `src/domain/workItem.ts`             | 工作项模型变更影响       | 下游组件是否兼容     |
| `src/data/projects.ts`               | 项目数据结构向后兼容     | 已有数据不丢失       |
| `src/App.tsx`                        | 路由和全局状态变更       | 现有页面不受影响     |

## Clause 6. 测试策略

### 6.1 [参考] 各模块测试要求

| 模块                                 | 测试类型 | 最低要求              |
| ------------------------------------ | -------- | --------------------- |
| `src/domain/projectStatusMachine.ts` | 单元测试 | 100% 状态流转路径覆盖 |
| `src/domain/workItem.ts`             | 单元测试 | 所有计算分支覆盖      |
| `src/data/projects.ts`               | 集成测试 | 工厂函数关键路径      |
| `src/App.tsx`                        | 手工验收 | 主要流程跑通          |

## Clause 7. 质量指标与监控

### 7.1 [参考] 每周记录指标

| 指标                      | 目标值 | 超限处理                               |
| ------------------------- | ------ | -------------------------------------- |
| AI 生成代码验收拒绝率     | < 30%  | 拒绝率过高则调整 Prompt 约束或更换模型 |
| 构建失败次数              | 0      | 每次构建失败记录根因                   |
| 缺少可解释交付物的提交    | 0      | 发现即要求补全                         |
| STABLE 模块未经申请被修改 | 0      | 触发红线，回滚并追责                   |

### 7.2 [参考] 双周架构健康度检查

**7.2.1 [参考]** 快速扫描以下信号：

- 是否有核心模块从 STABLE 变回 DRAFT？
- AI 是否新增了 `// eslint-disable` 注释？
- `package.json` 依赖是否无故增加？
- 是否有重复代码片段？
- 组件平均行数是否超过 300 行？
